17. セルフオーダー画面（お客さん向け） ​

セルフオーダー画面（/public/customer/menu.html）は、お客さんがスマートフォンでQRコードを読み取って注文する画面です。テーブルのQRコードを読み取ると、そのテーブル専用の注文画面が開きます。

17.0 はじめにお読みください ​

この章は「お客様がスマホでセルフ注文する」体験全体を解説した完全ガイドです。スタッフ・店長・オーナーは「お客様がどう触るか」を理解した上で、店内オペレーションを設計するために読みます。

この章を読む順番

  1. 17.0.1「全体フロー俯瞰」 で導線を把握
  2. 17.1〜17.3 で画面構成 + 注文の流れ
  3. 17.4〜17.7 で個別機能 (呼出・カート・LO・AI)
  4. 17.W で食事後アンケート (満足度評価)
  5. 17.X〜17.Z でトラブル・FAQ・技術補足

17.2 注文の流れ ​

17.2.1 ステップ 0: 操作前チェックリスト (お客様) ​

#項目補足

2スマホのカメラ or QR リーダーアプリiOS 標準カメラで十分
3スマホのインターネット接続店内 Wi-Fi または 4G/5G

17.2.2 ステップ 1: QR コードを読み取る ​

  1. テーブルに設置された QR コードをスマートフォンのカメラで読み取ります
  2. ブラウザが起動し、https://eat.posla.jp/public/customer/menu.html?store_id=xxx&table_id=xxx が開きます
  3. 内部で GET /api/customer/validate-token.php?store_id=xxx&table_id=xxx&token=xxx 軽量検証
  4. 店舗のブランドカラーとロゴが適用された画面が表示されます

URL パラメータ詳細 ​

パラメータ例必須説明
store_ids-shibuya-001✅店舗 ID
table_idt-001✅テーブル ID
tokenabc123...✅セッショントークン (4 時間有効)

17.2.3 ステップ 1.5: PIN 入力 (S6 — 2026-04-18 後) ​

S6 セキュリティ強化により、QR スキャン後に 4 桁の PIN 入力 が必須になりました (店舗ごとに ON / OFF 設定可)。

項目説明
PINスタッフが「テーブル開放」時に発行する 4 桁数字
入力場所テーブルのレシートホルダー or 卓上カード

失敗時5 回連続失敗で 10 分間ロック (pin-verify rate limit)
バイパス設定で OFF にすると PIN 不要 (旧仕様)

詳細は 25 章「テーブル開放」を参照。

17.2.7 注文のリトライ機能 (order-sender.js) ​

ネットワークエラーが発生した場合：

  • 自動的に最大 2 回リトライされます（2 秒間隔）
  • 冪等性キー（ランダムな 32 文字の 16 進数）により、重複注文が防止されます
  • 完全にオフラインの場合、注文はブラウザの localStorage に保存され、接続復旧時に自動送信されます
  • 24 時間以上経過した保留注文は自動破棄されます

17.2.8 注文のレートリミット ​

実装: api/customer/orders.php check_rate_limit('customer-order', 20, 600)

項目値
キーcustomer-order:<IP>
上限20 回
期間600 秒 (10 分)
超過時HTTP 429 「リクエスト回数の上限に達しました」

通常利用では到達しないが、短時間に連続で注文を送信すると制限される。10 分待てば自動解除。

17.3 注文ステータスのリアルタイム追跡 ​

注文送信後、画面上部に注文の進捗がリアルタイムで表示されます。

17.3.1 ポーリング間隔 ​

5 秒ごとに GET /api/customer/order-status.php?order_id=xxx でステータスを取得。order-tracker.js 実装。

17.3.2 ステータス表示 ​

ステータスバッジ色意味

17.3.3 プログレスバー ​

注文全体の進捗を視覚的に表示します。全品目が「served」になると完了です。

[●●●○○]  3/5 提供済み (60%)

17.3.4 リアルタイム調理進捗 (N-2) ​

KDS 側で調理開始 / 完了を打刻すると、客側に即座に反映：

  • 「調理開始から 5 分経過」のような時間表示
  • 「あとおよそ N 分」の推定時間表示 (店舗の平均調理時間ベース)
  • 提供完了時にバイブ + 通知音 (許可時のみ)

17.4 スタッフ呼び出し ​

画面のヘッダーにある🔔アイコンをタップすると、スタッフ呼び出し機能が利用できます。

17.4.1 呼び出しの種類 ​

選択肢reason 値説明
注文したい注文したい注文に関する呼び出し
お会計をお願いしますお会計をお願いします会計リクエスト
食器を下げてください食器を下げてください下げ物の依頼
その他(自由入力)自由入力 (100 文字制限)

17.4.2 呼び出しの流れ (call-staff.js) ​

  1. 🔔アイコンをタップします

  2. 呼び出しモーダルが開く

  3. 呼び出し理由を選択します (またはその他で自由入力)

  4. POST /api/customer/call-staff.php で送信

    {
  "store_id": "s-shibuya-001",
  "table_id": "t-001",
  "table_code": "A-3",
  "reason": "お会計をお願いします"
}

  5. サーバー側で call_alerts テーブルに INSERT (status='pending')

  6. 同テーブルの古い pending アラートは DELETE (重複防止)

  7. スタッフのハンディ POS と KDS にアラートが表示されます

  8. スタッフが「対応済み」をタップすると call_alerts.status='resolved' に

  9. お客さんの画面に 効果音（440Hz→523Hz） で通知されます

17.4.3 呼出履歴 (店舗側) ​

スタッフ側でアラートを履歴閲覧可能（dashboard.html >「テーブル」タブ）。status='pending' 件数で「未対応」表示。

17.4.4 クールダウン ​

呼び出し後、3 分間は次の呼び出しができません（カウントダウン表示）。これにより連打による誤発信を防止します。

17.4.5 サーバー側レートリミット ​

実装: check_rate_limit('call-staff', 10, 600)

  • 1 IP あたり 10 回 / 10 分
  • 超過時 HTTP 429
  • 通常運用では到達しない (UI のクールダウンで防止済み)

17.4.6 アラートステータス確認 API ​

GET /api/customer/call-staff.php?alert_id=xxx

レスポンス: { status: "pending" | "resolved" | "not_found" }

クライアントは送信後 5 秒間隔で polling し、resolved になったら効果音再生 + UI 更新。

17.5 カートの永続化 ​

カートの内容はブラウザの localStorage に保存されます。

17.5.1 保存仕様 ​

項目値
キーposla_cart_<store_id>_<table_id>
値JSON 配列 [{id, name, price, qty, options...}]
有効期限4 時間 (セッショントークンと同期)
クリア条件(1) セッショントークン変更時 (2) 注文送信成功時 (3) 24 時間経過時

17.5.2 動作 ​

  • ページをリロードしてもカートの内容は維持されます
  • セッショントークンが変わった場合（テーブルリセット後）、カートは自動クリアされます

17.5.3 迷い品目トラッキング (cart-event.php) ​

カートに追加した後に削除した品目は「迷い品目」として記録され、オーナーダッシュボードのクロス店舗レポートで分析できます。

POST /api/customer/cart-event.php
{
  "store_id": "s-shibuya-001",
  "table_id": "t-001",
  "session_token": "abc123",
  "menu_item_id": "m-uni",
  "event_type": "removed_after_added",  // or "added", "viewed"
  "qty": 1
}

これにより、お客さんが迷ったメニューの改善に活用できます (P1-33: cart_events テーブル)。

17.6 ラストオーダー ​

ラストオーダーが発動されると、セルフオーダー画面では新規注文がブロックされます。

17.6.1 LO 発動方法 ​

スタッフが dashboard >「テーブル」タブ → 該当テーブル → 「ラストオーダー」 ボタン or 「店舗全体ラストオーダー」 → API POST /api/store/last-order.php。

17.6.2 客側挙動 ​

状態挙動
LO バナー画面上部に赤色で「⚠ 注文受付は終了しました」表示
カートバー「注文する」ボタンが無効化 (灰色)
サーバー側LAST_ORDER_PASSED エラーで弾く (二重防御)
既存注文影響なし、提供は通常通り

17.6.3 LO の解除 ​

スタッフが LO 解除ボタンを押せば即時解除（ただし通常運用では LO 解除しない）。

17.7 AI ウェイター (ai-waiter.js) ​

17.7.1 機能概要 ​

お客様がメニューについて自然文で質問できる AI チャット機能。Gemini API (gemini-2.0-flash) をサーバープロキシ経由で呼び出します。

17.7.2 起動方法 ​

メニュー画面の右下フローティングボタン「💬 AI ウェイター」をタップ。

17.V セルフレジ (L-8) ​

17.V.1 機能概要 ​

お客様が スマホで自分で会計 できる機能。Stripe Checkout 経由でカード決済 (Apple Pay / Google Pay 対応)。

17.V.2 有効化条件 ​

項目必須設定場所
store_settings.self_checkout_enabled = 1✅dashboard >「設定」>「営業設定」
Stripe 設定 (Direct or Connect)✅21 章

17.V.3 客側フロー ​

  1. 食事終了 → 「お会計」ボタンタップ
  2. 注文一覧 + 合計金額表示
  3. 「セルフレジで支払う」ボタンタップ
  4. POST /api/customer/checkout-session.php で Stripe Session 作成
  5. Stripe Checkout 画面にリダイレクト
  6. カード入力 (or Apple Pay / Google Pay)
  7. 決済成功 → menu.html?checkout=success&stripe_session_id=* に戻る
  8. POST /api/customer/checkout-confirm.php で注文を paid に更新
  9. レシート表示 (receipt-view.php)

17.V.4 サーバー実装 (checkout-session.php) ​

ステップ処理
1session_token 検証 (hash_equals)

4合計金額算出
5Stripe Connect / Direct のどちらかで Session 作成
6success_url / cancel_url 組み立て
7200 OK + checkout_url 返却

17.Y よくある質問 ​

Q1. 客のスマホで言語切替は? A1. 画面右上の言語ボタンから (日 / 英 / 中 / 韓 / 繁体)。api/customer/ui-translations.php?lang=xx で UI 翻訳取得 + メニューは api/store/translate-menu.php 事前生成。

Q2. アレルギー表示 A2. メニュー詳細に ⚠ アイコン (menu_templates.allergens)。AI ウェイターで質問可。注文時にアレルゲンチップで重ねて申告すると KDS で大きく表示。

Q3. 満足度評価が低い時 A3. KDS 画面に低評価アラートが自動表示（kds/orders.php の low_rating_alerts）。詳細は 17.W 参照。

Q4. 途中で QR を再スキャン A4. 同じセッショントークンが再利用される (4 時間有効)。新しいタブで開いてもカートは localStorage 共有。

Q5. 複数人で別会計 A5. 個別 QR (F-QR1) 機能で相席用サブセッション発行可。sub_sessions テーブル + api/store/sub-sessions.php。詳細は 25 章。

Q6. スタッフ呼び出しが届かない A6. 確認順:

  1. ハンディの音量・通知設定 ON
  2. KDS のアラート表示 ON
  3. call_alerts に INSERT されているか (DB 直接確認)
  4. レートリミット 429 になっていないか

Q7. カートが消えてしまった A7. localStorage クリア / セッショントークン更新時に自動クリアされる仕様。スタッフが「テーブル開放」をやり直すと前のセッションは無効化されカートも消える。

Q8. 注文ボタンが押せない A8. 想定原因:

  • LO 発動中
  • カートが空
  • セッション期限切れ (4 時間経過、再 QR スキャン要)
  • 既に支払い済み

Q9. テーブル PIN を忘れた A9. スタッフに伝える → スタッフがテーブル開放画面で確認 (PIN は tables.pin_code に保存)。3 回以上忘れた場合は再発行 (新しい PIN を付与)。

Q10. 子供がスマホをいじって誤注文した A10. 注文後に staff 以上が dashboard >「注文」タブから「キャンセル」可能 (status='cancelled')。提供前のみ。

Q11. AI ウェイターが「答えられません」と言う A11. Gemini API のレートリミット (20 req / 1h / IP) または接続障害。1 時間待つ or POSLA 運営に連絡。

Q12. メニューに画像が表示されない A12. menu_templates.image_url が NULL or アクセス不可。dashboard >「メニュー」で画像をアップロード。S3 / Sakura object storage 経由 (api/store/upload-image.php)。

Q13. 品切れ品目が注文できてしまった** A13. タイミング差で UI と DB の is_sold_out がずれた瞬間に発生する可能性。サーバー側でも品切れチェックしているため MENU_NOT_AVAILABLE で弾かれるはず。万一通った場合はキッチンが「品切れ」と KDS に返す → スタッフが客に説明 → 別品提案。

Q14. 多言語翻訳がおかしい (中国語が英語になる等) A14. translation.php のキー解決ミス。dashboard >「メニュー」>「翻訳」で再生成 (Gemini API)。

Q15. 「セッションが無効です」と出る A15. 想定原因:

  1. 4 時間経過 (セッション期限切れ)
  2. スタッフがテーブルを「クローズ」して新セッション発行した
  3. token URL パラメータが欠損 / 改ざん スタッフがテーブル再開放 → 客が QR 再スキャン。

Q16. リオーダー (再注文) はどう使う？ A16. 過去注文した品目が画面上部に「もう一度」ボタン付きで表示。タップで即座にカート追加。

Q17. カートに 100 品入れたい A17. 上限なし (カート側)。ただし注文 API 側でサーバーリソース保護のため概ね 50 品程度を推奨。法人会食等は注文を分けることを推奨。

Q18. メニューカテゴリが英語表示にならない A18. カテゴリ名翻訳 (category_translations テーブル) が空。dashboard >「カテゴリ」>「翻訳」で多言語入力。

Q19. 「人気 #1」の判定基準は？ A19. 過去 30 日の注文数 TOP 3。menu.php のレスポンス popularity フィールドで返却。リアルタイム集計ではなく日次バッチ更新の場合あり。

Q20. おすすめバッジを変更したい A20. dashboard >「おすすめ管理」(daily_recommendations) で当日のおすすめ品目を 0〜5 件設定。AI が需要予測 + 在庫から自動提案する機能あり。

Q21. スタッフ呼び出しの効果音が鳴らない A21. ブラウザの自動再生制限。客が画面のどこかをタップ済みであれば鳴る。鳴らない場合はバイブ (vibrate API) でフォールバック。

Q22. 注文後にメニューを追加したい A22. カートを再構成して再度「注文する」を押す。注文ごとに別 order_id で記録される。同じセッション内なら全部まとめて会計可。

Q23. ウェルカムメッセージは設定できる？ A23. store_settings.welcome_message (実装によって異なる) でカスタマイズ可。多言語対応 (5 言語)。

Q24. 4 時間以上の長居でセッションが切れる A24. デフォルト 4 時間。長居前提の店舗 (カラオケ・カフェ等) は session_ttl_minutes を延長可。

Q25. 客のスマホがガラケー (フィーチャーフォン) だった A25. QR 読み取り後にブラウザが起動できないとセルフオーダー利用不可。スタッフが代理注文 (ハンディ) に切り替える。

Q26. 客が誤って店舗を変更してしまった A26. URL パラメータが他店舗 store_id になっていると別店舗のメニューが出る。テーブル QR は店舗固定 ID 入りなので通常起こらない。万一発生したら QR 貼り直し。

Q27. 「迷い品目」レポートは何のため？ A27. お客さんが「カートに入れたが削除した」品目を集計して、メニュー改善 (説明文追加 / 価格見直し / 写真追加) に活用。オーナーダッシュボードのクロス店舗レポート。

Q28. AI ウェイターのプライバシーポリシー A28. 質問内容は POSLA サーバーを経由して Google (Gemini) に送信される。個人特定可能情報は入力しないように UI に注意書き。

Q29. セルフレジを使わずスタッフ会計したい A29. 「お会計」ボタンで「スタッフを呼ぶ」を選択 → 通常のスタッフ呼び出しと同じ。スタッフがレジで処理。

Q30. テーブル QR を盗撮されて店外から注文されたら？ A30. S6 後は PIN 入力必須なので、店内にいる客以外は注文不可 (PIN は店内卓上にしか書いていない)。さらに validate-token.php のセッション期限 4 時間で抑止。
