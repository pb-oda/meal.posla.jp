8. 会計・POSレジ ​

POSレジ画面 (/kds/cashier.html) は、テーブルの会計処理を行うための画面です。現金・クレジットカード・QR コード決済に対応し、割引・個別会計・レジ開閉・取引履歴・レシート発行・返金 (C-R1) の機能を備えています。

8.0 はじめにお読みください ​

章の読み方ガイド

  • レジ担当: 8.1 → 8.3 (会計の基本フロー) が必読
  • 店長・オーナー: 8.9 (レジ開閉) + 8.10 (取引履歴) + 8.11 (返金) 必読
  • Stripe 連携設定: 21 章と合わせて読む
  • プリンタ設定: 26 章と合わせて読む
  • トラブル対応: 8.14 FAQ / 8.15 トラブル表

POS レジ画面の役割 ​

  • テーブルごとの会計処理 (現金・カード・QR)
  • Stripe Terminal 連携によるクレジット決済
  • 個別会計 (割り勘)
  • 割引適用
  • 返金処理 (C-R1、新機能)
  • レジ開け・締め
  • 取引履歴の確認・レシート再印刷
  • ドロア操作 (プリンタ連動、将来)

使える人 ​

  • owner / manager / staff / device
  • device ロールは GPS チェック必須 (23 章参照)

必要なもの ​

  • タブレット (推奨 10〜12 インチ)
  • 店舗 Wi-Fi
  • カードリーダー (Stripe Reader S700、カード決済時)
  • レシートプリンタ (26 章参照、任意)
  • 釣銭用の現金

8.0.1 操作前チェックリスト ​

POSレジを開く前に、以下を確認してください。会計開始後に発覚すると業務が止まります。

#確認項目確認方法NG 時の対応
1あなたのアカウントが owner / manager / staff / device いずれかログイン後のロール表示で確認該当しないロールはログイン不可

8.3 税金の計算 ​

POSLAは日本の消費税制度に対応し、品目ごとに税率を分けて計算します。

税率の区分 ​

税率対象条件
10%（標準税率）店内飲食デフォルト
8%（軽減税率）テイクアウトテイクアウト注文の場合
品目別税率個別設定品目にtaxRateが設定されている場合

税額の表示 ​

会計画面では税率区分ごとに内訳が表示されます。

10%対象  ¥1,100 （税 ¥100）
 8%対象    ¥540 （税  ¥40）
─────────────────────────
合計      ¥1,640

計算方法（税込価格からの逆算）：

税抜額 = 税込価格 ÷ (1 + 税率)  ※小数点以下切り捨て
税額 = 税込価格 − 税抜額

8.4 個別会計（割り勘） ​

1つのテーブルの注文を品目単位で分けて、複数回に分けて会計できます。

個別会計の操作 ​

  1. テーブルの「会計」ボタンをクリックして注文を表示します
  2. 品目リストのチェックボックスで、今回支払う品目を選択します
    • 「全選択」：全品目を選択
    • 「全解除」：全品目の選択を解除
    • 個別チェックボックス：品目ごとに選択/解除
  3. 一部のみ選択すると「個別会計」バッジが表示されます
  4. 選択した品目の合計が再計算されます
  5. 「個別会計する」ボタンで選択品目のみを会計します

個別会計後の状態：

  • 支払い済みの品目は「支払済」表示（グレー）になり、チェックボックスがロックされます
  • 残りの品目は次の会計を待ちます
  • 全品目が支払い済みになるとテーブルセッションが終了します

8.5 合流会計 ​

複数テーブルの注文をまとめて1回の会計で処理します。

8.5.1 操作手順 ​

  1. 左カラム上部の 「合流会計」 ボタンをクリック（トグル ON）
  2. 複数のテーブルカードを順にクリックして選択（チェック済になる）
  3. 選択したテーブルの全注文が右カラムにまとめて表示されます
  4. 通常どおり支払方法を選び、会計確定（PIN 入力含む）
  5. 全テーブルが同時に paid になりセッション終了

合流会計時の制約

  • テイクアウト注文との合流不可（在席テーブルのみ）
  • 合流相手テーブルの all 全注文 が会計対象。一部品目のみは個別会計（8.4）と組み合わせる必要あり
  • 合流後は片方のテーブルだけ後日「open に戻す」操作不可。誤合流は返金で対応

「クレカ」と「Stripe Terminal」の違い

クレカ = 別の決済端末（楽天ペイ等）で決済し、POSLA に金額だけ手入力で記録するモード。POSLA 側は単に「カードで支払われた」と記録するだけ。

Stripe Terminal = POSLA から直接 Stripe Reader S700 に決済命令を送り、その場で承認・記録する完全自動モード。返金も POSLA から実行可能。

将来的に Stripe Terminal が標準になり、「クレカ」モードはレガシー扱いになります。

8.6 レジ開閉操作 ​

POSレジでは、営業開始時の「レジ開け」と営業終了時の「レジ閉め」を行います。

レジ金庫操作は全て担当スタッフ PIN 必須（CB1d、2026-04-19 追加）

レジ開け / レジ閉め / 入金 / 出金 の 4 操作は、実行時に必ず 担当スタッフの PIN 入力が求められます。会計・返金と同じテンキーモーダルで、出勤中スタッフの PIN を照合します。

  • マネージャーも例外なし。バイパスはありません
  • 事前に 個人スタッフアカウントで管理画面にログイン → 出勤打刻しておくこと（退勤打刻したスタッフの PIN は無効）
  • 出金は店舗の現金がレジから出る最も危険な操作のため、誰が引き出したかを必ず記録します
  • 締め時の差額調査が「いつ・誰が・何のために動かした」をすべて追える状態になります
  • cash_log.user_id には PIN 認証されたスタッフ ID が入ります（ログイン中の device 端末ユーザーではありません）

8.6.1 差額調査の手順（締めで過不足が出た時） ​

差額が出た場合、以下の順序で監査します。

  1. 入出金履歴を順に読む

    • レジ開閉パネル下部の「本日の入出金履歴」を、開け → 入金 → 出金 → 売上 → 締めの順で確認
    • 各行のメモから直感的に判断（例：「仕入支払」と書いてあるのに領収書がない → 要追跡）

  2. cash_log テーブルを直接 SELECT（manager 以上、phpMyAdmin 等）

    sql
    SELECT cl.created_at, cl.type, cl.amount, cl.note, u.display_name
FROM cash_log cl
LEFT JOIN users u ON u.id = cl.user_id
WHERE cl.store_id = ? AND DATE(cl.created_at) = CURDATE()
ORDER BY cl.created_at;
    • 担当者（u.display_name）と時刻、金額が並ぶので「誰が」「いつ」「何のために」追える

  3. 監査ログ（audit_log）でエビデンス

    sql
    SELECT created_at, action, user_id, details
FROM audit_log
WHERE store_id = ? AND DATE(created_at) = CURDATE()
  AND action IN ('register_open','register_close','cash_in','cash_out','payment_complete','payment_refund')
ORDER BY created_at;
    • details (JSON) には pin_verified=1, cashier_user_id, cashier_user_name が入っており、PIN を介して動かした人を一意特定可

  4. 差額閾値を超える場合の対応

    • +/- ¥1,000 以内 → 翌日に持ち越し（朝礼で共有）
    • +/- ¥5,000 超 → owner / manager を呼んで監査ログ精査
    • 当該日の cash_out を全て点検 → 領収書突合
    • 該当スタッフから事情聴取（cashier_user_id で特定）

差額の責任範囲

差額（過不足）の補填責任は、店舗側の就業規則・労働基準に従ってください。POSLA は記録の正確性のみ保証します。安易に「スタッフ給与から天引き」運用は労働基準法違反の可能性あり、必ず社労士に相談を。

8.9 Stripe決済端末連携 ​

全契約者に標準提供され、Stripe Connect（POSLA 経由）または Stripe 自前キーが設定されている場合、カードリーダー（Stripe Terminal）での決済に対応します。

接続状態 ​

  • ヘッダーにカードリーダーの接続状態が表示されます（緑色/赤色のドット）
  • 「再検出」ボタンで再接続を試みます
  • Stripe Terminalが利用できない場合はステータス表示が自動的に非表示になります

カード決済の流れ ​

  1. 支払方法で「クレカ」を選択します
  2. 「会計する」ボタンをクリックします
  3. UIがロックされ、オーバーレイが表示されます
    • 「カードリーダーで決済準備中...」
    • 「カードをリーダーにかざしてください」
    • キャンセルボタン
  4. お客さんがカードをリーダーにかざします
  5. 決済が完了するとレシートが表示されます

内部処理の流れ（実装詳細）：

  1. fetch /api/store/terminal-intent.php（POST {amount}）
    • サーバー側: Stripe PaymentIntent を作成（Pattern A = テナント自前キー / Pattern B = Connect 経由 + application fee）
    • レスポンス: client_secret + payment_intent_id
  2. _terminal.collectPaymentMethod(client_secret)
    • S700 リーダーでカード読取（非接触 / 挿入 / スワイプ）
    • お客様がキャンセル可能（S700 画面の赤ボタン）
  3. _terminal.processPayment(paymentIntent)
    • Stripe 側で決済実行（3D セキュア等を含む）
    • 成功 or 失敗
  4. fetch /api/store/process-payment.php（POST）
    • payment_method='terminal', terminal_payment_id=pi_xxx
    • POSLA 側の orders.status='paid', payments レコード作成
    • audit_log に記録
  5. レシート表示・印刷・ドロア開け

8.10-b.4 PIN 不一致の場合 ​

8.10-b.5 個人アカウントでログインしている場合 ​

8.11 返金処理 (C-R1、新機能) ​

2026-04-18 に追加された返金機能。Stripe Refund API と連携し、カード決済済みの注文を部分返金 or 全額返金できます。

8.11.1 返金対象 ​

  • カード決済 (Stripe) のみ自動返金可能
  • 現金決済 は手渡し (POSLA 側は記録のみ)
  • QR 決済 は PayPay/d払いの管理画面から別途返金 (POSLA 側は記録のみ)

8.11.2 返金手順 (manager 以上) ​

  1. POSレジ画面の 【📋 取引履歴】 ボタン
  2. 返金対象の取引をタップ
  3. 【返金】 ボタン (赤色)
  4. 返金モーダル:
    • 返金額 (全額 or 部分)
    • 返金理由 (例: 「料理間違い」「アレルギー申告漏れ」)
  5. 【返金を確定】 → 担当スタッフ PIN 入力モーダル（2026-04-19 追加）
  6. PIN 確定 → Stripe Refund API 呼出
  7. 成功でトースト「返金完了 ¥XXX」

返金には担当スタッフ PIN が必須（2026-04-19 追加）

会計と同様、返金時にも 出勤中スタッフの PIN 入力が必須になりました。退勤打刻したスタッフの PIN は無効化されています。PIN 不一致の場合はエラー（PIN_INVALID）で返金は実行されません。

実装: POST /api/store/refund-payment.php の staff_pin パラメータ。詳細は 8.10-b を参照。

8.11.3 返金後の動作 ​

  • payments.refund_amount 更新
  • payments.refund_user_id に PIN で特定された担当スタッフ user_id を記録
  • audit_log に返金操作を payment_refund アクションで記録（誰が・いつ・いくら・理由）
  • お客様のカードに数営業日後返金
  • レシート再印刷時に「返金済 ¥XXX」表示

8.11.4 二重返金防止 ​

  • 同じ payment_intent に対して 2 回返金ボタンを押しても冪等 (Stripe 側で排他)
  • UI 上は返金ボタンが _refunding フラグで disabled になる

8.11.5 権限 ​

  • staff: 返金ボタンを見ることはできない
  • manager: 部分返金・全額返金 OK
  • owner: 全操作 + 高額返金 OK

8.14 よくある質問 40 件 ​

A. 会計基本（5 件） ​

Q1. 会計待ちテーブルが出ない A1. KDS 画面で「全品提供済」になっていない可能性。KDS で提供済にする。または、テーブルセッションが終了してしまっている（開店時に再オープン）。

Q2. 現金決済でお釣り計算は？ A2. 預かり金額を入力するとお釣りが自動表示。テンキーまたはクイック金額ボタン（¥1,000 / ¥5,000 / ¥10,000 / ぴったり）使用。

Q3. 会計中にお客様が追加注文した A3. いったん会計画面で別テーブルを選び直し → 元のテーブルを再選択して最新情報を取得。または _pollTimer 5 秒間隔で自動更新される。

Q4. 会計金額を手動で変更したい A4. 割引機能（8.7）を使う。金額（円）または %（パーセント）で割引適用。直接金額変更は不可（不正防止）。

Q5. 会計後にお客様が戻ってきた（クレーム等） A5. 返金機能（8.11）を使う。Stripe Terminal 決済のみ自動。現金は手渡し対応。

B. PIN 認証（7 件） ​

Q6. 自分の PIN を忘れた A6. dashboard >「アカウント設定」>「担当 PIN」>「PIN を変更」で新しい PIN を設定。古い PIN は不要（自分のアカウントなら自分で再設定可）。

Q7. PIN モーダルが表示されない（会計確定でそのまま動く） A7. ブラウザキャッシュが古い可能性。Cmd+Shift+R（または Ctrl+Shift+R）で強制リロード。cashier-app.js?v=20260419-cb1d のクエリ確認。

Q8. PIN を入力したのに「PIN_INVALID」エラー A8. 1) 出勤打刻していない 2) 退勤打刻済 3) PIN が一致しない、のどれか。dashboard でシフト・勤怠を確認、出勤打刻してから再試行。

Q9. 同じ店舗に同じ PIN を持つスタッフが 2 人以上いる場合は？ A9. 早く一致したスタッフが採用される（順序不定）。店舗内で PIN は一意になるよう運用（オーナー/マネージャーが重複チェック）。

Q10. マネージャー / オーナーは PIN なしで操作できないの？ A10. CB1d 以降、できない。マネージャー / オーナーも自分の PIN を設定し、出勤打刻してから操作。例外なし。

Q11. PIN を 4 桁にしたい / 8 桁にしたい A11. 4〜8 桁の範囲で自由に設定可。短いほど推測されやすい、長いほど入力が面倒。6 桁推奨。

Q12. 弱い PIN を弾く仕組みは？ A12. 1234 0000 1111 123456 等の連番・全桁同じは登録時に拒否（api/store/set-cashier-pin.php）。

C. カード決済（5 件） ​

Q13. Stripe Terminal が反応しない A13. 電源 + Wi-Fi 確認。Stripe ダッシュボード > Terminal Readers でオンライン状態確認。【再検出】ボタンで再ペアリング。

Q14. カード決済が途中で切れた A14. Stripe ダッシュボード > Payments で最終状態確認。POSLA 側は webhook で自動同期されるが、念のため取引履歴で確認。失敗していれば再実行。

Q15. 暗証番号入力失敗を繰り返した A15. Stripe Terminal 側で 3 回失敗するとカード一時ロック。別カードまたは現金で再会計。お客様にカード会社問い合わせを案内。

Q16. タッチ決済（NFC）が動かない A16. Terminal の NFC 機能 ON 確認。S700 設定 > 決済方法 > 非接触有効化。カード側も Visa Touch / Master Contactless 対応か確認。

Q17. 海外カードで決済したい A17. Stripe で受付可能なカード（Visa / Master / AMEX 等）なら OK。JCB は別途 Stripe 側で有効化必要。中国銀聯（UnionPay）は要 Stripe 申請。

D. レジ開け・締め（CB1d 含む）（6 件） ​

Q18. レジ開けしないと会計できない？ A18. 開け不要でも会計可能。ただしレジ締めでつじつまが合わなくなる（開始在高 0 円扱い）。毎営業日の最初に必ず開け推奨。

Q19. レジ締めで差額が出た A19. 8.6.1「差額調査の手順」参照。cash_log + audit_log を SELECT して「いつ・誰が・何のために」を追跡。¥1,000 以内は翌日持ち越し、¥5,000 超は manager 監査。

Q20. 1 日に複数回レジ締めしたい A20. 可能。開け → 締め → 再度開け のサイクル。シフト交代時に締めて引き継ぎ運用も推奨。

Q21. 出金時にメモを書き忘れた A21. cash_log.note は INSERT 後の編集不可。audit_log に経緯を補足するため、後続の cash_log に「先ほどの ¥3,000 出金は仕入支払」と記録 or owner-dashboard の「監査ログ」で備忘テキスト追加。

Q22. レジ開閉ボタンが押せない（disabled） A22. すでに営業終了（最後のイベントが close）。再度「レジ開け」セクションが表示されているはず。表示されていなければリロード。

Q23. 入金 / 出金の最大金額は？ A23. システム上は INT の最大値（¥21 億超）まで OK。実運用は ¥100,000 程度を上限に運用ルール化推奨。

E. 個別会計・合流会計・割引（5 件） ​

Q24. 4 人で均等割りしたい A24. 現バージョンは「品目選択 → 個別会計」のみ。均等割りは手計算 → 個別会計で順に支払い実行。将来「均等割りボタン」追加予定。

Q25. 品目指定で分けたい A25. 8.4 参照。チェックボックスで支払う品目を選択 → 「個別会計する」。

Q26. 端数が合わない A26. 自動で 1 円単位まで計算。割引で 1 円単位調整可。

Q27. 何種類の割引が使える？ A27. 金額割引（固定円） / % 割引 の 2 種類。両方を組み合わせる場合は事前に手計算で 1 つの値に。

Q28. 合流会計したテーブルを切り離したい A28. 会計確定前なら【合流会計】トグル OFF で個別状態に戻る。確定後は不可、誤合流は返金で対応。

F. 返金（5 件） ​

Q29. 返金したお金は誰に戻る？ A29. Stripe Terminal 決済はお客様カードに 5〜10 営業日後返金。現金は店舗から手渡し（POSLA 側は記録のみ）。

Q30. 返金履歴を見たい A30. 取引履歴で該当取引を開く →「返金済 ¥XXX」表示。詳細は監査ログ（payment_refund イベント）。

Q31. 間違えて返金した A31. 全額返金後の取消は不可。Stripe ダッシュボードで返金キャンセルできる場合あり（処理待ちの間のみ）。完了済なら新たに同額を再課金（別取引）して相殺。

Q32. 部分返金は可能？ A32. API 上は可能（refund-payment.php に amount パラメータ）。UI は全額返金がデフォルト、入力欄で金額編集可。

Q33. 返金時に PIN を求められない（古いキャッシュ） A33. ブラウザ強制リロード（Cmd+Shift+R）。cashier-app.js?v=20260419-cb1d の ?v= バージョンが新しいことを確認。

G. 決済方法・QR（3 件） ​

Q34. PayPay で決済したい A34. POSLA は PayPay 専用 SDK 統合なし。お客様スマホで PayPay アプリ操作 → POSLA 側は支払方法【QR】で記録のみ。

Q35. d払い / au PAY / 楽天ペイは？ A35. 上記同様、外部アプリで決済 → POSLA は記録のみ。将来的に Stripe 経由の各種 QR 統合検討中。

Q36. 複数の決済方法を併用したい（半分カード + 半分現金） A36. 個別会計（8.4）で品目を 2 グループに分け、片方カード・片方現金で 2 回決済。

H. エラー・トラブル（4 件） ​

Q37. 画面がフリーズした A37. Cmd+Shift+R（Mac）/ Ctrl+Shift+R（Win）で強制リロード。改善しなければブラウザ再起動 → タブレット再起動。

Q38. 取引履歴が昨日以前に遡れない A38. 当日分のみ表示。過去は dashboard >「注文履歴」タブまたは owner-dashboard >「売上レポート」で。

Q39. 「PIN 機能のマイグレーションが未適用です」エラー A39. サーバー側に migration-cp1-cashier-pin.sql + migration-l3-shift-management.sql の適用が必要。POSLA 運営に連絡。

Q40. Stripe Terminal が決済中に画面切替で止まった A40. 別タブ・別アプリへ移動すると S700 SDK が一時停止する場合あり。決済完了まで POSレジ画面を表示し続ける。スリープ防止設定推奨（タブレット設定 > ディスプレイ > スリープなし）。

「PIN_REQUIRED」エラーPIN モーダル未通過リロード（古いキャッシュで PIN モーダルが出ない場合あり）
「PIN_INVALID」エラー出勤打刻なし / PIN 不一致dashboard >「シフト・勤怠」で出勤打刻 → 再試行
「MIGRATION」エラーサーバー側 SQL 未適用POSLA 運営に連絡（migration-cp1-cashier-pin.sql）
[ALREADY_PAID] / [CONFLICT] (409、S3 #3)別端末・別スタッフが先に同じ伝票を会計済み画面リロードで最新状態確認。重複課金は発生していない（サーバーが排他制御で防止）。担当スタッフ間で口頭確認
[AMOUNT_MISMATCH] (409、S3 #4)端末表示額とサーバー再計算額が不一致画面リロードで最新メニュー価格を取得 → 再会計。原因として直前のメニュー価格更新あり
[STRIPE_MISMATCH] (409、S3 #4)Stripe Terminal の retrieve 結果と DB 上の payment_intent_id が一致しないPOSLA 運営に連絡。Stripe Dashboard 側の event_id を併せて伝える
