19. トラブルシューティング ​

よくある問題と対処法をまとめています。

19.0 エラー番号の調べ方（2026-04-19 追加） ​

POSLA の全エラー（API レスポンス・画面表示）には E から始まる 4 桁の番号（E1xxx〜E9xxx）が振られています。何かが失敗したとき、画面には以下のような形式でエラーが表示されます。

[E3024] PIN が一致しません。再度入力してください。

19.0.1 エラー番号を見たらまず何をするか ​

  1. 番号をそのまま控える（紙にメモ・スクショ・口頭でマネージャーに伝える）
  2. 管理画面の右下にある オレンジ色 ? ボタン（AI ヘルプデスク FAB） を開く（14.6 参照）
  3. 質問欄に [E3024] と貼り付けて送信
  4. AI がエラーカタログと該当章を照合して対処方法を返す

エラー番号の意味と対処方法は 章 99「エラーカタログ」（自動生成、docs/manual/tenant/99-error-catalog.md）に全件一覧化されています。AI ヘルプデスクが裏でこのカタログを参照しています。

19.0.2 エラー番号の桁の意味（参考） ​

範囲系統
E1xxx認証・セッション系
E2xxx入力バリデーション系
E3xxx会計・PIN・決済系
E4xxx注文・KDS 系
E5xxx在庫・メニュー系
E6xxx予約系
E7xxx印刷・ハードウェア系
E8xxx外部 API（Stripe / Gemini / Google）系
E9xxxサーバー内部エラー（POSLA 運営対応案件）

19.0.3 エラー番号がない古い画面に遭遇したら ​

[Exxxx] が付かないエラー（旧 UI からのレガシーメッセージ）は、エラーカタログ採番より前の実装か、JS 側のバリデーションです。動作には影響ありません。同じ症状で困った場合は文章そのものを AI ヘルプデスクに貼り付けてください。

19.1 ログイン関連 ​

ログインできない ​

症状原因対処法
「ユーザー名またはパスワードが正しくありません」入力ミス大文字/小文字を確認。ユーザー名は全て小文字の場合が多い
「このアカウントは無効化されています」アカウント無効化オーナーまたはマネージャーに連絡してアカウントを有効化してもらう
「このテナントは無効化されています」テナント自体が無効POSLA運営に連絡
「通信エラーが発生しました」インターネット未接続Wi-Fi/モバイル通信を確認して再試行
「サーバーが空のレスポンスを返しました」サーバー障害しばらく待って再試行。繰り返す場合はPOSLA運営に連絡

セッションが頻繁に切れる ​

  • 25分間操作がないとタイムアウト警告が表示されます
  • 30分で自動ログアウトされます
  • ブラウザのクッキーを削除した場合もログアウトされます
  • 解決策：操作を継続するか、警告が出たら「操作を続ける」ボタンを押す

パスワード変更ができない ​

症状原因対処法
「パスワードは8文字以上で入力してください」8文字未満8文字以上のパスワードに変更
「パスワードには英字を1文字以上含めてください」英字なし（数字だけ等）tanaka123 のように英字を混ぜる
「パスワードには数字を1文字以上含めてください」数字なし（英字だけ等）tanaka123 のように数字を混ぜる
「現在のパスワードが正しくありません」現在のパスワード入力ミス現在のパスワードを確認。忘れた場合はオーナー / マネージャーにリセット依頼
「新しいパスワードと現在のパスワードが同じです」同じパスワード必ず別のパスワードに変更
パスワード変更後に他端末がログアウトされる仕様（セキュリティ強化、P1-5）他端末で再ログインすれば復帰可能

「AIサービスは現在利用できません」と表示される ​

症状原因対処法
AIアシスタントタブで「AIサービスは現在利用できません。POSLA運営にお問い合わせください」サーバー側でGemini APIキー未設定 / API障害POSLA運営に連絡。テナント側で設定するキーはありません

19.2 注文関連 ​

セルフオーダーで注文できない ​

症状原因対処法
「注文する」ボタンが無効ラストオーダー中マネージャーがラストオーダーを解除する（フロアマップの「LO解除」）
QRコードを読み取れないQRコードが古い管理ダッシュボードで新しいQRコードを生成・印刷

ハンディPOSで注文が送信されない ​

症状原因対処法
「カートが空です」カートに品目がないメニューから品目を選択してカートに追加
「テーブルを選択してください」テーブル未選択ドロップダウンからテーブルを選択
通信エラーネットワーク問題Wi-Fi接続を確認。安定した接続を確保

19.3 KDS関連 ​

KDSに注文が表示されない ​

症状原因対処法
全く注文が表示されないステーション選択の問題「全品目」ボタンを選択して確認
一部の品目だけ表示されないステーションのカテゴリ割当店舗設定でステーションの担当カテゴリを確認
更新が遅いポーリング3秒間隔で自動更新されます。手動更新はありません
音が鳴らない音声未許可KDS起動時の「タップして開始」を確実に行う

19.4 会計・レジ関連 ​

レジで会計できない ​

症状原因対処法
会計待ちテーブルが表示されない未会計の注文がない注文が入っているか確認
「会計する」ボタンが押せない処理中二重押し防止。しばらく待つ
カードリーダーが接続できないStripe Terminal問題「再検出」ボタンで再接続。端末の電源を確認

PIN を求められる操作の一覧（CB1d、2026-04-19 更新） ​

会計・返金・レジ金庫操作（開け / 閉め / 入金 / 出金） の全てで担当スタッフ PIN が必須です。マネージャー/オーナーも例外なし。

操作PIN 必須
会計確定（現金 / カード / QR / Terminal）✅
返金処理✅
レジ開け / レジ閉め / 入金 / 出金（CB1d 新規）✅

「PIN が一致しないか、対象スタッフが出勤中ではありません」エラーが出る場合は: ① 自分の出勤打刻を確認 / ② PIN 設定済みか確認 / ③ 入力ミスがないか確認、の順で対処してください。

レジの過不足が大きい ​

  • レジレポートでキャッシュログを確認し、入金・出金の記録漏れがないか確認
  • 出金を記録せずに現金を取り出していないか確認（CB1d 以降は出金にも PIN 必須なので、PIN なし出金は物理的に不可能）
  • 閾値（デフォルト¥500）を超える過不足は自動で警告表示されます
  • CB1d 以降、レジ開け/閉め/入金/出金には誰が操作したかが audit_log に記録されているので、差額調査時に「いつ・誰が・何のために動かしたか」を追跡できます

19.5 ネットワーク関連 ​

オフライン警告が表示される ​

POSLAはクラウド型システムのため、常時インターネット接続が必要です。

  • 画面上部に赤いバナーで「オフライン」警告が表示されます
  • オフライン中は新規注文の送信やデータ取得ができません
  • 接続が復旧すると自動的に緑のバナーが表示され、通常動作に戻ります

対処法：

  1. Wi-Fiルーターの状態を確認
  2. モバイル通信に切り替えて試す
  3. 有線LAN接続を検討（安定性向上）
  4. バックアップ用のモバイルルーターを準備

セルフオーダーの注文がオフライン時に保留される ​

お客さんのスマートフォンがオフラインの場合、注文はブラウザに保留されます。接続が復旧すると自動的に送信されます。24時間以上経過した保留注文は自動破棄されます。

管理ダッシュボードでメニューを変更した後、各画面への反映タイミングは以下のとおりです。

画面反映タイミング

19.9 POSLA運営への問い合わせ ​

上記で解決しない場合は、POSLA運営（プラスビリーフ）にお問い合わせください。

問い合わせ時に以下の情報をお伝えいただくと、迅速な対応が可能です。

  • テナント名・店舗名
  • 問題が発生した日時
  • 問題の症状（エラーメッセージがあれば正確に）
  • 使用しているブラウザとOS
  • 問題が発生する画面（管理ダッシュボード/KDS/ハンディPOS/セルフオーダー等）

19.10 緊急対応フロー（重大障害時） ​

19.10.1 すぐ確認すべき 5 つの項目 ​

  1. POSLA は動いている？ — https://eat.posla.jp/api/monitor/ping.php を Chrome で開く → {"ok":true} が返れば OK
  2. 店内 Wi-Fi は生きている？ — タブレットの Wi-Fi マークが扇形 3 本表示されているか
  3. インターネットに繋がってる？ — Chrome で https://google.com を開いて表示されるか
  4. 電源は来てる？ — タブレット・ルーター・プリンタ・カードリーダーの電源ランプ確認
  5. 特定の端末だけ？ — 別の端末で同じ操作を試してみる

19.10.2 障害判別フローチャート ​

1 端末だけ動かない
    ↓
   端末再起動 → 改善？
    ↓ NO
   ブラウザリロード（Cmd+Shift+R） → 改善？
    ↓ NO
   別ブラウザ・別端末で同じ操作 → 動く?
    ↓ YES
   元端末の問題 → 端末交換 or サポート連絡

全端末動かない
    ↓
   POSLA 全体障害の可能性 → POSLA 運営に連絡
   または、店舗 Wi-Fi 障害の可能性 → ルーター再起動

19.10.3 営業継続のためのバックアップ手順 ​

POSLA が完全に止まった場合の 営業継続手順:

  1. 手書き伝票 で注文を取る
  2. 電卓 で会計
  3. 現金 メインで対応（カード決済は不可）
  4. レシートは手書き or 後日発行
  5. 復旧後、手書き伝票を POSLA に手動入力（売上記録のため）

手書き伝票テンプレートを準備

営業中の障害に備えて、手書き伝票（カーボン紙）を最低 50 枚は店舗に常備してください。

19.13 トラブル切り分けの基本フローチャート ​

POSLA で「動かない」と言われたとき、まずこのフローを上から順に確認してください。

    ├─ Step 1: スコープ判定 ───────────────────────

    │   ┌─ ブラウザリロード (Cmd+Shift+R) → 直る? → 終了

    │   └─ 全部 NG → POSLA 運営連絡
    │
    ├─ Step 3: 店舗単位問題
    │   ┌─ ルーター再起動 (電源 30 秒オフ) → 直る? → 終了
    │   ├─ https://google.com 開ける? → No なら回線会社へ
    │   ├─ POSLA 設定が誤っていないか (タイムアウト/権限) → 修正
    │   └─ 全部 NG → POSLA 運営連絡
    │
    └─ Step 4: 全体障害の可能性
        ┌─ https://eat.posla.jp/api/monitor/ping.php
        │   → {"ok":true} 返る? → No なら POSLA 全停止 (運営に連絡)
        └─ POSLA 運営の Slack/メール通知を確認
            → 通知あり → 復旧待ち
            → 通知なし → POSLA 運営に第一報

19.14 ブラウザ・キャッシュ・セッション・GPS・Wi-Fi の切り分け ​

「ログイン後すぐ画面が真っ白」「ボタンを押しても何も起きない」など、現象が漠然としているときは以下 5 点を順に切り分けます。

19.14.1 ブラウザキャッシュ ​

症状確認方法対処
古い UI が表示されるDevTools → Network → JS の Status: 200 (from disk cache)Cmd+Shift+R で強制リロード
キャッシュバスター (?v=xxx) を更新しても古い同上DevTools → Application → Storage → Clear site data
iOS Safari でしつこいキャッシュキャッシュバスターも効かない設定アプリ → Safari → 履歴と Web サイトデータを消去

19.14.2 セッション ​

症状確認方法対処
操作ごとに「ログインしてください」DevTools → Application → Cookies で PHPSESSID 確認Cookie ブロック解除 / サードパーティ Cookie 許可
25 分でログアウト警告仕様 (CLAUDE.md セッション TTL)操作継続 or 「操作を続ける」クリック
別端末でログインしたら旧端末が落ちたパスワード変更直後旧端末で再ログイン (P1-5 仕様)

19.14.3 GPS ​

症状確認方法対処
「位置情報の取得に失敗しました」ブラウザ設定 → 位置情報 → eat.posla.jp 許可許可に変更
「店舗から離れすぎています」16 章で店舗 GPS 半径 (デフォルト 200m) を確認半径を 300-500m に拡大 / 店舗位置を再設定
屋内で GPS が取れない商業ビル地下 / 鉄筋ビル中央Wi-Fi 位置情報を ON / 半径を 500m 以上
device で GPS 強制が効かないgps_required カラム値device は常に強制 (kds-auth.js 参照)

19.14.5 ブラウザ DevTools の使い方（基本） ​

  1. Chrome で問題画面を開く
  2. F12 または Cmd+Option+I で DevTools を開く
  3. 上部タブから:
    • Console: JS エラー (赤字) を確認
    • Network: API レスポンス (/api/...) のステータス確認
    • Application → Storage: Cookie / localStorage / Cache 確認
  4. POSLA 運営に問い合わせる時は Console のスクリーンショット を添付すると 5 倍速く解決する

19.16 実例ベースのトラブル対応（matsunoya / torimaru） ​

19.16.1 matsunoya 渋谷店: ピーク時に KDS が固まった ​

症状: 19:00 のピーク時、KDS 画面の更新が止まり、新規注文が表示されなくなった。

切り分け:

  1. ✅ 1 店舗のみ → Step 3 (Wi-Fi/設定) へ
  2. ✅ ルーター再起動 → 改善せず
  3. ✅ https://google.com は開ける → 回線は生きている
  4. ✅ 別タブで eat.posla.jp/api/monitor/ping.php → {"ok":true} → POSLA は生きている
  5. ✅ DevTools Console → Failed to fetch /api/kds/orders.php が連発 → サーバー側の負荷
  6. → POSLA 運営に連絡 → 5 分後復旧 (DB 接続プール枯渇が原因)

学び: ピーク時のフリーズは個別店舗側で解決しないことが多い。POSLA 運営が監視 (I-1) で気づくが、報告ありの方が早い。

19.16.2 torimaru: パスワードを忘れて全員入れない ​

症状: 個人店、オーナーが旅行中に唯一のスタッフがパスワードを忘れた。

切り分け:

  1. ✅ ロール: staff → manager / owner にリセット依頼が必要
  2. ✅ owner と LINE 通話 → owner のスマホからログイン → スタッフのパスワードリセット
  3. ✅ 仮パスワードをスタッフに口頭で伝える → スタッフ自身が変更

学び: owner の連絡先は店舗内に必ず控えておく。1 オーナー店舗は特にリスクが高い。

19.17 よくある質問 (FAQ) ​

Q1. 「動かない」と思ったら最初に何をすべき?

A. ブラウザリロード (Cmd+Shift+R)。次にエラー番号をメモして AI ヘルプデスク FAB に貼り付ける。

Q2. POSLA 運営の電話番号は?

A. 電話窓口は提供していません。info@posla.jp メール、または POSLA 内蔵の「お問い合わせ」フォームを使ってください。

Q3. POSLA が完全に止まったら売上はどうなる?

A. 19.10.3 のバックアップ手順で手書き伝票運用 → 復旧後手動入力。手書き伝票は最低 50 枚常備推奨。

Q4. AI ヘルプデスクの回答が間違っていることがある

A. AI は確率的なので 100% 正確ではありません。重要な操作は本マニュアルの該当章を必ず確認してください。

Q5. エラー番号が [Exxxx] 形式じゃない場合は?

A. レガシーメッセージです。文章そのままを AI ヘルプデスクに貼り付けるか、19.X の各章別トラブル参照へ。

Q6. 「権限が不足しています」が頻発する

A. ロール (staff/manager/owner) と所属店舗を確認。3 章 3.18 のロール別タブ可視性表を参照。

Q7. デバイス端末 (KDS / レジ / ハンディ) でトラブル

A. device アカウントは特殊です。23 章 (KDS デバイス端末ガイド) と 2 章 2.5 (デバイス管理) を参照。

Q8. レポートの数字が他と合わない

A. 集計タイミングの差です。13 章「レポート」の集計仕様を参照。日次集計は翌日朝に確定。

Q9. プリンタが印刷されない

A. 26 章「プリンタ連携」の 26.11 を参照。IP プリンタ専用なので USB / Bluetooth は対象外。

Q10. Stripe / スマレジ連携で詰まった

A. 21 章 (Stripe) / 22 章 (スマレジ、internal) のトラブル節を参照。決済関連は POSLA 運営でも事象再現が難しいので、エラー番号 + Stripe Dashboard のスクショ必須。

Q11. メニュー編集が反映されない

A. 19.6 参照。3 分のメニューバージョンチェック後反映。急ぎはブラウザリロード。

Q12. 予約・テーブル開放のトラブル

A. 24 章 (予約) / 25 章 (テーブル開放認証) のトラブル節を参照。

Q13. PIN を求められる操作で何度も弾かれる

A. CB1d 仕様: ① 自分の出勤打刻を確認 / ② PIN 設定済みか確認 / ③ 入力ミス。3 回連続失敗で 10 分ロック。

Q14. iOS Safari で音声コマンドが動かない

A. 仕様 (Web Speech API continuous モード非対応)。Chrome を使ってください。

Q15. Android 古機種で位置情報が取れない

A. Android 8 以下は非対応。新しい端末に交換してください。

19.S3 S3 セキュリティ修正で追加された新エラーコード（2026-04-19） ​

S3 セキュリティ修正一式で新規追加されたエラー文字列です。E 番号は次回 catalog 再生成で採番されます。詳細は 18.W S3 セキュリティ修正一式 を参照。

エラー文字列HTTP発生 API意味と対処
INVALID_ITEM400handy-order, takeout-orders注文に存在しないメニュー ID が含まれる → 端末再読込で最新メニューを取得
INVALID_OPTION400handy-order, takeout-orders注文に存在しないオプション ID が含まれる → 端末再読込
INVALID_QUANTITY400handy-order, takeout-orders数量が 0 / 負数 / 上限超過 → 数量を確認して再送信
ALREADY_PAID409process-payment, checkout-confirm別スタッフが先に会計済み → 画面リロードで最新状態確認。重複課金は発生していない
AMOUNT_MISMATCH409process-payment, handy-orderクライアント表示金額とサーバー再計算金額が不一致 → 端末再読込で最新価格取得
STRIPE_MISMATCH409process-paymentStripe Terminal の retrieve 結果が DB の payment_intent と不一致 → POSLA 運営に連絡
PAYMENT_NOT_CONFIRMED402signup/activateStripe Subscription が active/trialing でない → カード登録を完了してから再試行
INVALID_TOKEN404signup/activatesignup_token が DB に存在しない → メール本文の URL を再確認
CONFLICT409process-payment, reservation-create, checkout-confirmレースコンディション検出 → 画面リロード後に再操作
CAPACITY_EXCEEDED409takeout-ordersテイクアウト時間枠の容量を超過 → 別の時間枠を選択
WEBHOOK_NOT_CONFIGURED503signup/webhookstripe_webhook_secret が POSLA 運営側で未設定 → 運営に連絡

CONFLICT / ALREADY_PAID が出たら

焦らず画面をリロードしてください。多くの場合、別のスタッフ（あるいは別の端末）が同じ操作を一足先に完了しています。サーバー側の排他制御が正しく機能している証拠なので、データ破損は発生していません。
