# POSLA管理画面 運用強化機能案

作成日: 2026-04-25
対象: `public/posla-admin/`
位置付け: 未実装の設計メモ。manual 正本にはまだ入れない。

---

## 1. 目的

POSLA管理画面を「設定画面」から一段上げて、POSLA運営が以下を素早く判断できるようにする。

- 今どのテナントが危ないか
- 新規テナントがどこまで立ち上がっているか
- 誰がどの設定を変えたか
- 問い合わせ時にどの順で切り分ければよいか

前提:

- 既存の主要タブは `ダッシュボード / テナント管理 / テナント作成 / API設定 / PWA / Web Push / 管理者ユーザー / 顧客サポート / サポートセンター / 運用補助センター`
- 既存 API は `api/posla/dashboard.php`, `tenants.php`, `settings.php`, `monitor-events.php`, `customer-support.php`, `admin-users.php`, `push-vapid.php`
- 単一プラン運用なので、`plan` 差分ではなく `hq_menu_broadcast` と稼働状態を見る

## 3. まず入れるべき3機能

## 3.1 テナント健全性スコア

### 目的

テナント一覧を見た瞬間に「正常 / 要確認 / 危険」が分かる状態にする。

### 置き場所

- `ダッシュボード`
  - 「危険テナント 上位5件」カードを追加
- `テナント管理`
  - 一覧に `健全性` 列を追加
  - 詳細モーダルに `健全性内訳` セクションを追加

### スコア算出方針

100 点満点から減点。

- -40: `tenants.is_active = 0`
- -20: `stores` が 0 件
- -15: `users` が 0 件
- -10: owner ロールが 0 件
- -10: `store_settings` 欠落店舗あり
- -10: `payment_gateway='stripe'` なのに
  - `tenants.stripe_secret_key` 未設定
  - かつ `stripe_connect_account_id + connect_onboarding_complete=1` でもない
- -10: 直近 24 時間で `monitor_events` の `error/critical` が 3 件以上
- -5: 直近 24 時間で `reservation_notifications_log` 失敗あり
- -5: 直近 24 時間で `push_send_log` failure_rate が閾値超え

### 表示

- `80-100`: 緑 `正常`
- `50-79`: 黄 `要確認`
- `0-49`: 赤 `要対応`

### 補足

- `Gemini / Places / VAPID / Google Chat` は POSLA 共通設定なので、テナント個別の減点に直接入れず「共通依存」として別表示にする
- `Smaregi` は全テナント必須ではないので、スコア減点ではなく `任意連携` 表示にする

## 3.2 オンボーディング完了率

### 目的

新規テナントを作ったあと、「ログインはできるが検証を始められない」を減らす。

### 置き場所

- `テナント管理`
  - 一覧に `導入状況` 列を追加
- `テナント編集モーダル`
  - `オンボーディング進捗` セクション追加

### 判定ステップ

必須:

1. テナント作成済
2. 店舗 1 件以上
3. `store_settings` 作成済
4. owner アカウントあり
5. manager / staff / device の初期アカウントあり
6. テーブル 1 件以上
7. メニュー 1 件以上

任意:

8. Stripe 接続完了
9. Smaregi 接続完了
10. Push 購読あり

### 表示ルール

- 必須 7 項目で `xx%`
- 任意 3 項目は別バッジ
  - `決済 ready`
  - `Smaregi ready`
  - `Push ready`

### 初期目標

`テナント作成直後に owner / manager / staff / device が入る` ところまでは既に `api/posla/tenants.php` で実装済。
次は「テーブル / メニュー未設定」が一覧で見えるようにする。

---

## 3.3 危険テナントの上位表示

### 目的

ダッシュボードを見た瞬間に、POSLA運営が今日見るべき対象を特定できるようにする。

### 置き場所

- `ダッシュボード`
  - `危険テナント`
  - `立ち上げ途中テナント`
  - `直近エラー多発テナント`

### 抽出条件

- 危険テナント
  - 健全性スコア 50 未満
- 立ち上げ途中
  - 作成 7 日以内 and オンボーディング 100% 未満
- 直近エラー多発
  - `monitor_events` の `error/critical` が 24 時間で 3 件以上

- `subscription_events`
- `reservation_notifications_log`

用途:

- テナント問い合わせ時の時系列確認

## 6. DB設計案

## 6.1 Phase 1

追加なし。

既存テーブルだけで成立する。

- `tenants`
- `stores`
- `users`
- `store_settings`
- `monitor_events`
- `subscription_events`
- `reservation_notifications_log`
- `push_subscriptions`
- `push_send_log`

## 6.2 Phase 2: 設定変更履歴

新規テーブル推奨:

### `posla_admin_audit_log`

目的:

- POSLA管理画面で誰が何を変更したかを保存
- tenant/store の `audit_log` と混ぜない

カラム案:

- `id` VARCHAR(36) PK
- `admin_id` VARCHAR(36) NOT NULL
- `admin_email` VARCHAR(255) NOT NULL
- `action` VARCHAR(64) NOT NULL
- `target_type` VARCHAR(64) NOT NULL
- `target_id` VARCHAR(36) DEFAULT NULL
- `tenant_id` VARCHAR(36) DEFAULT NULL
- `before_json` JSON or TEXT
- `after_json` JSON or TEXT
- `ip_address` VARCHAR(64) DEFAULT NULL
- `user_agent` VARCHAR(255) DEFAULT NULL
- `created_at` DATETIME NOT NULL

主な action:

- `posla_settings_update`
- `tenant_update`
- `tenant_create`
- `admin_user_create`
- `admin_user_update`
- `admin_user_deactivate`
- `push_vapid_generate`

### 代替案

既存 `audit_log` を流用して `tenant_id='posla'` で入れる方法もある。
ただし tenant 向け監査と混ざるので、分離の方が安全。

## 7. 実装順

## Step 1

既存 `dashboard.php` / `tenants.php` を拡張して次を出す。

- `health_score`
- `onboarding_progress`
- `recent_error_count_24h`

メリット:

- 新規 API が最少
- UI 変更も少ない

## Step 2

`dashboard.html` / `posla-app.js` で以下を追加。

- ダッシュボードの危険テナントカード
- テナント一覧の健全性 / 導入状況列
- テナントモーダルの健全性内訳

## Step 3

必要になったら `tenant-timeline.php` を追加する。

用途:

- 問い合わせ対応
- 障害の一次切り分け

## Step 4

最後に `posla_admin_audit_log` を追加して設定変更履歴を入れる。

理由:

- 監査ログは価値が高い
- ただし Phase 1 より DB 変更の影響が大きい

---

## 10. 実装時の注意

- `Smaregi` は全テナント必須扱いにしない
- 共通設定不足 (`gemini_api_key`, `google_places_api_key`, `web_push_vapid_*`) はテナント個別スコアに混ぜず、共通依存として別表示
- `payment_gateway='none'` のテナントを即 NG 扱いしない
- `health_score` は万能判定ではなく、`flags` の中身を必ず並記する
- `tenant_id` / `store_id` の境界は既存ルールどおり厳守

---

## 11. 一言でいうと

POSLA管理画面に次に足すべきものは、
**「編集機能」より先に「見える化機能」** です。

最初の 3 つだけで、POSLA運営の判断速度はかなり上がる。
