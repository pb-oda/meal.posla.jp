## L-3 シフト管理

### 競合分析

| 競合 | 強み | 弱み |
|------|------|------|
| Airシフト（リクルート） | シフトボードとの連携、スタッフが自分でシフト希望を提出、自動シフト作成 | POS連携が薄い、売上データとの紐付けなし |
| スマレジ・タイムカード | スマレジPOSと統合、勤怠管理＋給与計算 | シフト「作成」機能は弱い、希望収集が手動的 |
| ジョブカン | 勤怠管理の老舗、複雑なシフトパターン対応、労基法チェック | 飲食特化ではない、POS連携なし |
| HANJO シフト管理 | 飲食特化、AI自動シフト作成 | 単体ツール、POSとは別契約 |

### POSLAの差別化ポイント

1. **POS売上データ連動のAIシフト提案** — 曜日×時間帯の売上実績から「この時間帯は3人必要」をAIが提案。競合はどこもPOS売上データとシフトを直接連動させていない
2. **スタッフ希望収集〜確定までワンストップ** — スタッフがスマホで希望提出→マネージャーがドラッグ&ドロップで調整→確定通知。外部ツール不要
3. **労働基準法コンプライアンスアラート** — 週40時間超過、6時間超の休憩未設定、深夜勤務の割増計算を自動チェック
4. **人件費リアルタイムシミュレーション** — シフト編集中に人件費が即座に反映。売上予測と対比して人件費率を表示
5. **マルチ店舗ヘルプ要請** — enterprise プランで店舗間のスタッフ貸し借りをシフト上で管理

### 実装スコープ

**Phase 1（MVP）**
- シフトテンプレート（週次パターン）
- スタッフ希望提出画面（モバイル対応）
- マネージャー用シフト編集画面（週次カレンダー）
- 勤怠打刻（出勤/退勤）— dashboard.html にボタン追加

**Phase 2（AI強化）**
- 売上データ連動のAIシフト提案（Gemini API）
- 人件費シミュレーション
- 労基法コンプライアンスチェック

**Phase 3（マルチ店舗）**
- 店舗間ヘルプ要請・承認フロー
- 統合シフトビュー（enterprise）

### DB設計（概要）※ 詳細は docs/L-3-shift-design.md 参照

```
shift_templates      — 週次テンプレート（day_of_week, start_time, end_time, required_staff, role_hint）
shift_assignments    — シフト割当（user_id, shift_date, start_time, end_time, break_minutes, role_type, status）
shift_availabilities — スタッフ希望（user_id, target_date, availability, preferred_start, preferred_end）
attendance_logs      — 勤怠打刻（user_id, clock_in, clock_out, break_minutes, status）
shift_settings       — 店舗別シフト設定（submission_deadline_day, default_break_minutes, auto_clock_out_hours）
```

## L-4 多言語対応

### 競合分析

| 競合 | 強み | 弱み |
|------|------|------|
| dinii | セルフオーダーの多言語対応、写真メニュー | 翻訳精度が機械的、メニュー説明文が不自然 |
| スマレジ | 基本的な多言語メニュー表示 | 対応言語が限定的、翻訳は手動入力 |
| Square | 英語UI標準、多言語レシート | 日本語→他言語の自動翻訳なし |
| Waiter.com / Toast | 英語圏POS、スペイン語対応 | アジア言語対応が弱い |

### POSLAの差別化ポイント

1. **Gemini APIによる高品質自動翻訳** — メニュー名・説明文を5言語（日/英/中簡/中繁/韓）に自動翻訳。手動入力不要。飲食用語に最適化したプロンプトで「焼き鳥」→「Yakitori (Grilled Chicken Skewers)」のように文化的コンテキスト付き
2. **セルフオーダー画面の完全多言語化** — メニュー名だけでなく、カテゴリ名、アレルゲン表示、注文確認画面まで全て翻訳
3. **スタッフ向けKDS二言語表示** — KDS画面で日本語＋英語を併記。外国人スタッフが働きやすい環境
4. **言語自動検出** — ブラウザの言語設定から自動で表示言語を切り替え（手動選択も可能）
5. **翻訳キャッシュ** — 一度翻訳した内容はDB保存。API呼び出しコストを最小化

### 実装スコープ

**Phase 1（MVP）**
- メニュー翻訳テーブル（menu_translations）
- Gemini API一括翻訳エンドポイント（メニュー登録/更新時に自動実行）
- セルフオーダー画面の言語切り替えUI（国旗アイコン）
- 対応言語: ja, en, zh-Hans, zh-Hant, ko

**Phase 2（拡張）**
- KDS二言語表示モード
- カテゴリ・アレルゲン・オプションの翻訳
- 翻訳の手動編集UI（オーナーが修正可能）

**Phase 3（高度）**
- AIによる料理説明文の自動生成（写真→説明文→翻訳）
- 音声案内の多言語化（Web Speech API）

### DB設計（概要）

```
menu_translations — メニュー翻訳キャッシュ（menu_item_id, lang, name, description, translated_at）
ui_translations   — UI文言翻訳（key, lang, value）— 静的に初期投入
```

## L-5 領収書・インボイス

### 競合分析

| 競合 | 強み | 弱み |
|------|------|------|
| スマレジ | インボイス制度完全対応、レシートプリンタ連携 | 物理プリンタ前提、電子発行が弱い |
| Airレジ | 無料でインボイス対応レシート発行 | PDF出力なし、メール送信なし |
| Square | 電子レシート（メール/SMS）、多通貨対応 | 日本のインボイス制度への対応が後追い |
| freee | 請求書・インボイス発行の本格ツール | POS連携が薄い、飲食向けではない |

### POSLAの差別化ポイント

1. **適格簡易請求書の完全自動生成** — 注文完了時にインボイス制度準拠のPDFを自動生成。登録番号（T+13桁）、税率区分（8%/10%）、税額計算を自動処理
2. **電子発行ファースト** — QRコードからPDFダウンロード or メール送信。レシートプリンタ不要でコスト削減
3. **軽減税率の自動判定** — メニューアイテムに「飲食料品（8%）」「酒類（10%）」「テイクアウト（8%）」のフラグを設定。混在注文でも自動で税率区分
4. **領収書の後発行** — 過去の注文から領収書を再発行可能。宛名の後入力にも対応
5. **電子帳簿保存法対応** — PDF保存にタイムスタンプ付与、検索要件（日付・金額・取引先）を満たすDB設計

### 実装スコープ

**Phase 1（MVP）**
- 適格簡易請求書PDF生成（TCPDF or DomPDF）
- 税率区分対応（8%/10%）
- メニューアイテムへの税率フラグ追加
- 領収書発行画面（注文詳細から「領収書発行」ボタン）
- QRコード付きPDFダウンロードリンク

**Phase 2（拡張）**
- メール送信機能
- 過去注文からの再発行
- 宛名入力（会社名・個人名）
- 月次売上の税率区分別集計レポート

**Phase 3（高度）**
- レシートプリンタ連携（Star/Epson WebPrint）
- 電子帳簿保存法の検索要件対応

### DB設計（概要）

```
receipts         — 領収書/インボイス発行記録（order_id, receipt_number, type, pdf_path, issued_at）
receipt_settings — 店舗の領収書設定（store_id, registration_number, business_name, address）
-- menu_items に tax_rate_division カラム追加（'8' or '10'）
```

## L-6 テイクアウト番号呼出

### 競合分析

| 競合 | 強み | 弱み |
|------|------|------|
| dinii | テイクアウト注文〜受取までのフロー完備 | 番号表示が単純、カスタマイズ性低い |
| Square | オンライン注文→店舗受取のフロー | 日本の飲食店向けUIではない |
| モバイルオーダー各社 | LINE連携で通知 | 店頭ディスプレイとの連動が弱い |
| マクドナルド等大手 | 大型ディスプレイでの番号表示 | 自社専用システム、汎用性なし |

### POSLAの差別化ポイント

1. **KDS連動の自動番号進行** — KDS で「提供完了」を押すと自動的に番号呼出画面に反映。手動操作ゼロ
2. **タブレット/TV表示ページ** — 店頭に置くタブレットやTVに全画面表示できる専用ページ。フォントサイズ・色・レイアウトをカスタマイズ可能
3. **音声読み上げ** — Web Speech API で「番号〇〇番のお客様、お料理ができました」を自動音声
4. **リアルタイム更新** — Server-Sent Events（SSE）で表示ページが自動更新。ポーリング不要
5. **多言語番号呼出** — L-4と連動して日本語/英語での音声呼出に対応

### 実装スコープ

**Phase 1（MVP）**
- テイクアウト注文に自動採番（日次リセットの連番）
- 番号呼出ディスプレイページ（`/public/takeout-display.html`）
- KDS「提供完了」→番号ステータス更新API
- SSEによるリアルタイム更新
- 準備中 / お呼び出し中 / 受取済み の3ステータス

**Phase 2（拡張）**
- 音声読み上げ（Web Speech API）
- 待ち時間表示（平均調理時間から推定）
- カスタマイズUI（フォントサイズ・背景色・ロゴ）

**Phase 3（高度）**
- スマホ通知（注文時のブラウザ通知許可→Push API）
- LINE通知連携

### DB設計（概要）

```
takeout_numbers — テイクアウト番号管理（order_id, number, status, called_at, picked_up_at）
-- orders テーブルに order_type カラム追加（'dine_in' / 'takeout'）
```

## L-8 セルフレジ・QR決済

### 競合分析

| 競合 | 強み | 弱み |
|------|------|------|
| スマレジ | セルフレジモード標準搭載、多決済対応 | ハードウェア前提（専用端末） |
| Square | QR決済、タッチ決済、オンライン決済統合 | セルフレジUIが飲食特化でない |
| dinii | テーブル会計→QRで決済完結 | 対応決済が限定的 |
| PayPay（店舗向け） | QRコード決済のデファクト | POS連携は別途必要、注文情報と紐付かない |
| Stripe | 柔軟なAPI、多決済手段対応 | 飲食向けUIは自分で構築する必要あり |

### POSLAの差別化ポイント

1. **テーブルQR→会計完結** — セルフオーダーの延長線上で「お会計」ボタン→Stripe決済画面→完了→テーブルセッション自動クローズ。スタッフ介在ゼロ
2. **割り勘対応** — 「2人で割る」「金額指定で分ける」「アイテム別に分ける」の3パターン
3. **複合決済** — 一部現金＋一部QR決済のような混合支払いに対応
4. **Stripe + PayPay二本立て** — カード決済（Stripe）+ QRコード決済（PayPay）で日本の主要決済を網羅
5. **レジ待ちゼロ体験** — 食事が終わったらテーブルのQRをスキャン→スマホで決済→そのまま退店。レジに並ぶ必要なし

### 実装スコープ

**Phase 1（MVP）**
- セルフ会計ページ（`/public/self-checkout.html`）— テーブルQRから遷移
- 注文内容確認→Stripe Checkout Session作成→決済→完了
- 決済完了時のテーブルセッション自動クローズ
- Webhook受信（支払い確定の非同期確認）

**Phase 2（拡張）**
- 割り勘機能（均等割り・金額指定・アイテム別）
- PayPay決済連携
- レシート自動発行（L-5連動）

**Phase 3（高度）**
- 複合決済（現金＋電子決済ミックス）
- ポイント/クーポン適用
- 決済分析レポート

### DB設計（概要）

```
payments — 決済記録（order_id, method, amount, stripe_session_id, status, paid_at）
-- orders テーブルに payment_status カラム追加（'unpaid'/'paid'/'partial'）
```

## L-9 予約管理

### 競合分析

| 競合 | 強み | 弱み |
|------|------|------|
| トレタ | 予約管理の定番、グルメサイト連携、顧客台帳 | 月額費用が高い（1.2万〜）、POS連携は別途 |
| ebica | 複数グルメサイト一元管理、ダブルブッキング防止 | 高価格帯、中小店舗にはオーバースペック |
| TableCheck | ノーショー対策（デポジット/クレカ保証）、直接予約で手数料削減 | 導入ハードルが高い |
| ホットペッパー予約台帳 | 無料、ホットペッパーからの送客 | ホットペッパー依存、独自サイト連携なし |
| POSLA現状 | フロアマップ・テーブル管理は既にある | 予約機能がまだない |

### POSLAの差別化ポイント

1. **フロアマップ統合** — 既存のフロアマップUI上で直接テーブルを予約に紐付け。視覚的に空席/予約済みが一目瞭然
2. **Web予約フォーム自動生成** — 店舗のWeb予約ページを自動生成。外部サービス不要で直接予約を受付（手数料ゼロ）
3. **ノーショー防止** — Stripe連携でデポジット徴収 or クレカ事前登録。L-8のインフラを再利用
4. **AI需要予測連動** — 過去の予約データと売上データからAIが混雑予測。「この時間帯は予約を制限すべき」をアラート
5. **顧客台帳自動蓄積** — 予約情報＋注文履歴＋来店回数を自動で顧客プロファイルに蓄積。リピーター管理の基盤
6. **SMS/LINE リマインダー** — 予約前日に自動リマインド送信。ノーショー率を大幅削減

### 実装スコープ

**Phase 1（MVP）**
- 予約テーブル（reservations）
- 予約管理画面（カレンダー＋リスト表示）— dashboard.html に「予約」タブ追加
- フロアマップとの連動（予約済みテーブルのハイライト）
- Web予約フォーム（`/public/reservation.html`）— 店舗ID付きURL

**Phase 2（拡張）**
- デポジット徴収（Stripe連携）
- SMS/メールリマインダー
- 顧客台帳（customer_profiles テーブル）
- Google ビジネスプロフィール連携

**Phase 3（高度）**
- AI需要予測連動のキャパシティ管理
- LINE公式アカウント連携
- グルメサイト連携（ホットペッパー/食べログ API）

### DB設計（概要）

```
reservations      — 予約（store_id, customer_name, phone, email, date, time, party_size, table_id, status, deposit_amount, notes）
customer_profiles — 顧客台帳（tenant_id, name, phone, email, visit_count, last_visit, total_spent, notes）
```

## 実装優先順位

### 評価基準

| 基準 | 説明 |
|------|------|
| デモインパクト | デモで見せたときの「おっ」感 |
| 実装難易度 | 開発コスト・期間 |
| 既存インフラ活用 | 現在のコードベースの流用度 |
| 顧客価値 | 飲食店オーナーにとっての実用性 |

### 推奨順序

| 順位 | 機能 | 理由 |
|------|------|------|
| **1** | **L-4 多言語対応** | Gemini API連携が既にあり実装が最速。セルフオーダー画面の多言語化はデモで即座にインパクト大。インバウンド需要に直結 |
| **2** | **L-5 領収書・インボイス** | 法的要件（インボイス制度）で全飲食店が必要。PDF生成は独立モジュールで既存に影響少。実用性が極めて高い |
| **3** | **L-6 テイクアウト番号呼出** | KDS連動で実装がシンプル。ディスプレイページは独立しておりリスク低。テイクアウト需要の取り込み |
| **4** | **L-8 セルフレジ・QR決済** | Stripe連携が既にpro計画にある。セルフオーダー→セルフ決済の体験完結はデモの目玉になる。ただしStripe審査に時間がかかる |
| **5** | **L-3 シフト管理** | 実装規模が大きいが、POS連動のAIシフトは差別化の柱。Phase 1（基本CRUD）は比較的シンプル |
| **6** | **L-9 予約管理** | 最も実装規模が大きい。ただしフロアマップ既存資産を活用できる。Web予約フォームは集客に直結 |

### 推定工数（Phase 1 MVP）

| 機能 | 推定 | 新規ファイル概算 |
|------|------|-----------------|
| L-4 多言語 | 3-4日 | API 2本、JS 1本、DBマイグレーション 1本 |
| L-5 領収書 | 4-5日 | API 3本、PDF生成ライブラリ、JS 1本、DBマイグレーション 1本 |
| L-6 番号呼出 | 2-3日 | API 2本、HTML 1本、JS 1本、SSEエンドポイント 1本 |
| L-8 セルフレジ | 5-7日 | API 3本、HTML 1本、JS 1本、Webhook 1本、DBマイグレーション 1本 |
| L-3 シフト管理 | 7-10日 | API 5本、JS 2本、HTML(モバイル) 1本、DBマイグレーション 1本 |
| L-9 予約管理 | 7-10日 | API 5本、HTML 2本、JS 2本、DBマイグレーション 1本 |
