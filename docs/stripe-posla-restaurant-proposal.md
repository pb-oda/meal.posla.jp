# POSLA x Stripe: 日本の飲食店決済を再設計する共同提案

Status: Draft  
Audience: Stripe Sales / Stripe Partner Ecosystem  
Last updated: 2026-04-26

## 1. Executive Summary

POSLAは、日本の飲食店向けに、注文、会計、予約、KDS、売上分析、シフト、AI運用支援を統合するSaaSです。

現状、日本の飲食店ではキャッシュレス決済手数料への抵抗が強く、Stripe標準料率のままでは、店内セルフ決済や日常会計を全面的にStripeへ集約する判断が難しい店舗が多いと見込んでいます。

一方で、StripeがPOSLA経由の飲食店に対して特別料率、Connect優遇、段階料率を提供できれば、POSLAは飲食店のセルフ決済、予約金、キャンセル料、テイクアウト事前決済、将来的な対面決済をStripeへ集約する強い導入理由を作れます。

本提案の目的は、POSLAとStripeが共同で日本の飲食店決済を再設計し、低手数料と業務統合をセットにした新しい導入モデルを作ることです。

POSLAは単なるSaaS導入だけを狙っていません。日本の飲食店決済市場そのものを、より低コストで、より業務に接続された形へ変えることを本気で目指しています。

そのため、POSLA本体を導入しない店舗に対しても、決済だけを先に販売する選択肢を検討します。Stripeの基盤をOEMまたは共同ブランドとして提供できる場合、仮称 `POSLA Pay` として飲食店向け決済サービスを展開し、決済を入口に店舗接点を獲得した後、POSLA本体へアップセルする戦略を取れます。

長期的には、POSLAはSaaSで終わりません。POSLA Payを起点に、飲食店の決済、仕入、発注、採用、送客までをPOSLA上でつなぐ「POSLA経済圏」を構築します。Stripeには、その中核となる決済インフラとして入ってほしいと考えています。

## 2. Problem

飲食店にとって、決済手数料は利益を直接圧迫するコストです。

POSLAはセルフ決済にStripe Checkout / Stripe Connectを利用できますが、標準料率のままでは、店舗側から次の懸念が出ます。

- 店内の日常会計まで3%台後半の手数料に乗せたくない
- 既存のPayPay、QR決済、カード端末の方が安い場合がある
- POSLAの業務価値は魅力でも、決済手数料が導入判断のブレーキになる
- セルフ決済を有効化すると、手数料負担が増えるように見える

そのため、POSLAでは現在、店内通常会計については既存の外部決済を残せる設計にしています。

- 現金、外部カード端末、外部QR決済をPOSLAレジに記録できる
- セルフ決済は管理画面からON/OFFできる
- Stripeは予約金、事前決済、セルフ決済など、オンライン決済が必要な場面に使う

これは安全な初期設計ですが、Stripe側から見ると、飲食店の日常決済流通を取り切れない状態でもあります。

## 3. Market Thesis

飲食店決済は巨大市場です。

日本全体のキャッシュレス決済額は拡大を続けており、飲食店はその中でも反復利用が多く、POS、注文、予約、顧客管理と決済が密接につながる領域です。

経済産業省によると、2025年の国内キャッシュレス決済額は162.7兆円、キャッシュレス決済比率は58.0%です。内訳は、クレジットカード134.6兆円、コード決済16.6兆円、電子マネー6.0兆円、デビットカード5.5兆円です。

日本フードサービス協会は、日本の外食産業市場規模を2023年時点で24兆1512億円としています。飲食店、喫茶店、居酒屋・ビヤホール、料亭・バー等の店舗型飲食に寄せても、2023年時点で約18.0兆円規模です。

飲食店だけのキャッシュレス決済額を直接示す公開統計は確認できていません。そのため、本資料では以下を市場規模の概算として置きます。

```text
広義の外食産業キャッシュレス決済TAM:
24.1512兆円 x 58.0% = 約14.0兆円

POSLA対象に近い店舗型飲食キャッシュレス決済TAM:
約18.0兆円 x 58.0% = 約10.4兆円
```

これは単年度の静的な推計です。2024年以降の外食需要回復、インバウンド、客単価上昇、キャッシュレス比率上昇を考えると、POSLAが狙う飲食店キャッシュレス決済は、少なくとも10兆円級の市場として扱えます。

POSLAが飲食店の業務基盤として導入されれば、決済は単独の決済端末ではなく、以下の業務フローに組み込まれます。

- 予約
- 着席
- 注文
- 厨房連携
- 提供
- 会計
- レシート
- 売上分析
- 再来店施策

この業務フロー上でStripeを標準決済基盤にできれば、単なる決済獲得ではなく、飲食店のオペレーション全体にStripeを組み込むことができます。

## 4. POSLA Current Payment Design

POSLAの現行方針は、決済方法を用途別に分けています。

| 決済シーン | 現行方針 |
|---|---|
| POSLA月額利用料 | Stripe Billing / Checkout |
| 予約金 | Stripe Checkout / Connect |
| キャンセル料 | Stripe連携を想定 |
| テイクアウト事前決済 | Stripe Checkout / Connect |
| 客席セルフ決済 | Stripe Checkout / Connect、店舗設定でON/OFF |
| 店内スタッフ会計 | 現金、外部カード、外部QRをPOSLAレジに記録 |
| 店舗既存QR決済 | POSLAは決済処理せず、支払済み記録としてDB保存 |

現状のPOSLAは、Stripeを無理に全会計へ強制しません。

これは飲食店にとって導入しやすい一方で、Stripeにとっては店内決済流通の獲得機会を限定します。

## 5. Why Pricing Matters

飲食店にとって、決済手数料差は月額SaaS費用と同等、またはそれ以上のインパクトがあります。

例:

```text
月間キャッシュレス売上: 2,000,000円
現行手数料: 3.6%
低料率案: 2.0%
差分: 1.6%

月間削減額: 32,000円
```

この削減額は、POSLAの月額利用料に近い、または上回る可能性があります。

つまり、POSLAが業務効率化だけでなく、決済コスト削減も同時に提示できれば、飲食店にとって導入理由が大きく変わります。

## 6. Proposal to Stripe

POSLA経由で登録された飲食店に対して、Stripeの特別パッケージを提供したいです。

仮称:

```text
POSLA Restaurant Package powered by Stripe
```

対象取引:

- 店内セルフ決済
- テイクアウト事前決済
- 予約金
- キャンセル料
- 将来的なStripe Terminal対面決済
- POSLA月額課金

希望条件:

- 国内カード決済の特別料率
- Stripe Connect追加コストの優遇
- POSLA経由店舗向けの段階料率
- 初期導入店舗向けの期間限定トライアル料率
- 一定GMV到達後の追加ディスカウント
- POSLAとStripeの共同導入事例化

料率イメージ:

| フェーズ | 条件 | 希望 |
|---|---|---|
| Pilot | 初期導入店舗 | 標準料率より明確に低い特別料率 |
| Growth | 月間GMV一定以上 | 2%台前半を目指す段階料率 |
| Scale | 大規模流通 | 個別協議 |

実際の料率は、Stripe側の原価構造、ブランドフィー、インターチェンジ、リスク、チャージバック率を踏まえて協議します。

## 7. Strategic Go-to-Market

POSLAの販売戦略は、SaaSを先に売る方法だけではありません。

飲食店は決済手数料に強く反応します。そのため、決済手数料を明確に下げられる提案を先に出せれば、POSLA本体よりも早く店舗との接点を作れる可能性があります。

戦略仮説:

```text
Step 1: POSLA Payとして低料率決済を販売
Step 2: 決済明細、売上、レジ締め、入金管理で店舗接点を持つ
Step 3: 注文、会計、KDS、予約、分析へPOSLA本体を提案
Step 4: 飲食店の業務基盤と決済基盤をPOSLA x Stripeに集約
```

このモデルでは、POSLA本体の導入前でもStripeの決済流通を獲得できます。

決済だけの販売形態:

| 形態 | 内容 |
|---|---|
| POSLA Pay単体 | POSLA本体を使わない店舗にも低料率決済を提供 |
| POSLA Pay + 簡易売上管理 | 決済明細、入金予定、支払方法別集計を提供 |
| POSLA Pay + POSLAレジ | 決済を入口にレジ、注文、KDSへ拡張 |
| POSLAフル導入 | 注文、会計、予約、分析、AI支援まで統合 |

この段階設計により、店舗は最初から業務システムを全面移行しなくても、決済コスト削減からPOSLAとの関係を開始できます。

POSLAが本気で狙うのは、単なる決済手数料の一部ではありません。飲食店の決済接点を押さえ、その後に業務全体をつなぐことです。

## 8. POSLA Economic Zone Vision

POSLAの長期構想は、飲食店向けSaaSから「飲食店経済圏」のインフラへ進化することです。

POSLAが売るのは、POS、KDS、予約、在庫、シフトといった個別機能ではありません。予約から会計、在庫減算、仕入、採用、送客までの業務フロー全体です。

将来的な経済圏の柱:

| 領域 | 構想 | Stripeとの接点 |
|---|---|---|
| POSLA Pay | 飲食店向け決済プラットフォーム | Checkout / Connect / Terminal / Billing |
| POSLA Order | 食材卸・備品仕入のB2B EC | 仕入代金のキャッシュレス決済、分割入金、請求管理 |
| POSLA Jobs | 飲食店特化の採用・シフト連携 | 採用課金、成功報酬、給与・報酬支払いの将来余地 |
| POSLA Map | 予約・送客・再来店導線 | 予約金、キャンセル料、送客課金 |

特に重要なのは、仕入・発注領域です。

飲食店は日々、食材、酒類、備品、消耗品を仕入れています。このB2B取引は、POSLAの注文データ、在庫データ、需要予測データと自然につながります。

将来像:

```text
注文データ → 在庫減算 → 需要予測 → 発注提案
→ POSLA Orderで仕入
→ POSLA Pay / Stripeでキャッシュレス決済
→ 入金・請求・支払履歴をPOSLAに集約
```

この構想が実現すると、Stripeが関わる決済は来店客の支払いだけではなくなります。

- 来店客から飲食店へのB2C決済
- 飲食店から食材卸・仕入先へのB2B決済
- 予約金、キャンセル料、テイクアウト事前決済
- 将来的な採用、送客、マーケットプレイス手数料

POSLAは、飲食店の売上側と仕入側の両方の決済接点を持つことを目指します。

これにより、StripeにとってPOSLAは単なるSaaSアカウントではなく、飲食業界のB2C / B2B決済フローを横断するプラットフォームになり得ます。

## 9. What POSLA Offers Stripe

POSLAはStripeに対して、単なる決済利用者ではなく、飲食店市場への業務導線を提供できます。

- 飲食店向けSaaSとしての導入接点
- 店舗ごとの注文、会計、予約、テイクアウト導線
- セルフ決済の利用促進
- Stripe BillingによるPOSLA月額課金
- Stripe Connectによる店舗別売上管理
- Stripe Checkoutによる予約金、テイクアウト、セルフ決済
- 将来的なStripe Terminal導入余地
- 飲食店向け共同マーケティングの可能性
- POSLA Pay単体販売による、POSLA未導入店舗の決済流通獲得
- 決済を入口にしたPOSLA本体へのアップセル導線
- POSLA Orderによる仕入・発注領域のB2B決済拡張余地
- POSLA経済圏全体での、B2C / B2B決済フロー獲得余地

POSLAが普及するほど、Stripeは飲食店の決済流通をPOSLA経由で獲得できます。

## 10. Partnership Models

### Model A: Custom Pricing

StripeがPOSLA経由店舗に特別料率を提供します。

POSLAは対象店舗にStripe利用を推奨し、セルフ決済、予約金、テイクアウト事前決済をStripe標準として案内します。

### Model B: POSLA Pay OEM / Co-brand

Stripeの決済基盤を、飲食店向けOEMまたは共同ブランドとして `POSLA Pay` の名称で提供します。

POSLA本体を導入していない店舗にも決済だけを販売し、決済明細、入金管理、売上集計を入口にPOSLA本体へ展開します。

### Model C: Platform Partnership

POSLAをStripeのSaaS Platform Partnerとして位置付けます。

Stripe ConnectをPOSLAの決済基盤として組み込み、オンボーディング、分配、返金、決済確認を標準化します。

### Model D: POSLA Economic Zone Infrastructure

StripeをPOSLA経済圏の決済インフラとして位置付けます。

初期はPOSLA Pay、予約金、セルフ決済から開始し、将来的にPOSLA Orderの仕入決済、送客課金、採用課金まで拡張します。

### Model E: Joint Go-to-Market

POSLAとStripeが飲食店向けに共同導入事例、共同資料、共同キャンペーンを展開します。

訴求:

```text
飲食店の注文、会計、予約、決済を1つにつなぐ
POSLA x Stripe Restaurant Payment Package
```

## 11. Pilot Plan

初期は大規模契約ではなく、検証可能なPilotとして始めます。

Pilot条件案:

- 対象: POSLA導入飲食店
- 店舗数: 10店舗から開始
- 期間: 3から6か月
- 対象決済: セルフ決済、予約金、テイクアウト事前決済
- 計測指標: GMV、決済件数、平均単価、返金率、チャージバック率、セルフ決済利用率

Pilotで検証すること:

- 低料率がセルフ決済ON率を上げるか
- 既存外部決済からStripeへ移行する店舗があるか
- POSLA本体未導入店舗に、決済単体で販売できるか
- 決済単体導入店舗をPOSLA本体へアップセルできるか
- 仕入・発注キャッシュレス化に対する店舗ニーズがあるか
- 食材卸・仕入先側にPOSLA経由発注/決済の提携余地があるか
- 飲食店にとって手数料削減が導入意思決定に効くか
- Stripe側のリスクと収益性が許容できるか

| POSLA PayからPOSLA本体への転換率 | 決済入口戦略の有効性 |
| 店舗継続率 | 決済 + SaaS の統合価値 |

## 13. Risks and Mitigations

| リスク | 対策 |
|---|---|
| 低料率でも店舗が既存決済を使い続ける | セルフ決済、予約金、テイクアウトなどStripeが強い用途から導入 |
| カード決済原価により料率下限がある | QRや既存外部決済を残し、Stripeは高付加価値決済に集中 |
| 決済単体販売でPOSLA本体に進まない | 決済明細、売上分析、レジ締めから業務課題を可視化し、自然にPOSLA本体へ接続 |
| OEMブランドの責任分界が曖昧になる | 契約、審査、サポート、返金、チャージバックの責任分界をPilot前に定義 |
| 仕入先開拓が進まない | 既存テナントの仕入先から始め、1社ずつ発注実績を作る |
| B2B決済の運用が複雑化する | 初期は請求・支払記録から始め、決済実行は段階的に導入 |
| 店舗ごとの審査負荷 | Connect onboardingを標準化 |
| チャージバックリスク | 飲食店カテゴリ、注文データ、予約データを活用してリスク管理 |
| POSLAの初期GMVが小さい | Pilotから始め、店舗数と利用シーンを段階的に拡大 |
