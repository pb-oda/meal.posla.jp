# 9. テイクアウト管理

テイクアウト管理では、持ち帰り注文の受付、受取時間、オンライン決済、注文ステータスを管理します。

## 利用開始前の設定

1. 店舗設定でテイクアウトを有効にします。
2. 受付時間、受取可能時間、注文上限を設定します。
3. テイクアウト対象商品を確認します。
4. 決済方法を確認します。
5. お客様用の案内URLまたはQRコードを掲示します。

## 注文受付の流れ

1. お客様がテイクアウトページを開きます。
2. 商品、数量、受取時間、連絡先を入力します。
3. 必要に応じて決済を行います。
4. 店舗側の管理画面に注文が表示されます。
5. 調理、受け渡し、完了のステータスを更新します。

## 営業中の確認ポイント

- 受取時間に無理がないか確認してください。
- 品切れ商品がテイクアウト画面に残っていないか確認してください。
- 受け渡し済みの注文は完了にしてください。
- 連絡先が必要な場合は、店舗の個人情報管理ルールに従って扱ってください。

## 誤会計の取消（POS レジで会計した場合、2026-04-21 Phase 4d-5c-bb-C）

POS レジで **ゲートウェイを経由しないで会計したテイクアウト**（現金 / 手動 card / 手動 qr）を、後から管理者が論理取消できます。取引ジャーナルに「会計取消」ボタンが表示されます。

**取消可能条件**:
- 注文が `status='paid'` になっている（POS レジで会計済み）
- 支払方法が現金 / 手動 card / 手動 qr（ゲートウェイ非経由）
- 分割会計ではない
- まだ返金・取消していない

**取消できないもの（本機能では）**:
- オンライン（Stripe）で決済したテイクアウト → 本機能ではなく**返金（refund）経路**で処理（下記「オンライン決済テイクアウトの返金」を参照）
- 分割会計 / 返金済み / 既に取消済み

操作場所と動作は POS レジ章「通常会計も取り消せます」を参照。現金会計の取消は `cash_log` に相殺行が入りレジ残高は合いますが、お客さんに現金を返す場合は手渡しで別途対応してください。

## オンライン決済テイクアウトの返金（2026-04-21 Phase 4d-5c-bb-D）

Stripe オンライン決済したテイクアウト注文の誤会計は、**返金（refund）** で処理します。「会計取消」ではなく既存の「返金」機能を使います。

### 実証済みの範囲

- **Stripe Connect Checkout で決済したテイクアウト**（POSLA のテナント口座に transfer される通常形態）で、お客様がクレジットカードで決済したケース
- 操作は POSLA 運営にご相談ください（テナント画面からの直接操作はまだ一般化していません）
- 返金するとお客様のカードに全額戻り、POSLA に渡っていた手数料も同時に戻します（canonical な destination charges refund）

### まだ一般化していない範囲（別対応 / 要確認）

- 店頭 POS レジの Stripe Terminal で決済したテイクアウト（実データ 0 件のため運用検証未実施）
- Stripe Direct 接続（テナントが自前で Stripe API key を設定している方式）のテイクアウト（ほぼ非運用）
- オンライン決済の「一部だけ返金」（部分返金 / partial refund）
- オンライン決済テイクアウトに「会計取消（void）」を使うこと（本機能は返金の責務、取消の責務は別）

これらが必要になった場合は、運用上 POSLA 運営に相談してください。技術的には追加実装・追加検証が必要です。

### 実行後のお客様への影響

- Stripe 側で返金処理が実行されると、お客様のカード明細に返金が反映されます（通常 5〜10 営業日）
- テナント側の売上レポートには「返金済」として表示され、実売上からは自動除外されます
- 注文自体の状態（調理・受取のステータス）は、返金しても自動では変わりません。必要に応じてテイクアウト管理画面で個別対応してください

## お客様画面でのエラー表示（2026-04-21 Phase 4d-5c-bb-F）

Stripe 決済が完了したのに、POSLA 側で注文の記録に失敗したごく稀なケースでは、お客様の画面に次のメッセージが表示されます。

```
決済確認に失敗しました
決済は Stripe 側で完了している可能性があります。
お手数ですが、下記の注文番号を添えて店舗まで直接お問い合わせください。
注文番号: XXXXXXXX
```

この場合、店舗（運営）側では次の手順で確認してください。

1. POSLA 運営に連絡してエラーログ（`php_errors.log` の `[4d-5c-bb-F][takeout-payment] persistence_failed` 行）の調査を依頼
2. order_id と Stripe PaymentIntent ID を取得
3. Stripe ダッシュボード上で PaymentIntent の decision 成否を確認
4. 必要に応じて手動で注文記録を作成、または Stripe 側で返金してお客様に連絡

※ キャンセル表示（オレンジ「決済がキャンセルされました」）とは別の表示です。お客様がキャンセル操作をした場合はキャンセル画面、決済完了しているのに記録失敗した場合は上記のエラー画面になります。

## 困ったとき

| 症状 | 対処 |
|---|---|
| 注文が入らない | テイクアウトが有効か、受付時間内か確認してください。 |
| 受取時間が選べない | 受付時間、準備時間、注文上限を確認してください。 |
| 決済が完了しない | 決済設定とお客様の入力状況を確認してください。 |
| お客様画面に「決済確認に失敗しました」と表示される | 上記「お客様画面でのエラー表示」を参照。POSLA 運営に調査依頼してください。 |
| 注文を停止したい | テイクアウト設定を無効にし、案内QRの掲示も確認してください。 |
