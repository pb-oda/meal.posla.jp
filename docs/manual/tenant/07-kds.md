# 7. KDS (キッチンディスプレイ)

KDS (Kitchen Display System) は、キッチンスタッフが注文をリアルタイムで確認し、調理の進捗を管理するための画面です。`/kds/` の URL で新しいタブとして開きます。

---

## 7.0 はじめにお読みください

::: tip 章の読み方ガイド
- **キッチンスタッフ**: 7.1 → 7.3 (画面構成) → 7.4 (注文カード) を読めば作業できます
- **店長・オーナー**: 7.10 (ステーション設定) + 7.11 (GPS チェック) は必読
- **音声コマンド使いたい人**: 7.8 を確認
- **トラブル対応**: 7.13 FAQ / 7.14 トラブル表 を参照
- **端末調達・配置・Wi-Fi**: 23 章「KDS デバイス端末ガイド」と併読
:::

### KDS 画面の役割

- 注文の**受付 → 調理中 → 提供待ち**のカンバン管理
- 品目単位でのステータス遷移 (全体一括 or 個別)
- **AI による調理優先度提案** (Gemini 2.0 Flash)
- 音声コマンドでのハンズフリー操作 (Web Speech API + Gemini)
- ステーション別表示 (大規模キッチン向け)
- スタッフ呼び出しアラート受信
- コースフェーズ管理
- 低評価アラート (お客様が 1〜2 星評価をした直後の品質確認)

### 使える人

- owner / manager / staff / **device** ロール (特に device 推奨: KDS 専用端末)
- device の場合は GPS チェック必須 (7.11 参照)

### 必要なもの

- **タブレット** (推奨: iPad 10" 以上、Android 11" 以上)
- **店舗 Wi-Fi** (5GHz 帯推奨)
- **Chrome 最新版** (音声コマンドは Chrome 必須)
- **マイク** (タブレット内蔵 or 外付け、音声コマンド使用時)
- **スピーカー** (効果音・ビープ音、タブレット内蔵で OK)
- **壁面スタンド or 吊り下げ金具** (調理中に手で持たなくて済むように)
- **device アカウント** (端末専用、店舗オーナー or マネージャーが事前作成)

### 所要時間

| 作業 | 目安 |
|---|---|
| 初回起動 + 音声 unlock | 30 秒 |
| 店舗選択 (複数店舗所属時) | 5 秒 |
| GPS チェック (device のみ) | 2〜10 秒 |
| 注文 1 件の調理開始 → 完成 → 提供済 | ワンタップ 3 回 (全体) or 品目ごと |
| ステーション切替 | 2 秒 |
| 音声コマンド 1 件 | 3〜5 秒 |
| AI キッチン分析 1 回 | 約 3 秒 (Gemini 応答待ち) |

## 7.1 操作前のチェックリスト

KDS を毎日起動する前に、以下 7 項目を必ず確認してください。1 項目でも NG だと「画面が出ない」「音が鳴らない」「店舗外エラー」のいずれかでつまずきます。

| # | 確認項目 | 確認方法 | NG 時の対応 |
|---|---------|---------|-----------|
| 1 | **タブレットが充電器に接続されている** | 画面右上のバッテリーアイコンが「充電中」アイコン | 電源ケーブルを差す。長時間営業中はバッテリー駆動禁止 |
| 2 | **店舗 Wi-Fi に接続されている** | 設定 > Wi-Fi > 店舗 SSID にチェック | 店舗 Wi-Fi に再接続。テザリング・公衆 Wi-Fi 不可 |
| 3 | **Chrome が最新版** | Chrome > メニュー > Chrome について | アップデートを適用 (古い Chrome は Web Speech API が不安定) |
| 4 | **画面の自動スリープが OFF** | 設定 > 画面 > スリープ「なし」 | 「なし」に変更。スリープすると通知音が鳴らない |
| 5 | **音量が最大** | 本体音量ボタンで MAX | キッチン内の環境音より大きく。普段の作業で「少し耳障り」程度が適正 |
| 6 | **device アカウントの ID/PW を確認** (device 端末のみ) | 端末横のラベルやスタッフノートで確認 | オーナー or マネージャーに再発行を依頼 (23 章参照) |
| 7 | **GPS が ON** (device 端末のみ) | 設定 > 位置情報サービス > ON | ON にする。OFF のままだと KDS 画面に進めない |

::: tip 朝のオープニング作業手順 (5 分)
1. タブレットの電源ボタン押下 → 起動
2. Chrome 起動 → ブックマーク「KDS」タップ (URL: `https://eat.posla.jp/public/kds/index.html`)
3. ログイン画面で device アカウントの ID/PW 入力
4. 店舗選択オーバーレイ (複数店舗所属時のみ) で当該店舗をタップ
5. GPS 許可ダイアログで「許可」(device のみ)
6. 起動オーバーレイの「KDSを開始」タップ → 音声 unlock 完了
7. 受付カラムに「なし」と表示されていれば OK
:::

::: warning device 端末を翌日以降ログアウトしない
device アカウントは「端末そのもの」のための共有アカウントです。終業時にログアウトすると翌朝のログインが必要になります。タブレットを電源 OFF せず、画面ロックで運用するのが推奨。
:::

## 7.2 初回起動 (音声 unlock + 店舗選択 + GPS)

### 7.2.1 起動フローの全体像

```
[1] KDS 画面アクセス (https://.../kds/index.html)
     ↓
[2] 認証チェック (KdsAuth.init)
     ├─ 未ログイン → admin/index.html にリダイレクト
     └─ ログイン済 → 続行
     ↓
[3] 複数店舗所属判定
     ├─ 1店舗のみ → スキップ
     └─ 複数店舗 → 店舗選択オーバーレイ表示
                   (localStorage:mt_kds_store に記憶)
     ↓
[4] device ロール判定
     ├─ device → GPS チェック (KdsAuth.checkGps)
     │           └─ 範囲外 → 「店舗外」エラーオーバーレイ
     └─ owner/manager/staff → スキップ
     ↓
[5] 起動オーバーレイ「KDSを開始」ボタン
     (sessionStorage:mt_kds_started にフラグ保存。
      同一セッション内の2回目以降はスキップ)
     ↓
[6] VoiceCommander.unlockAudio() 実行
     (Web Audio Context を resume + 無音バッファ再生)
     ↓
[7] KdsRenderer.init / VoiceCommander.init / startPolling
     ↓
[8] 通常運用開始
```

### 7.2.2 ステップごとの操作

#### ステップ 1: KDS 画面にアクセス

1. Chrome を開く
2. ブックマーク「KDS」をタップ、または URL を直接入力
3. URL 例: `https://eat.posla.jp/public/kds/index.html`

#### ステップ 2: 認証チェック

ログインしていない場合、自動で `https://eat.posla.jp/public/admin/index.html` にリダイレクトされます。device アカウントの ID と PW を入力してログインしてください。

#### ステップ 3: 店舗選択 (複数店舗所属時のみ)

複数店舗に所属しているアカウント (主にマネージャー・オーナー) は、画面中央に店舗選択オーバーレイが表示されます。

| 表示要素 | 説明 |
|---------|------|
| タイトル | 「店舗を選択してください」 |
| 店舗ボタン | 所属店舗の名前ボタンが縦に並ぶ |

選択した店舗 ID は `localStorage:mt_kds_store` に保存され、次回以降は自動で復元されます。店舗を切り替えたい場合は画面右上の「店舗切替」リンクから再選択。

#### ステップ 4: GPS チェック (device ロールのみ)

device アカウントでログインした場合、ブラウザが GPS 許可を求めます。詳細は 7.12 参照。

#### ステップ 5: 起動オーバーレイ「KDSを開始」

```
┌─────────────────────────────────┐
│             🍳                   │
│   KDS キッチンディスプレイ        │
│   松の屋 渋谷店                   │
│                                 │
│   ┌─────────────────────────┐   │
│   │      KDS を開始          │   │
│   └─────────────────────────┘   │
│                                 │
│   音声コマンド・通知音が          │
│   有効になります                  │
└─────────────────────────────────┘
```

::: warning 音声 unlock について
ブラウザのセキュリティ制限により、ユーザーの操作 (タップ/クリック) なしに音声を再生することはできません。KDS 起動時の「KDS を開始」は音声機能を有効にするために必要です。この手順をスキップすると、新規注文の通知音やアラート音が鳴りません。

また、ページを再読込 (Cmd+R) するとセッションがリセットされるため、再度オーバーレイが表示されます。同一セッション内 (タブを閉じずに別画面に遷移して戻る) の場合のみスキップされます。
:::

#### ステップ 6: メイン画面表示

「KDS を開始」をタップすると、3 カラムのカンバンボードが表示され、3 秒間隔のポーリングが始まります。受付カラムに「なし」と表示されていれば正常起動です。

### 7.3.2 3カラムのカンバンボード

KDS 画面は 3 つのカラム (列) で構成されています。注文カードは左から右へ進みます。

| カラム | 名称 | 内部値 | 内容 | 背景色 |
|--------|------|------|------|------|
| 左 | 受付 | `pending` | 新しく受け付けた注文 | 暗いグレー |
| 中央 | 調理中 | `preparing` | 調理を開始した注文 | やや明るいグレー |
| 右 | 提供待ち | `ready` | 調理が完了して提供を待っている注文 | 緑がかったグレー |

各カラムが空の場合は「なし」と表示されます。カラムヘッダーには現在の件数 `[N]` が表示されます。

### 7.3.3 ステーションバー

画面上部にステーション選択バーが表示されます。

| ボタン | 説明 |
|--------|------|
| **全品目** | 全てのカテゴリの注文を表示 (デフォルト、active) |
| **[ステーション名]** | そのステーションに割り当てられたカテゴリの品目のみ表示 |

**ステーションの例:**
- 「厨房」ステーション → メイン料理・前菜カテゴリを担当
- 「ドリンク」ステーション → ドリンクカテゴリを担当
- 「デザート」ステーション → デザートカテゴリを担当
- 「揚げ場」ステーション → 揚げ物カテゴリを担当

選択中のステーションはブラウザに記憶されます (`localStorage:mt_kds_station`)。タブレット 1 台 = 1 ステーションの運用が基本です。

::: tip ステーションの設定
KDS ステーションは管理ダッシュボード (`dashboard.html`) の「店舗設定」→「KDS ステーション」サブタブで設定します。詳しくは 7.11 を参照。
:::

### 7.3.4 コースフェーズバー

コース料理の注文がある場合、ステーションバーの下にコースフェーズバーが表示されます。

| 表示要素 | 説明 | 例 |
|---------|------|---|
| テーブルコード | コース注文のテーブル名 | `T03` |
| コース名 | コース料理の名前 | `松会席` |
| 現在フェーズ | 現在のフェーズ番号 | `P1` |
| 「次フェーズ」ボタン | 次のフェーズに進める | 緑色ボタン |

コース注文がない場合、このバーは非表示 (`:empty { display:none }`)。

### 7.3.5 ヘッダー要素詳細

| 要素 | クリック動作 | 表示条件 |
|------|------------|--------|
| `KDS` | (なし、タイトル) | 常時 |
| 店舗名 | (なし、表示のみ) | 常時 |
| **店舗切替** | 店舗選択オーバーレイを開く | 複数店舗所属時のみ |

## 7.4 注文カードの表示

### 7.4.1 カードの構成

各注文は以下の要素で構成されたカードとして表示されます。

**ヘッダー部分:**

| 要素 | 説明 |
|------|------|
| テーブルコード | 「5番」「T01」など。テイクアウトの場合は顧客名 |
| テイクアウトバッジ | テイクアウト注文の場合「テイクアウト」バッジ (青色) |
| 受取時間 | テイクアウトの場合、受取予定時刻 (「18:30」) |
| ハンディバッジ | ハンディから登録された注文に「H」バッジ |
| コースバッジ | コース注文の場合「コース P1:前菜」形式 (フェーズ番号＋フェーズ名) |
| 経過時間 | 注文からの経過時間 (「12m 34s」形式、`Utils.formatDuration`) |

**本文部分:**

| 要素 | 説明 |
|------|------|
| メモ | 注文メモがある場合に表示 (「ネギ抜き」等) |
| アレルゲン警告 | 「⚠️ アレルギー: 卵, 乳, 小麦」形式で表示 |
| 品目リスト | 品名 × 数量。オプションがある場合は小文字で表示 |

**フッター部分:**

| 要素 | 説明 |
|------|------|
| 合計金額 | 注文合計 (コース注文の場合は非表示) |
| アクションボタン | 注文全体のステータス変更ボタン |
| 取消ボタン | 注文をキャンセル |

### 7.4.2 経過時間と色の変化

注文カードは経過時間に応じて色が変わります。閾値は `kds-renderer.js` の `WARNING_THRESHOLD_SEC = 600` / `DANGER_THRESHOLD_SEC = 1200` で定義。

| 経過時間 | カードの状態 | CSS クラス | 意味 |
|---------|------------|----------|------|
| 10分未満 | 通常表示 | (なし) | 正常範囲 |
| 10分以上 | 黄色背景 | `kds-card--warning` | やや遅延 |
| 20分以上 | 赤色背景 | `kds-card--danger` | 大幅遅延 |

::: tip 経過時間の起点
経過時間は `orders.created_at` (注文作成時刻) を起点にします。お客様がセルフメニューで「注文確定」した瞬間、またはハンディスタッフが「送信」した瞬間です。実際の調理開始時刻 (`prepared_at`) ではない点に注意。
:::

### 7.4.3 品目ごとのステータス表示

各品目には個別のステータスと操作ボタンが表示されます。

| 品目ステータス | 内部値 | 表示 | アクションボタン |
|-------------|------|------|---------------|
| 受付 | `pending` | 品名と数量 | **[調理開始]** ボタン |
| 調理中 | `preparing` | 品名と数量 | **[完成]** ボタン |
| 提供待ち | `ready` | 品名と数量 | **[提供済]** ボタン |
| 提供済 | `served` | 取り消し線 + 「✓提供済」バッジ | — |
| キャンセル | `cancelled` | 取り消し線 + 「取消」バッジ | — |

`item_id` がない品目 (古い形式の注文) は個別ボタンが出ず、カードレベルの全体ボタンのみで操作します。

### 7.4.4 入力フィールド対応表 (このカードに反映される DB カラム)

KDS の注文カードには、複数のテーブルから情報が集約されて表示されます。

| 表示要素 | 由来テーブル/カラム | 由来 API |

| メモ | `orders.memo` | 同上 |
| アレルゲン | `order_items.allergen_selections` (JSON) | 同上 |
| 品名 | `order_items.name` | 同上 |
| 数量 | `order_items.qty` | 同上 |
| オプション | `order_items.options` (JSON) | 同上 |
| 品目ステータス | `order_items.status` | 同上 |
| 注文ステータス | `orders.status` | 同上 |
| コース名 | `course_templates.name` (joined) | 同上 |
| フェーズ番号 | `table_sessions.current_phase_number` | 同上 |
| フェーズ名 | `course_phases.name` (joined) | 同上 |

## 7.5 注文の操作

### 7.5.1 注文全体の操作 (カードレベル)

カードのフッターにあるボタンで、注文の全品目を一括操作できます。

| ボタン | 動作 | 表示条件 |
|--------|------|---------|
| **全品調理開始** | 全品目を pending → preparing に変更 | 注文ステータスが pending |
| **全品完成** | 全品目を preparing → ready に変更 | 注文ステータスが preparing |
| **全品提供済み** | 全品目を ready → served に変更 | 注文ステータスが ready |
| **取消** | 注文をキャンセル | 常時表示 |

#### 注文キャンセル手順 (クリック単位)

1. キャンセルしたい注文カードのフッターにある **[取消]** ボタンをクリック
2. 確認ダイアログ「この注文をキャンセルしますか？」が表示
3. **[OK]** をクリック
4. サーバー側で `orders.status = 'cancelled'` + 全 `order_items.status = 'cancelled'` に更新
5. カードが KDS から消える (次のポーリングで全カラムから除外)
6. 監査ログに `order_cancel` として記録

::: warning キャンセルは取り消せない
キャンセル操作は元に戻せません。誤キャンセルした場合は、お客様に再注文を依頼するか、レジ画面から手動で同じ品目を追加するしかありません。
:::

### 7.5.2 品目個別の操作 (アイテムレベル)

各品目の横にあるボタンで、品目ごとに個別操作できます。

| ボタン | 動作 | 内部 API |
|--------|------|---------|
| **調理開始** | その品目だけを pending → preparing に変更 | `PATCH /api/kds/update-item-status.php` |
| **完成** | その品目だけを preparing → ready に変更 | 同上 |
| **提供済** | その品目だけを ready → served に変更 | 同上 |

::: tip 個別操作と全体操作の使い分け
- 1 つの注文に複数品目がある場合、品目ごとに異なるタイミングで調理が進むことがあります
- 例: ドリンクは先に完成、メイン料理は後から完成
- このような場合は品目個別の操作ボタンを使います
- 全品目が同時に進む場合はカードレベルのボタンが効率的です
:::

### 7.5.3 親注文ステータスの自動連動

品目を個別操作した際、親注文の `orders.status` も自動で連動します (`api/kds/update-item-status.php` の集計ロジック)。

| 全品目の状態 | 親注文ステータス |
|------------|----------------|
| 全 served (cancelled 除く) | `served` |
| 全 ready 以上 (cancelled 除く) | `ready` |
| 1 つでも preparing | `preparing` |
| 全 pending | `pending` のまま |

### 7.5.4 楽観的更新 (Optimistic Update)

KDS では UI 更新を高速化するため、楽観的更新パターンを使用しています。

```
[1] スタッフがボタンクリック
[2] フロントエンド: _orders[id].status = 新ステータス → 即座に再描画
[3] フロントエンド: PATCH リクエスト送信 (バックグラウンド)
[4] サーバー: DB 更新成功 → 200 OK 返す
[5] 次のポーリング (3 秒後) で正規データ取得 → そのまま維持
```

API リクエストが失敗した場合、`PollingDataSource.forceRefresh()` で即座にポーリング → 正しい状態に巻き戻されます。

### 7.5.5 実例: matsunoya 渋谷店ランチタイムの 1 件

```
12:30:00 お客様が T01 でセルフメニューから注文確定
         items: [{ハンバーグ定食 ×1}, {サラダ ×1}, {コーヒー ×1}]
         orders.status = 'pending'

12:30:03 KDS の受付カラムに T01 カードが出現 (3 秒ポーリング)
         キッチン担当が「ハンバーグ定食」の [調理開始] をタップ
         → ハンバーグ定食のみ preparing カラムに移動

12:30:15 ドリンク担当が「コーヒー」の [調理開始] をタップ
         12:30:25 ドリンク担当が「コーヒー」の [完成] → 提供待ちに移動

12:32:00 ドリンク担当が「コーヒー」の [提供済] をタップ
         → コーヒーが served、KDS 上は取り消し線で表示

12:38:00 キッチン担当が「ハンバーグ定食」の [完成] をタップ
         → 提供待ちに移動。同時に call_alerts に product_ready が
            INSERT され、ハンディの担当者に「ハンバーグ定食 できました」通知

12:38:30 ホール担当がハンバーグ定食を運び、ハンディから [提供済] 
         → server で 全品目 served を確認 → orders.status = 'served'
         → 次回ポーリングで KDS から完全に消える
```

## 7.6 コース料理の操作

### 7.6.1 コースフェーズの進行

コース料理の注文は、フェーズ (段階) ごとに料理が提供されます。

**フェーズの進め方:**

1. コースフェーズバーの対象テーブルの **[次フェーズ]** ボタンをクリック
2. 確認メッセージ「次のフェーズに進みますか？(現フェーズの未完了品目は全て提供済みになります)」が表示
3. **[OK]** をクリックすると以下が実行
   - 現在のフェーズの未完了品目が全て `served` に変更される
   - フェーズ番号が 1 つ進む (例: フェーズ 1 → フェーズ 2)
   - 次のフェーズの品目が自動的に注文として生成され、KDS に表示される
4. ボタン表記が「処理中...」→ 「フェーズ 2 開始」に変化
5. ポーリングで新しい注文カードが受付カラムに出現

### 7.6.2 コース注文の表示上の特徴

- カードの左ボーダーが緑色 (3px、`.kds-card--course`)
- コースバッジ「コース P1:前菜」形式がヘッダーに表示 (`kds-badge-course`、緑背景)
- 合計金額は表示されない (コース料理は一括料金のため、`order.course_id` がある場合非表示)

### 7.6.3 フェーズ進行時のサーバー処理

```
POST /api/kds/advance-phase.php
Body: { store_id, table_id }
  ↓
[1] table_sessions.current_phase_number を取得
[2] 現フェーズの order_items すべてを served に UPDATE
[3] orders を served に UPDATE (連動)
[4] table_sessions.current_phase_number を +1
[5] course_phases から次フェーズの phase 定義を取得
[6] 次フェーズの items を orders + order_items に INSERT
[7] 全フェーズ完了なら { completed: true } を返す
```

::: warning フェーズは戻せない
コースフェーズは一方向のみ。誤って次フェーズに進めた場合、戻す機能はありません。お客様の食事ペースを確認してから操作してください。
:::

## 7.7 リアルタイム更新

### 7.7.1 ポーリング

KDS は定期的にサーバーから最新の注文データを取得します。

| 状態 | ポーリング間隔 | 定数 |
|------|-------------|------|
| 通常時 | 3 秒 | `_defaultPollMs = 3000` |
| オフライン時 | 30 秒 | `_offlinePollMs = 30000` |
| 接続復旧時 | 即座に 3 秒に復帰 + 強制リフレッシュ | `forceRefresh()` |

::: info ポーリング方式について
POSLA は WebSocket や Server-Sent Events ではなく、HTTP ポーリング方式を採用しています。3 秒間隔で十分なリアルタイム性を実現しつつ、サーバー構成 (素 PHP + MySQL) をシンプルに保っています。
:::

### 7.7.2 オフライン検知

インターネット接続が切断された場合 (`OfflineDetector` モジュールが検知):
- ポーリング間隔が 30 秒に延長
- 画面上にオフライン警告バナー (赤背景) が表示
- 接続が復旧すると自動的に通常のポーリングに戻る + `forceRefresh()` で即時更新

### 7.7.3 station_id フィルタリング

ステーションを選択している場合、ポーリング URL に `station_id` パラメータが付与されます。

```
GET /api/kds/orders.php?store_id=xxx&view=kitchen&station_id=yyy
                                                   ↑↑↑↑↑↑↑↑↑↑↑↑
                              指定ステーションが担当するカテゴリの品目のみ返す
```

サーバー側 (`api/kds/orders.php`) で `kds_routing_rules` テーブルを JOIN し、`station_id` に紐づく `category_id` に該当するメニュー品目だけをフィルタして返します。

## 7.8 スタッフ呼び出しアラート + 低評価アラート

### 7.8.1 スタッフ呼び出しアラートの表示

お客さんがセルフオーダー画面からスタッフを呼び出すと、KDS 画面にアラートが表示されます。

- 画面上部に赤色のバナーとして表示
- 5 秒ごとにチェック (`api/kds/call-alerts.php` ポーリング)
- アラート 1 件ごとに以下が表示

| 要素 | 説明 |
|------|------|
| 🔔アイコン | ベルアイコン |
| テーブルコード | 呼び出し元のテーブル |
| 呼び出し理由 | お客さんが入力した理由 (「お水ください」等) |
| 経過時間 | 呼び出しからの経過時間 |
| **[対応済み]** ボタン | アラートを消す (PATCH `/api/kds/call-alerts.php`) |

### 7.8.2 効果音

新しいアラートが検知されると効果音が鳴ります。

| 種類 | 周波数パターン | 音量 | API |
|------|--------------|------|-----|
| 新規アラート | 880Hz → 1100Hz (2音) | 30% | Web Audio API (サイン波) |
| 成功 | 600Hz → 800Hz | 30% | 同上 |
| エラー | 300Hz | 30% | 同上 |

### 7.8.3 アラートの種類別フィルタリング

KDS 画面では `product_ready` (料理完成通知) と `kitchen_call` (KDS 自身が送信したアラート) を除外して表示します (実装: `public/kds/js/kds-alert.js` L117-122)。

| アラートタイプ | KDS で表示 | ハンディで表示 | 用途 |
|--------------|----------|--------------|------|
| `staff_call` | ✅ | ✅ | お客様 → スタッフ (お水ください等) |
| `product_ready` | ❌ | ✅ | キッチン → ホール (料理完成通知) |
| `kitchen_call` | ❌ | ✅ | キッチン → ホール (キッチン内発信) |

### 7.8.4 低評価アラート (満足度評価との連動)

お客様が食事後に **1〜2 星の低評価** を送信すると、KDS 画面に **低評価アラート** が自動表示されます (実装: `api/kds/orders.php` L82-88 の `low_rating_alerts` 配列、レンダリングは `kds-renderer.js` の `renderLowRatingAlerts()`)。

- バナー色: 赤 (緊急)
- 表示内容: 絵文字 (😞=1星、😐=2星) + 品目名 + 評価 + 時刻 (HH:MM)
- 対応: すぐに品質確認 (同じ問題が他の客に出ていないか)、お客様がまだ店内にいればスタッフが伺う

詳細は [17 章 17.W 満足度評価](./17-customer.md) 参照。

## 7.9 AI 音声コマンド

KDS は Web Speech API を使用した音声コマンドに対応しています (Chrome 推奨)。コマンド体系は [docs/voice-commands.md](../../voice-commands.md) と完全一致。

### 7.9.1 音声コマンドの起動

1. KDS 画面右上の **🎤 音声OFF** ボタンをタップ
2. 「🎤 音声ON」に変化、ボタン枠が緑に
3. ブラウザのマイクアクセス許可が初回のみ求められる
4. 許可すると画面下部に「🎤 音声待機中 (例:「T01完成」「スタッフ呼んで」「コマンド一覧」)」と表示
5. コマンドを発話すると、ファストパスまたは Gemini で解析され実行

### 7.9.2 コマンド処理の 2 段階

```
[発話]
   ↓
[Web Speech API: 音声 → テキスト]
   ↓
[ノイズフィルタ: 2文字未満 / 60文字超は無視]
   ↓
[ファストパス判定]
   ├─ HIT → ローカル即実行 (Gemini API 不使用、高速)
   └─ MISS → Gemini API に送信して解析
       ↓
       [Gemini が JSON コマンドを返す]
       ↓
       [confidence: high → 即実行 / low → 5秒確認ダイアログ]
       ↓
       [実行]
```

### 7.9.3 ファストパス対応コマンド一覧

ローカルで即座に処理されるコマンド (Gemini 不要):

| カテゴリ | 発話例 | 動作 |
|---------|-------|------|
| **品目単位ステータス** | 「ハンバーグ開始」 | ハンバーグを調理中に移動 |
| | 「チキン南蛮完成」 | 提供待ちに移動 |
| | 「トンカツ提供済み」 | 提供済 |
| **スタッフ呼び出し** | 「スタッフ呼んで」「ホールに来て」 | ハンディに通知送信 |
| **品切れ一覧** | 「品切れ一覧」「今の品切れ」 | モーダル表示 |
| **テーマ切替** | 「ライトモード」「ダークモード」「明るく」「暗く」 | テーマ切替 |
| **スクロール** | 「下にスクロール」「上にスクロール」 | 画面スクロール |
| **ヘルプ** | 「コマンド一覧」「ヘルプ」「使い方」 | コマンド一覧モーダル (15秒で自動閉じ) |
| **AIシェフ** | 「シェフ」「ダッシュボード」 | AIシェフパネル開閉 |
| | 「シェフ閉じて」 | 閉じる |
| | 「シェフ更新」「最新にして」 | 再分析 |
| **ステーション切替** | 「全品目」「ドリンク」 | ステーション切替 (ボタンテキストで動的に認識) |
| **音声再起動** | 「音声再起動」「リスタート」「音声リセット」 | 音声認識を再起動 |

### 7.9.4 動作語の認識バリエーション

| ステータス | 認識される語 |
|-----------|-------------|
| 調理開始 (preparing) | 調理 / 開始 / 準備 / じゅんび |
| 完成 (ready) | 完成 / できた / できました / あがり / 上がり / アップ / レディ / ready |
| 提供済 (served) | 提供 / 済み / すみ / サーブ / サーヴ |

**品目名マッチング**: 「ハンバーグ定食」→「ハンバーグ」のようにサフィックス (定食/セット/弁当/丼/御膳/ランチ/プレート) を除去して照合。

### 7.9.5 Gemini 解析対応コマンド

ファストパスで処理できないコマンドは Gemini API (gemini-2.0-flash, temperature: 0) に送信されます。

| 発話例 | 動作 |
|-------|------|
| 「T01 完成」 | T01 の注文全体を提供待ちに |
| 「テーブル 3 のハンバーグ調理開始」 | テーブル 3 のハンバーグを調理中に |
| 「1 番キャンセル」 | 1 番テーブルの注文をキャンセル |
| 「トンカツ品切れ」 | トンカツを品切れに設定 |
| 「ハンバーグ販売再開」 | ハンバーグの品切れを解除 |

**テーブル番号認識**: T01 / t1 / テーブル1 / 1番 等の表記を認識 (正規表現: `/[TＴｔt]\s*0*\d{1,3}|テーブル\s*\d{1,3}|\d{1,3}\s*番/i`)。

### 7.9.6 confidence (確信度)

Gemini が返すコマンドには確信度が付与されます。

- **high**: 即座に実行
- **low**: 5 秒間の確認ダイアログを表示し、ユーザーが「実行する」を選択した場合のみ実行

### 7.9.7 実例: matsunoya 渋谷店での音声運用

```
シェフ「シェフ」 → AI キッチンパネル開く
シェフ「T01 ハンバーグ完成」 → Gemini 解析 → preparing→ready
ホール「コーヒー提供済み」 → ファストパス → ready→served
シェフ「コマンド一覧」 → モーダル表示 (15秒)
シェフ「シェフ閉じて」 → AI パネル閉じる
シェフ「ダークモード」 → ファストパス → 暗いテーマに切替
シェフ「全品目」 → ステーションフィルタ解除
```

::: warning Chrome 推奨
Web Speech API の continuous モード (連続認識) は Google Chrome でのみ安定動作します。Firefox、Safari、Edge では音声コマンド機能が使えない場合があります。
:::

::: warning Web Speech API の制約
- 造語はウェイクワードに使用不可 (「ハンバーグ」のような既存単語のみ)
- 英数字は日本語モードで認識されにくい (「T01」は「ティーゼロワン」と発音すると確実)
- interim 結果のタイマー確定 (1.2 秒) が必要 (final 未発火対策)
:::

## 7.10 AI キッチンダッシュボード

KDS 画面には AI による調理優先度の提案機能があります。

### 7.10.1 AI キッチンの設計思想

「料理を作る順番をテーブル単位で考える」 → 「料理品目単位で考える」への転換。AI が複数テーブルの全品目を横断して見て、「今すぐ着手すべき品目」を順位付けします。

### 7.10.2 パネルの表示

- 画面下部からスライドアップするパネル (画面の 40%、モバイルは 55% の高さ)
- ヘッダーの **🧑‍🍳 AIシェフ** ボタン or 音声「シェフ」で開く
- パネルを開くと自動で AI 分析が開始される
- **60 秒ごとに自動更新** (`REFRESH_INTERVAL = 60000`)
- **🔄 更新** ボタンで手動リフレッシュ
- **✕ 閉じる** で閉じると自動更新も停止

### 7.10.3 AI 分析フロー ASCII 図

```
[1] スタッフが「AIシェフ」ボタンをタップ
     ↓
[2] GET /api/kds/orders.php?view=kitchen
     → pending / preparing 注文取得
     ↓
[3] _expandItems() で注文を品目単位に展開
     [{order_id, table_code, item_name, qty, status, elapsed_seconds}, ...]
     ↓
[4] POST /api/kds/ai-kitchen.php { items }
     ↓
[5] サーバー: Gemini 2.0 Flash に分析依頼
     プロンプト: "以下の品目を urgent/next/waiting に分類し、
                  pace_status (good/normal/busy) と
                  summary (日本語要約) を返せ。
                  経過時間 + メニュー構成 + 過去調理時間データを考慮。"
     ↓
[6] Gemini レスポンス (JSON):
     {
       urgent:  [{item_name, qty, table_code, reason}, ...],
       next:    [{...}],
       waiting: [{...}],
       pace_status: "busy",
       summary:  "ピーク時間。揚げ場が混雑しています..."
     }
     ↓
[7] _renderResult(data) でパネルに描画
     ⚡ 今すぐ着手 (赤)  / 🔜 次に着手 (オレンジ) / ⏸ 待機中 (グレー)
     💬 サマリーコメント + 📊 ペースバッジ
```

### 7.10.4 AI の分類カテゴリ

| カテゴリ | 色 | アイコン | 意味 |
|---------|-----|---------|------|
| 今すぐ着手 | 赤 (`#d32f2f`) | ⚡ | 最優先で調理すべき品目 |
| 次に着手 | オレンジ (`#ffa000`) | 🔜 | 次に調理を開始すべき品目 |
| 待機中 | グレー | ⏸ | まだ着手不要な品目 |

### 7.10.5 AI からの追加情報

| 情報 | 内部値 | 表示色 |
|------|------|------|
| ペース「良好」 | `good` | 緑 (`#2e7d32`) |
| ペース「通常」 | `normal` | 黄 (`#f9a825`) |
| ペース「混雑」 | `busy` | 赤 (`#d32f2f`) |
| サマリーコメント | `summary` (string) | グレー背景 |

::: info AI 分析の API
AI キッチンダッシュボードは Gemini API を使用して分析を行います。API キーは POSLA 運営が一括管理しているため (`posla_settings.gemini_api_key`)、テナント側での設定は不要です。

エラー時の挙動:
- `AI_NOT_CONFIGURED` (503): POSLA 側未設定 → AI シェフボタン自動非表示
- 通信エラー: 「通信エラー: `<message>`」表示、再試行可能
:::

::: tip AI の提案を必ず守る必要はない
AI の提案はあくまで参考情報です。経験豊富なシェフの判断が優先されるべき場面 (急ぎの追加注文、お客様クレーム対応等) では無視して OK。
:::

## 7.11 KDS ステーションの設定

KDS ステーションは管理ダッシュボード (`dashboard.html`、店舗側) の「店舗設定」→「KDS ステーション」サブタブで管理します。

### 7.11.1 ステーション追加手順

1. owner / manager で `dashboard.html` にログイン
2. 上部タブから **店舗設定** をクリック
3. サブタブから **KDS ステーション** をクリック
4. 画面右上の **[+ ステーション追加]** ボタンをクリック
5. モーダルが開きます。以下を入力

| フィールド | 必須 | 文字種 | 説明 | 例 |
|-----------|------|------|------|----|
| **ステーション名** | ✅ | 全角 OK、最大 50 文字 | 日本語名 | 「厨房」「ドリンクバー」 |
| **ステーション名 (英語)** | 任意 | 半角英数 | 多言語表示用 | 「Kitchen」「Drink Bar」 |
| **担当カテゴリ** | 任意 | チェックボックス複数選択 | このステーションが担当するメニューカテゴリ | メイン料理 ✅ サラダ ✅ |
| **並び順** | 任意 | 整数 | ステーションバーの表示順 | 1, 2, 3 ... |

6. **[保存]** ボタンをクリック (`POST /api/store/kds-stations.php`)
7. ステーション一覧に新しいステーションが追加されます

### 7.11.2 ステーション編集

1. ステーション一覧から該当ステーションの **[編集]** ボタン
2. 名前 / カテゴリ / 並び順を変更
3. **[保存]** で `PATCH /api/store/kds-stations.php?id=xxx` を送信

### 7.11.3 ステーション有効/無効の切替

- ステーション一覧の **[有効/無効]** トグル
- 無効 (`is_active = 0`) にすると KDS のステーションバーに表示されなくなる
- 過去のルーティング設定は保持される (再有効化で復帰)

### 7.11.4 ステーション削除

1. **[削除]** ボタン
2. 確認ダイアログ「ステーション "xxx" を削除しますか？」 → **[OK]**
3. `DELETE /api/store/kds-stations.php?id=xxx` で完全削除
4. `kds_routing_rules` の関連レコードも CASCADE で削除

### 7.11.5 ステーション運用の例

#### 大規模キッチンの場合

| ステーション名 | 担当カテゴリ | KDS タブレット配置場所 |
|--------------|------------|-------------------|
| 厨房 (メイン) | メイン料理、前菜、サラダ | メイン調理場 |
| 揚げ場 | 揚げ物 | フライヤー横 |
| ドリンク | ドリンク、デザート | ドリンクバー |
| 寿司場 | 寿司、刺身 | 寿司カウンター内 |

各タブレットで KDS を開き、該当するステーションを選択すると、そのセクション担当の品目だけが表示されます。

#### 中規模キッチンの場合

| ステーション名 | 担当カテゴリ | 配置 |
|--------------|------------|------|
| キッチン | メイン、前菜、サラダ、揚げ物 | メイン調理場 |
| ドリンク・デザート | ドリンク、デザート | カウンター内 |

#### 小規模キッチンの場合

ステーション設定をせず「全品目」のまま使用すれば、全ての注文が 1 つの画面に表示されます。1 タブレット運用の最小構成。

### 7.11.6 実例: matsunoya 渋谷店のステーション構成

```
matsunoya 渋谷店 (s-matsu-shibuya-001)
├─ ステーション: 厨房 (sort_order=1)
│   担当: 定食、丼、麺
├─ ステーション: 揚げ場 (sort_order=2)
│   担当: 揚げ物、トンカツ
├─ ステーション: ドリンク (sort_order=3)
│   担当: ドリンク、アルコール
└─ (デザートはドリンクが兼任)

→ KDS タブレット 3 台、各ステーション専用
→ 厨房タブレットには「揚げ場」「ドリンク」品目は表示されない
```

## 7.12 GPS チェック (device ロール専用)

P1a で追加された device ロール (KDS / レジ端末専用アカウント) は、GPS 位置情報で店舗内からのアクセスを強制できます。

### 7.12.1 目的

- 店舗外から KDS に不正アクセスされるリスクを物理的に排除
- スタッフが自宅から KDS 画面を見られないようにする
- レジ操作 (返金等) を店舗内に限定

### 7.12.2 GPS チェックのフロー

```
[1] device アカウントでログイン
[2] KDS 画面に遷移しようとする
[3] KdsAuth.checkGps(storeId, callback) 実行
     ↓
[4] navigator.geolocation.getCurrentPosition() で現在地取得
     ├─ 許可拒否 → エラーオーバーレイ
     ├─ タイムアウト (15 秒) → エラーオーバーレイ
     └─ 取得成功
         ↓
[5] kds-auth.js 内の _haversine() で店舗座標との距離を計算 (クライアント側判定)
     → `gps_radius_meters` (デフォルト 200m) 以内か判定
     → サーバー API は呼ばない (`gps-check.php` は実装されていない)
     ↓
[6] allowed=true → KDS 表示 OK (callback 実行)
    allowed=false → エラーオーバーレイ「店舗外からのアクセスは
                    許可されていません (距離: 350m)」
```

### 7.12.3 設定

管理画面 (`dashboard.html`) → **店舗設定** → **シフト設定** サブタブ:

| フィールド | 説明 | デフォルト |
|----------|------|----------|
| **店舗緯度** (`store_lat`) | 店舗座標 (DECIMAL(10,7)) | NULL |
| **店舗経度** (`store_lng`) | 店舗座標 (DECIMAL(10,7)) | NULL |
| **GPS 許容半径** (`gps_radius_meters`) | 半径 N m 以内なら許可 | 100m |

::: tip device は gps_required 設定に関係なく常時 GPS チェック
通常スタッフ (manager/owner/staff) には `gps_required` 設定があり、ON でなければ GPS チェックをスキップします。しかし **device ロールは設定に関係なく常に GPS チェックされます** (P1a 仕様)。これは「端末は店舗内に固定する」という運用前提に基づきます。
:::

### 7.12.4 GPS 取得失敗時のエラー表示

| 原因 | 表示メッセージ | 対処 |
|------|------------|------|
| 許可拒否 (`PERMISSION_DENIED`) | 「GPS 許可がありません。ブラウザ設定から許可してください」 | Chrome 設定 > サイト情報 > 位置情報「許可」 |
| タイムアウト (`TIMEOUT`) | 「位置情報の取得に時間がかかっています。再試行してください」 | 屋外で再試行、Wi-Fi 接続確認 |
| 精度不足 (`accuracy > 100m`) | 「GPS 精度が低すぎます。屋外で再取得してください」 | 窓側に移動 |
| 範囲外 | 「店舗外からのアクセスは許可されていません (距離: XXm)」 | 店舗内に戻る、または店舗座標 / 半径設定を確認 |

### 7.12.5 manager / owner / staff の場合

これらのロールは `gps_required = 0` の場合 GPS チェックなしで KDS にアクセス可能 (既存仕様、店舗外からの確認も許可)。`gps_required = 1` に設定された場合は device と同じ挙動。

### 7.12.6 実例: 店舗座標の設定

```
matsunoya 渋谷店:
  store_lat = 35.6595
  store_lng = 139.7005
  gps_radius_meters = 100

→ device 端末で KDS にアクセスする際、現在地が
   (35.6595, 139.7005) から 100m 以内なら許可。
→ 店舗外 (例: 自宅から) アクセスすると
   「店舗外からのアクセスは許可されていません (距離: 5500m)」
```

## 7.13 よくある質問 (35 件以上)

### A. 操作

**Q1.** 起動時にタップしないと音が鳴らない
**A1.** ブラウザのセキュリティ制限 (Web Audio unlock)。「KDS を開始」必須。

**Q2.** 複数店舗で運用したい
**A2.** KDS は localStorage で単一店舗に紐付く。複数店舗は別タブ / 別タブレット。

**Q3.** カードが勝手に消える
**A3.** 全品目が `served` or `cancelled` になった注文は自動で消える (仕様)。

**Q4.** 注文が赤くなった
**A4.** 20 分以上経過で赤色警告。注文漏れや遅延の可能性、確認を。

**Q5.** ステーションを切り替えたら注文が減った
**A5.** ステーションが担当するカテゴリの品目のみ表示。「全品目」に戻すと全注文表示。

**Q6.** 全体ボタンと個別ボタンの使い分け
**A6.** 全品が同じタイミングで進む → 全体ボタン。品目ごとに違うタイミング → 個別ボタン。

**Q7.** 間違えて「提供済」にしてしまった
**A7.** 戻す機能なし。次回注文時に注意 (UI 改善予定)。

**Q8.** 1 つの品目だけキャンセルしたい
**A8.** 品目個別のキャンセルは未対応。注文全体を取消してから再発注するしかない (現状)。

**Q9.** アレルギー警告はどこから来る?
**A9.** お客様側のセルフメニュー「アレルギー選択」 (N-3) で入力されたもの。`order_items.allergen_selections` に JSON で保存。

**Q10.** カードのサイズが小さくて見にくい
**A10.** Chrome のズーム (Ctrl+`+`) で拡大可能。タブレット本体側のディスプレイ拡大率も調整可。

### B. 音声

**Q11.** 音声コマンドが動かない
**A11.** Chrome 必須 + 「KDS を開始」必須 + マイク許可必要。Firefox/Safari では動作保証なし。

**Q12.** 音声が誤認識する
**A12.** マイクが遠い or キッチンノイズが大きい。外付けマイク or ヘッドセット推奨 (23 章参照)。

**Q13.** 「T01 完成」が認識されない
**A13.** Web Speech API は英数字混じりが弱い。「テーブル 1 完成」や「1 番完成」が確実。

**Q14.** 確認ダイアログが出る
**A14.** Gemini の confidence が low の場合、5 秒確認ダイアログを表示 (誤動作防止)。

**Q15.** 雑談を拾われる
**A15.** ノイズフィルタ (2 文字未満 / 60 文字超は無視) + コマンドパターン検出 (動作語含む発話のみ) で対応済み。それでも誤検出する場合は音声 OFF を活用。

**Q16.** 音声 ON のままページ遷移して戻ったら ON のまま?
**A16.** はい。`localStorage:mt_kds_voice_active = 1` で次回起動時に自動復帰。

### C. AI キッチン

**Q17.** AI 調理優先度は当たる?
**A17.** 経過時間 + メニュー構成 + 過去の調理時間データで判定。精度は向上中。

**Q18.** AI パネルがうっとうしい
**A18.** パネル右上の **[✕ 閉じる]** で閉じる。60 秒自動更新も停止。

**Q19.** AI の提案と違う順番で作りたい
**A19.** 無視して OK。AI は参考情報で、強制ではない。

**Q20.** AI シェフボタンが消えた
**A20.** `AI_NOT_CONFIGURED` エラーで自動非表示。POSLA 運営に Gemini API キーの設定を依頼してください。

**Q21.** 自動更新の間隔を変えたい
**A21.** 現状は 60 秒固定 (`REFRESH_INTERVAL = 60000`)。手動更新ボタンで即座に再分析可能。

### D. 品切れ

**Q22.** 品切れ管理画面はどこ?
**A22.** ヘッダー右の **[品切れ管理]** リンク → `sold-out.html` に遷移。

**Q23.** 品切れにしたのに KDS に注文が来た
**A23.** 既に注文済みの品目は影響しない。新規注文だけお客様メニューから消える。

**Q24.** 「トンカツ品切れ」と音声で言ったが動作しない
**A24.** Gemini 解析が必要なコマンド。AI 設定を確認、または品切れ管理画面で手動操作。

**Q25.** 品切れ復活はどうやる?
**A25.** 品切れ管理画面で対象品目の「販売再開」ボタン、または音声「ハンバーグ販売再開」「唐揚げ復活」。

### E. ステーション

**Q26.** ステーションを増やしたい
**A26.** dashboard.html → 店舗設定 → KDS ステーションサブタブ → [+ ステーション追加]。詳細は 7.11。

**Q27.** ステーションを切り替えたら別タブレットの表示も変わる?
**A27.** いいえ。ステーション選択は端末 (localStorage) ごと。各タブレットで独立。

**Q28.** ステーション名を日本語と英語両方表示したい
**A28.** 現状は日本語のみ表示。英語名 (`name_en`) は将来の多言語対応用。

**Q29.** カテゴリ未割当のステーション
**A29.** 「全品目」と同じく、全カテゴリが表示されます (フィルタなし)。

### F. デバイス

**Q30.** device アカウントの作り方
**A30.** dashboard.html → スタッフ管理 → **[+ デバイス追加]** ボタン。詳細は 23 章。

**Q31.** GPS 許可ダイアログが毎回出る
**A31.** Chrome 設定で「常に許可」に。サイト情報 > 位置情報「許可」固定。

**Q32.** device アカウントでログアウトできない
**A32.** 画面右上のログアウトボタン (赤色) は常に表示されます。タップで `DELETE /api/auth/logout.php`。

**Q33.** device で店舗切替できない
**A33.** device は 1 端末 1 店舗の運用が前提。複数店舗の device は別途作成が必要。

### G. トラブル

**Q34.** 新規注文の通知音が聞こえない
**A34.** 音声 unlock が未済 → 画面をタップ後に再試行。タブレット本体音量も最大に。

**Q35.** ネット切断したら?
**A35.** オフラインバナー表示 + ポーリング 30 秒間隔に延長。復旧で自動復帰 + 即時更新。

**Q36.** 遅延の原因は?
**A36.** Wi-Fi 混雑、サーバー負荷、ポーリング間隔。原則 3 秒以内。

**Q37.** タブレットのスリープで表示が消える
**A37.** タブレット設定で「画面常時点灯」を ON。充電接続必須。

**Q38.** 注文が多すぎて画面に収まらない
**A38.** 自動スクロール機能あり (U-5)。新規注文が増えたカラムは最下部にスムーズスクロール。

**Q39.** コース注文がカードで見分けにくい
**A39.** 左ボーダー緑色 + 「コース P1:前菜」バッジが目印。

**Q40.** フェーズを戻したい
**A40.** 戻す機能なし (一方向のみ)。誤操作防止のため確認ダイアログあり。

**Q41.** コース中に単品注文が入ったら?
**A41.** 別カードで表示される (通常の注文と同じ扱い)。

**Q42.** Wi-Fi が頻繁に切れる
**A42.** 5GHz 帯の SSID に接続。2.4GHz は電子レンジ干渉が多い。23 章参照。

### 7.16.2 API エンドポイント (2026-04-19 時点の実ファイル)

- `GET /api/kds/orders.php?store_id=xxx&view=kitchen&station_id=xxx` — KDS カンバン用注文取得 (`view=accounting` は POS レジ用)
- `PATCH /api/kds/update-status.php` — 注文全体ステータス変更 (pending → preparing → ready → served → paid)
- `PATCH /api/kds/update-item-status.php` — 品目単位ステータス変更 (全品 served で親注文も自動 served)
- `POST /api/kds/advance-phase.php` — コースフェーズ進行 (`table_sessions.current_phase_number` 更新、次フェーズ注文を自動生成)
- `POST /api/kds/close-table.php` — テーブルクローズ (レガシー、現行は `process-payment.php` に統合)
- `GET /api/kds/call-alerts.php?store_id=xxx` — スタッフ呼び出しアラート取得 (KDS は `product_ready` と `kitchen_call` を除外、実質 `staff_call` のみ表示)
- `PATCH /api/kds/call-alerts.php` — アラート対応 (`acknowledged` に更新)
- `POST /api/kds/call-alerts.php` — アラート発信 (キッチン → スタッフの `kitchen_call`)
- `POST /api/kds/ai-kitchen.php` — AI 調理優先度分析 (Gemini 2.0 Flash、temperature: 0.3、urgent/next/waiting 分類)
- `GET/PATCH /api/kds/sold-out.php` — 品切れ管理 (`source=template` なら `store_menu_overrides`、`source=local` なら `store_local_items` 更新)
- `GET/POST /api/kds/cash-log.php` — レジ出納帳
- GPS 距離判定はクライアント側 (`public/kds/js/kds-auth.js` の `_haversine()`) で実施。サーバー API は存在しない (P1a)
- `GET/POST/PATCH/DELETE /api/store/kds-stations.php` — KDS ステーション CRUD (manager 以上)

### 7.16.4 音声 unlock の実装

```js
document.getElementById('kds-startup-btn').addEventListener('click', function () {
  VoiceCommander.unlockAudio();
  sessionStorage.setItem('mt_kds_started', '1');
  overlay.style.display = 'none';
  callback();
});
```

`VoiceCommander.unlockAudio()` の中身:

```js
function unlockAudio() {
  var ctx = new (window.AudioContext || window.webkitAudioContext)();
  ctx.resume();
  // 無音バッファを再生して確実に unlock
  var buffer = ctx.createBuffer(1, 1, 22050);
  var source = ctx.createBufferSource();
  source.buffer = buffer;
  source.connect(ctx.destination);
  source.start(0);
}
```

### 7.16.8 GPS チェック (kds-auth.js)

```js
KdsAuth.checkGps(storeId, function (ok) {
  if (!ok) {
    _showGpsOverlay('店舗外からのアクセスは許可されていません');
    return;
  }
  // KDS メイン画面に進む
});
```

内部処理:
1. `navigator.geolocation.getCurrentPosition` で現在地取得 (timeout: 10s)
2. `kds-auth.js` 内の `_haversine()` で店舗座標との距離をクライアント側計算
3. `gps_radius_meters` (デフォルト 200m) 以内なら `onSuccess()` を実行、超過時はオーバーレイで距離表示
   ※ サーバー側 API (`gps-check.php`) は存在しない

### 7.16.9 マルチテナント境界

KDS の全 API は以下のチェックを行う (`api/lib/auth.php`):

```php
require_auth();                  // セッション確認
require_store_access($storeId);  // 該当 store_id にアクセス可能か
```

`require_store_access()` 内部:
- ユーザーの所属 store_id 一覧を取得
- リクエストの $storeId が含まれるかチェック
- 含まれなければ 403 Forbidden

### 7.16.10 監査ログ

KDS 操作はすべて `audit_logs` テーブルに記録される:

| 操作 | action 値 | resource_type |
|------|---------|--------------|
| 注文ステータス変更 | `order_status` | `order` |
| 注文キャンセル | `order_cancel` | `order` |
| 品目ステータス変更 | `item_status` | `order_item` |
| アラート対応 | `alert_ack` | `call_alert` |
| AI 分析 | `ai_kitchen` | (なし、ログのみ) |

device ロールの操作も記録される (`actor_user_id` に device の users.id)。
