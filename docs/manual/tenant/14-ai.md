# 14. AI アシスタント

管理ダッシュボードの「AI アシスタント」タブでは、AI を活用した **SNS 投稿生成 / 売上分析 / 競合調査 / 需要予測** の 4 機能を利用できます。**マネージャー以上のロール限定**で、スタッフ（バイト）は使えません（理由は 14.0.4 参照）。

::: tip AI ヘルプデスクは「右下の橙色 ? ボタン」に移動しました（2026-04-19）
従来「AI アシスタント」タブの上部にあった **AI ヘルプデスク** セクションは、画面右下に常駐する **フローティングボタン (FAB)** に移行しました。タブを開かなくても、管理画面のどこからでも質問できます。詳細は [14.6](#146-ai-ヘルプデスク管理画面共通-fab) を参照。
:::

AI には Gemini 2.0 Flash API が使用されており、API キーは POSLA 運営が一括管理しているため、テナント側での設定は不要です。

::: info サーバープロキシ経由の呼び出し（P1-6）
管理画面の AI アシスタント機能は、API キーがブラウザに渡らないよう **サーバー側プロキシ経由** で Gemini API を呼び出します。
- フロントエンド (`public/admin/js/ai-assistant.js`) → サーバー (`api/store/ai-generate.php`) → Gemini API
- ブラウザの DevTools で API キーが見えることはありません
- AI サービスが利用できない場合は「AI サービスは現在利用できません。POSLA 運営にお問い合わせください」と表示されます
:::

### 14.0.4 スタッフは使えない（重要）

::: warning AI アシスタントタブはバイトスタッフに見せない設計
- ロール: manager / owner のみ表示（dashboard.html のタブ表示制御）
- 理由 1: AI 出力には経営判断（値上げ・メニュー削除・採用判断）に直結する情報が含まれる
- 理由 2: バイトが SNS 生成を勝手に投稿すると店舗ブランドを毀損するリスク
- 理由 3: 競合調査結果を外部に持ち出されると競合戦略が漏洩
- 理由 4: 需要予測は人件費・発注に直結する経営情報

スタッフが操作で困った場合は、**マネージャーに口頭で聞く** のが基本フロー。マネージャーが管理画面の AI ヘルプデスク FAB (14.6) で回答を引き出します。スタッフ用端末（KDS / レジ / ハンディ）には FAB を置きません。
:::

## 14.1 SNS 投稿生成

InstagramやX（旧Twitter）向けの投稿文を自動生成します。

### 14.1.1 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | manager 以上 |
| 2 | おすすめメニューが決まっている |
| 3 | 店舗 SNS アカウントが用意されている（投稿先） |
| 4 | （任意）アピールポイントが言語化できている |

### 14.1.2 操作手順（クリック単位）

1. dashboard.html → タブ **「AI アシスタント」** をクリック
2. 上部の **「SNS 投稿生成」** セクションへスクロール
3. **店舗名** を入力（既定: レシート設定の店舗名が自動入力）
4. **今日のおすすめメニュー** を入力（複数なら読点区切り）
5. （任意）**アピールポイント** を入力
6. **【Instagram】** または **【X (Twitter)】** ボタンをクリック
7. 「AI が生成中...」が表示される（1〜10 秒）
8. 結果カード（紫グラデーション）に投稿文が表示される
9. **【📋 コピー】** ボタンで全文クリップボードへ
10. SNS アプリに貼り付けて投稿

### 14.1.3 入力フィールドの詳細表

| # | フィールド | 入力例 | 必須 | 形式 | 補足 |
|---|---|---|---|---|---|
| 1 | 店舗名 | `松の屋 渋谷店` | ✅ | 全角OK | 既定: 店舗設定の receipt_store_name |
| 2 | 今日のおすすめメニュー | `特製焼き魚定食、季節の天ぷら盛り合わせ` | ✅ | 自由文 | 複数は読点区切り |
| 3 | アピールポイント | `本日限定！数量限定でご提供` | 任意 | 自由文 | 強調したい点 |

### 14.1.4 プラットフォーム別の生成

| プラットフォーム | 特徴 | 既定 temperature |
|---|---|---|
| Instagram | 絵文字使用、ハッシュタグ 5 つ以上、複数行の読みやすい形式 | 0.8 |
| X (Twitter) | テキストのみ、簡潔（140〜280 文字以内） | 0.8 |

### 14.1.5 AI パラメータ

| パラメータ | 値 |
|---|---|
| モデル | Gemini 2.0 Flash |
| temperature | 0.8（クリエイティブ） |
| max_tokens | 1024 |

## 14.2 売上・注文トレンド分析

店舗の売上データを AI が分析し、トレンドや改善点を提案します。

### 14.2.1 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | manager 以上 |
| 2 | 期間内に少なくとも 50 件の paid 注文（少ないと AI が「データ不足」と回答） |
| 3 | 13 章「レポート」の売上レポートが正常に表示される |

### 14.2.2 操作手順

1. dashboard.html → **「AI アシスタント」** タブ
2. **「売上・注文トレンド分析」** セクションまでスクロール
3. （owner のみ）**店舗ドロップダウン** で対象店舗を選択
4. **期間ボタン** を選択：今日 / 今週 / 今月（既定: 今週）
5. **【AI に分析させる】** ボタンをクリック
6. 「AI が分析中...」が表示される（5〜20 秒）
7. 結果カード（緑グラデーション）に分析が表示される

### 14.2.3 入力フィールドの詳細表

| # | フィールド | 既定 | 選択肢 | 補足 |
|---|---|---|---|---|
| 1 | 店舗 (owner のみ) | 最初の店舗 | dropdown | manager は自店舗固定 |
| 2 | 期間 | 今週 | today / week / month | クリックで切替 |

### 14.2.4 分析の内部フロー

```
1. 売上レポート API からデータを自動取得
   - トップ10 品目
   - 時間帯別売上
   - サービス時間データ
2. JSON を整形してプロンプトに埋め込む
3. /api/store/ai-generate.php に POST
4. Gemini が分析
5. 結果を整形（テーブル、見出し、箇条書き）して表示
```

### 14.2.5 AI からの出力（典型例）

| セクション | 内容 |
|---|---|
| 全体サマリー | 期間内の売上・注文・客単価の総括 |
| トレンド分析 | 増加 / 減少傾向、ピーク時間帯 |
| ベストセラー特定 | TOP 3-5 品目とその寄与度 |
| 改善が必要な領域 | キャンセル多発、提供時間長すぎ等 |
| 具体的改善提案 | 3-5 個のアクション提案 |

### 14.2.6 AI パラメータ

| パラメータ | 値 |
|---|---|
| モデル | Gemini 2.0 Flash |
| temperature | 0.5〜0.8（分析向け） |
| max_tokens | 1024 |

### 14.2.7 実例（週次分析）

```
週間 ¥1,184,000 / 542 件 / ¥2,184 / 前週比 +8%
TOP 焼き魚定食 148/¥162,800、唐揚げ 132、ハンバーグ 108
木曜が他平日より -15%
提案: ①木曜の集客強化 ②TOP 3 品の在庫アラート 1.5 倍
③12 時人員 +1 ④ハンバーグ × 大盛セット化 ⑤焼き魚 SNS 週 4 へ
```

## 14.3 周辺競合調査

周辺の競合店舗を Google Maps / Places API で取得し、Gemini で差別化戦略を提案します。

### 14.3.1 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | manager 以上 |
| 2 | **店舗住所が登録済み**（必須） |
| 3 | POSLA 運営が Google Places API キーを設定済み |

::: warning 店舗住所未登録時の挙動
住所が未登録の場合、画面に **オレンジの警告バナー**「店舗住所が未登録です。競合調査には住所情報が必要です。」が表示され、その横の **【店舗設定へ】** リンクで設定タブにジャンプできます。
:::

### 14.3.2 操作手順

1. dashboard.html → **「AI アシスタント」** タブ
2. **「周辺競合調査」** セクションまでスクロール
3. （owner のみ）**店舗ドロップダウン** で対象店舗を選択 → 住所が自動取得される
4. **調査半径ボタン** を選択：500m / 1km / 2km（既定: 1km）
5. **【周辺の競合店を調査する】** ボタンをクリック
6. 「Places API 検索中...」→「AI が分析中...」（合計 10〜30 秒）
7. 結果カード（青グラデーション）に競合店情報と差別化提案が表示される

### 14.3.3 入力フィールドの詳細表

| # | フィールド | 既定 | 選択肢 |
|---|---|---|---|
| 1 | 店舗 (owner のみ) | 最初の店舗 | dropdown |
| 2 | 調査半径 | 1000m | 500 / 1000 / 2000 |

### 14.3.4 内部フロー（Places API + Gemini）

```
1. フロント → /api/store/places-proxy.php?store_id=xxx&radius=1000
   - サーバーが Google Places Nearby Search API を呼び出す
   - API キーは POSLA 共通 (posla_settings.google_places_api_key)
   - CORS 回避のためサーバープロキシ
2. 上位 10-20 店舗の名前・評価・口コミ件数・住所を取得
3. 結果 JSON を整形してプロンプトに埋め込む
4. /api/store/ai-generate.php に POST
5. Gemini が分析
6. 結果を整形して表示
```

### 14.3.5 AI からの出力

- 競合店舗名と数値データを含む分析
- 価格帯・評価・推定客数の比較表
- 自店舗の強み / 弱みの SWOT 風整理
- **今週中に実行可能な 3 つの具体的アクション**
- 脅威度バッジ表示（高: 赤 / 中: オレンジ / 低: 緑）

### 14.3.6 エラーコード

| エラーコード | 意味 | 対処 |
|---|---|---|
| `EXTERNAL_SERVICE_ERROR` | Google 側の問題 | 数分待って再実行 |
| `PLACES_NOT_CONFIGURED` | POSLA 管理画面でキー未設定 | POSLA 運営に連絡 |
| `STORE_ADDRESS_MISSING` | 店舗住所未登録 | 設定 > レシート設定で住所入力 |

### 14.3.7 実例（半径 1km）

検出: 大戸屋 4.0/¥1,200、やよい軒 3.8/¥1,000、串カツ田中 4.2/¥3,500、ねぎし 4.1/¥1,500

AI 提案（脅威度: 中）:
1. 「本日の魚」ボード店頭設置（チェーンにできない差別化）
2. 大戸屋・やよい軒との価格比較を Web 掲載
3. ねぎしより安いボリューム強調を SNS 投稿

## 14.4 需要予測・発注提案

過去の売上データとレシピ情報を組み合わせて、今後の需要を予測します。

### 14.4.1 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | manager 以上 |
| 2 | 過去 7 日以上の売上データがある |
| 3 | （理想的には）レシピ情報・原材料データが入力済み |

### 14.4.2 操作手順

1. dashboard.html → **「AI アシスタント」** タブ
2. **「需要予測・発注提案」** セクションまでスクロール
3. （owner のみ）**店舗ドロップダウン** を選択
4. **【需要予測を実行】** ボタンをクリック
5. 「データ取得中...」→「AI が予測中...」（10〜30 秒）
6. 結果カード（オレンジグラデーション）に予測が表示される

### 14.4.3 入力フィールドの詳細表

| # | フィールド | 既定 | 選択肢 |
|---|---|---|---|
| 1 | 店舗 (owner のみ) | 最初の店舗 | dropdown |

期間や閾値は API 側で固定（過去 7 日 → 今後 7 日）。

### 14.4.4 データソース

| ソース | 取得元 |
|---|---|
| 過去 7 日間の売上データ | `orders` + `order_items` |
| 品目別の販売数量 | order_items.qty 集計 |
| 曜日別パターン | DAYOFWEEK(created_at) |
| レシピ情報 | `recipes` + `ingredients`（任意） |
| 既存在庫 | `inventory.current_stock`（任意） |

### 14.4.5 AI からの出力

| セクション | 内容 |
|---|---|
| 日別需要予測 | 今後 7 日の品目別販売予測（テーブル） |
| 原材料発注推奨 | 不足見込みの原材料と推奨発注量 |
| 低在庫アラート | 閾値を下回る見込みの原材料リスト |
| ポーション調整 | 使い切り・廃棄削減のための提案 |

### 14.4.6 AI パラメータ

| パラメータ | 値 |
|---|---|
| モデル | Gemini 2.0 Flash |
| temperature | 0.5（分析向け） |
| max_tokens | 8192（詳細予測のため大きめ） |

### 14.4.7 実例（7 日予測）

日別予測（主要品目のみ）:
- 焼き魚: 月 28、火 26、水 30、木 24、金 32、土 38、日 24（週合計 224 食）
- 唐揚げ: 週合計 190 食 / ハンバーグ: 147 食

発注推奨:
- 桜鱒 (焼き魚) 30kg を週末納品
- 鶏もも 25kg を火曜納品 / 牛挽肉 20kg を木曜納品

低在庫アラート: たけのこ 残 2kg / 想定 8kg → 即発注

ポーション提案: ハンバーグ 130g → 110g で提供時間短縮 / 木曜限定「半額デザート」で集客。

### 14.5.2 機能

## 14.5-b AI 予約（お客様向け、自由文入力）

## 14.6 AI ヘルプデスク（管理画面共通 FAB）

このマニュアル全体を知識ベースとして参照する **AI ヘルプデスク** を、管理画面右下の **オレンジ色（#ff6f00）の円形フローティングボタン (FAB、60×60px)** から呼び出せます。タブ移動なしで、どの画面からでも 1 タップで質問できます。

### 14.6.1 設置範囲（重要）

| 画面 | FAB | 理由 |
|---|---|---|
| 管理 dashboard / owner-dashboard / posla-admin | ✅ | 管理職の調べ物用 |
| POS レジ / KDS / ハンディ POS | ❌ | 業務中の操作妨害防止 |
| セルフオーダー | ❌ | お客様向けは AI ウェイター (14.5) |

::: info スタッフは AI ヘルプデスクを使わない運用
バイトスタッフが操作で困った場合は、**マネージャーに口頭で聞く** のが基本フローです。マネージャーが管理画面の AI ヘルプデスクで回答を引き出します。スタッフ用端末（KDS / レジ / ハンディ）には FAB を置かないことで、操作迷子と業務妨害を防止しています。
:::

### 14.6.2 機能概要

- **質問例**:
  - 「パスワードはどうやって変えますか？」
  - 「予約金の設定方法は？」
  - 「no-show になった時の処理は？」
  - 「**E3024 とは？**」（エラー番号で直接質問）
  - 「**[E5101] 在庫不足エラーの対処は？**」
- **回答**: マニュアル 26 章 (tenant) + Part 0-8 (operations) + 運営資料 (internal、POSLA 運営モード限定) の該当箇所から引用して回答
- AI モデル: Gemini 2.0 Flash
- 2 モード:
  - **tenant モード**: `helpdesk-prompt-tenant.txt` (~800KB、tenant 26 章 + operations Part 0-8 + voice-commands + 99-error-catalog)
  - **internal モード**: `helpdesk-prompt-internal.txt` (~1.1MB、上記 + internal 8 章 + SYSTEM_SPECIFICATION)

### 14.6.3 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | manager 以上のロール |
| 2 | 管理画面（dashboard / owner-dashboard / posla-admin）にいる |
| 3 | POSLA 運営が helpdesk-prompt-*.txt をビルド・配置済み |

### 14.6.4 使い方（クリック単位）

1. 管理画面の **右下にあるオレンジ色の ? ボタン**（60×60 円形 FAB）を 1 タップ
2. チャットパネル（380×600px）が開く
3. 上部に「🤖 POSLA AI ヘルプデスク」のヘッダー
4. 中央のログエリアにシステムメッセージ「画面に「エラー E2017」のような番号が表示されたら、そのまま入力すると意味と対処方法をご案内します。」
5. 下部にサジェストチップ：
   - E3024 とは？
   - 会計手順
   - 返金手順
   - 出勤打刻
6. **入力欄** に自然な日本語で質問（または画面に表示されたエラー番号 `[Exxxx]` をそのまま貼り付け）
7. **【送信】** ボタンまたは Enter キー（Shift+Enter で改行）
8. 「AI が回答を準備中…」のローダー表示（5〜15 秒）
9. AI がマニュアル該当箇所を引用して回答
10. 回答を読んで実行 → 必要なら追記質問（過去 4 ターンを文脈として保持）

### 14.6.7 実装

- フロント: `public/shared/js/posla-helpdesk-fab.js`（60×60 円形 FAB、自己挿入型 IIFE）
- API: `api/store/ai-generate.php` （POST、manager 以上、サーバープロキシ経由）
- プロンプト構築: `scripts/build-helpdesk-prompt.php` が全 md + 99-error-catalog.md を結合
- 事前ビルド済み `.txt` がサーバー上に配置されている
- マニュアル更新時に **再ビルド + デプロイ必須**

### 14.6.8 mode の指定

POSLA 管理者用の internal モードを使うには HTML 側で:

```html
<script>window.POSLA_HELPDESK_MODE = 'helpdesk_internal';</script>
<script src="../shared/js/posla-helpdesk-fab.js?v=cb1c"></script>
```

サーバー側で POSLA 管理者かどうかを `is_posla_admin || tenant_id === 'posla'` で検証。違反は 403 FORBIDDEN。

### 14.6.9 回答品質を上げるコツ

- **エラー番号で聞く**: `[E3024]` を見たらそのままコピペ → カタログ + 対処方法をまとめて回答
- **具体的に質問する**: ❌「支払いどうするの」 → ✅「現金会計の流れは？」
- **章番号を引用する**: 「8 章の会計手順を教えて」
- **過去文脈を使う**: 同じパネルで連続質問すると過去 4 ターンが文脈として送られる
- **英語で質問してもOK**: Gemini が解釈して日本語で回答

### 14.6.10 トラブル時の挙動

| エラー | 表示 | 対処 |
|---|---|---|
| `KNOWLEDGE_BASE_MISSING` (503) | 「ヘルプデスク knowledge base が未生成です。POSLA 運営に連絡してください」 | POSLA 運営に再ビルドを依頼 |

| `FORBIDDEN` (403) | 「POSLA運営モードはPOSLA管理者のみ利用可能です」 | internal モード指定を外す |

## 14.X FAQ 27 件

**Q1.** AI 機能の使用料は？
**A1.** POSLA 運営が Gemini / Google Places API 費用を負担。テナント無料。

**Q2.** AI が間違えた時は？
**A2.** 参考情報として扱い、最終判断は人間。特に SNS 投稿は必ず投稿前に内容確認。

**Q3.** AI が使える場所は？
**A3.** AI アシスタントタブ (4 機能) / KDS (調理優先度) / メニュー翻訳 / 予約 / セルフオーダー (ウェイター) / 管理画面共通 FAB (ヘルプデスク)。

**Q4.** 音声コマンドが動かない。
**A4.** Chrome 必須。マイク許可が必要。Safari は Web Speech API 制限あり。

**Q5.** AI 停止時の影響。
**A5.** 必須機能には使わない設計。AI OFF でも通常運用可（売上集計・KDS は AI 不要）。

**Q6.** Gemini が「マニュアルに記載がないため、POSLA 運営にお問い合わせください」と返す。
**A6.** マニュアル更新後にプロンプト再ビルドされていない。POSLA 運営に依頼。

**Q7.** AI 出力に絵文字が含まれて困る。
**A7.** SNS 用は絵文字含む設計。X / Twitter で絵文字を減らしたい場合は出力後に手動編集。

**Q8.** SNS 投稿生成のハッシュタグ数を変えたい。
**A8.** 現状は AI 任せ（5+）。プロンプトカスタマイズ機能は Phase 2。

**Q9.** 売上分析の期間を 6 ヶ月にしたい。
**A9.** 現状は今日 / 今週 / 今月の 3 択。Phase 2 でカスタム期間予定。

**Q10.** 競合調査の半径をもっと広く（5km）したい。
**A10.** 現状は 500m / 1km / 2km の 3 択。広域調査は Google Maps を直接使うことを推奨。

**Q11.** 需要予測の対象を 14 日に伸ばしたい → 7 日固定（長期は精度低下のため）。

**Q12.** AI ウェイター「カート追加」が反映されない → Cookie / localStorage 確認。シークレットモード不可。

**Q13.** AI 予約が日付を誤解 → 「明日」は JST 解釈。曖昧表現は「2026-04-25 19:00」明示形式へ案内。

**Q14.** AI ヘルプデスクが遅い → KB ~1MB のため初回 5-15 秒。2 回目以降は同セッションで高速化。

**Q15.** ヘルプデスクで「FORBIDDEN」 → internal モードなのに POSLA 管理者でない。`POSLA_HELPDESK_MODE = 'helpdesk_tenant'` に戻す。

**Q16.** スタッフ端末から FAB を使いたい → 設計上不可（KDS / レジ / ハンディには FAB なし）。マネージャー経由。

**Q17.** AI が間違った価格を言う → AI は実価格 DB 未参照。価格は手動確認。

**Q18.** SNS で店名が間違う → 入力欄の店舗名を正しく。既定は receipt_store_name。

**Q19.** 競合調査の「住所未登録」警告が消えない → 住所登録後にリロード必要。

**Q20.** 需要予測を仕入先システム連携 → 現状不可。コピペ運用。

**Q21.** ヘルプデスク会話履歴 → 同パネル開いている間のみ。閉じるとリセット。

**Q22.** 予約 AI を OFF にしたい → 【予約管理】→【予約設定】→「AI チャット予約」トグル OFF。

**Q23.** AI ウェイターが英語で答えない → 既定日本語。多言語は別途翻訳機能を検討。

**Q24.** Gemini モデルは変わる？ → ハードコード (`ai-generate.php` L68)。差し替えは POSLA 運営判断。

**Q25.** AI 出力が途中で切れる → max_tokens 上限 (既定 1024、最大 8192)。需要予測は 8192。

**Q26.** ヘルプデスクのプロンプト編集 → 入力欄に直接書ける。Shift+Enter で改行。

**Q27.** 競合調査を CSV 出力 → 現状はテキストのみ。コピペで Excel へ。

## 14.Z 技術補足

### 14.Z.1 共通仕様

Gemini 2.0 Flash 固定 (`gemini-2.0-flash`) / API キーは `posla_settings.gemini_api_key`（POSLA 運営一括負担）/ サーバープロキシ経由で API キー露出なし / temperature: SNS 0.8 / 分析 0.5-0.8 / コマンド 0.0 / キッチン 0.3 / ヘルプデスク 0.2

### 14.Z.2 `api/store/ai-generate.php` 詳細仕様（汎用 AI プロキシ）

POSLA 管理画面の AI 機能（SNS 投稿生成・売上分析・競合調査・需要予測）はすべてこのエンドポイント経由で Gemini にアクセスします。

#### リクエスト

```
POST /api/store/ai-generate.php
Body: {
  prompt: string,           // プロンプト本文（必須）
  temperature?: number,     // 0-2 にクランプ（未指定=0.7→0.8）
  max_tokens?: number,      // 1-8192 にクランプ（未指定=1024）
  mode?: string             // 'default' | 'helpdesk_tenant' | 'helpdesk_internal'
}
```

#### レスポンス

```
{ "ok": true, "data": { "text": "生成結果テキスト" } }
```

#### エラー

| エラーコード | HTTP | 意味 |
|---|---|---|
| `MISSING_PROMPT` | 400 | プロンプトが空 |
| `KNOWLEDGE_BASE_MISSING` | 503 | helpdesk-prompt-*.txt 未生成 |
| `FORBIDDEN` | 403 | internal モードで POSLA 管理者でない |
| `AI_NOT_CONFIGURED` | 503 | `posla_settings.gemini_api_key` 未設定 |
| `FETCH_FAILED` | 502 | Gemini API 接続失敗 |
| `PARSE_FAILED` | 502 | Gemini レスポンス JSON 解析失敗 |
| `AI_ERROR` | 502 | Gemini 側エラー |
| `EXTERNAL_SERVICE_ERROR` | 502 | Gemini 側障害 |

#### 権限

`require_auth()` （manager 以上、helpdesk_internal は POSLA 管理者のみ）

### 14.Z.3 mode='helpdesk_*' 時の挙動

- `mode='helpdesk_tenant'` または `'helpdesk_internal'` の場合
- 該当 KB ファイル (`scripts/output/helpdesk-prompt-*.txt`) を読み込み
- プロンプト = KB + ユーザー質問 + 「マニュアルに記載がないなら『記載なし』と答えて」指示
- temperature を 0.2 に強制（正確性優先）
- max_tokens を 2048 に強制

### 14.Z.4 関連 AI エンドポイント

| エンドポイント | 用途 | レートリミット |
|---|---|---|
| `/api/store/ai-generate.php` | 汎用 AI 生成（SNS / 分析 / 予測 / ヘルプデスク） | なし（manager 以上のみ） |
| `/api/store/places-proxy.php` | Google Places プロキシ（競合調査） | なし |
| `/api/customer/ai-waiter.php` | お客様 AI ウェイター | 20 req / 1h |
| `/api/customer/reservation-ai-parse.php` | 予約自由文解析 | 5 req / 60s |
| `/api/kds/ai-kitchen.php` | AI 調理優先度 (urgent/next/waiting) | なし |

### 14.Z.5 プロンプト設計ガイドライン (CLAUDE.md より)

- 「〜のみ出力。前置き・解説・補足は禁止」を必ず含める
- Gemini の前置き応答（「はい、承知いたしました」等）は `_cleanResponse()` で自動除去する
- temperature: SNS 生成 = 0.8、データ分析 = 0.8、コマンド解析 = 0、ヘルプデスク = 0.2
