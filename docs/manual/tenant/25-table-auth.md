# 25. テーブル開放認証 (新着席フロー)

POSLA は **スタッフが物理的にテーブルを「開放」した後でしかお客様が QR から着席できない**仕組み (ホワイトリスト方式) を採用しています。これにより、**店外から QR の写真を撮って遠隔注文する攻撃**を物理的に防ぎます。同時に、予約客は自動マッチングでスタッフ操作ゼロで着席できます。

---

## 25.0 はじめにお読みください

::: tip 章の読み方ガイド
- **はじめての方**: 25.0 → 25.1 → 25.3 の順で読むと全体が分かります
- **PIN 方式 / 自動 auto-create から移行した方**: 25.2 で変化点をチェック
- **予約メインの店舗**: 25.4 (自動マッチング) が重要
- **技術担当 / POSLA 運営**: 25.11 データベース構造まで
:::

### この認証方式の目的

- ❌ 防ぎたい: 店外から QR の写真を使った遠隔注文
- ❌ 防ぎたい: SNS で拡散された QR 画像による第三者攻撃
- ❌ 防ぎたい: 元従業員が覚えている URL からの攻撃
- ✅ 維持したい: お客様の UX (PIN 入力させない / GPS 許可させない)
- ✅ 維持したい: 予約客はスタッフ操作なしで着席

### 3 つの認証方式 (状況で使い分け)

| 方式 | 対象 | 店員操作 | 客操作 |
|---|---|---|---|
| **自動マッチング** | 予約客 (時刻±30分+テーブル割当済) | ★ 不要 | QR 読む |
| **スタッフ開放** | Walk-in 客 | 「📲 QR開放」1 タップ | QR 読む |
| **手動着席** | QR 読めないお客様 | ハンディの「着席」ボタン | 不要 |

### 必要なもの

- タブレットまたはスマホ (ハンディとして使用)
- 店内 Wi-Fi (ハンディがサーバーと通信)
- テーブルに貼った QR コード (既存のもので OK)

### 所要時間

| 作業 | 目安 |
|---|---|
| Walk-in 客を開放 → 着席 | **10 秒** (スタッフ開放 + 客が QR 読み) |
| 予約客を着席 (自動マッチ) | **0 秒** (スタッフ操作不要) |
| 例外対応 (手動着席) | **20 秒** |

## 25.1 全体像: 3 つの認証方式

### 25.1.1 方式 1: 自動マッチング (予約客)

```
[お客様] テーブル T03 の QR を読む
  ↓
[サーバー] 内部で自動判定:
  ・T03 に紐付く予約あり?
  ・現在時刻は予約時刻 ±30 分以内?
  ・ステータス confirmed/pending?
  ↓ 全部 YES
[自動着席成立]
  ・table_sessions 作成 (status='seated')
  ・予約 status='seated' に更新
  ・顧客台帳 visit_count +1
  ↓
[お客様] セルフメニュー画面が開く → 注文 → 会計
```

**スタッフ操作: 0**。予約を入れておけば、予約時刻付近に客が来て QR を読むだけで完結。

### 25.1.2 方式 2: スタッフ開放 (Walk-in 客)

```
[お客様来店] → スタッフが T03 へご案内
  ↓
[スタッフ] ハンディで「📲 QR開放 (5分)」をタップ
  ↓
[サーバー] tables.next_session_token 発行 (5分有効、ワンタイム)
  ↓
[お客様] T03 の QR を読む
  ↓
[サーバー] next_session_token 検証 → OK
  ・table_sessions 作成
  ・next_session_token 消費 (次の客には使えない)
  ↓
[お客様] セルフメニュー画面が開く
```

**スタッフ操作: 1 タップ**。次客用のトークンは使い切りなので、攻撃者に漏れても再利用不可。

### 25.1.3 方式 3: 手動着席 (例外)

```
[状況] お客様が QR を読むのが苦手 (高齢者等) / QR 汚れで読み取れない
  ↓
[スタッフ] ハンディの「着席」ボタン (従来機能)
  ・人数入力 / コース選択 / メモ等
  ↓
[table_sessions 起票]
```

この方式はお客様がセルフメニューを使わない (スタッフがハンディから注文取る) 運用時にも使います。

### 25.1.4 3 方式の使い分けフロー

```
お客様来店
  ↓
予約あり?
 ├ YES → 時刻 ±30分 以内?
 │        ├ YES → 【方式 1: 自動マッチング】(客が QR 読む)
 │        └ NO  → 【方式 3: 手動着席】 or ガントで ✏️変更
 └ NO → Walk-in
          ↓
         お客様が QR 使える?
          ├ YES → 【方式 2: スタッフ開放】
          └ NO  → 【方式 3: 手動着席】
```

## 25.2 旧方式との比較 (PIN / auto-create からの移行)

### 25.2.1 歴史的経緯

| 時期 | 方式 | 問題 |
|---|---|---|
| 〜2026-04-17 | **auto-create** (客が QR 読むと即セッション作成) | 店外から注文可能、攻撃リスク |
| 2026-04-17 (S6) | **PIN 方式** (スタッフが PIN 発行 → 客が 4 桁入力) | UX 悪い (入力面倒)、店員に PIN 伝える運用負荷 |
| **2026-04-18〜 現行** | **ホワイトリスト方式** (スタッフ開放 + 自動マッチング) | ✅ 最適化完了 |

### 25.2.2 PIN 方式との違い

| 項目 | PIN 方式 (旧) | ホワイトリスト方式 (現行) |
|---|---|---|
| 店員操作 | PIN 発行 (自動) + 客に口頭伝達 | ハンディで「📲 QR開放」1 タップ |
| 客操作 | QR 読む + **4 桁 PIN 入力** | **QR 読むだけ** |
| 予約客 | PIN 必要 | **自動マッチで PIN 不要** |
| 認証強度 | 中 (PIN ブルートフォース可) | 高 (物理的に店内操作必要) |

### 25.2.3 自動 auto-create との違い

| 項目 | auto-create (旧) | ホワイトリスト方式 (現行) |
|---|---|---|
| 店員操作 | なし | 1 タップ (walk-in) / 0 (予約) |
| 攻撃耐性 | ❌ 店外から遠隔注文可 | ✅ 物理操作必要 |
| UX | ◎ すぐ使える | ◎ 差は 1 タップのみ |

### 25.2.4 移行時の対応 (既存店舗)

S6 から移行する場合、**店舗側で特別な設定変更は不要**。

- 予約管理機能を ON にしていれば、予約時の自動マッチングが自動有効
- walk-in 時はハンディの「📲 QR開放」ボタンが新しく出現
- 旧 PIN 表示 UI は自動的に非表示

お客様への周知も不要 (操作感は同じか簡単化)。

## 25.3 スタッフ開放の手順 (Walk-in 客の場合)

### 25.3.1 前提条件

- ハンディ (スマホ / タブレット) にログイン済み
- 店舗 Wi-Fi に接続済み
- 対象テーブルに QR コードが物理的に貼ってある
- そのテーブルが空席状態 (直前客が退店済)

### 25.3.2 ステップ 1: お客様をテーブルにご案内

1. 入口でお客様を出迎え: 「いらっしゃいませ、何名様ですか?」
2. 人数に合った空席テーブルを特定 (例: T03 の 4 名席)
3. 「こちらのテーブルへどうぞ」とご案内
4. お客様が着席するまで待つ (まだハンディは操作しない)

### 25.3.3 ステップ 2: ハンディで「テーブル状況」を開く

1. ハンディを取り出す
2. 画面上部のタブから **【テーブル状況】** をタップ
3. テーブル一覧がグリッド表示される

```
┌────────────────────────────┐
│ 店内注文 テイクアウト テーブル状況 予約 注文履歴 │
├────────────────────────────┤
│ ┌──────┐ ┌──────┐ ┌──────┐  │
│ │T01   │ │T02   │ │T03   │  │
│ │4名   │ │4名   │ │2名   │  │
│ │[着席]│ │[着席]│ │[着席]│  │
│ │[QR開放]│[QR開放]│[QR開放]│
│ └──────┘ └──────┘ └──────┘  │
│ ┌──────┐ ┌──────┐           │
│ │T04   │ │T05   │           │
│ │6名   │ │6名   │           │
│ │[着席]│ │[着席]│           │
│ │[QR開放]│[QR開放]│           │
│ └──────┘ └──────┘           │
└────────────────────────────┘
```

### 25.3.4 ステップ 3: 「📲 QR開放 (5分)」ボタンをタップ

1. 該当テーブル (例: T03) のカードを見つける
2. カード下部の **【📲 QR開放 (5分)】** (緑色ボタン) をタップ
3. 確認ダイアログが表示される:

```
┌────────────────────────────┐
│ テーブル T03 を客向けに開放します。│
│ 5分以内に客が QR を読み込むと    │
│ 自動着席します。                 │
│ よろしいですか?                  │
│                                  │
│        [キャンセル]  [OK]         │
└────────────────────────────┘
```

4. **【OK】** をタップ
5. 画面下部にトースト通知: 「テーブル T03 を開放しました (5分有効)」

### 25.3.5 ステップ 4: お客様が QR を読む

1. お客様に「お席の QR コードからメニューをご覧ください」と一言
2. お客様がスマホで T03 の QR コードを読み込む
3. お客様のスマホで セルフメニュー画面が自動で開く
4. 同時に、ハンディのテーブル状況が **「着席中」(緑色)** に自動更新される (数秒後)

### 25.3.6 ステップ 5: セルフ注文開始

1. お客様が自分のスマホで注文
2. 注文が送信されると KDS (キッチン) に即座に表示
3. 以降は通常の会計フロー

### 25.3.7 開放の失敗パターンと対処

| 症状 | 原因 | 対処 |
|---|---|---|
| 「テーブルは使用中です」エラー | 他のセッションが残っている | ガントで該当セッション確認、会計完了 or キャンセル |
| ボタンが表示されない | 既に別客が着席中 | カードが緑色なら退店/会計後に再試行 |
| 5分後に「期限切れ」 | 客が QR を読まなかった | もう一度「📲 QR開放」を実行 |
| 開放したのに客が読めない | QR 汚れ / スマホカメラ不調 | 手動着席 (25.5) に切替 |

## 25.4 自動マッチング (予約客の場合)

### 25.4.1 事前条件 (予約段階で満たすべきこと)

予約登録時に以下の 3 条件が満たされていないと、自動マッチングは作動しません:

1. **テーブル割当済** — 予約の `assigned_table_ids` にテーブルが入っている (通常は予約作成時に自動で割り当てられる)
2. **時刻が明確** — `reserved_at` が具体的な日時
3. **ステータス確定** — `confirmed` または `pending` (キャンセル / no-show 不可)

### 25.4.2 来店時フロー

```
[1] お客様来店 → 「鈴木 様でご予約の」と確認
[2] スタッフが予約のテーブル (ガントで確認) へ ご案内
    ※ スタッフのハンディ側では何も操作不要
[3] お客様が座席の QR を読む (スマホで)
[4] サーバーが自動判定 (0.5秒以内):
    ・T03 に紐付く予約あり → 鈴木様の予約
    ・現在時刻 19:02 は予約 19:00 ±30分 内 → OK
    ・status='confirmed' → OK
    → 自動マッチ成立
[5] 即座に table_sessions 起票 + 予約紐付け
[6] お客様のスマホでセルフメニューが開く
[7] 同時に 管理画面のガントが「着席中」に自動更新
```

### 25.4.3 自動マッチしないケース (フォールバック)

| ケース | 解説 | 対処 |
|---|---|---|
| 予約客が早く着いた (30分以上前) | ±30分範囲外 | スタッフが **ガントで「🪑 着席」ボタン** で強制着席 |
| 予約客が遅刻 (30分以上) | 同上 | 同上 |
| テーブル変更 (予約は T01 だが T03 へ案内) | assigned_table_ids に T03 なし | ガントで **「✏️ 変更」→ テーブルを T03 に** → 客が T03 の QR 読み直し |
| お客様が QR を読むのが苦手 | QR スキャンできない | スタッフが **【🪑 着席】 ボタン** で手動着席 |

### 25.4.4 時刻範囲の設定 (±30分固定)

現在、時刻範囲は **±30分** でハードコード。今後、店舗設定で可変化する可能性あり。

- 19:00 予約 → 18:30〜19:30 の間に QR 読込で自動マッチ
- 早着・遅刻の 30 分以内なら自動対応

::: tip 30 分は短い?
多くの飲食店では予約時刻の前後 30 分が一般的な許容範囲。
もっと広くしたい場合は POSLA 運営にお問い合わせください。
:::

## 25.5 手動着席 (QR が使えないお客様)

### 25.5.1 使用場面

- 高齢のお客様で QR スキャンが苦手
- スマホを持っていない / バッテリー切れ
- QR コードが汚れている / 剥がれている
- セルフオーダーを使わない運用 (スタッフが全て手取り)

### 25.5.2 ハンディから手動着席

1. ハンディの **【テーブル状況】** タブ
2. 該当テーブルカードの **【着席】** ボタン (青色、QR 開放ボタンとは別)
3. モーダル表示:

```
┌────────────────────────────┐
│ T03 に着席                 │
├────────────────────────────┤
│ 人数        [4]            │
│ プラン      [なし ▼]       │
│ コース      [なし ▼]       │
│ メモ        [              ]│
├────────────────────────────┤
│        [戻る]     [着席]    │
└────────────────────────────┘
```

4. 人数を入力
5. 必要ならプラン / コースを選択
6. メモ (アレルギー等) を入力
7. **【着席】** ボタン → セッション起票 + ガント更新

### 25.5.3 予約客の手動着席 (ガント経由)

予約客で自動マッチしない場合:

1. 管理画面の **【予約管理】** タブ
2. ガントで該当予約ブロックをタップ
3. モーダルの **【🪑 着席】** ボタン
4. QR モーダル表示 (印刷 or URL コピー可)
5. 閉じる → ガントが緑色に更新

## 25.6 着席後の動き

### 25.6.1 着席完了時に内部で起きること

方式 1/2/3 のいずれでも、着席完了時に以下が実行されます:

1. `table_sessions` に新規レコード INSERT
   - status='seated'
   - started_at=NOW()
   - reservation_id (予約マッチの場合)
   - guest_count (人数)
2. `tables.session_token` を新規発行 (4時間 TTL)
3. `tables.next_session_token` をクリア (ワンタイム消費、方式 2 の場合のみ)
4. 予約マッチング成立時:
   - `reservations.status='seated'`, `seated_at=NOW()`, `table_session_id` セット
   - `reservation_customers.visit_count +1`, `last_visit_at=NOW()`

### 25.6.2 お客様のスマホ側の動き

- URL: `https://eat.posla.jp/public/customer/menu.html?store_id=...&table_id=T03`
- `table-session.php` が呼ばれる
- レスポンスに `sessionToken` (4時間有効) を含む
- スマホにトークン保持 → 以降 4 時間はこのトークンで注文 API 呼出

### 25.6.3 セッションの状態遷移

```
seated (着席)
  ↓ 注文があれば
eating (食事中)
  ↓ 「お会計」ボタンが押されたら
bill_requested (会計待ち)
  ↓ 会計完了で
paid (会計済)
  ↓ 5分後 or 手動で
closed (終了、テーブル再開放可)
```

### 25.6.4 同一セッションの持続時間

- `tables.session_token_expires_at` = 4 時間
- 4 時間を過ぎたら新規発行になり、リロード時にエラー表示

## 25.7 会計後のクールダウン (S2)

### 25.7.1 目的

会計完了 (status='paid') 直後に、お客様がうっかりリロードしてしまうことで、**新しいセッションが作られてしまう誤操作事故**を防ぐ。

### 25.7.2 仕組み

```
[1] 会計完了 → table_sessions.status='paid' + closed_at=NOW
[2] 5分以内にお客様が同じ QR を読む (or リロード)
[3] サーバー判定: 「直近 5 分以内に paid セッションあり」
[4] → SESSION_COOLDOWN エラー
    「お会計が完了しています。新しいご注文はスタッフにお声がけください」
```

### 25.7.3 クールダウン期間

- 現状 5分 固定 (ハードコード)
- 5分 経過後は通常どおり新規開放可能

### 25.7.4 例外: 次の予約客

クールダウン中であっても、次の予約の自動マッチング条件を満たせば着席可能 (連続予約店舗の配慮)。

```
例: 19:00 予約A が会計完了 (19:45)
    19:50 予約B が来店して QR を読む
    → クールダウン中 (5分以内) だが、予約B のマッチング条件を満たすので着席OK
```

| 予約 | 当日予約リスト → QR開放 / 即着席 / no-show |
| 注文履歴 | 当日の注文一覧 |

### 25.8.3 KDS (kds/index.html)

- 着席したテーブルからの注文が自動表示される
- 着席自体は KDS 画面から操作しない (表示のみ)

### 25.8.4 お客様のスマホ (menu.html)

- QR 読み込みで自動的にアクセス
- 認証成功で注文画面
- 認証失敗時は以下のエラー画面:
  - `NO_ACTIVE_SESSION`: 「スタッフが着席操作を行うまでお待ちください」
  - `SESSION_COOLDOWN`: 「お会計が完了しています。スタッフにお声がけください」

## 25.9 セキュリティ防御の詳細

### 25.9.1 防げる攻撃一覧

| 攻撃シナリオ | 防御メカニズム |
|---|---|
| 店外から QR 画像を読んで遠隔注文 | スタッフ開放 (next_session_token) なし → セッション作成不可 |
| SNS で拡散された QR 画像悪用 | 同上 |
| 元従業員が URL を覚えていて攻撃 | 同上 |
| 会計後のリロード悪用 | 5分クールダウン (S2) |
| 大量注文テロ | per-session 注文上限 + IP レートリミット (S1) |
| ブラウザ変えて多重セッション | table_sessions.status='seated' の排他制御 |
| 予約時刻 前後の悪用 | ±30分 で制限 |

### 25.9.2 残るリスクと緩和策

| リスク | 発生可能性 | 被害 | 緩和策 |
|---|---|---|---|
| 店内でスタッフ開放直後の 5 分で攻撃 | 低 | 数件の注文 | KDS 目視で却下、注文金額上限 |
| 予約時刻に合わせた攻撃 (予約データ漏洩+来店風) | 極低 | 1 セッション分 | 予約 LP のフォームで SQL インジェクション対策 |
| 内部犯行 (スタッフが共犯) | 低 | 不明 | audit_log で操作追跡 |

### 25.9.3 レートリミット (S1 併用)

以下の API にレートリミット適用:

- `table-session.php`: 30 回/10分/IP
- `orders.php` (客側): 10 回/10秒/IP
- `reservation-create.php`: 5 回/60秒/IP

## 25.10 よくある質問 20 件

### A. 仕組みに関する質問

**Q1.** 旧 PIN 方式から移行するのに特別な設定変更は必要?
**A1.** 不要。自動的に新方式に切り替わります。

**Q2.** 予約なしで来店した客も自動着席できる?
**A2.** 不可。スタッフが **📲 QR開放** をタップする必要があります (1 タップ)。

**Q3.** 予約客にも PIN を聞かれたりする?
**A3.** いいえ、聞かれません。予約データが認証の代わりになります。

**Q4.** 1 つのテーブルで複数客を受けるには?
**A4.** 相席は **F-QR1 (サブセッション QR)** 機能を使います (既存実装、別章)。

**Q5.** 5分の開放期限が短すぎる?
**A5.** 通常は客が席についてすぐ QR を読むので十分。長すぎると攻撃リスクが上がります。

### B. 運用に関する質問

**Q6.** テーブル開放したけど客が来なかったら?
**A6.** 5 分後に自動失効。次の客に改めて **📲 QR開放** をすれば OK。

**Q7.** 間違えて違うテーブルを開放した
**A7.** 5 分で自動失効するので放置で OK。もう一度正しいテーブルを開放。

**Q8.** 予約客が違うテーブルに座りたがる
**A8.** ガントで予約を ✏️ 変更 → 別テーブル割当。客がその新テーブルの QR を読めば自動マッチ。

**Q9.** 複数スタッフが同じテーブルを開放した
**A9.** 最後の開放が上書き (next_session_token は 1 つだけ)。特に問題なし。

**Q10.** 開放してから 5 分を超えてしまった
**A10.** ハンディでもう一度 **📲 QR開放** を実行してください。

### C. 技術的質問

**Q11.** next_session_token はどう生成される?
**A11.** `bin2hex(random_bytes(24))` で 48 文字のランダム文字列 (暗号論的に安全)。

**Q12.** 同じテーブルで 2 回連続開放したら古いトークンはどうなる?
**A12.** 上書き (古いトークンは無効化)。

**Q13.** トークンはどこに保存?
**A13.** `tables.next_session_token` カラムに保存。使用後クリア。

**Q14.** セッションの有効期限は?
**A14.** `tables.session_token_expires_at` = 着席時刻から 4 時間後。

**Q15.** 4 時間を超えた場合 (長居の宴会)?
**A15.** セッションは期限切れ。お客様がリロードすると 4 時間後に SESSION_COOLDOWN (直前 paid/closed がない限り) で弾かれる。スタッフが table_sessions.expires_at を延長する機能は現在なし (将来追加予定)。

### D. トラブル系

**Q16.** 開放ボタンを押してもトーストが出ない
**A16.** ネットワークエラーの可能性。ハンディ → 別タブで管理画面を開いて確認。

**Q17.** 客が QR を何度読んでも画面が変わらない
**A17.** 同じ URL 読み込みを繰り返しても既に着席済のセッションが継続。問題なし。

**Q18.** 「テーブルは既に使用中です」と出る
**A18.** 前の客の会計が未完了。レジで会計完了してから再度開放。

**Q19.** ハンディから開放後、管理画面のガントに反映されない
**A19.** 客が QR を読むまでは反映されない (予約なし walk-in の場合、開放だけではガント表示なし)。

**Q20.** device ロール (KDS / レジ専用アカウント) から開放できる?
**A20.** staff 以上の権限が必要。device は開放できません。

### 25.12.1 tables テーブル拡張 (migration-l9-table-auth.sql)

```sql
ALTER TABLE tables
  ADD COLUMN next_session_token VARCHAR(64) DEFAULT NULL
    AFTER session_token_expires_at,
  ADD COLUMN next_session_token_expires_at DATETIME DEFAULT NULL
    AFTER next_session_token,
  ADD COLUMN next_session_opened_by_user_id VARCHAR(36) DEFAULT NULL
    AFTER next_session_token_expires_at,
  ADD COLUMN next_session_opened_at DATETIME DEFAULT NULL
    AFTER next_session_opened_by_user_id;

  ADD INDEX idx_next_token_expires (next_session_token_expires_at);
```

### 25.12.2 API エンドポイント

#### POST /api/store/table-open.php

body: `{ store_id, table_id, expires_minutes=5 }`

処理:
1. require_role('staff')
2. テーブル存在 + is_active チェック
3. 既存 table_sessions で status NOT IN ('paid','closed') があれば 409 TABLE_IN_USE
4. bin2hex(random_bytes(24)) で token 生成
5. `tables.next_session_token` + `expires_at` + `opened_by_user_id` + `opened_at` を UPDATE
6. 200 OK で expires_at を返す

#### DELETE /api/store/table-open.php

query: `?store_id=&table_id=`

処理: tables の next_session_* を NULL にリセット (誤開放のリカバリー)

#### GET /api/customer/table-session.php (修正)

QR 読込時の分岐ロジック:

```
1. アクティブセッションあれば PIN チェック → 既存トークン返却
2. なければ:
   a. next_session_token が有効? → YES なら table_sessions 起票 + token 消費
   b. 予約マッチング? → YES なら table_sessions 起票 + 予約紐付け
   c. いずれも NG → 直前 5分 以内 paid? → YES なら SESSION_COOLDOWN
                                         → NO なら NO_ACTIVE_SESSION
```

コアコード (table-session.php 内):

```php
// (A) ホワイトリスト検証
$nsStmt = $pdo->prepare('SELECT next_session_token, next_session_token_expires_at FROM tables WHERE id = ?');
$nsStmt->execute([$tableId]);
$nsRow = $nsStmt->fetch();
if ($nsRow && $nsRow['next_session_token']
    && strtotime($nsRow['next_session_token_expires_at']) > time()) {
  // セッション作成 + トークン消費
  $pdo->prepare("INSERT INTO table_sessions (...) VALUES (...)")->execute(...);
  $pdo->prepare('UPDATE tables SET next_session_token = NULL, ... WHERE id = ?')->execute(...);
}

// (B) 予約マッチング
$rsStmt = $pdo->prepare(
  "SELECT id, party_size FROM reservations
   WHERE store_id = ? AND status IN ('confirmed','pending')
     AND JSON_SEARCH(assigned_table_ids, 'one', ?) IS NOT NULL
     AND reserved_at BETWEEN DATE_SUB(NOW(), INTERVAL 30 MINUTE)
                         AND DATE_ADD(NOW(), INTERVAL 30 MINUTE)
   ORDER BY ABS(TIMESTAMPDIFF(SECOND, reserved_at, NOW())) ASC LIMIT 1");
```

### 25.12.4 トークンライフサイクル

```
生成: random_bytes(24) → 48文字 hex
保存: tables.next_session_token カラム (VARCHAR 64)
期限: tables.next_session_token_expires_at = NOW + 5min
検証: 客の QR 読み込み時にサーバー側で現在時刻と比較
消費: 検証成功で即クリア (NULL) → ワンタイム
消失: 5分経過で検証時に拒否
```

### 25.12.5 予約マッチングの SQL 最適化

```sql
CREATE INDEX idx_status (store_id, status, reserved_at)
  ON reservations;
```

`JSON_SEARCH` は InnoDB の generated column でインデックスを貼れるが、現状は full scan。テーブル数が少ない (〜100) なら問題なし。1000+ テーブル規模では generated column + インデックス検討。
