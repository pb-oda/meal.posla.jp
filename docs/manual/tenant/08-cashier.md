# 8. 会計・POSレジ

POSレジ画面 (`/kds/cashier.html`) は、テーブルの会計処理を行うための画面です。現金・クレジットカード・QR コード決済に対応し、割引・個別会計・レジ開閉・取引履歴・レシート発行・**返金 (C-R1)** の機能を備えています。

- **店長・オーナー**: 8.9 (レジ開閉) + 8.10 (取引履歴) + 8.11 (返金) 必読

### POS レジ画面の役割

- テーブルごとの**会計処理** (現金・カード・QR)
- Stripe Terminal 連携によるクレジット決済
- 個別会計 (割り勘)
- 割引適用
- **返金処理 (C-R1、新機能)**
- レジ開け・締め
- 取引履歴の確認・レシート再印刷
- ドロア操作 (プリンタ連動、将来)

### 使える人

- owner / manager / staff / device
- device ロールは GPS チェック必須 (23 章参照)

### 必要なもの

- **タブレット** (推奨 10〜12 インチ)
- **店舗 Wi-Fi**
- **カードリーダー** (Stripe Reader S700、カード決済時)
- **レシートプリンタ** (26 章参照、任意)
- **釣銭用の現金**

### 所要時間

| 作業 | 目安 |
|---|---|
| 初回起動 + 音声 unlock | 10 秒 |
| 会計 1 件 (現金) | 30 秒 |
| 会計 1 件 (カード) | 60 秒 (カードタップ + 印刷含む) |
| 個別会計 (割り勘) | 60〜90 秒 |
| レジ開け / 締め | 30 秒 |

### 8.0.1 操作前チェックリスト

POSレジを開く前に、以下を確認してください。会計開始後に発覚すると業務が止まります。

| # | 確認項目 | 確認方法 | NG 時の対応 |
|---|---|---|---|
| 1 | あなたのアカウントが **owner / manager / staff / device** いずれか | ログイン後のロール表示で確認 | 該当しないロールはログイン不可 |
| 2 | 担当スタッフの **PIN が設定済み**（CP1） | dashboard >「アカウント設定」>「担当 PIN」で確認 | 未設定なら 8.10-b.6 参照（自分で設定可） |
| 3 | 担当スタッフが **本日出勤打刻済み**（PIN 必須操作の前提） | dashboard >「シフト・勤怠」タブで自分の clock_in 状況を確認 | 出勤打刻していない PIN は無効。打刻してから操作 |
| 4 | 店舗の **Stripe Connect オンボーディング完了**（カード決済使う場合） | owner-dashboard >「決済設定」タブで `charges_enabled=true` 確認 | カード決済不可。21 章 21.4 参照 |
| 5 | **Stripe Reader S700** がペアリング済み（カード決済使う場合） | レジ画面ヘッダー右上 `terminal-status` が緑ドット | 21.5 でペアリング |
| 6 | レシートプリンタが **接続・電源 ON**（自動印刷使う場合） | 26 章プリンタ動作確認 | 印刷せずに完了は可。後で再印刷可 |
| 7 | **釣銭準備金**（現金会計） | 紙幣・硬貨の在高をカウント | 不足は出勤前に補充 |
| 8 | **タブレットの GPS 許可**（device ロールのみ） | ブラウザ位置情報許可ダイアログで「許可」 | 範囲外で開くと cashier.html ロックされる |

::: tip まずはこれだけ覚えれば動かせる
最低限「ログイン → テーブル選択 → 支払方法選択 → 金額入力 → 担当 PIN 入力 → 会計確定 → レシート」の 7 ステップ。これで 95% の現場業務が回ります。
:::

:::

## 8.2 会計の流れ

### ステップ1：テーブルを選択する

1. 左カラムの会計待ちテーブル一覧から、会計するテーブルの「会計」ボタンをクリックします
2. 右カラムにそのテーブルの注文内容が表示されます

### ステップ2：注文内容を確認する

右カラムに表示される注文品目を確認します。

**品目の表示形式：**

| 要素 | 説明 |
|------|------|
| チェックボックス | 個別会計用（後述） |
| テーブル名 | 「[テーブル5]」形式 |
| 品名 | メニュー名 |
| 軽減税率ラベル | テイクアウト品目に「軽減」ラベル（8%対象） |
| オプション | 選択されたオプション（小文字） |
| 数量 | 「×2」形式 |
| 価格 | 品目の合計金額 |
| 支払済み | 一部支払い済みの品目は「支払済」表示（グレー） |

### ステップ3：割引を適用する（必要な場合）

割引が必要な場合は、決済前に割引を設定します。

**割引の種類：**

| タイプ | ラジオボタン | 入力例 | 説明 |
|--------|------------|--------|------|
| 金額割引 | 「金額」 | 500 | 固定金額（円）を割引 |
| パーセント割引 | 「%」 | 10 | 合計の○%を割引 |

**割引の操作：**

1. 割引タイプ（金額 or %）を選択します
2. 値を入力します
3. 「適用」ボタンをクリックします
4. 合計金額の下に「割引 -¥500」のように表示されます
5. 割引を取り消す場合は「取消」ボタンをクリックします

### ステップ4：支払方法を選択する

3つの支払方法から選択します。

| 方法 | ボタン | 動作 |
|------|--------|------|
| 現金 | 「現金」 | テンキーで受取金額を入力 |
| クレジットカード | 「クレカ」 | 金額が自動入力される |
| QRコード | 「QR」 | 金額が自動入力される |

### ステップ5：金額を入力する（現金の場合）

現金を選択した場合、テンキーとクイック金額ボタンで受取金額を入力します。

**テンキー配置：**
```
 7   8   9
 4   5   6
 1   2   3
00   0   C
```

- **C**：入力をクリア
- 最大7桁まで入力可能

**クイック金額ボタン：**

| ボタン | 動作 |
|--------|------|
| ¥1,000 | 受取金額に1,000円を設定 |
| ¥5,000 | 受取金額に5,000円を設定 |
| ¥10,000 | 受取金額に10,000円を設定 |
| ぴったり | 合計金額と同額を自動入力 |

**おつりの自動計算：**
受取金額を入力すると、おつりが自動計算されて表示されます。

```
受取金額：¥5,000
おつり　：¥1,360
```

### ステップ6：会計を確定する

1. **「会計する」** ボタンをクリックします
2. **担当スタッフ PIN モーダル**が表示されます（CP1、2026-04-19 から必須）
3. 自分の PIN（4〜8 桁）を入力 → **「確定」**
4. 会計処理が実行されます
5. レシートプレビューモーダルが表示されます
6. 注文ステータスが `paid` に更新されます
7. テーブルセッションが終了します
8. 左カラムからそのテーブルが消えます

**会計処理中：**
二重送信防止のため、「会計する」ボタンは処理中に無効化されます。`_isSubmitting=true` がフラグ。

### 8.2.1 入力フィールド早見表（POSレジ右カラム）

| フィールド名 | DOM ID | 例 | 必須 | 文字種 | 補足 |
|---|---|---|---|---|---|
| 割引タイプ | `name="ca-discount-type"` ラジオ | 金額 / % | 任意 | ラジオ | デフォルトなし |
| 割引値 | `ca-discount-input` | 500 (金額) / 10 (%) | 任意 | 半角数字 | 0 を許容 |
| 受取金額 | テンキー入力（`_receivedInput`） | 5000 | 現金時必須 | 半角数字 | 最大 7 桁 |
| 担当スタッフ PIN | `ca-pin-input` | 1357 | **必須**（PIN 必須操作） | 半角数字 4〜8 桁 | 伏字表示 |
| 返金理由 | 返金モーダル `<input>` | 料理間違い | 任意 | テキスト | 監査ログに残る |

### 8.2.2 実例：matsunoya 渋谷店 テーブル 5 で 2 品会計

```
状況: 14:00 着席、14:30 注文（唐揚げ定食 ¥980 + ハイボール ¥485 × 2 = ¥1,950）
14:55 提供完了、15:10 お客様が会計を依頼

操作:
1. レジ画面で「テーブル 5番」カードの【会計】ボタン押下
2. 右カラムに 3 品 表示、合計 ¥1,950
3. 支払方法 【現金】 選択
4. お客様から ¥2,000 預かり → クイック ¥1,000 + テンキー 「1000」 = 受取 ¥2,000
5. おつり自動表示 ¥50
6. 【会計する】 押下
7. PIN モーダル → 「1357」（田中花子の PIN）入力 → 【確定】
8. レシート印刷モーダル表示 → 【印刷】 → 退店
```

DB 反映:
- `orders.status = 'paid'`、`orders.staff_id = (PIN認証された田中花子の user_id)`
- `payments` 1 件（amount=1950, payment_method='cash', user_id=田中花子）
- `cash_log` 1 件（type='cash_sale', amount=1950）
- `audit_log` 2 件（`payment_complete` + `cashier_pin_used`）

## 8.3 税金の計算

POSLAは日本の消費税制度に対応し、品目ごとに税率を分けて計算します。

### 税率の区分

| 税率 | 対象 | 条件 |
|------|------|------|
| 10%（標準税率） | 店内飲食 | デフォルト |
| 8%（軽減税率） | テイクアウト | テイクアウト注文の場合 |
| 品目別税率 | 個別設定 | 品目に`taxRate`が設定されている場合 |

### 税額の表示

会計画面では税率区分ごとに内訳が表示されます。

```
10%対象  ¥1,100 （税 ¥100）
 8%対象    ¥540 （税  ¥40）
─────────────────────────
合計      ¥1,640
```

**計算方法（税込価格からの逆算）：**
```
税抜額 = 税込価格 ÷ (1 + 税率)  ※小数点以下切り捨て
税額 = 税込価格 − 税抜額
```

## 8.4 個別会計（割り勘）

1つのテーブルの注文を品目単位で分けて、複数回に分けて会計できます。

### 個別会計の操作

1. テーブルの「会計」ボタンをクリックして注文を表示します
2. 品目リストのチェックボックスで、今回支払う品目を選択します
   - 「全選択」：全品目を選択
   - 「全解除」：全品目の選択を解除
   - 個別チェックボックス：品目ごとに選択/解除
3. 一部のみ選択すると「個別会計」バッジが表示されます
4. 選択した品目の合計が再計算されます
5. 「個別会計する」ボタンで選択品目のみを会計します

**個別会計後の状態：**
- 支払い済みの品目は「支払済」表示（グレー）になり、チェックボックスがロックされます
- 残りの品目は次の会計を待ちます
- 全品目が支払い済みになるとテーブルセッションが終了します

## 8.5 合流会計

複数テーブルの注文をまとめて1回の会計で処理します。

### 8.5.1 操作手順

1. 左カラム上部の **「合流会計」** ボタンをクリック（トグル ON）
2. 複数のテーブルカードを順にクリックして選択（チェック済になる）
3. 選択したテーブルの全注文が右カラムにまとめて表示されます
4. 通常どおり支払方法を選び、会計確定（PIN 入力含む）
5. 全テーブルが同時に `paid` になりセッション終了

### 8.5.2 実例：torimaru 新宿店 4 名グループ × 2 卓合流

```
状況: テーブル 3番 と 4番 を 1 グループとして 4 名利用
       3番: 焼鳥 5本盛 ¥1,200 + 生ビール ¥500 × 2 = ¥2,200
       4番: 唐揚げ ¥800 + ハイボール ¥485 × 2 = ¥1,770
       合計予定 ¥3,970（カード払い）

操作:
1. 【合流会計】 ON
2. テーブル 3番 カード タップ → ✓
3. テーブル 4番 カード タップ → ✓
4. 右カラムに 6 品 + 合計 ¥3,970 表示
5. 【クレカ】 選択 → 受取金額自動入力 ¥3,970
6. 【会計する】 → PIN 入力（管理者の山田太郎、PIN「2468」）
7. Stripe Reader S700 にカードタップ → 承認
8. レシート 1 枚に 2 卓分まとめて印刷
```

サーバー側送信:
```json
{
  "merged_table_ids": ["table-3-uuid", "table-4-uuid"],
  "payment_method": "card",
  "staff_pin": "2468"
}
```

::: warning 合流会計時の制約
- **テイクアウト注文との合流不可**（在席テーブルのみ）
- 合流相手テーブルの **all 全注文** が会計対象。一部品目のみは個別会計（8.4）と組み合わせる必要あり
- 合流後は片方のテーブルだけ後日「open に戻す」操作不可。誤合流は返金で対応
:::

## 8.5-b 支払方法比較表（4 種類）

POSLA は 4 つの支払方法を提供。それぞれ運用と返金の扱いが異なります。

| 項目 | 現金 (cash) | クレカ (card) | QR (qr) | Stripe Terminal (terminal) |
|---|---|---|---|---|
| API `payment_method` | `cash` | `card` | `qr` | `terminal` |
| 物理機器 | レジ金庫 | 任意（手入力） | お客様スマホ | Stripe Reader S700 |
| お釣り計算 | ✅ 自動 | ❌ なし | ❌ なし | ❌ なし |
| 受取金額入力 | 必須 | 自動（合計と同額） | 自動 | 自動 |
| 決済確認 | 即時 | 端末依存 | 別アプリ確認 | S700 表示 |
| 返金（システム） | ❌ 手動現金 | ❌（Stripe ダッシュボード） | ❌（PayPay 管理画面） | ✅ 自動（refund-payment.php） |
| 領収書印鑑 | 必要に応じて | 不要 | 不要 | 不要 |
| Stripe 連携 | なし | なし（記録のみ） | なし | あり（PaymentIntent） |
| `payments.gateway_name` | NULL | NULL | NULL | `stripe_terminal` または `stripe_connect_terminal` |
| `cash_log` 自動 INSERT | ✅ (`cash_sale`) | ❌ | ❌ | ❌ |

::: tip 「クレカ」と「Stripe Terminal」の違い
**クレカ** = 別の決済端末（楽天ペイ等）で決済し、POSLA に金額だけ手入力で記録するモード。POSLA 側は単に「カードで支払われた」と記録するだけ。

**Stripe Terminal** = POSLA から直接 Stripe Reader S700 に決済命令を送り、その場で承認・記録する完全自動モード。返金も POSLA から実行可能。

将来的に Stripe Terminal が標準になり、「クレカ」モードはレガシー扱いになります。
:::

## 8.6 レジ開閉操作

POSレジでは、営業開始時の「レジ開け」と営業終了時の「レジ閉め」を行います。

::: warning レジ金庫操作は全て担当スタッフ PIN 必須（CB1d、2026-04-19 追加）
**レジ開け / レジ閉め / 入金 / 出金** の 4 操作は、実行時に必ず **担当スタッフの PIN 入力**が求められます。会計・返金と同じテンキーモーダルで、出勤中スタッフの PIN を照合します。

- マネージャーも例外なし。**バイパスはありません**
- 事前に **個人スタッフアカウントで管理画面にログイン → 出勤打刻**しておくこと（退勤打刻したスタッフの PIN は無効）
- 出金は店舗の現金がレジから出る最も危険な操作のため、誰が引き出したかを必ず記録します
- 締め時の差額調査が「いつ・誰が・何のために動かした」をすべて追える状態になります
- `cash_log.user_id` には PIN 認証されたスタッフ ID が入ります（ログイン中の device 端末ユーザーではありません）
:::

### レジ閉め

クリック単位の手順（CB1d 後）：

1. レジ内の現金を全て数えます（紙幣・硬貨）
2. 「レジ閉め」セクションの **「実際の現金額（円）」** に入力（例: `85230`）
3. 入力中、リアルタイムに「予想在高」との差額が表示されます
   - 緑「ぴったり」/ 赤「過剰」/ 赤「不足」
4. **「レジを閉める」** ボタンをクリック
5. ブラウザ確認ダイアログ「レジを閉めますか？ 実際: ¥85,230 予想: ¥85,200 差額: +¥30」 → **OK**
6. **担当スタッフ PIN モーダル** → PIN 入力 → **「確定」**
7. トースト「レジを閉めました」、パネル下部の履歴に「閉め ¥85,230」追加
8. レジ操作（入金・出金）が無効化されます（営業終了）

| 項目 | 説明 |
|------|------|
| 実際 | 入力した現金残高 |
| 予想 | システム計算の期待残高（開始金額 + 現金売上 + 入金 − 出金） |
| 差額 | 実際 − 予想（過不足） |

| 記録データ | 値 | 保存先 |
|---|---|---|
| タイプ | `close` | `cash_log.type` |
| 金額 | 入力した金額（円） | `cash_log.amount` |
| メモ | 「レジ閉め（差額: ±XXX）」 | `cash_log.note` |
| 担当ユーザー | PIN 認証されたスタッフ | `cash_log.user_id` |
| 監査ログ | `register_close` イベント | `audit_log` |

### 8.6.1 差額調査の手順（締めで過不足が出た時）

差額が出た場合、以下の順序で監査します。

1. **入出金履歴を順に読む**
   - レジ開閉パネル下部の「本日の入出金履歴」を、開け → 入金 → 出金 → 売上 → 締めの順で確認
   - 各行のメモから直感的に判断（例：「仕入支払」と書いてあるのに領収書がない → 要追跡）

2. **`cash_log` テーブルを直接 SELECT**（manager 以上、phpMyAdmin 等）
   ```sql
   SELECT cl.created_at, cl.type, cl.amount, cl.note, u.display_name
   FROM cash_log cl
   LEFT JOIN users u ON u.id = cl.user_id
   WHERE cl.store_id = ? AND DATE(cl.created_at) = CURDATE()
   ORDER BY cl.created_at;
   ```
   - 担当者（`u.display_name`）と時刻、金額が並ぶので「誰が」「いつ」「何のために」追える

3. **監査ログ（`audit_log`）でエビデンス**
   ```sql
   SELECT created_at, action, user_id, details
   FROM audit_log
   WHERE store_id = ? AND DATE(created_at) = CURDATE()
     AND action IN ('register_open','register_close','cash_in','cash_out','payment_complete','payment_refund')
   ORDER BY created_at;
   ```
   - `details` (JSON) には `pin_verified=1`, `cashier_user_id`, `cashier_user_name` が入っており、PIN を介して動かした人を一意特定可

4. **差額閾値を超える場合の対応**
   - +/- ¥1,000 以内 → 翌日に持ち越し（朝礼で共有）
   - +/- ¥5,000 超 → owner / manager を呼んで監査ログ精査
   - 当該日の `cash_out` を全て点検 → 領収書突合
   - 該当スタッフから事情聴取（`cashier_user_id` で特定）

::: warning 差額の責任範囲
差額（過不足）の補填責任は、店舗側の就業規則・労働基準に従ってください。POSLA は記録の正確性のみ保証します。安易に「スタッフ給与から天引き」運用は労働基準法違反の可能性あり、必ず社労士に相談を。
:::

### 8.6.2 入金・出金の追加例

入金・出金パネル（`#ca-reg-cashin-btn` / `#ca-reg-cashout-btn`）の運用例。

| シナリオ | type | 金額 | メモ例 | 監査での見え方 |
|---|---|---|---|---|
| 朝、銀行から両替持ってきた | `cash_in` | 10000 | 両替 | `cash_in` イベント、PIN 認証 |
| 13:00、ランチ売上が増えてきて千円札補充 | `cash_in` | 5000 | つり銭補充 | 同上 |
| 仕入業者に小口現金で支払い | `cash_out` | 3000 | 仕入支払（魚屋） | `cash_out` イベント、領収書を別途保管 |
| バイトへの交通費精算 | `cash_out` | 1500 | 交通費精算 鈴木 | 同上、PIN は manager の PIN 推奨 |
| 21:00、売上を一部金庫へ | `cash_out` | 50000 | 金庫保管 | manager 必須レベル |

::: danger 出金の運用ルール（推奨）
- ¥10,000 を超える出金は **manager 以上の PIN** で実行
- メモ欄を必ず記入（後の監査で追えなくなる）

:::

::: info レジ操作は監査ログにも記録されます（2026-04-19 CB1d で拡張）
**レジ開け / レジ締め / 入金 / 出金** の 4 操作はキャッシュログに加え、`audit_log` テーブルにもアクション `register_open` / `register_close` / `cash_in` / `cash_out` として記録されます。CB1d 以降、各レコードには `pin_verified=1`、`cashier_user_id`、`cashier_user_name` が含まれ、device 端末ログイン中でも「実際に金庫を動かした人間」が特定できます。「店舗設定」→「監査ログ」サブタブから「誰が・いつ・いくら・何のために」を後から監査できます。
:::

## 8.9 Stripe決済端末連携

全契約者に標準提供され、Stripe Connect（POSLA 経由）または Stripe 自前キーが設定されている場合、カードリーダー（Stripe Terminal）での決済に対応します。

### 接続状態

- ヘッダーにカードリーダーの接続状態が表示されます（緑色/赤色のドット）
- 「再検出」ボタンで再接続を試みます
- Stripe Terminalが利用できない場合はステータス表示が自動的に非表示になります

### カード決済の流れ

1. 支払方法で「クレカ」を選択します
2. 「会計する」ボタンをクリックします
3. UIがロックされ、オーバーレイが表示されます
   - 「カードリーダーで決済準備中...」
   - 「カードをリーダーにかざしてください」
   - キャンセルボタン
4. お客さんがカードをリーダーにかざします
5. 決済が完了するとレシートが表示されます

**内部処理の流れ（実装詳細）：**

1. `fetch /api/store/terminal-intent.php`（POST `{amount}`）
   - サーバー側: Stripe PaymentIntent を作成（Pattern A = テナント自前キー / Pattern B = Connect 経由 + application fee）
   - レスポンス: `client_secret` + `payment_intent_id`
2. `_terminal.collectPaymentMethod(client_secret)`
   - S700 リーダーでカード読取（非接触 / 挿入 / スワイプ）
   - お客様がキャンセル可能（S700 画面の赤ボタン）
3. `_terminal.processPayment(paymentIntent)`
   - Stripe 側で決済実行（3D セキュア等を含む）
   - 成功 or 失敗
4. `fetch /api/store/process-payment.php`（POST）
   - `payment_method='terminal'`, `terminal_payment_id=pi_xxx`
   - POSLA 側の `orders.status='paid'`, `payments` レコード作成
   - `audit_log` に記録
5. レシート表示・印刷・ドロア開け

実装: `public/kds/js/cashier-app.js` L2225-2331 `_processTerminalPayment()`。S700 SDK 初期化は L2115-2138 `_initStripeTerminal()`。

### Stripe Terminal 接続トークン

S700 SDK は独自の接続トークンを必要とします。POSLA は `api/connect/terminal-token.php` を POST（5 分有効）してトークンを供給します。

### 8.9.1 Terminal 決済 詳細フロー（クリック単位）

```
[1] スタッフ: 支払方法 【クレカ】 ボタン押下
    └─ _paymentMethod = 'card', _receivedInput = 合計額
[2] スタッフ: 【会計する】 ボタン押下
    └─ PIN モーダル → PIN 入力 → 確定
[3] サーバー: PIN 検証 OK → terminal-intent.php 呼出
    └─ Stripe PaymentIntent 作成 (client_secret 返す)
[4] フロント: _terminal.collectPaymentMethod(client_secret)
    └─ S700 が "カードをかざしてください" 表示
[5] お客様: カードを S700 にタップ / 挿入
    └─ S700 が読取・送信
[6] フロント: _terminal.processPayment(paymentIntent)
    └─ Stripe 認証実行（3D セキュア等）
[7] フロント: process-payment.php (POST { payment_method:'terminal', terminal_payment_id })
    └─ orders.status='paid', payments INSERT, audit_log
[8] フロント: レシートプレビュー → 印刷ダイアログ
```

### 8.9.2 Terminal 決済 トラブル対処（拡充版）

| 症状 | 切り分け方 | 対処 |
|---|---|---|
| ヘッダーが赤ドット | 接続切れ | `terminal-status` エリアの【再検出】ボタンを押す。改善しなければ S700 の電源を入れ直す |
| 「カードをかざしてください」のまま反応なし | NFC 干渉 / ICチップ汚れ | 1) 別カードで試す 2) 挿入式に切替 3) スタッフカードで動作確認 |
| `card_declined` エラー | カード側の限度額・残高不足 | 別カード or 現金で再会計 |
| `processing_error` | Stripe 側の一時不調 | 1〜2 分待って再試行。続くなら Stripe Status ページ確認 |
| ペアリング切れ → 操作中に切断 | Wi-Fi 不安定 | Wi-Fi ルーター再起動。S700 の Wi-Fi 設定で 2.4GHz 接続を試す |
| 接続トークン期限切れ（5 分） | アイドル放置 | UI に再ログイン or リロード |

### 8.9.3 Pattern A vs Pattern B の Terminal での違い

| 項目 | Pattern A（自前 Stripe キー） | Pattern B（POSLA 経由 Connect） |
|---|---|---|
| `payments.gateway_name` | `stripe_terminal` | `stripe_connect_terminal` |
| 認証キー | テナント自前の `sk_live_xxx` | POSLA プラットフォームキー（`posla_settings`） |
| Connect Account 必須 | ❌ | ✅ (`stripe_connect_account_id`) |
| application_fee | なし | 1.0% (`application_fee_amount`) |
| 返金実装 | `create_stripe_refund()` | `create_stripe_connect_refund()` |
| 入金先 | テナント自身の Stripe 残高 | テナントの Connect 残高 |

実装場所: `api/store/refund-payment.php` の L128 `if ($payment['gateway_name'] === 'stripe_connect_terminal')` 分岐。

## 8.10-b 担当スタッフ PIN 認証 (CP1、2026-04-19 追加)

### 8.10-b.1 機能概要

POSレジで会計確定時に **担当スタッフの 4〜8 桁 PIN 入力** を求めることで、誰がレジを操作したかを正確に記録します。device アカウントでレジにログインしていても、実際に会計を実行した「人」を `payments.user_id` に記録できる仕組み。

**目的**: 不正防止 + 担当者追跡の強化（複数スタッフがレジを入れ替わりで使う運用想定）

### 8.10-b.2 PIN 入力モーダル

会計確定ボタン押下後、確認ダイアログ「OK」を押すと PIN 入力モーダルが表示されます。

```
┌─────────────────────────────────┐
│   担当スタッフ PIN 入力          │
│                                 │
│  レジを操作するスタッフの PIN を │
│  入力してください                │
│                                 │
│      ┌───────────────┐          │
│      │  • • • •       │          │
│      └───────────────┘          │
│                                 │
│   1  2  3                       │
│   4  5  6                       │
│   7  8  9                       │
│   C  0  ⌫                       │
│                                 │
│  [キャンセル] [    確定    ]    │
└─────────────────────────────────┘
```

- **テンキー入力**（タッチ操作向け）+ キーボード入力どちらも可
- **Enter** キーで確定、**Esc** キーでキャンセル
- 入力中は伏字（••••）で表示、覗き見対策

### 8.10-b.3 認証の仕組み

1. 入力された PIN（4〜8 桁数字）をサーバーに送信
2. サーバー側で **「現在出勤中の店舗スタッフ」** の中から PIN 一致を検索
   - 出勤中 = `attendance_records.clock_out IS NULL` AND 当日 `clock_in` あり
   - 退勤打刻したスタッフの PIN は **無効化**（不正悪用防止）
3. 一致したスタッフの user_id を `payments.user_id` に記録
4. 監査ログに `cashier_pin_used` イベント記録（誰が・いつ・いくらの会計を実行したか）

### 8.10-b.4 PIN 不一致の場合

| ケース | エラー |
|---|---|
| PIN 桁数違反（3 桁以下 or 9 桁以上） | `INVALID_PIN` (400) |
| 出勤中スタッフの PIN と一致しない | `PIN_INVALID` (401) |
| 該当スタッフが退勤済み | `PIN_INVALID` (401) |
| 該当スタッフが PIN 未設定 | `PIN_INVALID` (401) |

エラー時は会計実行されず、再入力を促す。失敗回数は `audit_log` の `cashier_pin_failed` イベントに記録（ブルートフォース攻撃の調査用）。

### 8.10-b.5 個人アカウントでログインしている場合

PIN 入力欄に何も入れずにキャンセルすると会計は実行されません。**device アカウント運用が原則** ですが、緊急時に個人スタッフアカウントで POSレジに直接ログインしている場合も、必ず PIN 入力が必要（自分の PIN）。

### 8.10-b.6 PIN の設定・変更

→ [2 章 2.6 担当スタッフ PIN](./02-login.md#26-担当スタッフ-pin-cp1) 参照

### 8.10-b.7 PIN が要求される操作（2026-04-19 CB1d で完全網羅）

| 操作 | PIN 必須 | エンドポイント |
|---|---|---|
| 会計確定（現金 / カード / QR / Terminal） | ✅ | `POST /api/store/process-payment.php` |
| 返金処理（部分・全額） | ✅ | `POST /api/store/refund-payment.php` |
| **レジ開け** | **✅（CB1d 新規）** | `POST /api/kds/cash-log.php` (`type=open`) |
| **レジ閉め** | **✅（CB1d 新規）** | `POST /api/kds/cash-log.php` (`type=close`) |
| **入金（Cash In）** | **✅（CB1d 新規）** | `POST /api/kds/cash-log.php` (`type=cash_in`) |
| **出金（Cash Out）** | **✅（CB1d 新規）** | `POST /api/kds/cash-log.php` (`type=cash_out`) |
| 現金売上自動記録（`cash_sale`） | ❌（process-payment 側で会計時に PIN 認証済み） | `POST /api/kds/cash-log.php` (`type=cash_sale`) |

::: tip マネージャーも例外なし
レジ金庫操作（CB1d）は **マネージャー / オーナーであっても PIN バイパス不可**。マネージャーも事前に出勤打刻 → PIN 設定が必要です。「現金が動くすべての操作は、PIN 認証された担当者と紐づく」というのが CB1d の設計原則です。
:::

## 8.11 返金処理 (C-R1、新機能)

2026-04-18 に追加された返金機能。Stripe Refund API と連携し、カード決済済みの注文を部分返金 or 全額返金できます。

### 8.11.1 返金対象

- **カード決済** (Stripe) のみ自動返金可能
- **現金決済** は手渡し (POSLA 側は記録のみ)
- **QR 決済** は PayPay/d払いの管理画面から別途返金 (POSLA 側は記録のみ)

### 8.11.2 返金手順 (manager 以上)

1. POSレジ画面の **【📋 取引履歴】** ボタン
2. 返金対象の取引をタップ
3. **【返金】** ボタン (赤色)
4. 返金モーダル:
   - 返金額 (全額 or 部分)
   - 返金理由 (例: 「料理間違い」「アレルギー申告漏れ」)
5. **【返金を確定】** → **担当スタッフ PIN 入力モーダル**（2026-04-19 追加）
6. PIN 確定 → Stripe Refund API 呼出
7. 成功でトースト「返金完了 ¥XXX」

::: warning 返金には担当スタッフ PIN が必須（2026-04-19 追加）
会計と同様、返金時にも **出勤中スタッフの PIN 入力**が必須になりました。退勤打刻したスタッフの PIN は無効化されています。PIN 不一致の場合はエラー（`PIN_INVALID`）で返金は実行されません。

実装: `POST /api/store/refund-payment.php` の `staff_pin` パラメータ。詳細は [8.10-b](#810-b-担当スタッフ-pin-認証-cp12026-04-19-追加) を参照。
:::

### 8.11.3 返金後の動作

- `payments.refund_amount` 更新
- `payments.refund_user_id` に PIN で特定された担当スタッフ user_id を記録
- `audit_log` に返金操作を `payment_refund` アクションで記録（誰が・いつ・いくら・理由）
- お客様のカードに数営業日後返金
- レシート再印刷時に「返金済 ¥XXX」表示

### 8.11.4 二重返金防止

- 同じ payment_intent に対して 2 回返金ボタンを押しても冪等 (Stripe 側で排他)
- UI 上は返金ボタンが `_refunding` フラグで disabled になる

### 8.11.5 権限

- **staff**: 返金ボタンを見ることはできない
- **manager**: 部分返金・全額返金 OK
- **owner**: 全操作 + 高額返金 OK

## 8.14 よくある質問 40 件

### A. 会計基本（5 件）

**Q1.** 会計待ちテーブルが出ない
**A1.** KDS 画面で「全品提供済」になっていない可能性。KDS で提供済にする。または、テーブルセッションが終了してしまっている（開店時に再オープン）。

**Q2.** 現金決済でお釣り計算は？
**A2.** 預かり金額を入力するとお釣りが自動表示。テンキーまたはクイック金額ボタン（¥1,000 / ¥5,000 / ¥10,000 / ぴったり）使用。

**Q3.** 会計中にお客様が追加注文した
**A3.** いったん会計画面で別テーブルを選び直し → 元のテーブルを再選択して最新情報を取得。または `_pollTimer` 5 秒間隔で自動更新される。

**Q4.** 会計金額を手動で変更したい
**A4.** 割引機能（8.7）を使う。金額（円）または %（パーセント）で割引適用。直接金額変更は不可（不正防止）。

**Q5.** 会計後にお客様が戻ってきた（クレーム等）
**A5.** 返金機能（8.11）を使う。Stripe Terminal 決済のみ自動。現金は手渡し対応。

### B. PIN 認証（7 件）

**Q6.** 自分の PIN を忘れた
**A6.** dashboard >「アカウント設定」>「担当 PIN」>「PIN を変更」で新しい PIN を設定。古い PIN は不要（自分のアカウントなら自分で再設定可）。

**Q7.** PIN モーダルが表示されない（会計確定でそのまま動く）
**A7.** ブラウザキャッシュが古い可能性。`Cmd+Shift+R`（または `Ctrl+Shift+R`）で強制リロード。`cashier-app.js?v=20260419-cb1d` のクエリ確認。

**Q8.** PIN を入力したのに「PIN_INVALID」エラー
**A8.** 1) 出勤打刻していない 2) 退勤打刻済 3) PIN が一致しない、のどれか。dashboard でシフト・勤怠を確認、出勤打刻してから再試行。

**Q9.** 同じ店舗に同じ PIN を持つスタッフが 2 人以上いる場合は？
**A9.** 早く一致したスタッフが採用される（順序不定）。**店舗内で PIN は一意になるよう運用**（オーナー/マネージャーが重複チェック）。

**Q10.** マネージャー / オーナーは PIN なしで操作できないの？
**A10.** **CB1d 以降、できない**。マネージャー / オーナーも自分の PIN を設定し、出勤打刻してから操作。例外なし。

**Q11.** PIN を 4 桁にしたい / 8 桁にしたい
**A11.** 4〜8 桁の範囲で自由に設定可。短いほど推測されやすい、長いほど入力が面倒。**6 桁推奨**。

**Q12.** 弱い PIN を弾く仕組みは？
**A12.** `1234` `0000` `1111` `123456` 等の連番・全桁同じは登録時に拒否（`api/store/set-cashier-pin.php`）。

### C. カード決済（5 件）

**Q13.** Stripe Terminal が反応しない
**A13.** 電源 + Wi-Fi 確認。Stripe ダッシュボード > Terminal Readers でオンライン状態確認。【再検出】ボタンで再ペアリング。

**Q14.** カード決済が途中で切れた
**A14.** Stripe ダッシュボード > Payments で最終状態確認。POSLA 側は webhook で自動同期されるが、念のため取引履歴で確認。失敗していれば再実行。

**Q15.** 暗証番号入力失敗を繰り返した
**A15.** Stripe Terminal 側で 3 回失敗するとカード一時ロック。別カードまたは現金で再会計。お客様にカード会社問い合わせを案内。

**Q16.** タッチ決済（NFC）が動かない
**A16.** Terminal の NFC 機能 ON 確認。S700 設定 > 決済方法 > 非接触有効化。カード側も Visa Touch / Master Contactless 対応か確認。

**Q17.** 海外カードで決済したい
**A17.** Stripe で受付可能なカード（Visa / Master / AMEX 等）なら OK。JCB は別途 Stripe 側で有効化必要。中国銀聯（UnionPay）は要 Stripe 申請。

### D. レジ開け・締め（CB1d 含む）（6 件）

**Q18.** レジ開けしないと会計できない？
**A18.** 開け不要でも会計可能。ただしレジ締めでつじつまが合わなくなる（開始在高 0 円扱い）。**毎営業日の最初に必ず開け推奨**。

**Q19.** レジ締めで差額が出た
**A19.** 8.6.1「差額調査の手順」参照。`cash_log` + `audit_log` を SELECT して「いつ・誰が・何のために」を追跡。¥1,000 以内は翌日持ち越し、¥5,000 超は manager 監査。

**Q20.** 1 日に複数回レジ締めしたい
**A20.** 可能。開け → 締め → 再度開け のサイクル。シフト交代時に締めて引き継ぎ運用も推奨。

**Q21.** 出金時にメモを書き忘れた
**A21.** `cash_log.note` は INSERT 後の編集不可。`audit_log` に経緯を補足するため、後続の `cash_log` に「先ほどの ¥3,000 出金は仕入支払」と記録 or owner-dashboard の「監査ログ」で備忘テキスト追加。

**Q22.** レジ開閉ボタンが押せない（disabled）
**A22.** すでに営業終了（最後のイベントが `close`）。再度「レジ開け」セクションが表示されているはず。表示されていなければリロード。

**Q23.** 入金 / 出金の最大金額は？
**A23.** システム上は `INT` の最大値（¥21 億超）まで OK。実運用は ¥100,000 程度を上限に運用ルール化推奨。

### E. 個別会計・合流会計・割引（5 件）

**Q24.** 4 人で均等割りしたい
**A24.** 現バージョンは「品目選択 → 個別会計」のみ。均等割りは手計算 → 個別会計で順に支払い実行。将来「均等割りボタン」追加予定。

**Q25.** 品目指定で分けたい
**A25.** 8.4 参照。チェックボックスで支払う品目を選択 → 「個別会計する」。

**Q26.** 端数が合わない
**A26.** 自動で 1 円単位まで計算。割引で 1 円単位調整可。

**Q27.** 何種類の割引が使える？
**A27.** 金額割引（固定円） / % 割引 の 2 種類。両方を組み合わせる場合は事前に手計算で 1 つの値に。

**Q28.** 合流会計したテーブルを切り離したい
**A28.** 会計確定前なら【合流会計】トグル OFF で個別状態に戻る。確定後は不可、誤合流は返金で対応。

### F. 返金（5 件）

**Q29.** 返金したお金は誰に戻る？
**A29.** Stripe Terminal 決済はお客様カードに 5〜10 営業日後返金。現金は店舗から手渡し（POSLA 側は記録のみ）。

**Q30.** 返金履歴を見たい
**A30.** 取引履歴で該当取引を開く →「返金済 ¥XXX」表示。詳細は監査ログ（`payment_refund` イベント）。

**Q31.** 間違えて返金した
**A31.** 全額返金後の取消は不可。Stripe ダッシュボードで返金キャンセルできる場合あり（処理待ちの間のみ）。完了済なら新たに同額を再課金（別取引）して相殺。

**Q32.** 部分返金は可能？
**A32.** API 上は可能（refund-payment.php に amount パラメータ）。UI は全額返金がデフォルト、入力欄で金額編集可。

**Q33.** 返金時に PIN を求められない（古いキャッシュ）
**A33.** ブラウザ強制リロード（Cmd+Shift+R）。`cashier-app.js?v=20260419-cb1d` の `?v=` バージョンが新しいことを確認。

### G. 決済方法・QR（3 件）

**Q34.** PayPay で決済したい
**A34.** POSLA は PayPay 専用 SDK 統合なし。お客様スマホで PayPay アプリ操作 → POSLA 側は支払方法【QR】で記録のみ。

**Q35.** d払い / au PAY / 楽天ペイは？
**A35.** 上記同様、外部アプリで決済 → POSLA は記録のみ。将来的に Stripe 経由の各種 QR 統合検討中。

**Q36.** 複数の決済方法を併用したい（半分カード + 半分現金）
**A36.** 個別会計（8.4）で品目を 2 グループに分け、片方カード・片方現金で 2 回決済。

### H. エラー・トラブル（4 件）

**Q37.** 画面がフリーズした
**A37.** `Cmd+Shift+R`（Mac）/ `Ctrl+Shift+R`（Win）で強制リロード。改善しなければブラウザ再起動 → タブレット再起動。

**Q38.** 取引履歴が昨日以前に遡れない
**A38.** 当日分のみ表示。過去は dashboard >「注文履歴」タブまたは owner-dashboard >「売上レポート」で。

**Q39.** 「PIN 機能のマイグレーションが未適用です」エラー
**A39.** サーバー側に `migration-cp1-cashier-pin.sql` + `migration-l3-shift-management.sql` の適用が必要。POSLA 運営に連絡。

**Q40.** Stripe Terminal が決済中に画面切替で止まった
**A40.** 別タブ・別アプリへ移動すると S700 SDK が一時停止する場合あり。決済完了まで POSレジ画面を表示し続ける。スリープ防止設定推奨（タブレット設定 > ディスプレイ > スリープなし）。

### 8.15.1 画面・表示

### 8.15.2 会計処理

| 症状 | 原因 | 対処 |
|---|---|---|
| 「お釣りがマイナス」 | 預かり額不足 | 正しい預かり額入力 |
| 会計ボタン押しても動かない | 通信エラー / `_isSubmitting=true` 残留 | 1) ネットワーク確認 2) リロード 3) DevTools Network で `process-payment.php` ステータス確認 |
| 支払方法ボタンが無効 | 注文 0 円 / 全額割引 | 注文があるか・割引で 0 円になっていないか確認 |

| 「PIN_REQUIRED」エラー | PIN モーダル未通過 | リロード（古いキャッシュで PIN モーダルが出ない場合あり） |
| 「PIN_INVALID」エラー | 出勤打刻なし / PIN 不一致 | dashboard >「シフト・勤怠」で出勤打刻 → 再試行 |
| 「MIGRATION」エラー | サーバー側 SQL 未適用 | POSLA 運営に連絡（`migration-cp1-cashier-pin.sql`） |
| `[ALREADY_PAID]` / `[CONFLICT]` (409、S3 #3) | 別端末・別スタッフが先に同じ伝票を会計済み | 画面リロードで最新状態確認。**重複課金は発生していない**（サーバーが排他制御で防止）。担当スタッフ間で口頭確認 |
| `[AMOUNT_MISMATCH]` (409、S3 #4) | 端末表示額とサーバー再計算額が不一致 | 画面リロードで最新メニュー価格を取得 → 再会計。原因として直前のメニュー価格更新あり |
| `[STRIPE_MISMATCH]` (409、S3 #4) | Stripe Terminal の `retrieve` 結果と DB 上の `payment_intent_id` が一致しない | POSLA 運営に連絡。Stripe Dashboard 側の event_id を併せて伝える |

### 8.15.3 カード決済（Stripe Terminal）

| 症状 | 原因 | 対処 |
|---|---|---|
| ヘッダー terminal-status 赤ドット | S700 接続切れ | 1) S700 電源 → Wi-Fi 確認 2) 【再検出】 3) Stripe Dashboard でオンライン確認 |

### 8.15.4 レジ開閉・出納（CB1d）

| 症状 | 原因 | 対処 |
|---|---|---|
| 「レジ開け」を押すと PIN モーダルが出ない | キャッシュ古い | Cmd+Shift+R で強制リロード |
| `cash_log` に user_id が NULL | PIN なしの旧運用 | CB1d 適用後は必ず PIN 経由のため新規発生せず。旧データはそのまま |

| 差額がマイナス大幅 | 出金記録漏れ | `cash_log` SQL 確認、領収書突合 |

| 症状 | 原因 | 対処 |
|---|---|---|
| 自動印刷されない | 設定 OFF | 店舗設定で `printer_auto_receipt` ON |

### 8.15.6 返金

| 症状 | 原因 | 対処 |
|---|---|---|
| 返金ボタンがない | staff 権限 | manager に依頼 |
| 「返金済」と出る | 既に返金処理済 | 取引履歴で確認 |

## 8.16 データベース構造 (POSLA 運営向け)

::: tip 通常テナントは読まなくて OK

### 8.16.1 関連テーブル

- **orders** — 注文本体 (06 章参照)
- **payments** — 決済情報 (payment_method / amount / stripe_payment_intent_id)
- **refunds** — 返金履歴 (C-R1 新規、migration-cr1-refund.sql)
- **cash_register_events** — レジ開け/締めイベント
- **payments_journal** — 日次レジジャーナル

### 8.16.2 API エンドポイント（2026-04-18 時点の実ファイル）

#### 会計系
- `GET /api/kds/orders.php?view=accounting&store_id=xxx` — 会計待ち一覧
- `POST /api/store/process-payment.php` — 会計実行（現金/カード/QR/Terminal 統合エンドポイント、manager 以上）
- `POST /api/customer/checkout-session.php` — セルフレジ会計セッション開始
- `POST /api/customer/checkout-confirm.php` — セルフレジ会計確定
- `POST /api/store/refund-payment.php` — 返金 CR1（manager 以上）
- `GET /api/store/payments-journal.php` — 取引ジャーナル

#### レジ開閉・出納
- `GET /api/kds/cash-log.php?store_id=xxx` — レジ出納帳取得
- `POST /api/kds/cash-log.php` — レジ開け / 閉め / 入金 / 出金（`type` パラメータ: `open` / `close` / `cash_in` / `cash_out` / `cash_sale`）

#### Stripe 連携
- `POST /api/connect/terminal-token.php` — Stripe Terminal 接続トークン取得（5 分有効）
- `POST /api/store/terminal-intent.php` — Terminal PaymentIntent 作成（Pattern A = 自前キー / Pattern B = Connect 経由）

`terminal-capture.php` / `cash-register-open.php` / `cash-register-close.php` は存在しません（旧設計の名残）。代わりに上記の統合エンドポイントを使います。
