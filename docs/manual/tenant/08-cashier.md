# 8. 会計・POSレジ

POSレジでは、店内注文やテイクアウト注文の会計、割引、個別会計、領収書、返金、レジ開閉を行います。現金が動く操作では、担当スタッフのPIN入力が必要です。

::: tip 推奨端末
POSレジはAndroidタブレット + Google Chrome最新版を **標準環境** としています。PC + Chrome最新版でもご利用いただけます。iPad/Safariは決済端末（Stripe Reader）連携やレシート印刷の挙動が不安定なためサポート対象外です。
:::

## 会計の基本手順

1. POSレジ画面を開きます。
2. 会計するテーブルまたは注文を選びます。
3. 注文内容、数量、割引、合計金額を確認します。
4. 支払い方法を選びます。
5. 必要に応じて担当スタッフのPINを入力します。
6. 会計を完了し、レシートまたは領収書を発行します。

## 支払い方法

| 支払い方法 | 確認事項 |
|---|---|
| 現金 | 預かり金、お釣り、ドロア操作を確認してください。 |
| クレジットカード | 決済端末の接続と決済完了表示を確認してください。 |
| QR決済 | 店舗の運用ルールに従って入金確認を行ってください。 |
| その他 | 店舗で定めた支払い方法名と集計方法を確認してください。 |

## 個別会計

同じテーブル内で別々に支払う場合は、個別会計を利用します。対象商品、数量、割引の適用範囲を確認してから会計してください。

## 返金

返金は金額と理由を確認し、店舗責任者のルールに従って実施してください。返金後は、レジ締め時の差額確認も行ってください。

## レジ開閉

営業開始時にレジを開け、営業終了時にレジを閉めます。現金残高、入金、出金、売上金額を確認し、差額がある場合は理由を記録してください。

## 困ったとき

| 症状 | 対処 |
|---|---|
| 会計対象が出てこない | 注文が提供済みになっているか確認してください。 |
| 金額が合わない | 注文内容、割引、個別会計、返金履歴を確認してください。 |
| 決済端末が反応しない | 端末の電源、Wi-Fi、ペアリング状態を確認してください。 |
| PINが通らない | 出勤打刻、PIN設定、入力ミスを確認してください。 |
| 通常会計ボタンが押せない (黄色バナー) | **緊急レジモード** を使ってください (下記) |

## 緊急レジモード (通信障害・API障害時の会計記録)

通信が不安定な時・POSLA の通常会計が使えない時に、**会計した事実を端末内に記録して後から POSLA へ同期する** モードです。

::: danger この機能が代替するのは「記録」だけです
- POSLA がカード決済を代替するわけではありません。**カード決済は必ず外部決済端末 (Airpay / Square / Stripe Terminal / 電子決済端末等) で処理してください。**
- POSLA には **カード番号・有効期限・セキュリティコードを絶対に入力しないでください**。入力欄そのものを用意していません。
- 緊急レジで記録した会計は「POSLA 未同期」です。通常の売上レポートには、管理者が確認してから反映されます。
:::

### いつ使うか

- レジ画面上部に黄色いバナー「古いデータを表示中」が出て、通常会計ボタンが押せない時
- 通常会計を押したら「通信エラー」「最新の注文状態を取得できませんでした」と出た時
- お店の Wi-Fi / モバイル回線が不安定で、会計だけは記録を残したい時

### 起動方法

レジ画面ヘッダ右上の 🔴 **「緊急レジ」** ボタンを押してください。

- バッジ (赤丸の中の数字) が出ている時は、未同期の緊急会計がその件数ぶん端末内に残っています。
- 通常運用でも店長判断で手動起動できます (検証・訓練用)。

### 使える支払方法

| 選択肢 | 意味 |
|---|---|
| **現金** | お客様から現金を受領済み。受領額とお釣りを入力 |
| **外部カード端末** | 決済端末 (Airpay 等) でカード決済を完了させてから、控え番号 / 承認番号を記録 |
| **外部 QR / 電子決済** | 外部の QR 決済端末 / 電子マネー端末で完了させてから記録 |
| **その他外部決済** | 上記以外 (商品券、事前振込 等)。メモで詳細を残す |

::: warning カード決済の流れ
1. **お客様のカードは外部決済端末に通す (POSLA では処理しない)**
2. 外部端末で承認 → 承認番号・控え番号が出る
3. 外部端末の **控えを必ず保管** (紙 or 写真)
4. POSLA の「緊急レジ」で支払方法 = 「外部カード端末」を選び、控え番号 / 承認番号を記録
5. 通信復帰後、POSLA 側で自動同期される
:::

### 記録後の画面

記録すると「POSLA 未同期の緊急会計控え」が画面に表示されます。これには:
- 緊急会計番号 (端末内一意 ID)
- テーブル名 / 合計金額 / 支払方法
- 外部端末の控え番号・承認番号 (入力した場合)
- 担当者名 / 記録時刻

印刷ボタンで紙に残すこともできます。**この控えは「POSLA 未同期」と明示されている点が、通常のレシートとの違いです。**

### 未同期件数の確認と同期

- ヘッダの 🔴 ボタン横のバッジ = **未同期 + 要確認件数の合計**
  - `3` のように数字だけ → 未同期のみ
  - `!3` のように `!` 付き → **要確認 (conflict / pending_review) が含まれています**。管理者に連絡してください
  - tooltip (長押し / マウスホバー) で「未同期: N 件 / 要確認: M 件」の内訳表示
- 🔴 ボタン → 「未同期会計一覧」で状態を確認
- 状態の意味:
  - **pending_sync** (オレンジ): 端末にあり、まだ送信していない
  - **syncing** (水色): 送信中
  - **synced** (緑): サーバー到達済み、正常
  - **conflict** (赤): 同じ注文が通常会計でも処理されている可能性あり。**管理者確認必須**
  - **pending_review** (紫): 金額に差異あり、要手動確認
  - **failed** (濃赤): ネットワークエラー等。同期再試行ボタンで再送

### 未同期会計があるときにやってはいけないこと

::: danger 未同期の間は絶対やらないでください
- ❌ ブラウザのキャッシュ削除
- ❌ ブラウザデータ削除
- ❌ ログアウト
- ❌ POSLA アプリの再インストール
- ❌ 端末の初期化

→ これらをすると **IndexedDB が消え、未同期の会計記録が永久に失われます**。

通信復帰後、未同期件数が 0 になってから行ってください。
:::

### 通信復帰後の確認

1. 通信が復帰すると自動で同期が試みられます (上部の緑バナー「通信が復帰しました」)
2. 🔴 ボタンのバッジが 0 になれば同期完了
3. 0 にならない件がある場合は、🔴 → 「未同期会計一覧」で状態を確認
4. conflict / pending_review がある場合は、**管理者 (owner/manager) に連絡** して POSLA 管理画面で精査してもらう

### 管理者: 「緊急会計台帳」で確認する (Phase 4b-2 / 4c-1)

manager / owner の方は、管理画面で店舗単位の緊急会計を一覧確認 + 判断記録ができます。

**場所**: 管理画面 → レポート → 緊急会計台帳

**見える内容**:
- 期間フィルタ (from / to、最大 92 日)
- status フィルタ (すべて / synced / conflict / pending_review / failed)
- KPI 8 タイル: 件数・合計金額・要確認件数 (conflict + pending_review)・PIN 未検証件数 + **未解決 / 有効確認済 / 重複扱い / 無効扱い** (Phase 4c-1 追加)
- 一覧表 (日時 / status / **解決** / テーブル / 金額 / 支払方法 / 外部端末控え番号 / 担当者 / PIN 検証 / **解決者・日時** / conflict_reason)
- 行をクリックすると詳細モーダル (明細 / 注文 ID / メモ / **管理者解決操作**)

::: warning Phase 4c-1 時点でできること / できないこと
- ✅ 管理者判断の**記録**: 「有効確認 / 重複扱い / 無効扱い / 保留」ボタンで判断を DB に残せる (詳細は次節)
- ❌ **売上レポートへの反映**: 判断を記録しても `payments` テーブルや売上レポートには反映されません (**Phase 4c-2 以降で実装予定**)
- ❌ **修正・取消**: 一度記録した「確定判断 (有効確認 / 重複扱い / 無効扱い)」の取消はまだできません (**Phase 4c-2 以降で取消フロー実装予定**)
- conflict / pending_review があれば、**外部決済端末の控え** (レシート紙・端末画面) と必ず突き合わせてから判断してください
- カード決済の真偽は POSLA では確定できません。外部端末の承認記録が正です
:::

**よくあるケース**:
- `order_already_paid`: 通常会計で既に処理済。緊急会計側は無効扱い
- `emergency_duplicate_order`: 別端末で同じ注文を緊急会計していた → どちらを正とするか管理者判定
- `manual_entry_without_context`: テーブル選択なしの手入力モード → 外部端末控えと突合して精査
- `pin_entered_but_not_verified`: PIN 入力したが再試行で送信されなかった → 担当者を本人に確認

### 通常会計も取り消せます (2026-04-21 Phase 4d-5c-ba / 4d-5c-bb-A)

POS レジ（`kds/cashier.html`）から普段どおりに行った会計も、取引履歴パネルから管理者が論理取消できます。誤会計の **記録だけを無効化**する操作で、調理や卓状態は動かしません。

**操作手順**:
1. POS レジ > 右上「取引履歴」ボタンで取引ジャーナルを開く
2. 対象行の右側に表示される **「会計取消」** ボタンを押す（表示条件は下記）
3. **取消理由（必須）** を入力して実行

**取消ボタンが表示される条件**（4d-5c-bb-C 時点、AND 条件）:
- まだ取消していない
- まだ返金していない
- `is_partial=0`（分割会計でない）
- ゲートウェイ非経由（Stripe Connect などの外部決済を通していない）
  - 対象の支払方法: 現金 / 手動 card（端末を経由しない手入力の card） / 手動 qr
  - 対象外: Stripe Connect 経由の card など（引き続き返金で対応）
- 対象の注文種別: 店内（dine_in） / ハンディ（handy） / **POS レジで会計したテイクアウト**（4d-5c-bb-C で追加）
  - 対象外: オンライン決済済みテイクアウト（Stripe 決済が完了したもの。API 側の status ガードで弾く）

**取消後の挙動**:
- 取引ジャーナル: 「✗ 取消済」赤バッジ。合計金額は取消分を差し引いた **netTotal** で表示され、KPI に「取消済: N 件 / -¥N」が追加される
- 現金の場合のみ: レジ分析の cashLog に `cash_out` 相殺行が自動で 1 行追加され、期待残高が相殺される（手動 card / qr は cash_log に入らない）
- 売上レポート（sales-report / turnover-report / register-report）: 取消済 payments を含む注文は集計から除外される
- 注文そのもの（orders.status）: `paid` のまま維持。卓状態・レシート・KDS 表示は変更なし
- レシート再印刷（GET）: 既発行分は引き続き閲覧可能
- レシート **新規発行**（POST）: 取消済 payments に対しては 409 `VOIDED_PAYMENT` で拒否
- 返金（refund-payment）: 取消済 payments に対しては 409 `ALREADY_VOIDED` で拒否（取消と返金は排他）

**本 API ではまだ取り消せないもの**（取消ボタンが出ません / API 側で 409 で弾く）:
- 分割会計（is_partial=1）
- Stripe Connect などゲートウェイ経由の card/qr — **取消 (void) ではなく返金 (refund)** で処理。Connect Checkout (`stripe_connect`) の takeout は Phase 4d-5c-bb-D で返金経路実証済（テイクアウト章「オンライン決済テイクアウトの返金」を参照）
- オンライン決済テイクアウト — 同上。取消 (void) の対象外、返金 (refund) 経路を使う
- 返金済み payments
- 緊急会計転記由来（手入力計上 / 注文紐付き転記）— 既存の緊急会計台帳から取消

**注意**:
- 取消理由は必須・監査ログに残ります
- 二重に取り消そうとしても安全（idempotent）
- cash は cash_out 相殺で会計は合いますが、**お客さんへの現金返却は手渡しで別途実施**してください

### 注文紐付きの緊急会計転記も取り消せます (2026-04-21 Phase 4d-5b)

「売上へ転記する」（青ボタン、Phase 4c-2）で通常会計に反映した記録も、後から取り消せるようになりました。

**操作手順**:
1. 緊急会計台帳で対象の行をクリック
2. 「売上転記」セクションに **「緊急会計転記を取り消す」** ボタン（オレンジ色）が表示される
3. **取消理由（必須）** を入力
4. 「緊急会計転記を取り消す」を押す → 確認ダイアログ → 取消実行

**取消後の挙動**:
- 取引ジャーナル: 「取消済み」バッジ
- レジ分析 > 現金売上 / 期待残高: 現金は `cash_out` 相殺行が自動で入り、期待残高は差分 0
- レジ分析 > 外部分類別: voucher 等の集計から除外
- 緊急会計台帳: 「✗ 取消済み」バッジ + 取消日時 / 理由

**ここが手入力計上の取消とは違います（重要）**:
- **売上レポート（注文ベース集計）に「残ります」**。通常会計と同じく「会計済み注文」として表示されます
- **回転率レポートにも残ります**
- **テーブル状態は戻りません**。お客さんが退店済み・卓クローズ済みの後で取り消す前提です
- **QR コード（session_token）は戻りません**。新しいお客さんが座る前に必ず通常の卓オープン手順で再開してください

**いつ使うか**:
- 外部端末控えとの不一致が後から判明したとき
- 誤って二重に転記してしまったとき
- 本来 `other_external` で分類すべきだったが先に `external_card` で転記してしまったとき（取消後に正しい分類で再転記）

**いつ使わないか**:
- 通常会計（POS レジから直接会計したもの）→ 取引ジャーナル（取引履歴パネル）の「**会計取消**」ボタンから取り消します（§「通常会計も取り消せます」参照）。本ボタン（緊急会計転記の取消）ではありません。
- 営業中で次のお客さんが座っている卓 → ❌ **テーブルセッションは戻らない**ので、運用上の問題が発生

**注意**:
- 売上レポート（sales-report / turnover-report / register-report）は **Phase 4d-5c-a で「注文に active payment が 1 件も無ければ除外」に統合済み**。取消済の注文は自動で売上集計から落ちます。特例の追跡は原則不要です。
- 取消理由は必須・記録に残ります
- 二重に取り消そうとしても安全（idempotent）

### 手入力計上は取り消せます (2026-04-21 Phase 4d-5a)

誤って「売上として計上する（手入力）」を押してしまった場合、**管理者が後から取り消せます**。

**操作手順**:
1. 緊急会計台帳で対象の行をクリック（詳細モーダル）
2. 「売上転記」セクションの下に **「手入力計上を取り消す」** ボタン（赤）が出る
3. **取消理由（必須）** を入力
4. 「手入力計上を取り消す」を押す → 確認ダイアログ → 取消実行

**取消後の挙動**:
- 取引ジャーナル: 「取消済み」バッジが付く（行自体は残る）
- レジ分析 > 外部分類別: voucher/bank_transfer 等の集計から除外される
- レジ分析 > 現金売上 / 期待残高: 現金の場合は `cash_out` 相殺行が自動で入り、期待残高差分が 0 になる
- 台帳: 緊急会計記録には「取消済み」バッジが付く、取消日時と理由が表示される

**この「手入力計上の取消」では取り消せないもの**（それぞれ別の経路で取り消せます）:
- 注文紐付きで転記された通常会計（売上として計上する青いボタン経由）→ §「注文紐付きの緊急会計転記も取り消せます」
- 通常の POS レジで会計した payments → §「通常会計も取り消せます」（POS レジ > 取引履歴の「会計取消」ボタン）
- ❌ 取消そのものの取り消し（= 再計上）は全系統で不可

**注意**:
- 取消理由は必須です
- **同じ記録を 2 回取り消そうとしても安全です**（2 回目は何もせず成功扱い）
- 取消操作は**元に戻せません**。外部端末控えと金額を必ず再確認してから押してください

### 手入力売上は売上レポートには出ません (2026-04-21 Phase 4d-4a)

管理者が台帳から「**売上として計上する（手入力）**」を押した記録は、以下に反映されます。

| 画面 | 反映 |
|---|---|
| 取引ジャーナル | ✅ 出ます |
| レジ分析 > レジログ・現金売上・期待残高 | ✅ 現金の手入力だけ反映 |
| レジ分析 > 外部分類別（商品券・事前振込等） | ✅ 外部分類済みは反映 |
| **売上レポート** | ❌ **出ません** |
| 領収書 | △ 明細が入っている記録のみ発行可能 |
| 返金（レジ画面） | ❌ 構造上システム返金できません。手渡し返金 or 外部端末返金で運用 |

**大事なポイント**:
- 売上レポート（注文ベース集計）には出ないため、**手入力売上を含めた日次売上の確認はレジ分析タブを併用**してください
- 誤って計上した場合の取消は**今はできません**（後続バージョンで対応予定）。計上ボタンを押す前に必ず外部端末控えと合わせて金額を確認してください

### その他外部決済は分類を選んでください (2026-04-21 Phase 4d-2)

「その他外部決済」を選んだ場合、**5 つの分類から必ず 1 つ選択**してください。未選択のまま記録ボタンを押しても止まります。

| 分類 | いつ使うか |
|---|---|
| **商品券** | 紙の商品券、電子ギフト券、クーポン券を使ったとき |
| **事前振込** | 予約時の銀行振込、団体予約・ケータリングの前金が入っていたとき |
| **売掛** | 接待・固定客の付け払い、法人請求で後日集金するとき |
| **ポイント充当** | 店舗独自ポイントで全額または一部を相殺したとき |
| **その他** | 上記に当てはまらない特殊ケース (メモ欄に詳細を残す) |

**どれに当てはまるかわからないとき**:
- 最初の 4 つに当てはまりそうならそれを選ぶ
- どうしても当てはまらない場合だけ「その他」を選び、**メモ欄** に理由を書く
- 台帳を確認する管理者が後追いで判断できるように詳細を残すのが安全です

**注意**:
- Phase 4d-2 時点では、選んだ分類は**台帳表示と管理者確認のため**に使います。売上レポートへの自動反映は後続の管理者処理で行います
- 旧データ (2026-04-20 以前) の「その他外部」記録は「その他外部（未分類）」と表示されます。管理者が 4d-3 以降で分類できるようになります

### 緊急レジの PIN は任意です (2026-04-20 Phase 4d-1)

緊急レジの「担当スタッフ PIN」欄は **任意** です。空欄でも緊急記録は残せます。

**PIN 欄の使い方**:
- スタッフ本人の PIN を入力して記録 → 台帳 PIN 列が「**✓ 確認済**」（緑）になります
- 空欄のまま記録 → 台帳 PIN 列は「**入力なし**」（グレー）になります
- 間違った PIN を入力 → 台帳 PIN 列は「**入力不一致**」（オレンジ）になります
- 古い記録 (2026-04-20 以前) → 台帳 PIN 列は「**未確認**」（グレー） のままになります

**「確認済」と「自己申告」の違い**:
- **確認済** = PIN 検証が通り、どのスタッフか DB で特定できている状態
- **自己申告** = 担当欄にユーザー名は入っているが、PIN 検証なしでクライアントが自己報告したもの。台帳上で「担当」列に `(自己申告) 〇〇` と表示され、紐付け保証はありません

PIN を運用したい場合はオーナーダッシュボード「スタッフ管理」タブで各スタッフに PIN を事前登録 + 勤怠打刻しておいてください。

### 管理者の判断を記録する (Phase 4c-1)

2026-04-20 以降、manager / owner は緊急会計台帳の詳細モーダルから **判断結果** を記録できます。

**記録できる判断 (4 種類)**:

| ボタン | 意味 | note |
|---|---|---|
| **有効確認** | この緊急会計は正しい会計記録と確認した | 任意 |
| **重複扱い** | 通常会計 / 別の緊急会計と重複していたため売上計上対象外 | **必須** |
| **無効扱い** | 誤入力・テスト・無効記録 | **必須** |
| **保留にする** | まだ判断できない (外部端末の日締め待ち等) | 任意 |

**操作手順**:
1. 緊急会計台帳で対象の行をクリック → 詳細モーダルが開く
2. 下部の「管理者解決」エリアで note を入力 (重複扱い / 無効扱いは必須)
3. ボタンをクリック → 確認ダイアログ → 記録

::: warning 売上レポートには反映されません
- **「有効確認」を記録しても、売上レポートには自動で反映されません**
- 管理者判断を台帳に残すまでが今回 (Phase 4c-1) の範囲です
- `payments` テーブルへの転記と売上レポート反映は **Phase 4c-2 以降** で実装予定
- 店舗の日次売上に緊急会計を計上する運用は、外部決済端末の控えを別途管理する運用 (Phase 4a 時点と同じ) を継続してください
:::

::: tip 同じ判断の再送はエラーになりません
- 同じ判断を誤って 2 回押しても「既に記録済み」として正常に処理されます (idempotent)
- ただし**異なる判断に変更する操作はできません** (例: 有効確認 → 無効扱いへの変更)。変更したい場合は POSLA 運営へ連絡してください (Phase 4d 以降で取消フロー実装予定)
:::

### 売上へ転記する (Phase 4c-2)

2026-04-20 以降、**「有効確認」済みの緊急会計は、管理者が「売上へ転記」ボタンを押すと通常の売上に計上されます**。

**前提条件** (すべて満たすとき転記ボタンが出ます):
- 管理者解決が「有効確認 (confirmed)」
- まだ転記していない (syncedPaymentId が空)
- 同期失敗 (status=failed) ではない
- 対象注文 (orderIds) が 1 件以上ある
- 支払方法が「その他外部決済」ではない (現金 / 外部カード / 外部 QR のみ)

**操作手順**:
1. 緊急会計台帳で有効確認済みの行を開く
2. 詳細モーダル下部の「売上転記」ブロックに「売上へ転記する」ボタンが表示される
3. メモ (任意) を入力
4. ボタンを押すと確認ダイアログで「合計・支払方法・テーブル」が表示される
5. **外部決済端末の控えと金額・承認番号が一致することを必ず確認**
6. 確定 → 通常 payments に記録され、売上レポート・レジ締めに反映

**転記後の扱い**:
- 詳細モーダルに「✓ 転記済み: payment_id xxx」と表示
- 通常の売上レポートで「緊急会計由来の売上」として集計される
- 同じ緊急会計に対して再度ボタンを押しても二重転記されない (idempotent)

::: danger 転記取消はまだできません
- 一度「売上へ転記」した緊急会計を取り消す UI は Phase 4d 以降で実装予定
- 誤転記した場合は POSLA 運営へ連絡して手動対応が必要
- 転記前の外部端末控えとの突合を必ず行ってください
:::

::: tip 支払方法の違い
- **現金**: 現金残高と一致する想定 → 通常レポートで「現金売上」に計上
- **外部カード端末**: 通常レポートで「カード売上」に計上 (内部では gateway_name='external_card' として識別可能)
- **外部 QR / 電子マネー**: 通常レポートで「QR 売上」に計上 (gateway_name='external_qr')
- **その他外部決済**: Phase 4d で設計予定。転記ボタンが出ません
:::

### 手入力モードの売上反映

緊急会計の「手入力モード」(テーブル未選択で記録した会計) は、**現時点で売上転記できません**。

理由: payments テーブルは注文 ID (orderIds) と強く結び付いているため、注文 ID がない手入力の会計は通常売上への自動反映設計が難しい。

対応:
- Phase 4d で「手入力売上」専用テーブルまたは payments 側の manual_entry 対応を設計予定
- それまでは、手入力モードの緊急会計は **POSLA 運営に連絡して手動で売上計上** してもらう必要があります

### 管理者が未同期会計がある端末を扱う時の注意

スタッフが未同期件数バッジを「!N」表示 (要確認あり) と報告してきた場合:
1. 端末の 🔴 → 「未同期会計一覧」で状態を確認
2. pending_sync / syncing / failed が残っていれば通信復帰を促す
3. conflict / pending_review があれば、管理画面の「緊急会計台帳」で精査
4. 端末の IndexedDB を**絶対に消さない**よう、スタッフに再周知

### よくある質問

**Q. 緊急レジで記録した会計は、自動的に売上レポートに乗りますか？**
A. いいえ。Phase 4a/4b (2026-04-20 時点) では「台帳保存 + 管理者の読み取り確認」までです。通常売上レポートへの反映は管理者が手動で行います (**Phase 4c 以降で自動化予定**)。

**Q. 同じテーブルで同じ会計を 2 回記録してしまったら？**
A. 各緊急会計には端末内で一意の ID が付くので、1 回の記録で POST ボタンを連打しても二重登録はされません。ただし手順自体を 2 回繰り返した場合は 2 件記録されます。未同期一覧で確認し、不要な方を管理者に相談してください。

**Q. 複数端末で同じテーブルを同時に緊急会計したら？**
A. サーバー側で注文 ID の重複を検出し、2 件目は `conflict` 状態になります。管理者が精査してください。

**Q. 外部端末の控え番号を間違えて記録してしまった**
A. 現時点では緊急会計の修正は不可です。メモ欄に訂正情報を書き、管理者に連絡してください (**Phase 4c 以降で「承認」「承認取消」を設計予定**)。

**Q. カード番号を入力したい**
A. **入力しないでください**。POSLA の緊急レジはカード情報を一切扱わない設計です (PCI-DSS 範囲外)。カード決済は必ず外部端末で完結させ、その控え番号のみを記録してください。
