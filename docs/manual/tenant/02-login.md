## 2.1 初回ログインの手順

### 事前準備

ログインするには、オーナーまたはマネージャーが事前にアカウントを作成している必要があります。アカウント作成時に以下が決まります。

- **ユーザー名**（POSLA 全体（テナント横断）で一意。例：matsunoya_tanaka、momonoya_suzuki01、kds-shibuya-01）
  - 2026-04-19 の S3 セキュリティ修正以降、ユーザー名はグローバル一意になりました。他店舗・他テナントと衝突する場合はテナント略号や店舗名をプレフィックスに付けてください
- **初期パスワード**（作成者が設定し、口頭やメモでスタッフに伝えます）
- **ロール**（owner / manager / staff / **device**）
- **所属店舗**（ownerは全店舗、manager/staff/deviceは指定された店舗のみ）

POSLAのアカウントには **4種類のロール** があります。

| ロール | 用途 | ログイン後の遷移先 | 例 |
|---|---|---|---|
| **owner** | テナントオーナー（全店舗統括） | オーナーダッシュボード | 経営者本人 |
| **manager** | 店長（担当店舗の管理） | 管理ダッシュボード | 田中店長 |
| **staff** | 一般スタッフ（個人ログイン） | 管理ダッシュボード or スタッフ画面 | ホール鈴木さん、調理山田さん |
| **device** | KDS / レジ / ハンディ端末専用（人間ではない） | KDS / POSレジ / ハンディ画面に直接 | 厨房KDS-1、レジ-1、ハンディ-1 |

::: tip メールアドレスは不要です
POSLAのログインにメールアドレスは必要ありません。飲食店スタッフがメールアドレスを持っていない・教えたくないケースに配慮した設計です。ユーザー名とパスワードだけでログインできます。
:::

::: warning device ロールは特殊です
device は「人間スタッフ」ではなく「端末そのもの」に与えるアカウントです。シフト・人件費・スタッフ評価レポートには含まれません。詳しくは **2.5 デバイスアカウント** を読んでください。
:::

### ログイン前のチェックリスト

ログインボタンを押す前に、以下 5 項目を確認してください。

| # | 確認項目 | 確認方法 | NG 時の対応 |
|---|---|---|---|
| 1 | ブラウザは Chrome / Edge / Safari の最新版か | ブラウザのバージョン確認画面で確認 | 古いブラウザは XSS / WebSpeech 対応が不完全。最新版に更新 |
| 2 | URL は正しいか（`https://eat.posla.jp/admin/`） | アドレスバーで `https://` から始まることを確認 | `eat-posla.jp`（ハイフンあり）は誤り。`eat.posla.jp` |
| 3 | ユーザー名・初期パスワードを上司から受領済みか | 紙のメモ・社内チャット等で確認 | owner / manager に確認、口頭 or 社内チャットで再受領 |
| 4 | Caps Lock は OFF か | キーボード LED で確認 | 大文字小文字は厳密に区別される。Caps Lock を OFF |
| 5 | テナントが有効化されているか | 不明な場合は POSLA 運営に確認 | `TENANT_DISABLED` エラー時は 2.7 の対処を参照 |

### ログイン手順（クリック単位）

1. ブラウザで管理画面 URL を開く
   - 本番例：`https://eat.posla.jp/admin/`
2. ログイン画面が表示される（下記 ASCII 図）

```
┌──────────────────────────────────────────────┐
│            POSLA 管理画面ログイン             │
│      ユーザー名とパスワードでログイン         │
│                                              │
│  ユーザー名                                  │
│  ┌────────────────────────────────────────┐  │
│  │ shibuya-staff                          │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  パスワード                                  │
│  ┌────────────────────────────────────────┐  │
│  │ ●●●●●●●●●●                              │  │
│  └────────────────────────────────────────┘  │
│                                              │
│         ┌──────────────────┐                 │
│         │     ログイン     │                 │
│         └──────────────────┘                 │
└──────────────────────────────────────────────┘
```

3. **ユーザー名** 欄に、アカウント作成時に決めたユーザー名を入力する
4. **パスワード** 欄に、初期パスワード（または変更済みパスワード）を入力する
5. **「ログイン」** ボタンを押す
6. ログイン中は **「ログイン中...」** に表示が変わる（API 呼び出し中）
7. 成功すると自動的に該当画面へリダイレクトされる:
   - **owner** → `/admin/owner-dashboard.html`
   - **manager** / **staff** → `/admin/dashboard.html`
   - **device** → `userVisibleTools` に応じて `handy/index.html` > `kds/cashier.html` > `kds/index.html` の順で振り分け

### 入力フィールドの詳細

| フィールド | 例 | 必須 | 文字種 | 補足 |
|---|---|---|---|---|
| **ユーザー名** | `tanaka`, `shibuya-staff`, `kds-shibuya-01` | ✅ | 半角英数 + ハイフン + アンダースコア（3〜50 文字） | メールアドレスでも後方互換でログイン可。大文字小文字を厳密区別 |
| **パスワード** | （表示されない） | ✅ | 8 文字以上 + 英字 1 文字 + 数字 1 文字 | 変更後は **2.2 パスワード変更** 参照 |

```

```

### ログインに失敗した場合

| エラーメッセージ | エラーコード | HTTP | 原因 | 対処法 |
|---|---|---|---|---|
| 「ユーザー名またはパスワードが正しくありません」 | `INVALID_CREDENTIALS` | 401 | ユーザー名 or パスワードの入力ミス | 入力内容を確認。大文字小文字を区別。Caps Lock OFF |
| 「ユーザー名とパスワードは必須です」 | `MISSING_FIELDS` | 400 | どちらかの欄が空 | 両方入力する |
| 「このアカウントは無効化されています」 | `ACCOUNT_DISABLED` | 403 | `users.is_active = 0` | owner / manager にアカウント有効化を依頼 |
| 「このテナントは無効化されています」 | `TENANT_DISABLED` | 403 | `tenants.is_active = 0`（解約・未払い等） | POSLA 運営に連絡。Stripe Billing 状態を確認 |
| 「セッションがタイムアウトしました。再度ログインしてください。」 | `SESSION_TIMEOUT` | 401 | 30 分以上無操作 | 再度ログインする（2.3 参照） |
| 「セッションが無効化されました。再ログインしてください。」 | `SESSION_REVOKED` | 401 | manager+ がリモートログアウトを実行 | 別ロールで再ログイン or 管理者に確認 |
| 「通信エラーが発生しました」 | （クライアント側） | — | ネットワーク接続不可 | Wi-Fi / モバイル回線を確認、再試行 |
| 「サーバーが空のレスポンスを返しました」 | （サーバ 502/504 等） | — | サーバ側の障害 | 数分待って再試行、続く場合は POSLA 運営に連絡 |
| 「リクエストの JSON が不正です」 | `INVALID_JSON` | 400 | クライアント実装バグ | ブラウザ拡張・古いキャッシュを無効化、Ctrl+F5 で再読込 |

::: tip ブルートフォース対策の現状（2026-04-19）
ログイン API 自体には**回数制限が未実装**です（`api/auth/login.php`）。ただし：

- パスワードは bcrypt（`PASSWORD_DEFAULT`）でハッシュ化されており、漏洩しても平文復元は困難
- 連続失敗を検知するには `audit_log` の `event='login'` レコード数 + `error_log.code='INVALID_CREDENTIALS'` 件数を運営側で確認可能
- 認証以外（テーブル PIN、サインアップ、デバイス登録）には個別レートリミットあり（[18 章 18.Z.1](./18-security.md#18-z-1-全レートリミット一覧-s-1) 参照）
- 将来的にログイン API にもレートリミット追加予定
:::

## 2.2 パスワード変更

パスワード変更には3つの経路があります。

### 自分のパスワードを変更する（推奨）

各ユーザーは画面右上の**「パスワード変更」ボタン**から自分のパスワードを変更できます。

1. ログイン後、画面右上のユーザー名の横にある**「パスワード変更」**ボタンを押す
   - 管理ダッシュボード（manager / staff）：右上の「ログアウト」ボタンの隣
   - オーナーダッシュボード（owner）：右上の「ログアウト」ボタンの隣
2. パスワード変更モーダルが開きます：
   - **現在のパスワード**：いま使っているパスワードを入力
   - **新しいパスワード**：新しいパスワードを入力（パスワードポリシー要件あり、下記）
   - **新しいパスワード（確認）**：もう一度同じものを入力
3. **「変更する」**を押す
4. 成功メッセージが表示され、画面はそのまま使えます
5. **他の端末でログイン中のセッションは自動的にログアウトされます**（いま使っている端末はそのまま継続）

::: tip パスワードポリシー
新しいパスワードは以下の要件を全て満たす必要があります（`api/lib/password-policy.php` `validate_password_strength()`）。

- **8文字以上**
- **英字を1文字以上**含む（A-Z または a-z）
- **数字を1文字以上**含む（0-9）

例：`tanaka123`、`Yamada2026`、`shibuya2024`

要件を満たさない場合、違反した項目に応じて以下のいずれかのエラーが返ります（HTTP 400 / `WEAK_PASSWORD`）。

- 8文字未満 → 「パスワードは8文字以上で入力してください」
- 英字なし → 「パスワードには英字を1文字以上含めてください」
- 数字なし → 「パスワードには数字を1文字以上含めてください」

チェック順は「長さ → 英字 → 数字」で、最初に失敗した項目のメッセージが1件返ります。
:::

::: warning 現在のパスワードが分からない場合
何度試してもログインできない場合や、現在のパスワードを忘れてしまった場合は、自分でパスワードを変更することはできません。下記の「オーナー/マネージャーが他のユーザーのパスワードを変更する」手順で、上司にリセットを依頼してください。
:::

### オーナーがスタッフのパスワードを変更する場合

スタッフがパスワードを忘れた場合や、新規スタッフの初期パスワードを設定する場合の手順です。

1. オーナーダッシュボードの「ユーザー管理」タブを開く
2. 対象のユーザーの**編集ボタン**を押す
3. 編集モーダルが開く
4. 「パスワード」欄に新しいパスワードを入力する（パスワードポリシーを満たすこと）
5. **保存**ボタンを押す
6. パスワードが更新される（新しいパスワードを口頭でスタッフに伝えてください）

### マネージャーがスタッフのパスワードを変更する場合

1. 管理ダッシュボードの「店舗設定」→「スタッフ管理」サブタブを開く
2. 対象のスタッフの**編集ボタン**を押す
3. 「パスワード」欄に新しいパスワードを入力する（パスワードポリシーを満たすこと）
4. **保存**を押す

::: warning 注意
マネージャーが変更できるのは自店舗のスタッフのパスワードのみです。他店舗のスタッフや他のマネージャーのパスワードは変更できません。
:::

## 2.6 担当スタッフ PIN（CP1、2026-04-19 追加）

POSレジで会計を実行する際、**4〜8 桁の PIN** で「誰が今レジを操作したか」を特定する仕組み。device アカウント運用でも実際の人間スタッフを記録できる。

### 2.6.1 PIN の用途

- POSレジ画面の **会計確定モーダル** で入力 → 一致したスタッフの user_id が `payments.user_id` に記録
- 入力 PIN は **出勤中のスタッフ**（`attendance_records.clock_out IS NULL`）の中から検索
- 退勤打刻したスタッフの PIN は自動無効化（不正悪用防止、CP1 設計のキモ）

### 2.6.2 PIN の仕様

| 項目 | 値 |
|---|---|
| 桁数 | **4〜8 桁の数字のみ** |
| ハッシュ | bcrypt（`users.cashier_pin_hash` に保存、平文不可） |
| 弱 PIN 拒否 | `1234`, `12345`, `0000`, `00000`, ゾロ目 (`1111` 等) は登録不可 |
| 対象ロール | `staff`, `manager`, `owner`（device は不可） |

### 2.6.3 PIN の設定方法

#### 設定前のチェックリスト

| # | 確認項目 | NG 時 |
|---|---|---|
| 1 | 自分または対象スタッフが `staff` / `manager` / `owner` のどれか | device は不可（`VALIDATION` エラー） |
| 2 | 設定者は **自分の PIN なら誰でも**、他人の PIN なら **manager 以上** | staff が他人を設定すると `FORBIDDEN`（403） |
| 3 | manager の場合、対象スタッフは **自店舗** の所属 | 他店舗のスタッフを設定すると `FORBIDDEN`（403） |
| 4 | PIN は 4〜8 桁の数字のみ（弱 PIN は拒否） | `INVALID_PIN` / `WEAK_PIN`（400） |

#### 自分で設定（誰でも自分の PIN は設定可）

1. **dashboard.html** にログイン
2. 上部タブから **「店舗設定」** を開く
3. サブタブから **「スタッフ管理」** を選択
4. 自分の行の **「編集」** ボタンを押す
5. 編集モーダルが開く
6. **「レジ担当 PIN」** 欄に 4〜8 桁の数字を入力
7. **「保存」** ボタンを押す
8. 一覧の自分の行に **「✅ PIN 設定済み」** バッジが表示される

```
┌──── スタッフ編集 ────────────────────────────┐
│ 表示名:   [ 田中 太郎              ]          │
│ ユーザー名:[ shibuya-tanaka         ]         │
│ パスワード:[                        ]         │
│            （空欄で変更なし）                 │
│                                              │
│ レジ担当 PIN: [ 4-8 桁の数字 ]                │
│            ※退勤打刻すると自動無効化         │
│                                              │
│         ┌──────┐  ┌──────┐                   │
│         │ 保存 │  │キャンセル│                │
│         └──────┘  └──────┘                   │
└──────────────────────────────────────────────┘
```

#### 他のスタッフの PIN を設定（manager 以上）

1. 同じ手順で対象スタッフの **「編集」** ボタンを押す
2. **「レジ担当 PIN」** 欄に新しい PIN を入力
3. **「保存」** を押す
4. 設定したスタッフに PIN を口頭で伝える（メモ・チャットには残さない）

#### 入力フィールドの詳細

| フィールド | 必須 | 文字種 | 補足 |
|---|---|---|---|
| **PIN** | ✅ | **数字のみ 4〜8 桁** | `1234` / `0000` / ゾロ目（`1111` 等）は拒否 |

#### 実装詳細（POSLA 運営向け）

| 項目 | 値 |
|---|---|
| API | `POST /api/store/set-cashier-pin.php` |
| Body | `{ "user_id": "...", "pin": "1357" }` |
| 認証 | `require_auth()`（自分 PIN） / `require_role('manager')` 相当（他人 PIN） |
| ハッシュ | `password_hash($pin, PASSWORD_BCRYPT)` |
| 保存先 | `users.cashier_pin_hash` + `users.cashier_pin_updated_at` |
| 弱 PIN ブラックリスト | `1234`, `12345`, `123456`, `1234567`, `12345678`, `0000`, `00000`, `000000`, `00000000` |
| ゾロ目検出 | 正規表現 `/^(\d)\1+$/` |
| 監査ログ | `cashier_pin_set`（target_username / is_own を記録） |

### 2.6.4 PIN を忘れたとき

自分でリセットする手段はありません。以下のいずれか:
- **スタッフが忘れた**: 店長 or オーナーに依頼 → 編集画面で PIN を再設定
- **店長が忘れた**: オーナーに依頼
- **オーナーが忘れた**: POSLA 運営に依頼（DB 直接対応）

### 2.6.5 PIN 運用ルール

- ✅ **退勤時は必ず退勤打刻** → 退勤後は PIN が無効化される（C 案の核心）
- ✅ **PIN は他人に教えない**（パスワードと同じ重さで管理）
- ✅ **付箋でレジに貼らない**
- ✅ **半年〜1 年に 1 回は PIN 変更**
- ✅ **PIN 漏洩疑いがあれば即変更**
- ❌ 同じ PIN を複数スタッフで共有しない
- ❌ 推測しやすい PIN（誕生日・電話番号下 4 桁）を避ける

### 2.6.6 PIN 関連のエラーメッセージ

| メッセージ | 原因 | 対処 |
|---|---|---|
| 「PIN は 4〜8 桁の数字で入力してください」 | 桁数不足・英字混入 | 数字のみ 4〜8 桁で入力 |
| 「同じ数字の繰り返しは PIN として使えません」 | `1111`, `0000` 等 | 別の組み合わせに変更 |
| 「推測されやすい PIN は使えません」 | `1234`, `12345`, `0000` 等 | 別の組み合わせに変更 |
| 「PIN が一致しないか、対象スタッフが出勤中ではありません」 | PIN 間違い or 退勤済み | 出勤打刻を確認、PIN 再入力 |

### 2.6.7 監査ログ

PIN 関連操作は全て `audit_log` テーブルに記録:

| イベント | 記録内容 |
|---|---|
| `cashier_pin_used` | 会計時に PIN 一致（cashier_user_id, total, payment_method） |
| `cashier_pin_failed` | PIN 入力失敗（pin_length、ブルートフォース調査用） |
| `cashier_pin_set` | PIN 設定・変更（target_username, is_own） |

不正発覚時は監査ログで「いつ・誰の PIN が・どんな会計に使われたか」を完全追跡可能。

詳細は [8 章 8.10-b 担当スタッフ PIN 認証](./08-cashier.md) 参照。

---

### パスワード設定の運用ルール

- 推測されやすいパスワード（`12345678`、`password`、`tanaka2024` のような単純なパターン）は避けてください
- スタッフ間でパスワードを使い回さないでください
- 他人にパスワードを教えないでください（POSLA運営も問い合わせません）
- 退職者のパスワードは速やかにオーナーがリセット、またはアカウント自体を無効化してください

## 2.3 マルチデバイスログイン制御

POSLAでは、同じアカウントで複数の端末（PC、タブレット、スマートフォン等）から同時にログインできます。以下の制御が実装されています。

### 2.3.1 サーバー側で記録されている内容

ログイン時、`api/auth/login.php` が `user_sessions` テーブルに以下を記録します（2026-04-18 時点の実装）。

- `tenant_id` / `user_id` / `session_id`（PHP セッションID）
- `device_label`（User-Agent から簡易判定）
- `ip_address`（`REMOTE_ADDR` のみ。プロキシ経由の場合に備え `X-Forwarded-For` は信頼しない）
- `login_at`（ログイン時刻）
- `last_active_at`（各 API 呼び出しで自動更新）
- `is_active`（1=有効、0=無効化済み）

ログインAPIのレスポンスには、同じユーザーで **2 件以上** アクティブセッションがある場合に `warning` と `active_sessions`（件数）を返します。

```json
{
  "ok": true,
  "data": {
    "user": { ... },
    "stores": [ ... ],
    "warning": "他の端末でもログインしています",
    "active_sessions": 2
  }
}
```

::: warning UI 表示は現状未実装（2026-04-18 時点）
`login.php` は上記の `warning` / `active_sessions` を返しますが、`public/admin/index.html` のログイン後処理はこの警告を **画面に表示しません**（成功時は即リダイレクトのみ）。サーバー側に記録は残るので、管理者が `user_sessions` テーブルで監査はできます。警告モーダル化の UI タスクは未着手。
:::

### 2.3.2 セッションタイムアウト

POSLAのセッションには有効期限があります（実装: `api/lib/auth.php` L24、`public/js/session-monitor.js`）。

- **セッション Cookie TTL: 8時間**（`'lifetime' => 28800`）
- **操作なし 25 分で警告モーダル表示**（`session-monitor.js` L13 `WARNING_MS = 25 * 60 * 1000`）
  - モーダル文言: 「セッションタイムアウト」「5分以内に操作がないとログアウトされます」
  - ボタン: 「操作を続ける」 → `/api/auth/check.php` を叩いて `last_active_at` をリセット
- **操作を続ける** を押さないまま 30 分後にサーバーが `SESSION_TIMEOUT` / `SESSION_REVOKED` を返し、JS が `/admin/index.html` にリダイレクト
- リセット判定イベント: `click` / `keydown` / `scroll` / `touchstart`（同 L130）

### 2.3.3 アクティブセッション管理 API

manager 以上のロールに対して、以下の API が実装されています（実装: `api/store/active-sessions.php`）。

| HTTP | エンドポイント | 用途 | 権限 |
|---|---|---|---|
| GET | `/api/store/active-sessions.php?store_id=xxx` | アクティブセッション一覧 | manager+（owner はテナント全体、manager は所属店舗のみ） |
| DELETE | `/api/store/active-sessions.php` | 対象セッションを `is_active=0` に更新して無効化 | manager+ |

返却フィールド: `id`, `user_id`, `username`, `display_name`, `role`, `session_id`, `device_label`, `ip_address`, `login_at`, `last_active_at`。

::: warning 管理画面の UI は未実装（2026-04-18 時点）
API は稼働していますが、`public/admin/dashboard.html` にこれを使う UI（セッション一覧タブ・リモートログアウトボタン）は実装されていません（`getActiveSessions` / `activeSessions` を使うJS参照なし）。当面は POSLA 運営が `curl` または DB 直参照で対応する運用です。将来的にダッシュボードへ組み込み予定。
:::

::: tip 使いどころ（UI 実装後の想定）
- スタッフが退勤後にログアウトし忘れた場合
- 紛失した端末のセッションを無効化したい場合
- 不審なセッションを検出した場合
:::

## 2.4 ログアウト

ログアウトは画面右上の**ログアウトボタン**から行います。

1. 画面右上のユーザー名の横にある**ログアウト**ボタンを押す
2. セッションが終了し、ログイン画面に戻ります

::: warning 個人スタッフアカウントで共有端末に常時ログインしないでください
KDS（キッチンディスプレイ）・POSレジ・ハンディは **店舗が用意した備品端末** で、スタッフ全員で共有します。これに田中さん個人のアカウントで常時ログインしておくと、誰が会計したか・誰が調理完了を押したかが全部「田中」名義で記録されてしまいます。勤怠打刻・監査ログ・スタッフ評価レポートが全部おかしくなります。

**正しい運用は「デバイスアカウント」を作ること**です。次の **2.5 デバイスアカウント** を読んでください。
:::

::: info POSLA の業務端末は全て店舗備品が前提
スタッフの個人スマートフォンを業務に使う運用は想定していません。ハンディ／KDS／POSレジは **店舗が購入・管理する備品端末** で、全て device アカウントで運用するのが標準です。

個人スタッフアカウント (`role='staff'`) の主な用途:

- 管理ダッシュボード（予約管理・シフト管理の 2 タブのみ閲覧可）
- GPS 勤怠打刻（出退勤記録）
- 自分のシフト確認・希望提出

スタッフは個人アカウントでハンディ／KDS／POSレジ端末を直接操作する必要はありません。それらは device アカウントが常時ログインしている共有端末です。
:::

## 2.5 デバイスアカウント（KDS / レジ / ハンディ端末専用）

### 2.5.1 デバイスアカウントとは

POSLA には、人間スタッフのアカウントとは別に **「デバイスアカウント」** という端末専用ログインがあります。

- ロール: `device`（owner / manager / staff とは独立した第4のロール）
- 用途: **共有端末 1台 = 1 デバイスアカウント**（KDS タブレット／POSレジ端末／共有ハンディのそれぞれ）
- ログインしっぱなしを前提とした設計（共有端末を想定）
- **シフト・人件費・スタッフ評価レポートには一切含まれない**（人間ではないので）
- 監査ログにはちゃんと記録される（`role='device'` で識別可能）

::: tip なぜ「デバイス専用アカウント」が必要なのか
店内オペレーションには2種類のアカウントが必要だからです。

| 種類 | 例 | 用途 | ログイン頻度 |
|---|---|---|---|
| **個人スタッフ** | 田中さん、鈴木さん | 管理ダッシュボード（予約・シフト閲覧） / 勤怠打刻 | 出勤時にログイン → 退勤時にログアウト |
| **デバイス** | KDS-1、レジ-1、ハンディ-1 | 店舗備品端末そのもの（厨房タブレット／レジカウンター／ホール用ハンディ） | **常時ログインしっぱなし**（端末再起動時のみ再ログイン） |

店舗備品ハンディに「田中」でログインしたまま田中さんが退勤すると、夜勤の鈴木さんが触っても全部「田中」の操作として記録されます。これを防ぐために、端末そのものに「ハンディ-1」というアカウントを与えておく、という発想です。
:::

### 2.5.2 個人スタッフアカウントとの違い

| | スタッフアカウント | デバイスアカウント |
|---|---|---|
| ロール | `staff`（または `manager` / `owner`） | `device` |
| 想定ユーザー | 人間1人 | 端末1台 |
| ログイン後の遷移先 | 管理ダッシュボード（`dashboard.html`、`owner` は `owner-dashboard.html`） | **KDS画面 / POSレジ画面 / ハンディ画面に直接遷移**（管理画面には入れない。遷移先は `visible_tools` に応じて決まる、2.5.4 参照） |
| dashboard.html で使える機能 | **staff**: シフト管理タブのみ（マイシフト・希望提出・勤怠打刻）。予約管理タブは 2026-04-19 から非表示（manager+ のみ）。ハンディ/KDS/レジのヘッダーショートカットも **非表示**（`dashboard.html` L850-858 の `_staffToolIds`）<br>**manager**: 全タブ + ヘッダーショートカット **表示**<br>**owner**: `owner-dashboard.html` に即遷移するため dashboard.html には到達しない | — （管理画面に入れない） |
| 業務で触る端末 | 店舗備品の管理端末（PC/タブレット）で管理ダッシュボード・シフト・勤怠を操作 | 店舗備品の KDS / POSレジ / ハンディに常時ログイン |
| シフト管理対象 | ✅ | ❌ |
| 人件費計算対象 | ✅ | ❌ |
| スタッフ評価レポート | ✅ | ❌ |
| 出退勤打刻 | ✅ | ❌（端末は出勤・退勤しないため） |
| 監査ログ記録 | ✅ | ✅（`audit_log.role='device'` で識別） |
| 作成権限 | オーナー・マネージャー | オーナー・マネージャー |
| 1アカウントの単位 | 人間1人 | 端末1台 |

::: tip 実装参照 (2026-04-18 時点)
- 用途の許可値: `api/store/staff-management.php` `_validate_visible_tools()` — `['kds', 'register', 'handy']` のいずれか（カンマ区切りで複数可）
- device ログイン後の振り分け: `public/admin/dashboard.html` L534-554 — 優先度 `handy` > `register` > `kds`
- device のログイン先はスタッフの遷移先と完全に分離（管理画面 DOM に入れない）
:::

### 2.5.3 デバイスアカウントの作成手順

オーナー、または店長（マネージャー）が作成します。スタッフロールでは作成できません。

1. **管理ダッシュボード**にログイン（マネージャー or オーナー）
2. 上部タブから **「店舗設定」** を開く
3. サブタブから **「スタッフ管理」** を選択
4. 画面を下にスクロールすると **「デバイスアカウント（KDS / レジ / ハンディ端末）」** というセクションがあります
5. **「+ デバイス追加」** ボタンを押す
6. 「デバイスを追加」モーダルが開きます
7. 以下の項目を入力します:

| 項目 | 説明 | 例 |
|---|---|---|
| **表示名** | この端末に付ける呼び名（管理画面の一覧に出る） | `厨房KDS 1番`、`バー側KDS`、`レジ1`、`銀座店レジ` |
| **ユーザー名** | ログイン時に入力する名前。半角英数字・ハイフン・アンダースコアのみ、3〜50文字 | `kds-shibuya-01`、`register-shibuya-01`、`kds-bar` |
| **パスワード** | 8文字以上、英字+数字を含む | `kdsShibuya2026` |
| **用途** | チェックボックスで **KDS / POSレジ / ハンディ** から選択（少なくとも1つ、複数可） | KDS のみ、POSレジ のみ、ハンディ のみ、または複数の組み合わせ |

8. **「保存」** を押す
9. 一覧に追加されたら作成完了です

::: tip ユーザー名の付け方の推奨
後から「どの端末がどのアカウントだっけ？」と分からなくなるので、ユーザー名に **「役割 + 店舗 + 番号」** を含めると運用が楽です。

- ✅ `kds-shibuya-01`（渋谷店のKDS 1番）
- ✅ `register-ginza-01`（銀座店のレジ 1番）
- ✅ `kds-bar-shinjuku`（新宿店バー側KDS）
- ❌ `tablet1`（どの店舗の何の端末か不明）
- ❌ `kds`（複数店舗で使えない）

表示名は日本語OKなので、`厨房KDS 1番` のように人間向けの名前を入れておくと管理画面で見やすくなります。
:::

### 2.5.3-b デバイスのワンタイムトークン自動ログイン（F-DA1）

上記の「手動でユーザー名とパスワードを設定してログインさせる」手順に加えて、**ワンタイムトークンによる自動ログイン**の仕組みも提供されています（2026-04-18 追加）。新規タブレットを設置するときに便利です。

#### 使い方

1. マネージャーが管理画面 → スタッフ管理サブタブを開く
2. デバイスアカウントセクションで **【トークン発行】** ボタン（`btn-token-device`、`dashboard.html` L303）をタップ
3. ダイアログで「表示名」と「用途（visible_tools）」を入力して発行
4. `api/store/device-registration-token.php` がワンタイムトークンを生成し、セットアップ用 URL（`/public/admin/device-setup.html?token=...` 形式）を返す
5. その URL を新しい KDS / レジ / ハンディ端末の Chrome で開く（手動コピー or QR にして読み取り）
6. `device-setup.html` がページロード時に `POST /api/auth/device-register.php` を自動実行
7. サーバー側でトークンを検証し、**device ロールのユーザーを新規作成**（ユーザー名は `device_xxxxxxxxxxxx`、パスワードはランダム生成・自動破棄）
8. 端末にデバイスアカウントのセッション Cookie がセットされる → `visible_tools` の優先度（`handy` > `register` > `kds`）に従い、3 秒後に対応画面（`handy/index.html` / `kds/cashier.html` / `kds/index.html`）へ自動遷移

#### セキュリティ制御（実装: `api/auth/device-register.php`）

- ワンタイムトークン（`is_used=1` で使い切り、再利用不可）
- 有効期限あり（発行から `expires_hours` 時間、デフォルト数時間）
- レートリミット: `check_rate_limit('device_register:' . IP, 3, 3600)` — 同一 IP から 1 時間に 3 回まで
- 成功時に `audit_log` に `event='device:register'` を記録
- `SELECT ... FOR UPDATE` でレース条件防止、tenant / store の `is_active=1` チェック
- パスワードは `password_hash(PASSWORD_DEFAULT)` でハッシュ化のみ保存（平文は破棄）
- DB: `device_registration_tokens` テーブル

#### 利点

- 新端末設置時にパスワードを手入力する必要がない（タイポの心配なし）
- 既存のデバイスアカウントを選ぶ運用ではなく、**端末ごとに新しい device ユーザーが自動生成される**ため重複ログインや共有事故が起きない
- 本番環境で POSLA 運営がテナント所有者と離れた場所で設置作業をする場合に便利
- 端末を店舗に送付してから QR / URL で設定完了できる

### 2.5.4 用途（KDS / POSレジ / ハンディ）の選択

「用途」チェックボックスで、この端末アカウントがログインしたときにどの画面を開くかを決めます。複数選択した場合の遷移先は `public/admin/dashboard.html` L546-553 の優先度 **`handy` > `register` > `kds`** に従います。

| 用途の選択 | ログイン後の遷移先（実装: dashboard.html L546-552） |
|---|---|
| **ハンディ** を含む（単独・複数問わず） | `../handy/index.html` |
| **POSレジ** を含み、ハンディなし | `../kds/cashier.html` |
| **KDS のみ** | `../kds/index.html` |
| **何も選択なし** | KDS（デフォルト） |

::: warning 1端末1用途を強く推奨
複数チェックも技術的には可能ですが、運用上は **1端末1用途**（KDS のみ／POSレジ のみ／ハンディ のみ）にしてください。

- KDS / POSレジ / ハンディ は画面構成・操作頻度・設置場所が全く違います（厨房 vs レジカウンター vs 巡回）
- 同じ端末で兼用すると、KDS 画面表示中にお会計が来て切り替える→ピックアップ画面が消える、という事故が起きます
- 「KDS-1」「レジ-1」「ハンディ-1」と物理的にも別端末・別アカウントにしてください
:::

### 2.5.5 デバイスアカウントでログインしたときの動作

1. 通常のログイン画面（`/admin/`）でデバイスアカウントのユーザー名とパスワードを入力
2. ログイン成功後、**管理ダッシュボード（dashboard.html）には入らず**、即座に KDS / POSレジ / ハンディ画面にリダイレクトされます（`visible_tools` に応じて、優先度 `handy` > `register` > `kds`）
3. 以降は **画面が自動で更新**されるので、原則ログアウト不要
4. 端末を再起動・ブラウザを閉じた場合は再ログインが必要です

::: tip デバイスアカウントは管理画面に絶対入れません
これはセキュリティと UX の両面で意図的な設計です。

- KDS タブレットからメニュー編集や売上レポートに触れないようにする（誤操作・店内利用客から見られるリスクを排除）
- 厨房スタッフが「あれ？管理画面どこ？」と迷子にならないようにする
- 万が一デバイスアカウントのパスワードが流出しても、管理データには到達できない

管理画面を操作したいときは、必ず人間のオーナー/マネージャーアカウントで別途ログインしてください。
:::

### 2.5.6 どのスタッフが操作したか追跡したい場合

ハンディ／KDS／POSレジ は店舗備品で device アカウントが常時ログインしっぱなしのため、「誰がいま操作したか」を追跡する仕組みは以下のように用意されています。

| 追跡したいこと | 追跡方法 |
|---|---|
| 誰が会計したか（POSレジ） | POSレジ画面のヘッダー（`ca-staff-name`）で会計担当スタッフを選択 → `orders.staff_id` に記録。ログイン中の device アカウント ID ではなく、選択されたスタッフ ID を書き込む |
| 誰が注文を取ったか（ハンディ） | ハンディ画面で注文送信すると `orders.staff_id` に **現在ログイン中のアカウントの user_id** が記録される。device アカウントでログインしている場合は device ユーザー ID が入る。個人紐付けが必要なら、ハンディにも POSレジ同様のスタッフ切替 UI が必要（将来機能） |
| 誰がいつ出退勤したか | 個人スタッフアカウントでの**勤怠打刻**で記録（GPS 付き）。打刻は管理ダッシュボードからか、店舗備品の管理端末から行う |
| 誰が管理操作をしたか（メニュー編集・売り切れ切替等） | オーナー/マネージャー個人アカウントの**監査ログ**（`audit_log.user_id`） |

KDS の「調理開始」「完成」「提供済」ボタンはスループット計測用（料理が出るまでの時間）であって、スタッフ評価対象にはなりません。これは仕様です。

### 2.5.7 デバイスアカウントの編集・削除

#### 編集

1. 管理ダッシュボード → 店舗設定 → スタッフ管理 → デバイスアカウントセクション
2. 対象のデバイスの **「編集」** ボタンを押す
3. 以下を変更可能:
   - 表示名
   - ユーザー名
   - **新しいパスワード**（空欄なら変更なし）
   - 用途（KDS / POSレジ / ハンディ チェックボックス）
   - **有効** トグル（OFF にすると即座にログイン不可になる）
4. **「保存」** を押す

#### 削除

1. 対象のデバイスの **「削除」** ボタンを押す
2. 確認ダイアログで「OK」を押す
3. アカウントが削除され、その端末は次回操作時にログイン画面に飛ばされます

::: warning 削除前にやること
削除する前に、対象端末がいま稼働中でないか確認してください。営業時間中に削除すると、その端末は突然ログアウトされて KDS / レジ / ハンディ が止まります。

- 営業時間外の作業を推奨
- どうしても営業時間中に削除する必要がある場合は、まず代替端末を用意してから
:::

### 2.5.8 デバイスアカウント運用ルール

#### 必須ルール

- ✅ **1端末 = 1アカウント**（KDS-1 と KDS-2 を同じアカウントで使い回さない。ハンディ 3 台なら `handy-shibuya-01/02/03` のように別々に作成）
- ✅ **役割が違えばアカウントも分ける**（KDS / POSレジ / ハンディ は用途別の別アカウント）
- ✅ **店舗が違えばアカウントも分ける**（渋谷店のKDS-1 と 新宿店のKDS-1 は別アカウント、ユーザー名に店舗名を入れる）
- ✅ **パスワードは店舗の鍵と同じ重さで管理**（誰でも触れるところにメモしない）

#### 推奨ルール

- 💡 **半年〜1年に1回はパスワード変更**（人間アカウントと同じく）
- 💡 **端末が壊れた・廃棄するときは即座にアカウント削除**（または「有効」OFF）
- 💡 **端末交換時は新しいアカウントを作って古いものを削除**（同じアカウントを引き継がない）

#### やってはいけないこと

- ❌ デバイスアカウントを **スタッフ個人のスマホ**にログインさせる（そもそも業務に個人スマホを使わない運用。店舗が備品ハンディを用意すること）
- ❌ 1つのデバイスアカウントで **複数の端末**にログインする（管理画面側からどれが本物か区別できなくなる）
- ❌ デバイスアカウントの **パスワードを口頭で全スタッフに伝える**（必要ありません。ログインは初回設置時の1回だけ）

### 2.5.9 デバイスアカウントに関するよくある質問

**Q1. 既存のスタッフアカウントを KDS タブレットでログインしっぱなしにしてはダメですか？**

ダメではないですが、強く非推奨です。シフト・勤怠・スタッフ評価レポートが全部その人名義になってしまうため、シフト管理が機能しなくなります。デバイスアカウントを作ってください。

**Q2. デバイスアカウントは何個まで作れますか？**

上限はありません。店舗の KDS 台数 + レジ台数 + ハンディ台数の分だけ作ってください。例: 3 店舗・各店 KDS 2 台 + レジ 1 台 + ハンディ 3 台 = 18 個のデバイスアカウントになります。

**Q3. デバイスアカウントを作っても、シフト管理や人件費レポートに影響しないですか？**

影響しません。`role='device'` のユーザーはシフト・勤怠・人件費・スタッフ評価レポートのSQLクエリで完全に除外されます。

**Q4. デバイスアカウントは監査ログに残りますか？**

残ります。`audit_log.role='device'` で識別できるので、「KDS-1 がメニュー売り切れを ON にした」のような操作も追跡可能です。ただし KDS / POSレジ / ハンディ で発生する操作は限定的（注文ステータス変更、会計、売り切れ切替、注文送信など）なので、人間アカウントほど大量の監査ログは出ません。

**Q5. デバイスアカウントでハンディPOSにログインできますか？**

**できます。むしろそれが標準運用です**。POSLA ではスタッフ個人のスマートフォンを業務に使う想定はなく、ハンディも店舗が用意した備品端末を device アカウント（例: `handy-shibuya-01`）で常時ログインして使います。

`visible_tools` は `kds` / `register` / `handy` のいずれかを選択可能で、`handy` を含む場合は `../handy/index.html` に直接遷移します（`dashboard.html` L546）。店内に 3 台ハンディがあれば、それぞれ `handy-shibuya-01` / `handy-shibuya-02` / `handy-shibuya-03` のように個別アカウントを作ります。

注文時の「誰が取ったか」は、ハンディ画面左上のスタッフ選択で操作中のスタッフを切り替えるか、`orders.staff_id` に device ユーザー ID として記録されます（スタッフ個人紐付けが必要ならレジ側のスタッフ選択・勤怠ログと突合して追跡）。

**Q6. デバイスアカウントは何のロールで作成できますか？**

オーナー・マネージャーが作成できます。スタッフロールには作成権限がありません。

**Q7. 桃の屋のような複数店舗で、本部から全店舗のデバイスアカウントを一元管理できますか？**

オーナーアカウントなら全店舗のデバイスアカウントを管理できます。マネージャーアカウントは自分の所属店舗のデバイスのみ管理可能です。

**Q8. デバイスアカウントのパスワードを忘れたら？**

オーナー/マネージャーが「編集」画面から新しいパスワードを設定できます。デバイスアカウント自身が「自分でパスワードを変更する」機能は提供していません（人間ではないため）。

**Q9. 端末が盗まれたらどうすればいいですか？**

すぐに管理ダッシュボードからそのデバイスアカウントを **「有効」OFF** にするか、**削除**してください。盗難端末からはログイン画面に飛ばされ、それ以降ログインできなくなります。

**Q10. デバイスアカウントの「セッション有効期限」はどうなっていますか？**

人間アカウントと同じ8時間の有効期限が設定されています（`api/lib/auth.php` L24 `'lifetime' => 28800`）。KDS / POSレジ / ハンディ は通常の運用で API を頻繁に呼ぶため、営業時間中は事実上セッションが切れません。再起動した翌朝のみ再ログインが必要です。

#### データベース構造

```sql
-- P1a マイグレーション (2026-04-09)
ALTER TABLE users
    MODIFY COLUMN role ENUM('owner','manager','staff','device') NOT NULL DEFAULT 'staff';
ALTER TABLE audit_log
    MODIFY COLUMN role ENUM('owner','manager','staff','device') DEFAULT NULL;
```

`users.role='device'` のレコードを `user_stores` 経由で店舗に紐付けることで「この店舗のデバイス」として管理します。`visible_tools` は `kds` / `register` / `kds,register` のいずれか。

#### API エンドポイント

スタッフ管理APIを `?kind=device` パラメータで再利用しています:

| HTTP | URL | 用途 |
|---|---|---|
| GET | `/api/store/staff-management.php?store_id=xxx&kind=device` | デバイス一覧 |
| POST | `/api/store/staff-management.php?kind=device` | デバイス作成 |
| PATCH | `/api/store/staff-management.php?id=xxx&kind=device&store_id=xxx` | デバイス更新 |
| DELETE | `/api/store/staff-management.php?id=xxx&kind=device&store_id=xxx` | デバイス削除 |

`require_role('manager')` で権限チェック。`kind=device` を指定すると `users.role='device'` の絞り込みになります。

#### ログイン後リダイレクトロジック

`public/admin/dashboard.html` 内（L534-554、2026-04-18 時点）:

```javascript
// P1a: device ロールは KDS / レジ / ハンディ画面に直接リダイレクト
// dashboard.html（管理画面）には絶対に入れない
if (_role === 'device') {
  var firstStore = (stores && stores.length > 0) ? stores[0] : null;
  // visible_tools は store レベル: userVisibleTools (個別) > staffVisibleTools (店舗) > null
  var devTools = null;
  if (firstStore) {
    devTools = (firstStore.userVisibleTools !== undefined && firstStore.userVisibleTools !== null)
      ? firstStore.userVisibleTools
      : firstStore.staffVisibleTools;
  }
  var devToolsStr = devTools || '';
  if (devToolsStr.indexOf('handy') >= 0) {
    window.location.href = '../handy/index.html';
  } else if (devToolsStr.indexOf('register') >= 0) {
    window.location.href = '../kds/cashier.html';
  } else {
    // デフォルトは KDS（kds 指定 or 設定なし）
    window.location.href = '../kds/index.html';
  }
  return;
}
```

`visible_tools` の優先順位: `user_stores.visible_tools`（個別 = `userVisibleTools`）> 店舗デフォルト（`staffVisibleTools`）> なし。遷移先の優先順位: `handy` > `register` > `kds`（デフォルト）。

#### シフト管理から除外される仕組み

`api/store/shift/*.php` の各クエリは `WHERE u.role IN ('staff','manager','owner')` のように `role` で絞り込んでいるため、`device` ロールは自然に除外されます。

#### 監査ログ

`write_audit_log()` は `$user['role']` をそのまま `audit_log.role` に書き込むので、デバイスアカウントの操作は `role='device'` で記録されます。POSLA運営は監査ログ検索で `role='device'` フィルタを使えば「端末操作のみ」を抽出できます。

## 2.8 よくある質問 30 件

### A. アカウント作成・ロール（Q1〜Q5）

**Q1. メールアドレスは必須ですか？**
A. **不要です**。POSLA はユーザー名 + パスワードのみで運用できます。飲食店スタッフがメールアドレスを持っていない・教えたくないケースに配慮した設計です。`users.email` カラムは存在しますが、ログインは `username` を主とし、`email` は後方互換のために OR 条件で受け付けます（`api/auth/login.php`）。

**Q2. 共有アカウント（kitchen / hall 等）を作って全員で使い回してもいいですか？**
A. **禁止です**。POSLA の監査ログ・勤怠打刻・スタッフ評価・シフト管理が全て「個人」を前提に設計されています。1 人 1 アカウントを必ず作成してください。

**Q3. owner / manager / staff / device の違いを 1 行で言うと？**
A. **owner** = 経営者（全店舗）、**manager** = 店長（自店舗フル管理）、**staff** = 一般スタッフ（シフト管理タブのみ）、**device** = 端末専用（管理画面に入れない）。

**Q4. 1 つの店舗に owner と manager の両方が必要ですか？**
A. テナントには **必ず 1 人以上の owner** が必要です。manager は任意（owner だけで全店舗運営も可）。manager を作ると店舗ごとに権限を分散できます。

**Q5. owner を複数人作れますか？**
A. **作れます**。共同経営者・配偶者・経理担当を owner にすれば全員が全店舗にアクセス可能です。ただし監査追跡のため 1 人 1 アカウント原則を守ること。

### B. ログイン操作（Q6〜Q10）

**Q6. ログイン URL は？**
A. 本番テナント: `https://eat.posla.jp/admin/`（テナント側）/ `https://eat.posla.jp/posla-admin/`（POSLA 運営側）。テスト環境は運営から別途案内。

**Q7. スマホからログインできますか？**
A. **可能ですが推奨しません**。POSLA は店舗備品の PC / タブレット での運用を前提に設計されています。個人スマホで管理画面を触ると勤怠 GPS や監査ログが個人端末に紐付き混乱します。スタッフのシフト確認のみ個人スマホでも OK。

**Q8. ログインボタンを連打したら 2 重リクエストになりますか？**
A. なりません。`index.html` の JS が `loginBtn.disabled = true` で連打を防止し、テキストを「ログイン中...」に変えます。

**Q9. ブラウザのパスワードマネージャー（Chrome / 1Password 等）に保存できますか？**
A. **保存できます**。`autocomplete="username"` / `autocomplete="current-password"` 属性を入れているので、パスワードマネージャーが自動補完します。共有端末では保存しないこと。

**Q10. ログイン後にエラーが出ても再試行できますか？**
A. はい。エラーメッセージが表示された後でも、入力欄をクリックすれば自動的にエラー表示が消え、再試行できます（`hideError()` ハンドラ）。

### C. パスワード（Q11〜Q15）

**Q11. パスワード要件は？**
A. **8 文字以上 + 英字 1 文字以上 + 数字 1 文字以上**（`api/lib/password-policy.php` `validate_password_strength()`）。OK 例: `tanaka123` / `Yamada2026` / `shibuya2024`。NG 例: `abcdefgh`（数字なし） / `12345678`（英字なし） / `abc123`（短い）。

**Q12. パスワード変更後に他端末はどうなる？**
A. **自動的にログアウトされます**（次回 API 呼び出し時に `SESSION_REVOKED` 401）。これはセキュリティ仕様です（漏洩疑い時に即座に他端末を遮断するため）。

**Q13. パスワードリセットメールは届きますか？**
A. **届きません**。メール経由のリセット機能は意図的に未実装です（メールアドレスを持たないスタッフが多い前提）。リセットは上司（owner / manager）が直接 PIN リセットと同じ画面から行います。

**Q14. 同じパスワードを複数アカウントで使い回してもよい？**
A. **NG です**。1 人ごとに独自パスワードを設定してください。漏洩時の被害を最小化するためです。

**Q15. デフォルトの初期パスワードを「password」のように単純にしてもよい？**
A. システム側で `password` は弾かれます（数字なし → 8 文字未満ではないが英字のみ）。とはいえ初期パスワードでも **8 + 英字 + 数字** ポリシーを満たす必要があり、推測しやすいものは避けてください。

### D. PIN（CP1）（Q16〜Q20）

**Q16. PIN を設定しないとレジが使えませんか？**
A. **使えません**（CB1d 以降）。POSレジ会計・返金・レジ開け/閉め・現金入金/出金 の **全 6 操作** で PIN 必須化されました。事前に該当スタッフ全員に PIN を設定してください。

**Q17. PIN は誰が設定する？**
A. **自分の PIN は誰でも自分で設定可**。他人の PIN は **manager 以上** が設定可。staff が他人を設定しようとすると `FORBIDDEN`（403）。

**Q18. 退勤打刻したスタッフの PIN はどうなる？**
A. **自動無効化**されます（CP1 設計のキモ）。PIN 検索クエリは `attendance_logs.clock_out IS NULL`（出勤中）のスタッフのみを対象とするため、退勤後は PIN が一致しません。退職者の悪用も物理的に防げます。

**Q19. PIN を忘れたら？**
A. 自己リセット不可。owner / manager に依頼して再設定してもらう。owner も忘れたら POSLA 運営へ。

**Q20. device アカウントには PIN を設定できますか？**
A. **できません**。`api/store/set-cashier-pin.php` で `target.role === 'device'` のとき `VALIDATION` エラーで弾かれます（人間ではないため）。device 端末でも会計時は PIN 入力で「実際に操作した人間スタッフ」を特定する設計です。

### E. セッション・マルチデバイス（Q21〜Q25）

**Q21. 同じアカウントで PC とタブレットの両方からログインできますか？**
A. **できます**。`user_sessions` テーブルが各セッションを個別に追跡し、login.php が 2 件以上検知時にレスポンスに `warning: '他の端末でもログインしています'` を返します（UI 表示は未実装、サーバ側で監査可能）。

**Q22. セッションタイムアウトは？**
A. **30 分の無操作で自動ログアウト**。25 分時点で警告モーダル表示。Cookie TTL は **8 時間**。アクティブな端末（API を頻繁に呼ぶ KDS / レジ / ハンディ）は事実上タイムアウトしません。

**Q23. リモートログアウト（他端末を強制ログアウト）はできる？**
A. API は実装済（`DELETE /api/store/active-sessions.php`、manager 以上）。**UI は 2026-04-19 時点で未実装**のため、運営が `curl` で対応する運用です。

**Q24. セッションは何で識別されますか？**
A. PHP セッション ID（`session_id()`）+ `tenant_id` で `user_sessions` テーブルに記録。`device_label`（User-Agent から `Chrome/Mac` 等を抽出）と `ip_address`（`REMOTE_ADDR` のみ。`X-Forwarded-For` は信頼しない）も保存します。

**Q25. パスワード変更で他端末を遮断する仕組みは？**
A. パスワード変更 API が成功時に同じ `user_id` の `user_sessions.is_active = 0` を一括更新。次回 API 呼び出し時に `_check_session_validity()` が `SESSION_REVOKED` 401 を返し、JS が `/admin/index.html` にリダイレクトします。

### F. 運用・トラブル（Q26〜Q30）

**Q26. 退職者のアカウントはどうすればいい？**
A. **削除ではなく無効化**（`is_active=0`）してください。削除すると過去の監査ログ・勤怠記録の `username` 表示が崩れます。無効化なら履歴は保持されつつログイン不可になります。

**Q27. ログイン履歴を確認できますか？**
A. はい。`audit_log` テーブルの `event='login'` レコードで「いつ・誰が・どの IP からログインしたか」を確認できます。閲覧は管理ダッシュボード「店舗設定 → 監査ログ」サブタブから（manager 以上）。

**Q28. 不審なログイン履歴を見つけたら？**
A. ①該当ユーザーのパスワードを即変更（自分で変更すると他端末は自動ログアウト）②`active-sessions.php` で全セッションをリモートログアウト（API、運営経由）③IP アドレスを確認、心当たりがなければ POSLA 運営に連絡。

**Q29. 多要素認証（MFA / 2FA）はありますか？**
A. 現状未対応（2026-04-19 時点）。Phase 2 で SMS / TOTP の検討予定。当面は強パスワード + 1 人 1 アカウント + PIN（CP1）+ device ロール分離で運用してください。

**Q30. ログイン画面のロゴ・配色をカスタマイズできますか？**
A. 現状は POSLA 共通デザイン固定です。テナントごとのブランディングは将来機能（カスタムドメイン対応と合わせて検討）。
