# 2. ログインとアカウント

## 2.1 初回ログインの手順

### 事前準備

ログインするには、オーナーまたはマネージャーが事前にアカウントを作成している必要があります。アカウント作成時に以下が決まります。

- **ユーザー名**（店舗内で一意。例：tanaka、suzuki、yamada01、kds-shibuya-01）
- **初期パスワード**（作成者が設定し、口頭やメモでスタッフに伝えます）
- **ロール**（owner / manager / staff / **device**）
- **所属店舗**（ownerは全店舗、manager/staff/deviceは指定された店舗のみ）

POSLAのアカウントには **4種類のロール** があります。

| ロール | 用途 | ログイン後の遷移先 | 例 |
|---|---|---|---|
| **owner** | テナントオーナー（全店舗統括） | オーナーダッシュボード | 経営者本人 |
| **manager** | 店長（担当店舗の管理） | 管理ダッシュボード | 田中店長 |
| **staff** | 一般スタッフ（個人ログイン） | 管理ダッシュボード or スタッフ画面 | ホール鈴木さん、調理山田さん |
| **device** | KDS / レジ端末専用（人間ではない） | KDS画面 or POSレジ画面に直接 | 厨房KDS-1、レジ-1 |

::: tip メールアドレスは不要です
POSLAのログインにメールアドレスは必要ありません。飲食店スタッフがメールアドレスを持っていない・教えたくないケースに配慮した設計です。ユーザー名とパスワードだけでログインできます。
:::

::: warning device ロールは特殊です
device は「人間スタッフ」ではなく「端末そのもの」に与えるアカウントです。シフト・人件費・スタッフ評価レポートには含まれません。詳しくは **2.5 デバイスアカウント** を読んでください。
:::

### ログイン手順

1. ブラウザで管理画面のURLにアクセスします
   - URL例：`https://あなたのドメイン/admin/`
2. ログイン画面が表示されます
3. **ユーザー名**欄に、アカウント作成時に決めたユーザー名を入力します
   - 例：`shibuya-staff`
4. **パスワード**欄に、初期パスワードを入力します
5. **ログイン**ボタンを押します
6. ログインに成功すると、自動的にダッシュボードに遷移します
   - オーナー → オーナーダッシュボード
   - マネージャー・スタッフ → 管理ダッシュボード

### ログインに失敗した場合

| エラーメッセージ | 原因 | 対処法 |
|----------------|------|--------|
| 「ユーザー名またはパスワードが正しくありません」 | ユーザー名かパスワードの入力ミス | 入力内容を確認してやり直す。大文字/小文字の区別に注意 |
| 「このアカウントは無効化されています」 | アカウントが無効化された | オーナーまたはマネージャーに連絡してアカウントを有効化してもらう |
| 「このテナントは無効化されています」 | テナント全体が無効化された | POSLA運営に連絡する |
| 「通信エラーが発生しました」 | インターネット接続の問題 | ネットワーク接続を確認して再試行する |
| 「サーバーが空のレスポンスを返しました」 | サーバー側の問題 | 少し待ってから再試行する。繰り返す場合はPOSLA運営に連絡 |

## 2.2 パスワード変更

パスワード変更には3つの経路があります。

### 自分のパスワードを変更する（推奨）

各ユーザーは画面右上の**「パスワード変更」ボタン**から自分のパスワードを変更できます。

1. ログイン後、画面右上のユーザー名の横にある**「パスワード変更」**ボタンを押す
   - 管理ダッシュボード（manager / staff）：右上の「ログアウト」ボタンの隣
   - オーナーダッシュボード（owner）：右上の「ログアウト」ボタンの隣
2. パスワード変更モーダルが開きます：
   - **現在のパスワード**：いま使っているパスワードを入力
   - **新しいパスワード**：新しいパスワードを入力（パスワードポリシー要件あり、下記）
   - **新しいパスワード（確認）**：もう一度同じものを入力
3. **「変更する」**を押す
4. 成功メッセージが表示され、画面はそのまま使えます
5. **他の端末でログイン中のセッションは自動的にログアウトされます**（いま使っている端末はそのまま継続）

::: tip パスワードポリシー
新しいパスワードは以下の要件を全て満たす必要があります：

- **8文字以上**
- **英字を1文字以上**含む（A-Z または a-z）
- **数字を1文字以上**含む（0-9）

例：`tanaka123`、`Yamada2026`、`shibuya2024`

要件を満たさない場合は「パスワードは8文字以上で、英字と数字を含めてください」というエラーが表示されます。
:::

::: warning 現在のパスワードが分からない場合
何度試してもログインできない場合や、現在のパスワードを忘れてしまった場合は、自分でパスワードを変更することはできません。下記の「オーナー/マネージャーが他のユーザーのパスワードを変更する」手順で、上司にリセットを依頼してください。
:::

### オーナーがスタッフのパスワードを変更する場合

スタッフがパスワードを忘れた場合や、新規スタッフの初期パスワードを設定する場合の手順です。

1. オーナーダッシュボードの「ユーザー管理」タブを開く
2. 対象のユーザーの**編集ボタン**を押す
3. 編集モーダルが開く
4. 「パスワード」欄に新しいパスワードを入力する（パスワードポリシーを満たすこと）
5. **保存**ボタンを押す
6. パスワードが更新される（新しいパスワードを口頭でスタッフに伝えてください）

### マネージャーがスタッフのパスワードを変更する場合

1. 管理ダッシュボードの「店舗設定」→「スタッフ管理」サブタブを開く
2. 対象のスタッフの**編集ボタン**を押す
3. 「パスワード」欄に新しいパスワードを入力する（パスワードポリシーを満たすこと）
4. **保存**を押す

::: warning 注意
マネージャーが変更できるのは自店舗のスタッフのパスワードのみです。他店舗のスタッフや他のマネージャーのパスワードは変更できません。
:::

### パスワード設定の運用ルール

- 推測されやすいパスワード（`12345678`、`password`、`tanaka2024` のような単純なパターン）は避けてください
- スタッフ間でパスワードを使い回さないでください
- 他人にパスワードを教えないでください（POSLA運営も問い合わせません）
- 退職者のパスワードは速やかにオーナーがリセット、またはアカウント自体を無効化してください

## マルチデバイスログイン制御

POSLAでは、同じアカウントで複数の端末（PC、タブレット、スマートフォン等）から同時にログインできます。ただし、以下の制御が動作します。

### 複数端末ログイン時の動作

1. ログイン時に、同じアカウントが他の端末でもログイン中かをシステムが自動チェックします
2. 他の端末でもログイン中の場合、ログイン直後に**警告メッセージ**が表示されます
   - 「他の端末でもログインしています」
3. この警告は情報提供のみです。ログイン自体はブロックされません
4. アクティブなセッション数も表示されます（例：「アクティブセッション: 3」）

### セッションタイムアウト

POSLAのセッションには有効期限があります。

- **セッション有効時間：8時間**
- **操作なし25分で警告モーダル表示：**「セッションタイムアウト — 5分以内に操作がないとログアウトされます」
- 警告モーダルの**「操作を続ける」ボタン**を押すと、セッションがリセットされます
- 5分以内に操作がない場合、自動的にログアウトされ、ログイン画面にリダイレクトされます
- 通常の操作（クリック、キー入力、スクロール、タッチ）を行っている限り、タイマーは自動リセットされます

### アクティブセッション管理（リモートログアウト）

マネージャー以上のロールは、他の端末のセッションをリモートでログアウトさせることができます。

1. 管理ダッシュボードの「店舗設定」→「監査ログ」タブ、または専用のセッション管理画面を開く
2. アクティブセッション一覧が表示されます
   - セッションごとに：端末情報（ブラウザ/OS）、IPアドレス、ログイン日時、最終操作日時
3. 不要なセッションの**ログアウト**ボタンを押す
4. 確認ダイアログで「はい」を押す
5. 対象セッションが無効化され、その端末は次回操作時にログイン画面にリダイレクトされます

::: tip 使いどころ
- スタッフが退勤後にログアウトし忘れた場合
- 紛失した端末のセッションを無効化したい場合
- 不審なセッションを検出した場合
:::

## 2.4 ログアウト

ログアウトは画面右上の**ログアウトボタン**から行います。

1. 画面右上のユーザー名の横にある**ログアウト**ボタンを押す
2. セッションが終了し、ログイン画面に戻ります

::: warning 個人スタッフアカウントでKDS/レジ端末に常時ログインしないでください
KDS（キッチンディスプレイ）やPOSレジは厨房・レジに据え置かれた**共有端末**です。これに田中さん個人のアカウントで常時ログインしておくと、誰が会計したか・誰が調理完了を押したかが全部「田中」名義で記録されてしまいます。勤怠打刻・監査ログ・スタッフ評価レポートが全部おかしくなります。

**正しい運用は「デバイスアカウント」を作ること**です。次の **2.5 デバイスアカウント** を読んでください。
:::

## 2.5 デバイスアカウント（KDS / レジ端末専用）

### 2.5.1 デバイスアカウントとは

POSLA には、人間スタッフのアカウントとは別に **「デバイスアカウント」** という端末専用ログインがあります。

- ロール: `device`（owner / manager / staff とは独立した第4のロール）
- 用途: **KDS タブレット 1台 = 1 デバイスアカウント**、**POSレジ端末 1台 = 1 デバイスアカウント**
- ログインしっぱなしを前提とした設計（共有端末を想定）
- **シフト・人件費・スタッフ評価レポートには一切含まれない**（人間ではないので）
- 監査ログにはちゃんと記録される（`role='device'` で識別可能）

::: tip なぜ「デバイス専用アカウント」が必要なのか
店内オペレーションには2種類のアカウントが必要だからです。

| 種類 | 例 | 端末 | ログイン頻度 |
|---|---|---|---|
| **個人スタッフ** | 田中さん、鈴木さん | ハンディPOS（自分のスマホ or 共有iPod touch） | 出勤時にログイン → 退勤時にログアウト |
| **デバイス** | KDS-1、KDS-2、レジ-1 | 厨房タブレット、レジカウンタータブレット | **常時ログインしっぱなし**（再起動時のみ再ログイン） |

KDS タブレットに「田中」でログインしたまま田中さんが退勤すると、夜勤の鈴木さんが触っても全部「田中」の操作として記録されます。これを防ぐために、端末そのものに「KDS-1」というアカウントを与えておく、という発想です。
:::

### 2.5.2 個人スタッフアカウントとの違い

| | スタッフアカウント | デバイスアカウント |
|---|---|---|
| ロール | `staff` | `device` |
| 想定ユーザー | 人間1人 | 端末1台 |
| ログイン後の遷移先 | 管理ダッシュボード（dashboard.html） | **KDS画面 または POSレジ画面に直接遷移**（管理画面には入れない） |
| 使えるアプリ | ハンディPOS / KDS / POSレジ（個別設定可） | **KDS / POSレジのみ**（ハンディは使えない） |
| シフト管理対象 | ✅ | ❌ |
| 人件費計算対象 | ✅ | ❌ |
| スタッフ評価レポート | ✅ | ❌ |
| 出退勤打刻 | ✅ | ❌（端末は出勤・退勤しないため） |
| 監査ログ記録 | ✅（`role='staff'`） | ✅（`role='device'`） |
| 作成権限 | オーナー・マネージャー | オーナー・マネージャー |
| 1アカウントの単位 | 人間1人 | 端末1台 |

### 2.5.3 デバイスアカウントの作成手順

オーナー、または店長（マネージャー）が作成します。スタッフロールでは作成できません。

1. **管理ダッシュボード**にログイン（マネージャー or オーナー）
2. 上部タブから **「店舗設定」** を開く
3. サブタブから **「スタッフ管理」** を選択
4. 画面を下にスクロールすると **「デバイスアカウント（KDS / レジ端末）」** というセクションがあります
5. **「+ デバイス追加」** ボタンを押す
6. 「デバイスを追加」モーダルが開きます
7. 以下の項目を入力します:

| 項目 | 説明 | 例 |
|---|---|---|
| **表示名** | この端末に付ける呼び名（管理画面の一覧に出る） | `厨房KDS 1番`、`バー側KDS`、`レジ1`、`銀座店レジ` |
| **ユーザー名** | ログイン時に入力する名前。半角英数字・ハイフン・アンダースコアのみ、3〜50文字 | `kds-shibuya-01`、`register-shibuya-01`、`kds-bar` |
| **パスワード** | 8文字以上、英字+数字を含む | `kdsShibuya2026` |
| **用途** | チェックボックスで KDS / POSレジ を選択（少なくとも1つ） | KDS のみ、または POSレジ のみ、または両方 |

8. **「保存」** を押す
9. 一覧に追加されたら作成完了です

::: tip ユーザー名の付け方の推奨
後から「どの端末がどのアカウントだっけ？」と分からなくなるので、ユーザー名に **「役割 + 店舗 + 番号」** を含めると運用が楽です。

- ✅ `kds-shibuya-01`（渋谷店のKDS 1番）
- ✅ `register-ginza-01`（銀座店のレジ 1番）
- ✅ `kds-bar-shinjuku`（新宿店バー側KDS）
- ❌ `tablet1`（どの店舗の何の端末か不明）
- ❌ `kds`（複数店舗で使えない）

表示名は日本語OKなので、`厨房KDS 1番` のように人間向けの名前を入れておくと管理画面で見やすくなります。
:::

### 2.5.4 用途（KDS / POSレジ）の選択

「用途」チェックボックスで、この端末アカウントがログインしたときにどの画面を開くかを決めます。

| 用途の組み合わせ | ログイン後の遷移先 |
|---|---|
| **KDS のみ** ☑ | `kds/index.html`（カンバン画面） |
| **POSレジ のみ** ☑ | `kds/cashier.html`（レジ画面） |
| **両方** ☑☑ | `kds/cashier.html`（POSレジを優先） |

::: warning 1端末1用途を強く推奨
「両方チェック」も技術的には可能ですが、運用上は **1端末1用途**（KDS だけ、または POSレジ だけ）にしてください。

- KDS と レジ は画面構成・操作頻度・設置場所が全く違います（厨房 vs レジカウンター）
- 同じ端末で両方やると、KDS 画面表示中にお会計が来て切り替える→ピックアップ画面が消える、という事故が起きます
- 「KDS-1」「レジ-1」と物理的にも別端末・別アカウントにしてください
:::

### 2.5.5 デバイスアカウントでログインしたときの動作

1. 通常のログイン画面（`/admin/`）でデバイスアカウントのユーザー名とパスワードを入力
2. ログイン成功後、**管理ダッシュボード（dashboard.html）には入らず**、即座に KDS または POSレジ画面にリダイレクトされます
3. 以降は **画面が自動で更新**されるので、原則ログアウト不要
4. 端末を再起動・ブラウザを閉じた場合は再ログインが必要です

::: tip デバイスアカウントは管理画面に絶対入れません
これはセキュリティと UX の両面で意図的な設計です。

- KDS タブレットからメニュー編集や売上レポートに触れないようにする（誤操作・店内利用客から見られるリスクを排除）
- 厨房スタッフが「あれ？管理画面どこ？」と迷子にならないようにする
- 万が一デバイスアカウントのパスワードが流出しても、管理データには到達できない

管理画面を操作したいときは、必ず人間のオーナー/マネージャーアカウントで別途ログインしてください。
:::

### 2.5.6 どのスタッフが操作したか追跡したい場合

「KDS-1 でログインしっぱなしなら、誰が調理完了を押したか分からないのでは？」という疑問が出ますが、POSLA では **KDS の調理完了ボタン操作にスタッフ個人の紐付けは行いません**（そういう設計です）。

代わりに以下で個人を追跡します:

| 追跡したいこと | どこで取れるか |
|---|---|
| 誰が会計したか（POSレジ） | レジ画面で**スタッフ名を選択**してから会計するため、orders.staff_id に記録されます（POSレジ画面左上のスタッフ選択） |
| 誰が注文を取ったか（ハンディ） | ハンディPOSは個人スタッフアカウントでログインするため、自動記録 |
| 誰がいつ出退勤したか | 個人スタッフアカウントの**勤怠打刻**で記録（GPS付き） |
| 誰が管理操作をしたか | オーナー/マネージャー個人アカウントの**監査ログ** |

KDS の調理完了ボタンはスループット計測用（料理が出るまでの時間）であって、評価対象にはなりません。

### 2.5.7 デバイスアカウントの編集・削除

#### 編集

1. 管理ダッシュボード → 店舗設定 → スタッフ管理 → デバイスアカウントセクション
2. 対象のデバイスの **「編集」** ボタンを押す
3. 以下を変更可能:
   - 表示名
   - ユーザー名
   - **新しいパスワード**（空欄なら変更なし）
   - 用途（KDS / POSレジ チェックボックス）
   - **有効** トグル（OFF にすると即座にログイン不可になる）
4. **「保存」** を押す

#### 削除

1. 対象のデバイスの **「削除」** ボタンを押す
2. 確認ダイアログで「OK」を押す
3. アカウントが削除され、その端末は次回操作時にログイン画面に飛ばされます

::: warning 削除前にやること
削除する前に、対象端末がいま稼働中でないか確認してください。営業時間中に削除すると、その端末は突然ログアウトされて KDS / レジが止まります。

- 営業時間外の作業を推奨
- どうしても営業時間中に削除する必要がある場合は、まず代替端末を用意してから
:::

### 2.5.8 デバイスアカウント運用ルール

#### 必須ルール

- ✅ **1端末 = 1アカウント**（KDS-1 と KDS-2 を同じアカウントで使い回さない）
- ✅ **役割が違えばアカウントも分ける**（KDS と POSレジ は別アカウント）
- ✅ **店舗が違えばアカウントも分ける**（渋谷店のKDS-1 と 新宿店のKDS-1 は別アカウント、ユーザー名に店舗名を入れる）
- ✅ **パスワードは店舗の鍵と同じ重さで管理**（誰でも触れるところにメモしない）

#### 推奨ルール

- 💡 **半年〜1年に1回はパスワード変更**（人間アカウントと同じく）
- 💡 **端末が壊れた・廃棄するときは即座にアカウント削除**（または「有効」OFF）
- 💡 **端末交換時は新しいアカウントを作って古いものを削除**（同じアカウントを引き継がない）

#### やってはいけないこと

- ❌ デバイスアカウントを **個人スタッフのスマホ**にログインさせる（個人スタッフは個人アカウントで）
- ❌ 1つのデバイスアカウントで **複数の端末**にログインする（管理画面側からどれが本物か区別できなくなる）
- ❌ デバイスアカウントの **パスワードを口頭で全スタッフに伝える**（必要ありません。ログインは初回設置時の1回だけ）

### 2.5.9 デバイスアカウントに関するよくある質問

**Q1. 既存のスタッフアカウントを KDS タブレットでログインしっぱなしにしてはダメですか？**

ダメではないですが、強く非推奨です。シフト・勤怠・スタッフ評価レポートが全部その人名義になってしまうため、シフト管理が機能しなくなります。デバイスアカウントを作ってください。

**Q2. デバイスアカウントは何個まで作れますか？**

上限はありません。店舗のKDS台数+レジ台数の分だけ作ってください。3店舗・各店KDS2台+レジ2台なら全部で12個になります。

**Q3. デバイスアカウントを作っても、シフト管理や人件費レポートに影響しないですか？**

影響しません。`role='device'` のユーザーはシフト・勤怠・人件費・スタッフ評価レポートのSQLクエリで完全に除外されます。

**Q4. デバイスアカウントは監査ログに残りますか？**

残ります。`audit_log.role='device'` で識別できるので、「KDS-1 がメニュー売り切れを ON にした」のような操作も追跡可能です。ただし KDS / POSレジ で発生する操作は限定的（注文ステータス変更、会計、売り切れ切替など）なので、人間アカウントほど大量の監査ログは出ません。

**Q5. デバイスアカウントでハンディPOSにログインできますか？**

できません。デバイスアカウントは「用途」が KDS / POSレジ のみで、ハンディはチェックできない設計です。ハンディPOSは必ず**個人スタッフアカウント**でログインしてください（誰が注文を取ったか記録するため）。

**Q6. デバイスアカウントは何のロールで作成できますか？**

オーナー・マネージャーが作成できます。スタッフロールには作成権限がありません。

**Q7. 桃の屋のような複数店舗で、本部から全店舗のデバイスアカウントを一元管理できますか？**

オーナーアカウントなら全店舗のデバイスアカウントを管理できます。マネージャーアカウントは自分の所属店舗のデバイスのみ管理可能です。

**Q8. デバイスアカウントのパスワードを忘れたら？**

オーナー/マネージャーが「編集」画面から新しいパスワードを設定できます。デバイスアカウント自身が「自分でパスワードを変更する」機能は提供していません（人間ではないため）。

**Q9. 端末が盗まれたらどうすればいいですか？**

すぐに管理ダッシュボードからそのデバイスアカウントを **「有効」OFF** にするか、**削除**してください。盗難端末からはログイン画面に飛ばされ、それ以降ログインできなくなります。

**Q10. デバイスアカウントの「セッション有効期限」はどうなっていますか？**

人間アカウントと同じ8時間の有効期限が設定されていますが、KDS / POSレジ画面は3秒ごとに自動でAPIをポーリングしているため、営業時間中は事実上セッションが切れません。再起動した翌朝のみ再ログインが必要です。

### 2.5.10 デバイスアカウントのトラブルシューティング

| 症状 | 原因 | 対処 |
|---|---|---|
| デバイスでログインしたら管理画面が出てしまった | スタッフアカウントでログインしている | 一度ログアウトし、デバイスアカウント（`kds-xxx` 等）でログインし直す |
| デバイスアカウントでログインしてもKDSに行かずログイン画面に戻る | 「用途」に何も選択していない・visible_tools が空 | 編集画面を開いて KDS か POSレジ にチェックを入れて保存 |
| 「+ デバイス追加」ボタンが見当たらない | スタッフロールでログインしている、または「店舗設定」タブにいない | マネージャーかオーナーで再ログインし、店舗設定 → スタッフ管理 を開く |
| デバイスアカウントを削除したのに端末がまだ使えている | ブラウザにキャッシュされたページが残っている | 端末でページを再読み込みすると即座にログイン画面に飛ぶ |
| 「このユーザー名は既に使用されています」 | 他店舗のデバイスや人間アカウントと重複 | ユーザー名に店舗名を入れて重複を避ける（例: `kds-01` → `kds-shibuya-01`） |
| デバイスアカウントがシフト管理画面に出てこない | 仕様通り（シフト対象外） | これは正常動作。シフト・人件費レポートにも入りません |

### 2.5.11 内部仕様（POSLA運営向け）

::: tip 通常テナントは読まなくてOK
ここから先はデバッグ・問い合わせ対応用の技術詳細です。一般のオーナー・マネージャーは読み飛ばしてください。
:::

#### データベース構造

```sql
-- P1a マイグレーション (2026-04-09)
ALTER TABLE users
    MODIFY COLUMN role ENUM('owner','manager','staff','device') NOT NULL DEFAULT 'staff';
ALTER TABLE audit_log
    MODIFY COLUMN role ENUM('owner','manager','staff','device') DEFAULT NULL;
```

`users.role='device'` のレコードを `user_stores` 経由で店舗に紐付けることで「この店舗のデバイス」として管理します。`visible_tools` は `kds` / `register` / `kds,register` のいずれか。

#### API エンドポイント

スタッフ管理APIを `?kind=device` パラメータで再利用しています:

| HTTP | URL | 用途 |
|---|---|---|
| GET | `/api/store/staff-management.php?store_id=xxx&kind=device` | デバイス一覧 |
| POST | `/api/store/staff-management.php?kind=device` | デバイス作成 |
| PATCH | `/api/store/staff-management.php?id=xxx&kind=device&store_id=xxx` | デバイス更新 |
| DELETE | `/api/store/staff-management.php?id=xxx&kind=device&store_id=xxx` | デバイス削除 |

`require_role('manager')` で権限チェック。`kind=device` を指定すると `users.role='device'` の絞り込みになります。

#### ログイン後リダイレクトロジック

`public/admin/dashboard.html` 内（L469 付近）:

```javascript
// P1a: device ロールは KDS / レジ画面に直接リダイレクト
if (_role === 'device') {
  var devTools = (firstStore.userVisibleTools !== null)
    ? firstStore.userVisibleTools
    : firstStore.staffVisibleTools;
  if (devTools.indexOf('register') >= 0) {
    window.location.href = '../kds/cashier.html';
  } else {
    window.location.href = '../kds/index.html';
  }
  return;
}
```

`visible_tools` の優先順位: `user_stores.visible_tools`（個別設定）> 店舗デフォルト > なし。`register` が含まれていれば POSレジ、それ以外は KDS。

#### シフト管理から除外される仕組み

`api/store/shift/*.php` の各クエリは `WHERE u.role IN ('staff','manager','owner')` のように `role` で絞り込んでいるため、`device` ロールは自然に除外されます。

#### 監査ログ

`write_audit_log()` は `$user['role']` をそのまま `audit_log.role` に書き込むので、デバイスアカウントの操作は `role='device'` で記録されます。POSLA運営は監査ログ検索で `role='device'` フィルタを使えば「端末操作のみ」を抽出できます。
