# 5. テーブル・フロアマップ管理

管理ダッシュボードの「テーブル・フロア」タブでは、店舗のテーブル登録とフロアマップ（リアルタイム状況表示）を管理します。このタブは `manager` 以上のロールで利用できます。

---

## 5.0 はじめにお読みください

::: tip 章の読み方ガイド
- **新規開店**: 5.1 テーブル登録 → 5.1.6 QR 印刷 が中心
- **日常運用**: 5.2 フロアマップ + 5.3 セッション
- **相席運用**: 5.Y2 サブセッション QR（F-QR1）
- **客側 QR からの再開放**: 25 章 25-table-auth と合わせて読む
- **困ったとき**: 5.Y トラブルシューティング → 5.Z FAQ
:::

### 5.0.1 操作前のチェックリスト

| チェック項目 | 確認方法 |
|---|---|
| ログインしているロールが `owner` または `manager` であること | 画面右上のユーザーバッジ |
| 操作対象店舗が選択されていること | ヘッダーの店舗セレクター |
| QR を発行する場合、本番ドメインで動いているか | URL が `https://eat.posla.jp/...` か |
| フロアマップを使う場合、Wi-Fi が安定しているか | 5 秒ポーリングするため |
| 相席 QR を発行する場合、メインセッションが既に開放済みか | フロアマップで「着席」状態か |
| device ロール端末（ハンディ）から QR 印刷する場合は事前に GPS 許可 | `_user.role === 'device'` は GPS 強制 |

::: warning ロール権限
- `staff` ロールはテーブル管理 UI に到達できません（フロアマップ閲覧のみ）
- `device` ロール（KDS / レジ端末）はフロアマップ + サブセッション発行のみ
- テーブル CRUD（追加・編集・削除）は `manager` 以上
:::

## 5.1 テーブル管理

### 5.1.1 テーブル管理画面の構成

「テーブル・フロア」タブ → 「テーブル管理」サブタブを選択すると、登録済みテーブルの一覧が表示されます。

```
[テーブル管理] サブタブ
┌──────────────────────────────────────────────────────────────────┐
│  テーブル (12)                                  [+ テーブル追加]  │
├──────────────────────────────────────────────────────────────────┤
│  T01  [有効]   定員: 4名   [QR][個別QR][編集][削除]              │
│  T02  [有効]   定員: 4名   [QR][個別QR][編集][削除]              │
│  T03  [有効]   定員: 6名   [QR][個別QR][編集][削除]              │
│  C01  [有効]   定員: 1名   [QR][個別QR][編集][削除]   ← カウンター│
│  X01  [無効]   定員: 2名   [QR][個別QR][編集][削除]              │
└──────────────────────────────────────────────────────────────────┘
```

画面上部には登録テーブル数がカッコ内に表示されます（例：`(12)`）。テーブルが 1 つも登録されていない場合は「🪑 テーブルがありません」と表示されます。

各テーブルは以下の情報を持つカードとして表示されます。

| 表示要素 | 説明 |
|---------|------|
| テーブルコード | T01、A-1 などの識別コード |
| ステータスバッジ | 緑色「有効」/ 赤色「無効」 |
| 座席数 | 「4 名」のように表示 |
| QR ボタン | 通常 QR コードを表示 |
| 個別 QR ボタン（オレンジ） | サブセッション QR（F-QR1、5.Y2 参照） |
| 編集ボタン | テーブル情報を編集 |
| 削除ボタン | テーブルを削除 |

### 5.1.2 テーブルの追加

1. **「+ テーブル追加」**ボタンをクリックします
2. モーダルが開きます（タイトル：「テーブルを追加」）
3. 以下のフィールドに入力します

| フィールド | 種類 | 必須 | 初期値 | 説明 |
|-----------|------|------|--------|------|
| テーブルコード | テキスト入力 | ✅ | — | 店舗内で一意のコード（例：T01、A-1、カウンター 1）。プレースホルダーは「T01」 |
| 定員 | 数値入力 | — | 4 | テーブルの定員人数（capacity） |

4. **「保存」**ボタンをクリックします
5. 成功すると「テーブルを追加しました」トーストが表示され、一覧が更新されます

**バリデーション：**

- テーブルコードが空の場合：「テーブルコードは必須です」エラー
- 同じ店舗に同じコードが既にある場合：「このテーブルコードは既に使用されています」エラー（HTTP 409）
- 定員に文字列を入れた場合：自動で `parseInt` され、NaN なら 4 にフォールバック

**テーブル作成時に自動的に行われること：**

- UUID 形式の一意な ID が自動生成されます
- セッショントークン（64 文字のランダムな 16 進数文字列）が自動生成されます
- セッショントークンの有効期限が **4 時間後**に設定されます
- フロアマップ用の初期位置（X=0, Y=0）、サイズ（幅 80px, 高さ 80px）、形状（rect = 四角形）、フロア（1F）が自動設定されます

### 5.1.3 テーブルコードの命名規則（実例）

| 命名 | 例 | 用途 |
|---|---|---|
| 連番方式 | T01, T02, T03... | 小規模店舗 |
| エリア方式 | A-1, A-2, B-1, B-2... | 複数フロア / エリア分割（matsunoya 渋谷店） |
| 用途方式 | カウンター 1, テラス 1, 個室 A | 特殊テーブル |
| 番号 + 定員 | T04-4P, T06-6P | 大人数テーブルを目立たせる |

::: tip 実例: torimaru 銀座店
- カウンター席 8 席：`C01〜C08`（capacity=1、相席不可）
- テーブル席 6 卓：`T01〜T06`（capacity=4）
- 個室 2 室：`R01〜R02`（capacity=8、予約専用運用）
:::

### 5.1.4 テーブルの編集

1. 一覧で対象テーブルの**編集**ボタンをクリックします
2. モーダルが開きます（タイトル：「テーブルを編集」）
3. 以下のフィールドを変更できます

| フィールド | 種類 | 説明 |
|-----------|------|------|
| テーブルコード | テキスト入力 | 新しいコードに変更可能 |
| 定員 | 数値入力 | 定員の変更 |
| 有効 | チェックボックス | チェックを外すとテーブルが無効になり、フロアマップやセルフオーダーに表示されなくなります |

4. **「保存」**ボタンをクリックします
5. 成功すると「テーブルを更新しました」トーストが表示されます

### 5.1.5 テーブルの削除

1. 一覧で対象テーブルの**削除**ボタンをクリックします
2. 確認ダイアログが表示されます：「「T01」を削除しますか？」
3. 「OK」をクリックすると削除されます
4. 成功すると「テーブルを削除しました」トーストが表示されます

::: danger 削除ではなく無効化を推奨
テーブルを削除すると、そのテーブルに紐づく注文履歴やセッション履歴に外部キー制約上の影響があります。

過去データを残したい場合は、削除ではなく**「有効」のチェックを外して無効化**することを強く推奨します（フロアマップ・セルフオーダーから消えるが過去データは保持）。
:::

### 5.1.6 QR コードの表示と印刷

各テーブルにはセルフオーダー用の QR コードが生成できます。

1. 一覧で対象テーブルの**QR**ボタンをクリックします
2. モーダルが開きます（タイトル：「T01 QR コード」）
3. QR コード画像（200×200 ピクセル）が表示されます
4. QR コードの下に URL が表示されます
5. 「閉じる」ボタンでモーダルを閉じます

**QR コードの URL 形式：**

```
https://eat.posla.jp/public/customer/menu.html?store_id={店舗ID}&table_id={テーブルID}
```

実装: `TableEditor.showQR()` 内で `window.location.origin + basePath + '/public/customer/menu.html?store_id=' + storeId + '&table_id=' + id` を生成。

お客さんがこの QR コードをスマートフォンのカメラで読み取ると、そのテーブル専用のセルフオーダー画面が開きます。

::: tip QR コードの活用方法
- 印刷してテーブルに貼付する（A6 / 名刺サイズ推奨）
- アクリルスタンドに入れてテーブルに設置する
- メニューブックの裏表紙に貼付
- テーブルごとに異なる QR コードが生成されるため、注文がどのテーブルからのものかを自動的に識別できます
:::

**印刷時の注意：**

| 注意点 | 内容 |
|---|---|
| 解像度 | 200×200 px。A6 サイズ印刷で十分視認可能 |
| ブラウザ印刷 | モーダルが開いた状態で Ctrl+P すると QR + URL が印刷される |
| PDF 化 | ブラウザ印刷で「PDF として保存」を選択 |
| ラミネート | 防水・破れ防止のためラミネート推奨（耐用年数 +1 年） |
| 一括印刷 | 全テーブル分を 1 ページにまとめる場合は、別途 owner 側「テーブル QR シート」機能 |

### 5.1.7 セッショントークン

QR コードにはテーブルのセッショントークン（`session_token`、64 文字 hex）が紐づいています。トークンは以下のタイミングで自動更新されます。

- テーブル作成時（4 時間の有効期限付き）
- お客さんの会計完了時（精算処理でトークンが再生成される）
- 着席キャンセル時（セッション強制終了でトークンが再生成される）
- `next_session_token` 方式の開放完了時（25 章参照）

これにより、前のお客さんの QR コードでは次のお客さんが注文できないようになっています（前 QR の盗用攻撃防御）。

**実装上の照合：**

サーバー側では `hash_equals($dbValue, (string)$inputValue)` でタイミング攻撃に強い比較を行っています（2026-04-18 監査で全エンドポイント適用済み）。

## 5.2 フロアマップ

### 5.2.1 フロアマップ画面の構成

「テーブル・フロア」タブ → 「フロアマップ」サブタブを選択すると、リアルタイムのテーブル状況一覧が表示されます。

```
┌──────────────────────────────────────────────────────────────────────┐
│ [サマリーバー]                                                       │
│  全 12卓  空き 3  使用中 7  未注文 2  合計 28名  ¥42,000  [LO中]    │
├──────────────────────────────────────────────────────────────────────┤
│ [テーブルカードグリッド]                                             │
│  ┌─T01─────────┐ ┌─T02─────────┐ ┌─T03─────────┐ ┌─T04─────────┐    │
│  │ 着席         │ │ 食事中       │ │ 空き         │ │ 会計待ち     │    │
│  │ 4 名         │ │ 3 名 25分    │ │ 4 名         │ │ 2 名 70分    │    │
│  │              │ │ 残り 65分    │ │              │ │ ¥4,200       │    │
│  │              │ │ ▓▓░░ 進捗バー│ │              │ │ [メモ][取消] │    │
│  └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘    │
└──────────────────────────────────────────────────────────────────────┘
```

**データは 5 秒ごとに自動更新されます**（`POLL_INTERVAL = 5000`）。他のタブに切り替えると自動更新（ポーリング）は停止し、フロアマップタブに戻ると再開します。

### 5.2.2 サマリーバー

画面上部にはテーブル状況のサマリーが表示されます。

| 項目 | 表示例 | 説明 | 色 |
|------|--------|------|---|
| 全テーブル数 | 「全 12 卓」 | 有効なテーブルの総数 | 灰 |
| 空き | 「空き 3」 | 使用中でないテーブル数 | 緑 |
| 使用中 | 「使用中 7」 | セッションがあるテーブル数 | 青 |
| 未注文 | 「未注文 2」 | 着席しているが注文がまだないテーブル数（0 の場合は非表示） | オレンジ |
| 合計人数 | 「合計 28 名」 | 現在の全テーブルのゲスト合計 | 灰 |
| 合計売上 | 「¥42,000」 | 現在の全テーブルの未会計注文合計 | 太字 |
| ラストオーダー状態 | 「ラストオーダー中」/ 「LO 22:00」 | LO 状態 | 赤 / 灰 |
| ラストオーダーボタン | 「ラストオーダー」/「LO 解除」 | LO を手動で発動 / 解除 | — |

### 5.2.3 テーブルカードの表示内容

各テーブルはカード形式で表示されます。テーブルの状態に応じてカードの左ボーダーと背景色が変わります。

#### ステータスと色分け（実装値）

| ステータス（内部キー） | 色 | 背景 | バッジ | 条件 |
|-----------|-----|---|--------|------|
| `empty` | #43a047 緑 | #e8f5e9 | 空き | セッションなし、注文なし |
| `seated` | #1e88e5 青 | #e3f2fd | 着席 | セッションあり、status=seated |
| `no_order` | #ff9800 橙 | #fff8e1 | 未注文 | seated だが注文 0 件 |
| `eating` | #1565c0 濃青 | #e3f2fd | 食事中 | status=eating または注文あり |
| `bill_requested` | #e53935 赤 | #ffebee | 会計待ち | status=bill_requested |
| `overtime` | #e65100 濃橙 | #fff3e0 | 時間超過 | 経過時間 > 制限時間（食べ放題プラン） |

#### カード内に表示される情報

**空きテーブルの場合：**
- テーブルコード
- 「空き」バッジ（緑色）
- 定員（「4 名」）

**使用中テーブルの場合：**
- テーブルコード
- ステータスバッジ（色分け）
- ゲスト数（「4 名」）
- 経過時間（「25 分経過」）
- 残り時間（食べ放題プラン適用時）：「残り 65 分」
  - 時間超過の場合は赤文字で「5 分超過」
- 注文件数（「2 件の注文」）
- 合計金額（「¥4,200」）
- 調理進捗バー（後述）
- メモ（設定されている場合）
- 「メモ」ボタン
- 「取消」ボタン

#### 調理進捗バー

使用中テーブルのカードには、注文品の調理状況を示す横棒グラフが表示されます。4 色のセグメントで構成されています。

| セグメント | 意味 |
|-----------|------|
| 提供済 | お客さんに提供完了 |
| 完成 | 調理完了、提供待ち |
| 調理中 | 現在調理中 |
| 受付 | 注文受付、調理未開始 |

各セグメントにマウスを合わせると「提供済 3 品」「調理中 1 品」などのツールチップが表示されます。

### 5.2.4 ラストオーダー機能

ラストオーダーは、セルフオーダー画面での新規注文をブロックする機能です。

**ラストオーダーの発動：**

1. サマリーバーの**「ラストオーダー」**ボタンをクリックします
2. 確認ダイアログが表示されます：「ラストオーダーを発動しますか？\nセルフオーダー画面で注文がブロックされます。」
3. 「OK」をクリックすると発動します
4. ボタンが赤色の**「LO 解除」**に変わります
5. サマリーバーに「ラストオーダー中」が赤色で表示されます
6. お客さんのセルフオーダー画面で新規注文ができなくなります

**ラストオーダーの解除：**

1. 赤色の「LO 解除」ボタンをクリックします
2. 確認ダイアログが表示されます：「ラストオーダーを解除しますか？」
3. 「OK」をクリックすると解除されます
4. ボタンが通常の「ラストオーダー」に戻ります

::: info 自動ラストオーダー
店舗設定でラストオーダー時刻を設定している場合（例：22:00）、設定時刻になると自動的にラストオーダーが発動します。手動ボタンは、設定時刻に関わらずいつでも発動 / 解除できます。
:::

### 5.2.5 メモ機能

テーブルセッションにメモを追加できます。アレルギー情報や VIP 対応など、スタッフ間の情報共有に使います。

**メモの追加・編集：**

1. テーブルカードの**「メモ」**ボタンをクリックします
2. メモモーダルが開きます（タイトル：「T01 メモ」）
3. 6 つのプリセットタグが表示されます

| プリセットタグ |
|--------------|
| アレルギー注意 |
| VIP |
| お子様連れ |
| 誕生日 |
| 接待 |
| 車椅子 |

4. プリセットタグをクリックすると、テキストエリアにタグが追記されます（改行区切り）
5. 自由にテキストを入力することもできます
6. **「保存」**ボタンをクリックします
7. 保存中はボタンが「保存中...」に変わり無効化されます
8. 成功するとモーダルが閉じ、テーブルカードにメモが表示されます

メモはテーブルカードの下部に灰色のテキストとして表示され、スタッフ全員が確認できます。

### 5.2.6 着席のキャンセル

テーブルの着席（セッション）を強制的に終了させます。

1. テーブルカードの**「取消」**ボタンをクリックします
2. 確認ダイアログが表示されます：「T01 の着席をキャンセルしますか？」
3. 「OK」をクリックすると以下が実行されます
   - セッションが「closed」ステータスに変更されます
   - そのテーブルの未会計の注文が全て「cancelled」ステータスに変更されます
   - テーブルのセッショントークンが再生成されます（古い QR コードが無効になります）
4. テーブルカードが「空き」状態に戻ります

::: warning 注意
着席キャンセルは未会計の注文を全てキャンセルします。既に調理済みの料理がある場合は、先に会計処理を行ってからキャンセルしてください。
:::

## 5.3 テーブルセッション

テーブルセッションとは、お客さんが着席してから会計が完了するまでの一連の流れを管理するデータです。

### 5.3.1 セッションのライフサイクル

```
┌─────────┐  ┌─────────┐  ┌──────────────┐  ┌──────┐  ┌────────┐
│ seated  │→│ eating  │→│ bill_requested│→│ paid │→│ closed │
└─────────┘  └─────────┘  └──────────────┘  └──────┘  └────────┘
   ↑                              ↑                       ↑
   │ ハンディ / フロアマップ      │ お客さん「会計」      │ 取消 or 精算後
   │ から着席操作                  │ ボタン                │
```

各ステータスの意味は以下のとおりです。

| ステータス | 意味 | 遷移タイミング |
|-----------|------|-------------|
| `seated` | 着席済み | ハンディ POS やフロアマップから着席操作した時 |
| `eating` | 食事中 | 注文が入った時 |
| `bill_requested` | 会計待ち | お客さんが会計をリクエストした時 |
| `paid` | 支払済み | レジで会計が完了した時 |
| `closed` | 終了 | 支払い後にセッションが閉じられた時、または着席キャンセル時 |

### 5.3.2 セッションに記録される情報

| 情報 | 説明 |
|------|------|
| ゲスト数 | 着席人数 |
| 開始日時 | 着席した日時 |
| 終了日時 | 会計完了または強制終了した日時 |
| プラン | 食べ放題プランを選択した場合（プラン ID、制限時間、ラストオーダー時間、有効期限） |
| コース | コース料理を選択した場合（コース ID、現在のフェーズ番号、フェーズ開始日時） |
| メモ | テーブルメモ（アレルギー情報、VIP 等） |
| session_token | 64 文字 hex の認証トークン |
| sub_session_id | サブセッション ID（F-QR1） |

### 5.3.3 食べ放題プラン選択時の動作

着席時に食べ放題プランを選択すると、以下が自動計算されます。

- **有効期限** = 着席日時 + プランの制限時間（例：90 分）
- **ラストオーダー期限** = 着席日時 + （制限時間 − ラストオーダー閾値）（例：90 分 − 15 分 = 75 分後）

フロアマップのテーブルカードでは残り時間がカウントダウン表示され、制限時間を超えると「時間超過」ステータスになります。

### 5.3.4 コース料理選択時の動作

着席時にコース料理を選択すると、以下が自動で行われます。

1. 現在のフェーズが「フェーズ 1」に設定されます
2. フェーズ 1 に含まれる料理が自動的に注文されます（数量はコース定義に従い、価格は 0 円）
3. KDS（キッチンディスプレイ）にフェーズ 1 の料理が表示されます
4. フェーズの進行は KDS 画面から「次フェーズ」ボタンで操作します

::: info プランとコースの排他制御
食べ放題プランとコース料理は同時に選択できません。着席時にどちらか一方を選択するか、どちらも選択しない（通常のアラカルト注文）かを選びます。
:::

## 5.Y テーブル開放認証 (L-9-TableAuth / 25 章連携)

QR コードのトークンが漏れたとき、漏れた QR ではなく**「次の正規の QR」**でホワイトリスト方式に再開放する仕組み。

- `next_session_token` フィールドにあらかじめ次回トークンを格納
- ハンディ「📲 開放」ボタンで `next_session_token` を `session_token` に昇格
- 旧トークンは無効化
- 詳細: [25 章 25-table-auth](./25-table-auth.md)

| セッション強制終了 | ✓ | — |

| 漏洩時の緊急ローテーション | — | ✓ |

## 5.Y2 サブセッション（相席用個別 QR、F-QR1）

### 5.Y2.1 概要

同じ物理テーブルに着席する複数グループが**別会計**したい場合に、**サブセッション**（親テーブルセッションの子）を発行する仕組みです（2026-04-18 追加、F-QR1）。

- 実装: `api/store/sub-sessions.php`（GET / POST）
- DB: `table_sub_sessions` テーブル、`orders.sub_session_id` で紐付け
- 発行 UI: ハンディ「テーブル状況」 + admin「テーブル管理」両方で利用可

### 5.Y2.2 DB スキーマ（migration-fqr1-sub-sessions.sql）

```sql
CREATE TABLE table_sub_sessions (
  id VARCHAR(36) NOT NULL PRIMARY KEY,
  table_session_id VARCHAR(36) NOT NULL,    -- 親セッション
  store_id VARCHAR(36) NOT NULL,
  table_id VARCHAR(36) NOT NULL,
  sub_token VARCHAR(64) NOT NULL,           -- 個別 QR の認証トークン
  label VARCHAR(50) DEFAULT NULL,           -- 'A様', 'B様' 等
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  closed_at DATETIME DEFAULT NULL,
  INDEX idx_sub_session_token (sub_token),
  INDEX idx_sub_session_parent (table_session_id),
  INDEX idx_sub_session_store (store_id)
);

ALTER TABLE orders ADD COLUMN sub_session_id VARCHAR(36)
  DEFAULT NULL AFTER session_token;
```

### 5.Y2.3 運用フロー（実例：相席 T05 に A 2 名 + B 3 名）

```
[Step 1] T05 メインセッション通常開放
  ハンディ「📲 開放」 → 通常 QR 発行
       ↓
[Step 2] A グループ着席・注文
  メイン QR を読む → カート → 注文確定 (sub_session_id=NULL)
       ↓
[Step 3] B グループ来店、別会計希望
  ハンディ or admin「📟 個別QR」ボタン
       ↓
  sub-sessions.php POST → label='B様' で発行
       ↓
  オレンジ色の個別 QR モーダルが出る
       ↓
[Step 4] B グループが個別 QR を読む
  カート → 注文確定 (sub_session_id='xxx')
       ↓
[Step 5] レジで A / B 別会計
  POSレジ画面で「サブセッション選択」
  → A セッション会計 → B セッション会計
       ↓
[Step 6] テーブル開放
  両セッション closed → 親テーブル次回開放
```

### 5.Y2.4 個別 QR 発行 UI（admin 側）

「テーブル管理」サブタブの各テーブル行に**オレンジ色の「個別 QR」ボタン**があります。

1. 「個別 QR」ボタンを押す
2. モーダルが開く（タイトル：「T05 個別 QR 発行」）
3. アクティブセッション確認中... 表示
4. アクティブセッションがない場合：「このテーブルにアクティブなセッションがありません。着席後に個別 QR を発行できます。」
5. アクティブセッションがある場合：既存サブセッション一覧 + 「+ 個別 QR を発行」ボタン
6. 「+ 個別 QR を発行」ボタンを押す → label が自動付与（A 様 → B 様 → C 様 ...）
7. 発行完了で「B 様 の個別 QR を発行しました」トースト
8. 一覧に新しいサブセッション行が追加される
9. 各行の「URL コピー」ボタンで個別 QR の URL をクリップボードに保存

### 5.Y2.5 サブセッション QR の URL 形式

```
https://eat.posla.jp/public/customer/menu.html
  ?store_id={店舗ID}
  &table_id={テーブルID}
  &sub_token={サブトークン}
```

`sub_token` パラメータが付与されている点が通常 QR との違い。クライアント側はこれを保持して以降の注文 API に付与します。

### 5.Y2.6 運用上の注意

| 項目 | 注意点 |
|---|---|
| 親子関係 | サブセッションは親テーブルセッションに紐付き、親セッション解放で**全子セッションも解放** |
| 会計単位 | POSレジではサブセッション単位で会計を選択 |
| ガント台帳 | 1 テーブルに複数サブセッションを並列表示（色分け） |
| label 重複 | 同一テーブル内で同じ label は付与されない（A→B→C... と自動採番） |
| 漏洩時 | sub_token 単位で `closed_at` を入れて無効化（親は維持） |
| 印刷不要 | 紙 QR は不要。スタッフがお客さんに直接 URL を見せる / メールで送る運用 |

## 5.Z よくある質問（FAQ 30 件）

### A. テーブル登録・編集

**Q1. テーブルコードを後から変えても大丈夫?**
A. 可能。過去注文は ID（UUID）で紐付いているため影響なし。ただし印刷済 QR は再印刷不要（URL 内は ID のみ）。

**Q2. テーブルの削除と無効化、どっちを使う?**
A. **無効化推奨**。削除は外部キー制約で過去データに影響。閉店した席は無効化で十分。

**Q3. 定員（capacity）は何に使う?**
A. 予約システム（24 章）の自動割当ロジック。少人数予約は小卓に、大人数予約は連結を試みる。

**Q4. 同じテーブルコードを別店舗で使える?**
A. 可能。一意制約は (store_id, table_code) なので店舗をまたげば重複 OK。

**Q5. テーブル数の上限は?**
A. ソフト上限なし。フロアマップ表示は 100 卓くらいまでスムーズ。

### B. QR コード

**Q6. QR コードの再発行**
A. テーブル編集モーダルから「QR」ボタン。`session_token` も自動再生成。

**Q7. QR コードを一括印刷したい**
A. 各テーブルの QR モーダルからブラウザ印刷（Ctrl+P）。一括 PDF 機能は今後追加予定。

**Q8. QR コードを読んでも何も表示されない**
A. ① URL が HTTPS か ② store_id / table_id が正しいか ③ session_token 有効期限内か。

**Q9. 紙の QR を席間違いで貼ってしまった**
A. 注文時に「他のお客さんのテーブルに紐付く」事故が起きる。即時剥がして正しい QR に交換。

**Q10. 防水 QR を作りたい**
A. ラミネート推奨。または防水紙 + UV コーティング印刷。

### C. フロアマップ

**Q11. フロアマップの自動更新が止まった**
A. 別タブ表示で停止する仕様。フロアマップタブに戻ると 5 秒ポーリング再開。

**Q12. テーブルカードの色がよく変わる**
A. 5 秒ごとにステータス更新するため。注文 → 食事中 → 会計待ちの遷移が即時反映。

**Q13. 「未注文」って何?**
A. 着席（seated）したが注文 0 件のテーブル。ホール接客の優先度判定に使う。

**Q14. 「時間超過」が出た**
A. 食べ放題プランの制限時間を超えた状態。会計を促してください。

**Q15. ラストオーダーは自動でも発動する?**
A. はい、店舗設定の LO 時刻に自動発動。手動ボタンは追加で発動 / 解除可。

### D. メモ機能

**Q16. メモは誰に見える?**
A. 全スタッフ（manager / staff / device）。お客さんには見えません。

**Q17. メモはどこに保存される?**
A. `table_sessions.memo` カラム。セッション終了で破棄。

**Q18. プリセットタグを増やしたい**
A. 現状ハードコード（`MEMO_PRESETS = ['アレルギー注意', 'VIP', ...]`）。要望が多ければ設定化検討。

### E. セッション

**Q19. 着席キャンセルしても注文が残ってる**
A. 通常 cancel される。残っているなら `closeTableSession` API がエラー。エラー内容を確認 → POSLA 運営連絡。

**Q20. 一度 closed になったセッションを復活できる?**
A. 不可。新規セッションを開くしかない。過去レポートには残る。

**Q21. セッションの有効期限は?**
A. `session_token` のみ 4 時間で切れる。セッション本体は会計まで継続。

### F. サブセッション（F-QR1）

**Q22. サブセッションっていつ使う?**
A. 相席で別会計したいとき。例：合コン後の会計分け、商談で割り勘。

**Q23. サブセッションの label を変更したい**
A. 現状 UI なし。DB 直接編集で対応。要望が多ければ追加検討。

**Q24. サブセッションが何個まで作れる?**
A. ソフト上限なし。実用上 4〜5 個まで（A 様〜E 様）。

**Q25. メインセッション + サブセッションで会計が分かれる仕組み**
A. `orders.sub_session_id` で振り分け。NULL = メイン、UUID = サブ。POSレジで選択。

**Q26. サブセッション QR を印刷したい**
A. URL コピー → 別途 QR 生成サービスで作成 / プリンタ印刷も可。基本は画面表示で運用。

**Q27. 親セッション会計したらサブも自動で会計?**
A. 自動会計はしない。各サブセッションを個別に会計する必要あり。

### G. 関連機能

**Q28. テーブルとハンディ POS の連携は?**
A. ハンディから「着席」すると `table_sessions` に行が入りフロアマップに即時反映（5 秒以内）。

**Q29. 予約とテーブルの自動割当**
A. 24 章参照。予約 → 来店時間 + 人数 → 空きテーブルから自動割当。

**Q30. テーブル開放認証（25 章）と通常 QR の違い**
A. 通常 QR = 永続 URL（ただし `session_token` で会計後ローテ）。25 章方式 = 漏洩時の緊急 next_token 切替対応。

## 5.Y トラブルシューティング

| 症状 | 原因 | 対処 |
|---|---|---|
| QR を読んでも「テーブルが見つかりません」 | `session_token` 期限切れ / 無効化 | テーブル編集 → QR 再表示で再発行 |
| QR を読んだら他テーブルの画面が出る | 紙 QR を貼り間違えた | 即時交換 |
| フロアマップが更新されない | 別タブで開いている / Wi-Fi 切断 | フロアマップタブをアクティブに / Wi-Fi 確認 |
| 「ラストオーダー」ボタンが反応しない | 権限不足（staff） | manager 以上でログイン |
| 着席キャンセルが失敗する | 既に closed | 一覧をリロード |
| 個別 QR ボタンを押してもセッションがない | テーブル未着席 | 先にメインセッションを開放 |
| サブセッション QR が読めない | sub_token 不正 / 期限切れ | 再発行 |
| メモを保存しても消える | API エラー（500 系） | サーバー側ログ確認 / POSLA 運営連絡 |
| テーブル削除でエラー | 外部キー制約（過去注文あり） | 削除→無効化に変更 |
| QR モーダルに QR が出ない | qrcode.js 未ロード | キャッシュバスター v= をバンプ / Ctrl+F5 |
| 「合計売上」が実態と違う | 未会計のみ集計の仕様 | 会計済みは別レポート参照 |
| 進捗バーが全部「受付」 | KDS 側の調理開始ボタン未押下 | KDS スタッフに確認 |
| device 端末で個別 QR ボタンが出ない | 権限制御 | device は admin の「テーブル管理」に到達できない（ハンディから発行） |
