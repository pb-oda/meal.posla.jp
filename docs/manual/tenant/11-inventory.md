# 11. 在庫・レシピ管理

在庫管理機能は **全契約者に標準提供** されます（2026-04-09 以降、旧 pro プラン制限は撤廃）。管理ダッシュボードの「在庫・レシピ」タブから、食材の在庫管理、レシピ設定、棚卸し、AI 需要予測を行います。マネージャーのみ操作できます（`data-group="inventory"` が `data-role-max="manager"` のため owner は非表示）。

実体は `api/owner/ingredients.php` (CRUD) + `api/owner/recipes.php` (BOM) + `api/owner/ingredients-csv.php` / `api/owner/recipes-csv.php` (CSV) + `api/store/demand-forecast-data.php` (AI 用集計) + `api/lib/inventory-sync.php` (品切れ自動同期 S-5/S-6) で構成されます。詳細は 11.Z。

## 11.1 原材料一覧

「在庫・レシピ」タブ → 「原材料一覧」サブタブで、食材（原材料）の登録・管理を行います。

### 11.1.1 操作前のチェックリスト

| # | 確認項目 | NG 時の対応 |
|---|---------|-----------|
| 1 | あなたのアカウントが **manager** ロール | owner では非表示 → manager に依頼 |
| 2 | 「在庫・レシピ」タブが表示されている | dashboard.html を再読込 |
| 3 | テナントの `ingredients` テーブルが作成済み | `MIGRATION` エラーなら `sql/migration-c5-inventory.sql` 実行依頼 |
| 4 | 単位 (g / ml / 個 / 枚) を登録時に決めた | 後から変更不可 (Q2 参照) |
| 5 | レシピで使う原材料は事前に列挙済み | 紙やスプレッドシートで一覧整理 |

### 11.1.2 原材料登録のクリック単位手順

1. **「在庫・レシピ」** タブをクリック
2. **「原材料一覧」** サブタブをクリック (デフォルトで開いている場合あり)
3. 画面右上の **「+ 原材料追加」** ボタンをクリック
4. モーダルが開きます (画面遷移は 11.1.4)
5. 以下のフィールドに入力します (詳細表は 11.1.3)
6. **「保存」** ボタンをクリック
7. POST `/api/owner/ingredients.php` → 一覧に新規行追加
8. ソート順 (`sort_order`) と名前順で並ぶ

### 11.1.3 入力フィールドの詳細表

| フィールド | DB カラム | 種類 | 必須 | 入力例 | 制約 |
|-----------|----------|------|------|--------|------|
| 原材料名 | `name` | テキスト | ✅ | `鶏もも肉` | 1〜100 文字 |
| 単位 | `unit` | セレクト | ✅ | `g` | `g` / `ml` / `個` / `枚`、後から変更不可 |
| 在庫数量 | `stock_quantity` | 数値 (DECIMAL 10,2) | 任意 | `5000` | マイナス許容（仕様。在庫負債を表現可能） |
| 単価 (¥) | `cost_price` | 数値 (DECIMAL 10,2) | 任意 | `2.5` | 1 単位あたりの原価。粗利計算に使用 |
| 低在庫閾値 | `low_stock_threshold` | 数値 (DECIMAL 10,2) | 任意 | `500` | この数量以下で警告 + 自動品切れトリガー |
| ソート順 | `sort_order` | 整数 | 任意 | `10` | 一覧表示順 |

### 11.1.4 画面遷移 (ASCII)

```
[在庫・レシピタブ]
  └─[原材料一覧サブタブ]
        ├─[+原材料追加] ──→ [モーダル: 入力] ──保存──→ [一覧 (新規行追加)]
        │                                              │
        │                                          POST /api/owner/ingredients.php
        │
        ├─[行クリック] ──→ [編集モーダル] ──保存──→ [一覧更新]
        │                                              │
        │                                          PATCH /api/owner/ingredients.php?id=xxx
        │
        ├─[棚卸しモード ON] ──→ [全行が編集可能セル化]
        │                          │
        │                       [一括保存] ──→ PATCH ?action=stocktake
        │                                       └─ S-5: sync_sold_out_from_inventory(autoClear=true)
        │
        ├─[CSV エクスポート] ──→ [CSV ダウンロード]
        └─[CSV インポート] ──→ [モーダル: ファイル選択] ──→ POST → upsert
```

### 11.1.5 一覧表示の項目

| 表示要素 | DB | 説明 |
|---------|-----|------|
| 原材料名 | `name` | 編集可能 |
| 単位 | `unit` | g / ml / 個 / 枚 |
| 在庫数量 | `stock_quantity` | 現在の在庫量。低在庫時はハイライト表示 |
| 単価 (¥) | `cost_price` | 1 単位あたりの原価 |
| 低在庫閾値 | `low_stock_threshold` | — |
| レシピ使用数 | `(SELECT COUNT(*) FROM recipes WHERE ingredient_id = i.id)` | この原材料を参照しているレシピ件数 |
| ステータス | `is_active` | 1=有効、0=無効 |

### 11.1.6 在庫レベルの色分け

| 状態 | 条件 | 表示 |
|---|---|---|
| 通常 | `stock_quantity > low_stock_threshold` | 標準 (黒文字) |
| 低在庫 | `stock_quantity ≤ low_stock_threshold` | 黄色ハイライト |
| 在庫切れ | `stock_quantity ≤ 0` | 赤色ハイライト |

::: warning 自動品切れの条件
S-5 で導入された `sync_sold_out_from_inventory()` は **`stock_quantity ≤ COALESCE(low_stock_threshold, 0)`** を在庫切れ判定に使います。閾値を設定しないと「0 になるまで品切れにならない」運用になるので、特に肉/魚類は **閾値必須** を推奨。
:::

### 11.1.7 原材料の編集・削除

- **編集**: 一覧行クリック → モーダルで全フィールド変更可能
  - 単位は変更不可 (PATCH ペイロードに入れても更新されない設計推奨)
  - `is_active` をトグルで論理削除可能
- **削除**: 物理削除 (DELETE)。`recipes` テーブルの該当行も同時削除 (`ON DELETE CASCADE`)。
  - 推奨は **論理削除** (`is_active = 0`)。レシピが壊れない
  - 物理削除後は復活不可

| 玉ねぎ | g | 8000 | 0.5 | 2000 | 多数 |
| ご飯 | g | 50000 | 0.3 | 10000 | 全丼物 |
| 卵 | 個 | 120 | 30 | 30 | 親子丼/オムライス |
| 醤油 | ml | 5000 | 0.1 | 500 | 全料理 |
| カレールー | g | 2000 | 1.5 | 500 | カレー類 |

#### torimaru 渋谷店 (焼鳥、本部一括配信)

| 原材料名 | 単位 | 在庫 | 単価 | 閾値 |

| 鶏もも肉 (国産) | g | 6000 | 2.5 | 2000 |

### 11.1.9 トラブルシューティング (原材料登録)

| 症状 | 原因 | 対処 |
|---|---|---|
| `MIGRATION` エラー | `ingredients` テーブル未作成 | POSLA 運営に `migration-c5-inventory.sql` 実行依頼 |
| `MISSING_FIELDS` | name 空 | 必須フィールド入力 |
| 単位を後から変えたい | 仕様で不可 | 削除→新規登録、レシピも入れ直し |
| 在庫がマイナス表示 | 仕様 (在庫負債を表現) | 棚卸しで補正 |
| レシピ使用数 0 なのに削除できない | 物理削除は CASCADE で実行されるはず | コンソールで DELETE のレスポンス確認 |

## 11.2 レシピ設定

「在庫・レシピ」タブ → 「レシピ設定」サブタブで、メニューと原材料の紐づけ（レシピ = BOM = Bill of Materials）を管理します。実装: `api/owner/recipes.php`。

### 11.2.1 レシピとは

レシピは「メニュー1品あたりに使用する原材料と数量」の定義です。DB 上は `recipes (menu_template_id, ingredient_id, quantity)` の単純な紐付けテーブル。`UNIQUE (menu_template_id, ingredient_id)` で同じメニュー × 原材料の重複は不可。

**例: 「チキンカレー」のレシピ**
- 鶏もも肉: 200 g
- 玉ねぎ: 100 g
- カレールー: 50 g
- ご飯: 250 g

### 11.2.2 操作前のチェックリスト

| # | 確認項目 |
|---|---------|
| 1 | 原材料 (11.1) が必要分だけ登録済み |
| 2 | 対象メニューが `menu_templates` に登録済み (本部メニュー or 店舗メニュー) |
| 3 | 1 食あたりの分量を実測済み (キッチンスケールで計量推奨) |
| 4 | ドリンクメニューはレシピ不要 (氷・水は計算除外、Q5 参照) |

### 11.2.3 レシピ設定のクリック単位手順

1. **「在庫・レシピ」** タブ → **「レシピ設定」** サブタブ
2. 左カラムの **メニュー一覧** から対象メニューをクリック
3. 右カラムに **レシピ編集パネル** が表示
4. **「+ 原材料を追加」** ボタンをクリック
5. プルダウンから原材料を選択 (`is_active=1` のみ表示)
6. 使用量を入力 (例: `200`)
7. **「保存」** ボタンをクリック → POST `/api/owner/recipes.php`
8. 必要な原材料を全て追加したら、別のメニューに移動

### 11.2.4 入力フィールドの詳細表

| フィールド | 種類 | 必須 | 例 | 制約 |
|-----------|------|------|----|------|
| 原材料 | セレクト | ✅ | `鶏もも肉` | `is_active=1` のみ表示 |
| 使用量 | 数値 | ✅ | `200` | 0 < x、DECIMAL(10,2) |
| 単位 | 表示のみ | — | `g` | 原材料の単位を自動表示 |
| 現在の在庫 | 表示のみ | — | `4500g` | 参考表示 |

### 11.2.5 画面遷移 (ASCII)

```
[レシピ設定タブ]
  ┌──────────────────┬────────────────────────────────┐
  │ [メニュー一覧]    │ [レシピ編集 (チキンカレー)]      │
  │ ─ 親子丼          │                                │
  │ ─ カツ丼          │ 原材料        使用量  単位      │
  │ ★チキンカレー     │ ─────────────────────────       │
  │ ─ 唐揚げ定食      │ 鶏もも肉      200    g          │
  │ ─ 生姜焼き定食    │ 玉ねぎ        100    g          │
  │ ─ ハンバーグ定食  │ カレールー    50     g          │
  │ ─ 肉野菜炒め定食  │ ご飯          250    g          │
  │                  │                                │
  │                  │ [+原材料を追加] [保存]          │
  └──────────────────┴────────────────────────────────┘
```

### 11.2.6 1 メニュー複数原材料 (BOM 構築)

- 1 つのメニューに **複数の原材料** を紐付け可
- 1 つの原材料を **複数のメニュー** で使用可 (例: 鶏もも肉 → カツ丼/親子丼/唐揚げ)
- DB の `UNIQUE (menu_template_id, ingredient_id)` により、同一メニューに同じ原材料を複数行登録は不可

### 11.2.7 レシピの削除

- レシピ編集パネルで原材料の右側 **「× 削除」** ボタン
- 確認ダイアログなし、即座に DELETE
- 削除しても原材料マスタ (`ingredients`) には影響なし

### 11.2.8 実例ブロック

#### matsunoya 「カツ丼」レシピ

| 原材料 | 使用量 | 単位 |
|---|---|---|
| 鶏もも肉 (本来は豚) | 180 | g |
| 玉ねぎ | 80 | g |
| 卵 | 1 | 個 |
| ご飯 | 250 | g |
| 醤油 | 10 | ml |

→ 1 食原価: 180×1.2 + 80×0.5 + 1×30 + 250×0.3 + 10×0.1 = ¥216 + ¥40 + ¥30 + ¥75 + ¥1 = **¥362**

→ 売価 ¥900 - ¥362 = 粗利 ¥538 / 59.8%

#### torimaru 「もも串」レシピ

| 原材料 | 使用量 | 単位 |
|---|---|---|
| 鶏もも肉 (国産) | 50 | g |
| ねぎ | 0.3 | 本 |

→ 1 本原価: 50×2.5 + 0.3×80 = ¥125 + ¥24 = **¥149**
→ 売価 ¥250 - ¥149 = 粗利 ¥101 / 40.4%

### 11.2.9 トラブルシューティング (レシピ)

| 症状 | 原因 | 対処 |
|---|---|---|
| 原材料が選択できない | `is_active=0` | 11.1 で有効化 |
| 「重複しています」 | 同じ原材料を 2 行登録 | 既存行を削除して新規追加 |
| 数量を 0 にしたい | 仕様で 0 < x | 削除する (`× 削除`) |
| 保存後、需要予測に出ない | キャッシュ | F5 リロード |

## 11.3 棚卸し

棚卸し機能で、実際の在庫数量を確認・修正します。実装: `api/owner/ingredients.php?action=stocktake` (PATCH)。

### 11.3.1 操作前のチェックリスト

| # | 確認項目 |
|---|---------|
| 1 | 営業時間外 or 在庫変動が止まっている時間帯 |
| 2 | キッチンスケール / 計量カップ / カウンターを準備 |
| 3 | 原材料の保管場所を一覧化 (倉庫/冷蔵/冷凍) |
| 4 | 紙の棚卸し用紙を CSV エクスポート (11.5) で準備 |

### 11.3.2 棚卸しのクリック単位手順

1. **「在庫・レシピ」** → **「原材料一覧」** サブタブ
2. 画面右上の **「棚卸しモード」** トグルを **ON** に
3. 各原材料の在庫数量セルが編集可能な入力フィールドに変わる
4. 実際の在庫を数えて数量を入力 (キーボード Tab キーで次フィールド)
5. すべての原材料を入力し終わったら、画面下部の **「一括保存」** ボタンをクリック
6. 確認ダイアログで **「OK」**
7. PATCH `?action=stocktake` で全行を **トランザクション一括 UPDATE**
8. 成功時: トースト「N 件更新しました」+ S-5 自動品切れ同期 (autoClear=true)
9. 失敗時: 全件ロールバック (atomic)

### 11.3.3 入力フィールドの詳細表

| フィールド | 種類 | 例 | 制約 |
|-----------|------|----|------|
| `items[].id` | 文字列 | `ing-uuid-xxx` | tenant 所有の `ingredients.id` |
| `items[].stock_quantity` | 数値 | `1500.5` | DECIMAL (10,2)、null は無視 |

### 11.3.4 画面遷移 (ASCII)

```
[原材料一覧] [棚卸しモード OFF]
   ↓
[棚卸しモード ON トグル]
   ↓
[原材料一覧] [棚卸しモード ON]
  鶏もも肉:  [4500] g  ← 編集可能
  玉ねぎ:    [8000] g  ← 編集可能
  ご飯:      [50000] g ← 編集可能
   ↓
[実数を数えて入力]
  鶏もも肉:  [4350] g  (差分 -150g)
  玉ねぎ:    [7600] g  (差分 -400g)
  ご飯:      [48000] g (差分 -2000g)
   ↓
[一括保存] ボタン
   ↓
[確認ダイアログ: 3 件更新します。よろしいですか?]
   ↓ OK

  body: { items: [...] }
   ↓

   ↓
```

### 11.3.5 棚卸し差異の追跡

棚卸し前後の差分は audit_log に記録される予定 (Phase 2)。現状はマネージャーが手で CSV を保存して比較する運用。

::: tip 定期的な棚卸し
在庫の正確性を保つため、**週 1 回** または **月 1 回** の棚卸しを推奨します。棚卸し結果は在庫管理の基礎データとなり、AI 需要予測 (11.4) の精度にも影響します。
:::

### 11.3.6 実例: matsunoya 大宮店 (週次棚卸し)

```
2026-04-13 (日) 23:00 営業終了後

棚卸し対象: 18 原材料
担当: 山田 (社員)

差異が大きかった原材料:
  鶏もも肉:  予定 4500g → 実数 4350g  (-150g、廃棄ロス想定)
  玉ねぎ:    予定 8000g → 実数 7600g  (-400g、レシピ過剰の疑い)
  ご飯:      予定 50000g → 実数 48000g (-2000g、米の炊き具合誤差想定)

→ 翌週、玉ねぎのレシピを 100g → 80g に下方修正
```

### 11.3.7 トラブルシューティング (棚卸し)

| 症状 | 原因 | 対処 |
|---|---|---|
| `MISSING_FIELDS items が必要です` | items 配列空 | 1 件以上の id + stock_quantity を含める |
| `DB_ERROR 棚卸し更新に失敗` | トランザクション失敗 | エラーログ確認、再試行 |
| 一部だけ反映されない | 仕様: tenant 不一致は SKIP | tenant_id 確認 |
| S-5 同期エラー | 仕様: 同期失敗は棚卸し成功に影響しない | エラーログ確認、手動で `sync_sold_out_from_inventory()` 再実行 |

## 11.4 AI 需要予測

在庫管理が有効な場合（全契約者に標準提供）、AI による需要予測機能が利用できます。実装: `api/store/demand-forecast-data.php` (データ集計) + Gemini API (需要予測本体は AI Assistant タブで呼び出し)。

### 11.4.1 操作前のチェックリスト

| # | 確認項目 | NG 時の対応 |
|---|---------|-----------|
| 1 | 過去 7 日以上の `orders` (status=paid) がある | 7 日以上運用後に再試行 |
| 2 | 全メニューにレシピ (11.2) が紐付けられている | 紐付け不足のメニューは予測対象外 |
| 3 | 原材料に `low_stock_threshold` が設定済み | 「不足見込み」判定に必要 |
| 4 | ドリンク以外のカテゴリにメニューが分類されている | ドリンクは予測除外 |
| 5 | POSLA 運営の Gemini API キーが設定済み (`posla_settings.gemini_api_key`) | 未設定なら `AI_NOT_CONFIGURED` (503) |

### 11.4.2 AI 予測の実行手順

1. dashboard の **「AI アシスタント」** タブ (14 章)
2. **「需要予測」** カードをクリック
3. 内部で `GET /api/store/demand-forecast-data.php?store_id=xxx` を呼び、過去 7 日間の集計データを取得
4. データは `dailySales` / `weekdayPattern` / `inventory` / `recipes` / `today` を含む JSON
5. Gemini API へプロンプトを送信 → 5〜15 秒で応答
6. 結果を画面に表示 (4 セクション: 11.4.4)

### 11.4.3 AI 予測の仕組み (内部処理)

```
[GET /api/store/demand-forecast-data.php]
  │
  ├─ 過去 7 日間の orders.items を集計 (status=paid, ドリンク除外)
  │     → dailySales = [{date, items:[{name, qty}]}]
  │
  ├─ 曜日別パターン (orders.created_at から week-day グループ)
  │     → weekdayPattern = [{weekday, totalOrders, avgRevenue}]
  │
  ├─ ingredients 一覧 (在庫量・閾値)
  │     → inventory = [{id, name, unit, stockQuantity, costPrice, lowStockThreshold}]
  │
  ├─ recipes (BOM)
  │     → recipes = [{menuName, ingredientName, quantityPerDish, unit}]
  │
  └─ today (本日の曜日)
       → today = {date, weekday, weekdayNum}

[Gemini API プロンプト構築]
  「以下のデータから、今後 3 日間の需要予測と発注推奨を出力してください」
  + JSON データ

[Gemini レスポンス]
  → 4 セクションを構造化テキストで返す
```

### 11.4.4 AI からの出力 (4 セクション)

| セクション | 内容 | 例 |
|-----------|------|---|
| 日別需要予測 | 今後 3 日の品目別販売予測 | 「明日のカツ丼: 23 食 (信頼度 中)」 |
| 原材料発注推奨 | 不足が見込まれる原材料と推奨発注量 | 「鶏もも肉 1500g 不足、2kg 発注推奨」 |
| 低在庫アラート | 閾値を下回る見込みの原材料 | 「玉ねぎ 2 日後に閾値割れ」 |
| ポーション調整 | 使い切りのためのポーション調整提案 | 「葉物野菜が余剰、サラダボウル小サイズ追加検討」 |

### 11.4.5 実例: matsunoya 大宮店 (2026-04-12 実行)

#### 入力データ

- 在庫: 18 原材料

## 11.5 CSV インポート/エクスポート

原材料・レシピを CSV で一括管理できます。実装: `api/owner/ingredients-csv.php` / `api/owner/recipes-csv.php`。

### 11.5.1 操作前のチェックリスト

| # | 確認項目 |
|---|---------|
| 1 | CSV を **UTF-8** で保存 (Excel デフォの Shift-JIS は弾かれる) |
| 2 | BOM 付き UTF-8 でも OK (システムが BOM を自動検出してスキップ) |
| 3 | 必須カラム `name` が含まれている (原材料 CSV) |
| 4 | テナント間で共有しない (テナント越境は不可) |

### 11.5.2 原材料 CSV のフォーマット

```csv
id,name,unit,stock_quantity,cost_price,low_stock_threshold,sort_order,is_active
ing-uuid-001,鶏もも肉,g,4500,1.2,1000,10,1
ing-uuid-002,玉ねぎ,g,8000,0.5,2000,20,1
,新規原材料,g,1000,0.8,200,30,1
```

| カラム | 必須 | 説明 |
|---|---|---|
| id | 任意 | 既存 UUID (空なら新規作成) |
| name | ✅ | 原材料名 |
| unit | 任意 | デフォルト `個` |
| stock_quantity | 任意 | デフォルト 0 |
| cost_price | 任意 | デフォルト 0 |
| low_stock_threshold | 任意 | デフォルト null |
| sort_order | 任意 | デフォルト 0 |
| is_active | 任意 | デフォルト 1 |

### 11.5.3 エクスポート手順

1. **「在庫・レシピ」** → **「原材料一覧」**
2. 画面右上の **「CSV エクスポート」** ボタンをクリック
3. ブラウザがファイルダウンロード (`ingredients.csv`)
4. ファイルは BOM 付き UTF-8 で生成 (Excel で文字化けしない)

### 11.5.4 インポート手順

1. **「CSV インポート」** ボタンをクリック
2. モーダルでファイル選択
3. **「アップロード」** をクリック → POST (multipart/form-data)
4. レスポンスに `created` / `updated` / `errors` (行番号付きエラーリスト) が返る
5. 成功した行のみ DB 反映、エラー行は SKIP (atomic ではない)

### 11.5.5 レシピ CSV のフォーマット

```csv
menu_name,ingredient_name,quantity
カツ丼,鶏もも肉,180
カツ丼,玉ねぎ,80
カツ丼,卵,1
親子丼,鶏もも肉,150
```

| カラム | 必須 | 説明 |
|---|---|---|
| menu_name | ✅ | `menu_templates.name` と完全一致 |
| ingredient_name | ✅ | `ingredients.name` と完全一致 |
| quantity | ✅ | DECIMAL(10,2) |

### 11.5.6 画面遷移 (ASCII)

```
[原材料一覧]
  ├─[CSV エクスポート] ──→ [ファイルダウンロード]
  │                          ingredients.csv (BOM付き UTF-8)
  │
  └─[CSV インポート] ──→ [モーダル: ファイル選択]
                              │
                          [アップロード]
                              │
                          POST multipart/form-data
                              │
                          [パース] BOM スキップ → header → 行ループ
                              │
                          [トランザクション開始]
                          各行 UPSERT (id 一致なら UPDATE、不一致なら INSERT)
                              │
                          [コミット]
                              │

```

## 11.6 在庫→品切れ自動同期 (S-5 / S-6)

レシピ連動在庫が閾値以下になったメニューを自動で品切れに設定します。実装: `api/lib/inventory-sync.php` の `sync_sold_out_from_inventory()`。

### 11.6.1 同期タイミング

| トリガー | autoClear | 説明 |
|---|---|---|
| 棚卸し完了 (`?action=stocktake`) | true | 棚卸しは在庫上昇もありうるため解除も含む |
| 原材料単体 PATCH (`stock_quantity` 変更時) | true | 在庫更新の都度実行 |
| 注文確定時 (`recipe_deduct_stock_dine_in()` の後) | false | 在庫減少のみ、解除しない |
| 会計時 (`process-payment.php` の後) | false | 同上 |

### 11.6.2 自動品切れの判定ロジック

```sql
-- 在庫切れメニュー検索
SELECT DISTINCT r.menu_template_id
FROM recipes r
JOIN ingredients i ON i.id = r.ingredient_id
WHERE i.tenant_id = ?
  AND i.is_active = 1
  AND i.stock_quantity <= COALESCE(i.low_stock_threshold, 0)
```

- 1 つのメニューに紐付く原材料のうち **どれか 1 つでも在庫切れ** なら、そのメニューは品切れ扱い
- 結果: `store_menu_overrides.is_sold_out = 1` に UPSERT (店舗ごとに行を作成)

### 11.6.3 自動解除の判定ロジック (autoClear=true 時のみ)

```sql
-- 全原材料が在庫ありのメニュー検索
SELECT DISTINCT r.menu_template_id
FROM recipes r
JOIN ingredients i ON i.id = r.ingredient_id
WHERE i.tenant_id = ?
  AND i.is_active = 1
  AND r.menu_template_id NOT IN (
      -- 在庫切れメニューを除外
      SELECT DISTINCT r2.menu_template_id
      FROM recipes r2
      JOIN ingredients i2 ON i2.id = r2.ingredient_id
      WHERE i2.tenant_id = ?
        AND i2.is_active = 1
        AND i2.stock_quantity <= COALESCE(i2.low_stock_threshold, 0)
  )
```

→ 該当メニューの `store_menu_overrides.is_sold_out = 0` に UPDATE

### 11.6.4 store_menu_overrides との関係

- 自動同期は `store_menu_overrides` テーブルに `is_sold_out` カラムを書き込み
- セルフメニュー (お客さん側、`api/lib/menu-resolver.php`) はこのフラグを読んで「売り切れ」表示
- 手動品切れトグル (メニュー管理) も同じカラムを使うため、自動 ↔ 手動が混在しても OK

### 11.6.5 失敗時の挙動 (フェイルセーフ)

- `sync_sold_out_from_inventory()` は try-catch で例外を握り潰し
- recipes / ingredients / store_menu_overrides テーブル未作成時もエラーで止まらず、SKIP
- 棚卸し / 注文確定 / 会計などの主処理に影響しない

### 11.6.6 実例: matsunoya 大宮店

#### ケース 1: 鶏もも肉が閾値割れ

```
鶏もも肉:  stock_quantity=900, low_stock_threshold=1000
  → 900 <= 1000 → 在庫切れ判定

レシピで鶏もも肉を使うメニュー:
  - カツ丼 → store_menu_overrides.is_sold_out = 1 (品切れ表示)
  - 親子丼 → store_menu_overrides.is_sold_out = 1
  - 唐揚げ定食 → store_menu_overrides.is_sold_out = 1

セルフメニュー (お客さん画面):
  カツ丼: [売り切れ] (グレーアウト)
  親子丼: [売り切れ] (グレーアウト)
  唐揚げ定食: [売り切れ] (グレーアウト)
```

#### ケース 2: 棚卸しで鶏もも肉を 5000g に補充

```
棚卸しモード ON → 鶏もも肉 = 5000g 入力 → 一括保存
  → autoClear=true で sync_sold_out_from_inventory()
  → カツ丼/親子丼/唐揚げ定食 = is_sold_out=0 に解除
  → セルフメニューでも注文可能に戻る
```

### 11.6.7 トラブルシューティング (自動同期)

| 症状 | 原因 | 対処 |
|---|---|---|
| 在庫 0 でも品切れにならない | レシピ未紐付け | 11.2 でレシピ追加 |
| 在庫補充しても品切れ解除されない | 注文確定時は `autoClear=false` | 棚卸し (11.3) で再実行 or 手動トグル |
| 1 つの原材料切れで複数メニューが品切れ | 仕様 (BOM 連動) | 別レシピを設計 (代替原材料) |
| ドリンクも品切れになる | レシピ紐付けあり | レシピを削除 (ドリンクは在庫管理対象外) |

## 11.7 在庫運用の典型ワークフロー

### 11.7.1 月初: 原材料マスタの整備

1. 過去のメニュー変更を反映:
   - 新規追加されたメニューに対応する原材料が登録されているか
   - 廃止メニューの原材料が「無効化」されているか
2. CSV エクスポート → 棚卸用紙として印刷 → 業者発注リストとして利用
3. AI 需要予測を実行して、月初の発注計画を立案

### 11.7.2 週次: 棚卸し（実在庫確認）

1. 営業時間外 (深夜 or 開店前) に実施
2. **【棚卸しモード】** ON
3. 全原材料を実際に数えて入力
4. **【一括保存】**
5. システム上の在庫数と実数の差分が **棚卸差異** として記録される
6. 差異が大きい原材料は廃棄ロス・記録漏れの可能性 → 原因究明
7. レシピの過剰計上が見つかったらレシピ補正 (11.8)

### 11.7.3 日次: 営業中の自動消化

- メニュー注文 → レシピ紐付け済み原材料の在庫が自動減算 (`recipe_deduct_stock_dine_in()`)
- 例: カツ丼 1 食 注文 → 鶏もも肉 200g、玉ねぎ 100g、ご飯 250g、卵 1 個 が自動的に減る
- 在庫が **低在庫閾値** を下回るとアラート、**0 以下** で売り切れ表示 (S-5 自動同期)

### 11.7.4 発注タイミング

**手動発注の判断基準:**
- 低在庫閾値を下回った原材料
- 翌週の予約で大量消費が見込まれる原材料
- AI 需要予測で「不足見込み」と提案された原材料

**発注フロー:**
1. CSV エクスポート → 業者ごとの発注書を作成
2. メール / FAX / 業者管理画面で発注
3. 入荷時に在庫数を更新（手動入力 or CSV インポート）

### 11.7.5 月末: 振り返り

1. 月次の棚卸し差異の合計 → 廃棄率を算出
2. AI 需要予測の精度検証 (予測 vs 実数)
3. 原材料単価の見直し (業者見積もりと照合)
4. レシピの粗利率確認 → 不採算メニューの値上げ or レシピ見直し

## 11.8 在庫管理を活用するためのレシピ設計

### 11.8.1 レシピの精度

レシピを正確に登録すればするほど、自動消化の精度が上がります。

**精度の目安:**
| 精度 | 例 | 自動消化精度 |
|---|---|---|
| **高（推奨）** | 鶏もも肉 **200g** ピッタリ | ±5% |
| 中 | 鶏もも肉 **約 180-220g** | ±15% |
| 低 | 鶏もも肉 **適量** | 計測不能 |

### 11.8.2 例外ケース

- **手作りソースの原材料**: 1 食分の使用量が極小（例: 5g）の場合、1 食ずつ正確に計測しないので誤差発生
- **盛り付け野菜**: 葉物野菜は重量変動が大きい
- **ドリンク**: 氷・水は計算除外
- **トッピング**: 「卵追加 +¥100」などは別メニュー化推奨

これらは **「目安在庫」** として扱い、棚卸しで定期補正してください。

### 11.8.3 レシピ実測の手順

1. キッチンスケールを準備
2. 1 食調理する前に各原材料を計量、メモ
3. 5 食ぶん作って平均値を取る
4. レシピを更新 (11.2)
5. 翌週の棚卸しで差異が縮小しているか確認

### 11.8.4 廃棄ロスを在庫に反映

- 廃棄分は別途「廃棄」原材料として登録するのが理想だが、現状のスキーマでは未対応
- 暫定: 棚卸しで負の差異として記録 → audit_log で追跡

## 11.X よくある質問 30 件

### A. 基本仕様

**Q1. 在庫 0 で自動品切れになる?**
A1. レシピ紐付け済みの品目は **在庫自動連動で売り切れ表示** (S-5)。判定は `stock_quantity <= COALESCE(low_stock_threshold, 0)`。レシピ未紐付けのメニューは手動で売り切れトグル。

**Q2. 食材の単位変換は?**
A2. POSLA は **g / ml / 個 / 枚** から選択。kg は g に変換して登録（例: 1kg = 1000g）。1 つの原材料に対して単位を後から変えることはできないので、初期登録時に決める。

**Q3. レシピなしの品目の在庫管理は?**
A3. 手動品切れトグルのみ（メニュー管理から）。原材料管理は不要。

**Q4. 単価の小数点入力は?**
A4. DECIMAL(10,2) 形式で 2 桁まで。例: ¥3.55 はそのまま保存可能。

**Q5. 氷・水は在庫管理する?**
A5. 通常しない。ドリンクメニューにレシピを付けない運用が一般的。

### B. AI 需要予測

**Q6. 発注提案機能はある?**
A6. **AI 需要予測 (11.4)** で対応。過去売上 + レシピから「3 日後に〇〇が不足見込み」と提案。

**Q7. AI 予測の精度は?**
A7. 過去 30 日以上のデータが蓄積されてから精度が上がる。営業 1 ヶ月目は参考程度に。

**Q8. AI 予測がドリンクを除外する理由は?**
A8. ドリンクは在庫変動が読めない (氷溶けや継ぎ足し) ため、ノイズとなり予測精度が下がるため除外。

**Q9. AI 予測を毎日自動実行できる?**
A9. 現状は手動。Phase 2 でバッチ化検討。

**Q10. AI 予測結果を CSV エクスポートできる?**
A10. 現状不可。画面のテキストをコピペ運用。

### C. 棚卸し

**Q11. 棚卸し（スポット在庫調整）**
A11. 在庫画面で直接数値上書き可。差異は `audit_log` に記録予定 (Phase 2)。

**Q12. 棚卸しの履歴は残る?**
A12. 残ります。各原材料の更新履歴が `audit_log` に記録（`event='ingredient_update'`）。月次棚卸の差異もこれで追跡可能。

**Q13. 棚卸し中に営業を続けたい**
A13. 推奨しない。棚卸し中の注文も在庫を減らすので、棚卸し結果と実数がズレる可能性。

**Q14. 棚卸しを途中で中断したい**
A14. 「棚卸しモード」OFF にすれば未保存。一括保存ボタン押下前なら破棄可。

**Q15. 一括保存後に間違いに気付いた**
A15. 該当原材料を個別 PATCH で再修正。audit_log には両方の操作が残る。

### D. CSV / 一括操作

**Q16. CSV インポートで一括更新できる?**
A16. はい。**【CSV インポート】** ボタンから一括更新。同じ原材料 ID の行があれば上書き、新しいものは追加。

**Q17. レシピも CSV で管理できる?**
A17. はい。`/api/owner/recipes-csv.php` で同様に。

**Q18. CSV のフォーマット仕様は?**
A18. 11.5.2 (原材料) / 11.5.5 (レシピ) を参照。BOM 付き UTF-8 推奨。

**Q19. Excel で編集した CSV が文字化けする**
A19. Excel の保存形式を「CSV UTF-8 (BOM 付き)」に変更。

**Q20. CSV の sort_order を変更すると?**
A20. 一覧表示順が変わる。実運用に影響なし。

### E. レシピ・原価

**Q21. 1 つの原材料を複数メニューで使う**
A21. もちろん可能。例: 鶏もも肉 → カツ丼 / 親子丼 / 唐揚げ で同じ `ingredient_id` を参照。1 食ごとの消費量はメニュー別に設定。

**Q22. 在庫の単価は何に使う?**
A22. **粗利計算** に使用。売上レポート（13 章 13.2 商品別ランキング）で粗利が表示。例: カツ丼 ¥900 - 原価 ¥320 = 粗利 ¥580 / 64.4%

**Q23. 季節限定メニューのレシピは?**
A23. 通常メニューと同じく登録。シーズン終了後はメニュー側を非表示。

**Q24. レシピを変更すると過去の注文の原価集計は?**
A24. 過去の注文は変わらない (orders.items は JSON で保存済み)。レシピ変更は新規注文から適用。

**Q25. レシピ単位を換算する**
A25. 例えば「2 食分でレシピ作成」したい場合は、各原材料を 2 倍にして登録。1 食基準推奨。

### F. 自動連動・S-5/S-6

**Q26. 注文確定時にいつ在庫が減る?**
A26. 客側 `checkout-confirm.php` 確定時 (`recipe_deduct_stock_dine_in()`)、または店舗側 `process-payment.php` 会計時。両方で同じ関数を呼ぶが二重減算しないようガード済み。

**Q27. 在庫マイナスを禁止したい**
A27. 仕様で許容 (在庫負債を表現)。禁止オプションは未実装。

**Q28. 在庫アラートは通知される?**
A28. 現状、店舗の管理ダッシュボード上での表示のみ。Slack / メール通知は未対応。Phase 2 で対応予定。

**Q29. 1 つの原材料切れで複数メニューが品切れになるのが困る**
A29. 仕様 (BOM 連動)。代替原材料を別レシピで設計するか、在庫を補充。

**Q30. 原材料を削除したい**
A30. **論理削除（無効化）** を推奨。完全削除はレシピの参照が壊れるため非推奨。「無効」トグルで在庫表示から消える。物理削除は CASCADE で recipes も同時削除される。

## 11.Y トラブルシューティング (総合)

| 症状 | 原因 | 対処 |
|---|---|---|
| `MIGRATION` エラー | テーブル未作成 | POSLA 運営に migration-c5-inventory.sql 実行依頼 |
| 在庫が減らない | レシピ未紐付け | メニュー編集画面でレシピを設定 (11.2) |
| 在庫が想定より早く減る | レシピの量が大きすぎる | 実測してレシピ修正、または棚卸しで実数値補正 |
| 在庫が想定より減らない | レシピの量が小さすぎる | 実測 + レシピ補正 |
| 品切れが解除できない | 在庫残 0 + autoClear=false | 在庫を補充 + 棚卸し実行で解除 |
| 自動売切ロジックが効かない | レシピで原材料が複数の品目で共有 | 1 つでも在庫切れがあれば全品切れになる仕様 (Q29) |
| CSV インポートでエラー | 文字コード Shift-JIS | CSV を UTF-8 で再保存 |
| 単価入力が反映されない | 整数形式以外 | DECIMAL(10,2) で入力可、Excel で `'0.5` 入力推奨 |
| AI 需要予測が表示されない | データ不足 | 過去 7 日以上の売上データ蓄積後に再度試す |
| AI 需要予測が `AI_NOT_CONFIGURED` | Gemini キー未設定 | POSLA 運営に依頼 |
| 棚卸しモードトグルが見えない | manager 権限なし | manager アカウントでログイン |
| 「在庫・レシピ」タブが見えない | owner ロール (UI 仕様で非表示) | manager アカウントでログイン |
| 1 件だけ更新したいのに棚卸しモードが必要 | 個別 PATCH `?id=xxx` も使える | 行クリック → 編集モーダルで保存 |
| 在庫がマイナス値になっている | 仕様 (在庫負債) | 棚卸しで補正 |
| `recipe_deduct_stock` がエラーログに出る | テーブル未作成時 | フェイルセーフで握り潰し済み、注文は成功している |

## 11.Z データベース構造 (運営向け)

::: tip 通常テナントは読まなくて OK

### DB テーブル

| テーブル | 用途 | 主キー / UNIQUE |
|---|---|---|
| `ingredients` | 原材料マスタ (tenant 単位) | `id` (UUID) |
| `recipes` | メニュー × 原材料の BOM | `id` (UUID), UNIQUE `(menu_template_id, ingredient_id)` |
| `store_menu_overrides` | 店舗別 is_sold_out (S-5 が書き込む) | `id` (UUID), `(store_id, template_id)` |

### API（2026-04-19 時点の実ファイル）

- `GET/POST/PATCH/DELETE /api/owner/ingredients.php` — 原材料 CRUD（manager 以上）
  - `?action=stocktake` (PATCH) — 棚卸し一括更新 (atomic)
- `GET/POST /api/owner/ingredients-csv.php` — 原材料 CSV (BOM 付き UTF-8)
- `GET/POST/PATCH/DELETE /api/owner/recipes.php` — レシピ CRUD（manager 以上）
- `GET/POST /api/owner/recipes-csv.php` — レシピ CSV
- `GET /api/store/demand-forecast-data.php` — 需要予測データ取得（manager）

::: info 運用権限メモ
原材料・レシピ登録は `/api/owner/` 配下の API を使用しますが、UI は `dashboard.html` の「在庫・レシピ」タブで操作します（`data-group="inventory"` が `data-role-max="manager"` のため owner は非表示、manager が操作）。実装の整合性のため、将来 `/api/store/` に移設する可能性あり。
:::

### 自動品切れ同期 (S-5 / S-6)

- ライブラリ: `api/lib/inventory-sync.php`
- 関数: `sync_sold_out_from_inventory(PDO $pdo, string $tenantId, ?string $storeId, bool $autoClear)`
- 呼び出し箇所:
  - `api/owner/ingredients.php` 棚卸し直後 (autoClear=true)
  - `api/owner/ingredients.php` 個別 PATCH の stock 変更時 (autoClear=true)
  - `api/customer/checkout-confirm.php` 注文確定後 (autoClear=false)
  - `api/store/process-payment.php` 会計後 (autoClear=false)

### 注文時の在庫自動引落

- 関数 `recipe_deduct_stock_dine_in()` (`api/store/process-payment.php` 内)
- 関数 `recipe_deduct_stock()` (`api/customer/checkout-confirm.php` 内)
- どちらも `recipes` JOIN `ingredients` で `quantity` × `qty` 分だけ `stock_quantity -= ...` を実行
- テーブル未作成時は try-catch で握り潰し → 注文は成功

### マイグレーション

- `sql/migration-c5-inventory.sql` — `ingredients` + `recipes` テーブル作成 + low_stock_threshold パッチ
- `sql/seed-c5-inventory.sql` — テストデータ (matsunoya/momonoya 用)

### フロント

- `public/admin/js/inventory-manager.js` — 在庫管理 UI のロジック
- `public/admin/js/recipe-editor.js` — レシピ編集 UI
- `public/admin/js/ai-assistant.js` — 需要予測呼び出し (Gemini 経由)

### 関連 API エラーコード総覧

| コード | HTTP | 発生 API | 説明 |
|---|---|---|---|
| `MIGRATION` | 500 | ingredients / recipes | テーブル未作成 |
| `MISSING_FIELDS` | 400 | POST / PATCH | 必須フィールド欠落 |
| `MISSING_ID` | 400 | PATCH / DELETE | id パラメータ欠落 |
| `NOT_FOUND` | 404 | 同上 | 該当原材料/レシピなし |
| `NO_FIELDS` | 400 | PATCH | 更新項目なし |
| `DB_ERROR` | 500 | stocktake | トランザクション失敗 |
| `DELETE_FAILED` | 500 | DELETE | 削除失敗 |
| `NO_FILE` | 400 | CSV import | ファイル未アップロード |
| `EMPTY_CSV` | 400 | CSV import | データ行ゼロ |
| `MISSING_COLUMN` | 400 | CSV import | name カラム欠落 |
| `FILE_READ_ERROR` | 500 | CSV import | アップロードファイル読込失敗 |
| `AI_NOT_CONFIGURED` | 503 | demand-forecast | Gemini キー未設定 |

### マルチテナント境界

- 全 SQL に `WHERE tenant_id = ?` 必須
- ingredients は `FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON UPDATE CASCADE`
- recipes は menu_template 経由で間接的にテナント境界
- store_menu_overrides は store_id 経由でテナント境界
- CSV インポート時もテナント境界チェック (他テナントの id は SKIP)
