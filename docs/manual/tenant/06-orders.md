# 6. 注文管理 (ハンディPOS)

ハンディ POS (`/handy/`) は、スタッフがスマートフォンやタブレットから注文を入力するための画面です。店内飲食・テイクアウトの両方に対応し、予約管理・テーブル開放・メモ編集まで1台でこなせます。

---

## 6.0 はじめにお読みください

::: tip 章の読み方ガイド
- **新人スタッフ**: 6.0 → 6.1 → 6.3 (店内注文の流れ) の順で読むと操作が分かります
- **管理担当**: 6.9 (予約との連動) と 6.10 (テーブル開放) は必読
- **テイクアウト担当**: 6.4 を集中的に読む
- **トラブル対応**: 6.15 トラブルシューティングと 6.14 FAQ を参照
:::

### ハンディの役割

- **店内注文**: スタッフが席を回って注文を取る
- **テイクアウト注文**: 電話注文やカウンター対面注文
- **テーブル状況確認**: 空席・着席中・会計待ちの管理
- **注文履歴・編集**: 当日の注文一覧、未会計注文の変更
- **予約管理**: 当日予約の確認・即着席・no-show 記録
- **テーブル開放**: Walk-in 客のセルフ注文開放

### 使える人 (ロール別)

| ロール | 可能な操作 |
|---|---|
| owner | 全操作 |
| manager | 全操作 |
| staff | 注文入力 / テーブル状況 / メモ編集 / 予約確認 / QR開放 |
| device | 操作不可 (KDS / レジ用) |

### 必要なもの

- **スマートフォン or タブレット** (推奨: iPad mini / Android 10" タブレット)
- **店舗 Wi-Fi** (接続必須、4G でも可だが速度次第)
- **Chrome 最新版** (Safari 最新版でも可)
- **ログインアカウント** (staff 以上)

### 所要時間

| 作業 | 目安 |
|---|---|
| 初回ログイン | 1 分 |
| 店内注文 1 件入力 | 20〜40 秒 |
| テイクアウト注文 1 件 | 30〜60 秒 |
| 予約客の着席 (自動マッチ) | 0 秒 (スタッフ操作不要) |
| walk-in 開放 (📲 QR開放) | 5 秒 |
| メモ編集 | 10 秒 |

### 操作前のチェックリスト (全機能共通)

ハンディを開く前に、以下 6 項目を上から順に確認してください。

| # | 確認項目 | 確認方法 | NG 時の対応 |
|---|---------|---------|-----------|
| 1 | 自分のアカウントが **staff / manager / owner** いずれか | ログイン後、画面右上のユーザー名横にロール表示 | device ロールはハンディ専用画面に直接遷移するため、本章ではなく `/handy/` 直アクセス |
| 2 | 店舗 Wi-Fi に接続されている | スマホの設定アプリ → Wi-Fi で店舗 SSID と接続済みを確認 | テザリング 4G で代用可 (やや遅延) |
| 3 | ブラウザが **Chrome 最新版** または **Safari 最新版** | `chrome://version` または「Safari について」で確認 | 古い場合は更新。IE / 旧 Edge は不可 |
| 4 | プライベート (シークレット) モードで開いていない | URL バーの右上にプライベート印が出ていないこと | 通常モードで開き直す (localStorage が保存されない) |
| 5 | 端末が **GPS 許可** されている (ハンディ device の場合) | 初回アクセス時のブラウザ確認ダイアログで「許可」 | device ロールは GPS 拒否で操作不可 |
| 6 | 操作対象の店舗が選択済み | ヘッダー左の店舗名が現在の店舗 | 6.2.3 「店舗切替」を実施 |

::: warning device ロールでログインした場合
KDS / レジ / ハンディ専用 device アカウントは **シフト管理・スタッフレポートから除外** されます。注文を「誰が入力したか」追跡したいテーブル (例: 接客評価対象) は、必ず人スタッフ (staff/manager/owner) のアカウントで入力してください。
:::

## 6.1 画面構成

### 6.1.1 ヘッダー

| 位置 | 要素 | 説明 |
|---|---|---|
| 左 | 「ハンディ」タイトル + 店舗名 | 現在操作中の店舗 |
| 右 | 店舗切替リンク | 複数店舗所属の場合のみ表示 |
| 右 | ユーザー名 (🌙/☀ アイコン) | ログイン中のスタッフ名 + テーマ切替 |
| 右 | 【ログアウト】ボタン | 暗赤色、タップで即ログアウト |

### 6.1.2 メインタブ (5 つ)

```
┌─────────────────────────────────────────────┐
│ 店内注文 | テイクアウト | テーブル状況 | 予約 | 注文履歴 │
└─────────────────────────────────────────────┘
```

| タブ | 機能 | デフォルト表示 |
|---|---|---|
| **店内注文** | テーブル選択 + メニュー + カート | ✅ 開くと最初に表示 |
| **テイクアウト** | 顧客名 + 受取時刻 + メニュー | — |
| **テーブル状況** | 全テーブルの状態一覧 (カード形式) | — |
| **予約** (L-9) ★ | 当日予約リスト + 即着席 / QR開放 | — |
| **注文履歴** | 当日の注文一覧 + 編集 | — |

### 6.1.3 下部の常時表示エリア

- **カートバー** (店内 / テイクアウトタブ時): 固定フッターでカート合計金額・品目数
- **編集中バナー** (注文編集中のみ): オレンジ色で「修正中: テーブルA-1」表示

### 6.1.4 アラートバナー

スタッフ呼び出し・料理完成通知・キッチン連絡は、ヘッダーの下に赤色バナーでリアルタイム表示されます (5 秒毎ポーリング)。

## 6.2 初回アクセスと店舗選択

### 6.2.1 ログイン後の遷移

- staff / manager / owner → 管理ダッシュボードが開く → 上部「ハンディ」リンクをタップ → 新タブで `/handy/` 開く
- device (handy visible_tools あり) → ログイン直後に自動で `/handy/` に遷移

### 6.2.2 複数店舗所属スタッフの場合

初回アクセス時に **店舗選択オーバーレイ** が表示:

```
┌────────────────────────────────────┐
│      店舗を選択してください         │
├────────────────────────────────────┤
│  [松の屋 渋谷店]                    │
│  [松の屋 新宿店]                    │
│  [松の屋 池袋店]                    │
└────────────────────────────────────┘
```

1. 所属店舗の一覧から操作する店舗をタップ
2. 選択内容は `localStorage.mt_handy_store` に保存
3. 次回以降は自動で同じ店舗が開く

### 6.2.3 店舗切替

ヘッダー右の **【店舗切替】** リンクで再選択可能 (単一店舗所属の場合は非表示)。

### 6.2.4 タブの初期設定

初回表示時は **「店内注文」** タブがアクティブ。最後に使ったタブは記憶されません (毎回店内注文から)。

## 6.3 店内注文の流れ (全 6 ステップ)

### 6.3.1 ステップ 1: テーブルを選択

1. **【店内注文】** タブが選択されていることを確認
2. 画面上部の**テーブル選択ドロップダウン**をタップ
3. 一覧から操作するテーブルを選択

```
テーブル
  ↓ タップ
[A-1 (4名)    ▼]
[A-2 (4名)]
[A-3 (2名)]
[B-1 (6名)]
[カウンター1 (1名)]
```

表示形式: **`{テーブルコード} ({座席数}名)`**

テーブルは「テーブル管理」([5 章](./05-tables.md)) で登録されたものがすべて表示されます。

### 6.3.2 ステップ 2: カテゴリを選択

テーブル選択の下に **カテゴリタブ**が横スクロールで表示:

```
[ドリンク] [前菜] [メイン] [ご飯物] [デザート] [サイド]
```

- タップするとそのカテゴリのメニューが表示
- 選択中は青色 (#536dfe) でハイライト
- カテゴリ順序は「カテゴリ管理」で設定した並び順

### 6.3.3 ステップ 3: メニューを選択

カテゴリタブの下に **メニューグリッド**:

**レスポンシブ表示**:
| 画面幅 | 列数 |
|---|---|
| < 600px (スマホ) | 2 列 |
| 600〜900px (タブレット縦) | 3 列 |
| 900px 以上 (タブレット横 / PC) | 4 列 |

**メニューカードの内容**:

| 要素 | 説明 |
|---|---|
| 画像 | 高さ 80px、未登録時は灰色プレースホルダー |
| 品名 | オプションありは ⚙ アイコン付き |
| 価格 | 緑色 (#4caf50) で `¥XXX` 表示 |
| 限定バッジ | 限定メニューはピンク色「limited」 |
| カート数量 | 既にカート追加済なら右上に赤丸 + 数量 |
| 品切れ | 半透明 (opacity 0.4) + 「品切れ」オーバーレイ |

### 6.3.4 ステップ 4: オプション選択 (該当品目のみ)

オプショングループが設定されたメニューをタップすると**オプションモーダル**が開く:

```
┌────────────────────────────────┐
│ ラーメン                   ×   │
├────────────────────────────────┤
│ トッピング              [任意]  │
│ ○ チャーシュー 2枚 (+¥200)      │
│ ○ 味玉 (+¥150)                  │
│ ○ ネギ増し (+¥100)              │
│                                 │
│ 麺の硬さ                [必須]  │
│ ● 普通                          │
│ ○ 硬め                          │
│ ○ 柔らかめ                      │
├────────────────────────────────┤
│  合計: ¥1,350 [カートに追加]    │
└────────────────────────────────┘
```

**バリデーション**:
- 必須グループで未選択 → 「カートに追加」ボタンが disabled (グレーアウト)
- 合計金額は基本価格 + オプション追加料金で自動計算
- 任意グループは選択しなくても OK

**オプションなしのメニュー**:
- タップで即カート追加 (数量 1)
- 既にカートにある同じメニューなら数量 +1

### 6.3.5 ステップ 5: カートを確認

画面下部の**カートバー** (固定表示):

**折りたたみ時**:
```
┌────────────────────────────────────────┐
│ カート (3アイテム) 合計 ¥1,800 [注文送信]│
└────────────────────────────────────────┘
```

**展開時** (サマリー部タップで展開):
```
┌────────────────────────────────────────┐
│ カート (3アイテム) 合計 ¥1,800 [注文送信]│
├────────────────────────────────────────┤
│ ラーメン (チャーシュー+味玉)    ¥1,350  │
│                        [-] 1 [+]        │
│ ネギ増し トッピング             (+¥100) │
│ ライス                          ¥300    │
│                        [-] 2 [+]        │
│                                         │
│ メモ: [ネギ抜き等                      ]│
│                                         │
│ [クリア]  合計 ¥1,800    [注文送信]     │
└────────────────────────────────────────┘
```

**数量操作**:
- [+] で +1
- [-] で -1 (1 の時に [-] で削除)
- 同じメニューでもオプション違いは別カート行 (例: 「ラーメン(にんにく多め)」と「ラーメン(なし)」)

### 6.3.6 ステップ 6: 注文を送信

#### 送信前のチェックリスト (5 項目)

カートバーの **【注文送信】** をタップする前に必ず指差し確認してください。

| # | 確認項目 | 確認方法 |
|---|---------|---------|
| 1 | テーブルが選択されている (店内のみ) | カートバー上の「テーブル」ドロップダウンに現在のテーブルコードが表示されている |
| 2 | カート品目数が 1 以上 | カートバー左の `カート (N品)` の N が 1 以上 |
| 3 | オプション必須項目はすべて選択済 | 既にカート行になっていれば OK (未選択ならカートに入らない) |
| 4 | 数量が想定通り (誤タップで `qty=10` 等になっていない) | 展開カートで [+] / [-] 部分の数値を確認 |
| 5 | メモが現在のテーブルの注文内容と一致 | 別テーブルのメモを使い回していないか |

#### 送信操作

1. カートバーの **【注文送信】** ボタン (オレンジ色) をタップ
2. ボタンが「処理中...」になる (1〜2 秒、再送信防止のため自動 disabled)
3. 成功時: 画面下にトースト「注文を送信しました (¥XXXX)」
4. カート自動クリア + メニューの数量バッジもクリア
5. 注文は KDS にリアルタイムで表示 (3 秒ポーリング)

**送信前バリデーション** (クライアント側):
- カート空 → エラー「カートが空です」
- テーブル未選択 → エラー「テーブルを選択してください」
- テイクアウト時に受取時間未入力 → エラー「受取時間を入力してください」

**送信データ** (POST `/api/store/handy-order.php`、`require_auth()` 必須):

| フィールド | 型 | 必須 | 値の例 | 補足 |
|---|---|---|---|---|
| `store_id` | string | ★ | `s-momo-shibuya-001` | 現在の店舗 ID。`mt_handy_store` から自動セット |
| `table_id` | string | ★ (店内のみ) | `tbl-A-01` | 選択中のテーブル ID |
| `items` | array | ★ | `[{id, name, price, qty, options}]` | カート配列。各品目は `options` 任意 |
| `order_type` | enum | ★ | `handy` / `takeout` | タブにより自動切替 |
| `memo` | string | — | `アレルギー: 卵` | 注文全体メモ。最大 200 文字、空欄可 |
| `customer_name` | string | ★ (テイクアウト) | `田中` | テイクアウト時のみ必須 |
| `customer_phone` | string | ★ (テイクアウト) | `090-1234-5678` | 同上 |
| `pickup_at` | datetime | ★ (テイクアウト) | `2026-04-19 20:00:00` | `YYYY-MM-DD HH:mm:ss` |

**サーバー側で再チェックされる項目** (失敗するとエラー):

| エラーコード | HTTP | 発生条件 | 対処 |
|---|---|---|---|
| `MISSING_FIELDS` | 400 | `store_id` / `items` のいずれかが空 | カート + 店舗を確認 |
| `MAX_ITEMS_EXCEEDED` | 400 | `store_settings.max_items_per_order` 超過 | 分割注文。デフォルト 10 品 |
| `MAX_AMOUNT_EXCEEDED` | 400 | `store_settings.max_amount_per_order` 超過 | 分割注文。デフォルト ¥30,000 |
| `LAST_ORDER_PASSED` | 409 | ラストオーダー時刻を過ぎている | 6.13 / [O-3] 参照 |
| `SOLD_OUT` | 409 | 送信時点で品切れに変わった品目あり | 該当品目をカートから外し再送信 |
| `SESSION_ORDER_LIMIT` | 429 | 同一セッションで 50 件 or ¥200,000 超過 | スタッフ介在で分割注文 |

**送信後の状態**:
- `orders.status='pending'` (KDS で受付待ち)
- `orders.created_by_user_id` = ログインスタッフ ID (誰が入力したか追跡)
- `orders.order_type='handy'` または `'takeout'`
- `order_items` テーブルに品目単位レコードが INSERT (品目ステータス管理用)
- KDS に即座に表示 (3 秒ポーリング)
- L-15 連携が ON ならスマレジにベストエフォートで同期

#### 実例: matsunoya 渋谷店 (テーブル A-3 / 4 名)

ある日のディナータイム (2026-04-19 19:42)、スタッフ `tanaka` がテーブル A-3 から下記注文を入力した実例。

```
カート (5品)
  ハンバーグ (チーズ追加)        ¥1,580 × 2  = ¥3,160
  ライス (大盛)                 ¥350  × 2  = ¥700
  サラダ (ドレッシング: 和風)    ¥480  × 1  = ¥480
合計                                       ¥4,340
メモ: お子様用に取り分け皿2枚お願いします
```

送信後の DB 状態:
- `orders.id` = `ord_018f3a9c-xxxx` (UUID)
- `orders.status` = `pending`
- `orders.total_amount` = `4340`
- `orders.created_by_user_id` = `usr_tanaka`
- `order_items` に 3 行 INSERT (qty=2, qty=2, qty=1)

KDS には数秒以内に「テーブル A-3 / 5 品 / メモあり」として赤バナーで表示されます。

## 6.4 テイクアウト注文の流れ

### 6.4.1 タブ切替

**【テイクアウト】** タブをタップ → テーブル選択が消え、顧客情報入力欄が表示:

```
顧客情報
┌─────────────────┐ ┌─────────────────┐
│ 顧客名            │ │ 電話番号          │
└─────────────────┘ └─────────────────┘

受取時間
[2026-04-19 20:00  ▼]
```

### 6.4.2 入力フィールド

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| お客さん名 | テキスト | ★ | 受取人の名前 |
| 電話番号 | tel | ★ | 連絡先 |
| 受取時間 | datetime-local | ★ | 初期値は現在時刻 |

### 6.4.3 メニュー選択・カート操作

店内注文と**同一** (6.3.2〜6.3.5 参照)。

### 6.4.4 送信データの違い

```
store_id        → 店舗ID
items           → 品目リスト
order_type      → "takeout"
customer_name   → お客さん名
customer_phone  → 電話番号
pickup_at       → 受取日時 (YYYY-MM-DD HH:mm:ss)
memo            → メモ
```

### 6.4.5 テイクアウト送信後

- `orders.order_type='takeout'`
- KDS にテイクアウトマーク付きで表示
- レジ側で「テイクアウト」タグ表示
- オンライン決済不要 (店頭払いをデフォルト、設定により変更可)

### 6.4.6 実例: torimaru 池袋店 (電話注文)

電話で「20:00 受取、唐揚げ弁当 3 つ」の注文を受けた場合の操作手順:

| ステップ | 操作 | 入力値 |
|---|---|---|
| 1 | **【テイクアウト】** タブをタップ | — |
| 2 | 顧客名を入力 | `山本` |
| 3 | 電話番号を入力 | `090-3333-4444` |
| 4 | 受取時間欄をタップ → ピッカー操作 | `2026-04-19 20:00` |
| 5 | カテゴリ「弁当」をタップ | — |
| 6 | 「唐揚げ弁当」を 3 回タップ | (カート 1 行、qty=3) |
| 7 | カート展開 → 合計 ¥2,400 を確認 | — |
| 8 | メモ欄に「割り箸 3 膳」と入力 | — |
| 9 | **【注文送信】** をタップ | — |

トースト「注文を送信しました (¥2,400)」が出れば成功。テイクアウト一覧 (KDS の takeout タブ) に「20:00 / 山本 / 唐揚げ弁当 ×3」として表示されます。

### 6.4.7 受取時間のフォーマット

- HTML5 `datetime-local` 入力 → サーバー送信時に `toMySQLDatetime()` で `YYYY-MM-DD HH:mm:ss` に変換
- iPhone Safari ではネイティブのドラムロール UI、Android Chrome ではカレンダー + 時刻ピッカー
- **過去日時を入力するとサーバー側でエラーにならない** ので、19:00 のつもりで 09:00 (午前) を選んでしまわないよう注意

### 6.8.4 注文の編集手順

- **【↻ 更新】** ボタンで手動リフレッシュ
- 自動更新なし (ページ開いた時のみ取得)

### 6.8.7 編集時のサーバー制約

`PATCH /api/store/handy-order.php` (`require_auth()` + `require_store_access()` 必須) の挙動:

| 状況 | サーバー応答 | UI トースト |
|---|---|---|
| `paid` 注文を編集 | `409 CONFLICT 会計済み・キャンセル済みの注文は修正できません` | 「会計済みの注文は編集できません」 |
| `cancelled` 注文を編集 | 同上 | 同上 |
| 編集中に他スタッフが先に paid 化 | 409 | 「他スタッフが既に会計しました」 |
| `items` が空配列 | `400 VALIDATION 更新項目がありません` | 「品目が0件です」 |
| 別店舗の注文 ID を指定 | `403 FORBIDDEN` | 「アクセス権限がありません」 |

::: tip 編集 = items まるごと差し替え
PATCH では `items` 配列を**まるごと差し替え**ます。「1 品だけ追加」したい場合も、現状の全品目 + 追加品目を含めた新配列を送信する必要があります。ハンディ UI 側でカートに既存注文を自動ロードしているのはこのためです。
:::

### 6.8.8 編集の実例: トッピング追加忘れの救済 (matsunoya)

```
19:50  テーブル A-3 で「ハンバーグ」のみ注文送信 (¥1,580)
19:51  お客様「あ、チーズトッピングお願い」
19:52  ハンディ → 注文履歴 → 該当注文の【編集】タップ
       → カートに「ハンバーグ ¥1,580 × 1」が自動ロード
       → ハンバーグをタップ → オプションモーダル「チーズ +¥200」を選択
       → カートに「ハンバーグ (チーズ) ¥1,780 × 1」と元の「ハンバーグ ¥1,580 × 1」の 2 行
       → 元の「ハンバーグ ¥1,580」を [-] で削除
       → 【注文を更新】タップ (オレンジ色)
19:52  PATCH 成功 → トースト「注文を更新しました」
19:52  KDS 側でも品目が更新される (注文 ID は同じ)
```

### 6.8.9 編集できないケースの代替手段

| ケース | 代替策 |
|---|---|
| 会計済 + 一部返金したい | レジ画面 (cashier.html) の【返金】機能 (CR1)。8 章参照 |

## 6.10 テーブル開放 (📲 QR開放、L-9 連携)

### 6.10.1 Walk-in 客対応の新フロー

従来: 客来店 → スタッフが着席登録 → お客様スマホで QR 読む → 追加情報入力

新方式 (25 章参照): 客来店 → スタッフ **「📲 QR開放」** 1 タップ → 客が QR 読むだけ → 自動着席

### 6.10.2 QR開放の手順

1. **【テーブル状況】** タブ
2. 空席カード (緑色でない) の **【📲 QR開放 (5分)】** をタップ
3. 確認ダイアログ: 「テーブル T03 を客向けに開放します。5分以内に客が QR を読み込むと自動着席します。よろしいですか?」
4. **【OK】** をタップ
5. トースト「テーブル T03 を開放しました (5分有効)」
6. お客様が QR を読むと**数秒後にカード状態が「着席中」(緑) に自動更新**

### 6.10.3 開放失敗時

- テーブルが既に使用中 → エラー「テーブルは使用中です」
- ネット不通 → エラー「通信エラー」

### 6.10.4 予約客は開放不要 (自動マッチング)

予約時刻 ±30分以内 + テーブル割当済の予約客は、**スタッフ操作なしで自動着席**。ハンディの予約タブ・テーブル状況タブにリアルタイム反映。

### 6.10.5 QR開放のチェックリスト (実施前)

| # | 確認項目 | 確認方法 |
|---|---------|---------|
| 1 | テーブルカードが「空席」 (グレー) | テーブル状況タブで対象カードの色 |
| 2 | テーブル管理で `is_active=1` | 5 章テーブル管理タブで確認 |
| 3 | 同テーブルで他スタッフが操作中でない | テーブル状況の更新を 1 回待ってから操作 |
| 4 | お客様がスマホ + カメラ起動できる状態 | 店頭ですぐスキャンできるよう案内 |

### 6.10.6 開放後 5 分の挙動 (タイムライン)

```
19:30:00  スタッフ「📲 QR開放」タップ → トースト「開放しました」
19:30:05  お客様 QR 読み込み → /customer/menu.html?store=xxx&table=xxx 開く
19:30:10  PIN 入力画面 (店舗設定で PIN 必須なら) → 4 桁入力
19:30:15  table_sessions に INSERT (status='seated', auto_seated=1)
19:30:15  ハンディのテーブル状況カードが「食事中」(緑) に自動更新 (5秒ポーリング後)
   ︙
19:35:00  5 分経過 → token expires → これ以降は QR 読んでも 403 エラー
          スタッフが再開放するか、手動着席する
```

### 6.10.7 開放トラブルシュート

| 症状 | 原因 | 対処 |
|---|---|---|
| トースト「テーブルは使用中です」 | 既に着席中 | 一度【着席キャンセル】 → 再開放 |
| 5 分経過後も自動着席されない | 客が QR を読まなかった | 再度【📲 QR開放】タップ |
| QR 読んでも「セッション無効」表示 | token 期限切れ | 再開放。それでもダメなら 25 章参照 |
| 同じテーブルを連続開放 | 仕様上 OK (token 上書き) | 問題なし |

## 6.11 スタッフ呼び出しアラート

## 6.13 品切れ品目の扱い

### 6.13.1 表示

- 半透明 (opacity 0.4)
- 「品切れ」オーバーレイ表示
- タップしてもカート追加されない

### 6.13.2 サーバー側チェック

注文送信時にサーバーで再チェック。**カート追加後に品切れになった**場合:
- HTTP 409 エラー
- メッセージ: 「品切れの品目が含まれています: [品目名]」
- 該当品目をカートから削除して再送信

### 6.13.3 品切れの設定

ハンディでは品切れの **解除のみ** 可能 (限定メニューの在庫連動品目のみ)。オーナー・マネージャーは「メニュー管理」で品切れトグル。

### 6.13.4 品切れの3層判定

メニュー品切れは 3 つのデータソースから合成判定されます (`api/lib/menu-resolver.php`):

| 層 | テーブル | カラム | 優先度 |
|---|---|---|---|
| 本部マスタ | `menu_templates` | `is_sold_out` | 低 (店舗で上書き可) |
| 店舗オーバーライド | `store_menu_overrides` | `is_sold_out` | 高 (本部 < 店舗) |
| 店舗限定品 | `store_local_items` | `is_sold_out` | 単独 |

**合成ルール**:
- 店舗オーバーライドの `is_sold_out` が `1` → 品切れ
- 店舗オーバーライドが NULL かつ本部マスタの `is_sold_out=1` → 品切れ
- 店舗限定品の `is_sold_out=1` → 品切れ

### 6.13.5 売り切れ操作のロール権限

| ロール | 「メニュー管理」での操作 | ハンディでの操作 |
|---|---|---|
| owner | 全品目の売り切れトグル可 | 不可 (管理画面で操作) |
| manager | 全品目の売り切れトグル可 | 不可 |
| staff | 不可 | 不可 |
| device | 不可 | 不可 |

## 6.14 よくある質問 30 件以上

### A. 操作全般

**Q1.** テーブルを選ばずに注文できる?
**A1.** 不可。店内注文は必ずテーブル選択が必要。

**Q2.** カテゴリを変えるとカートがクリアされる?
**A2.** されない。カテゴリ移動してもカートは保持。

**Q3.** 同じ品目を数量 5 でカートに入れたい
**A3.** 5 回タップ、または展開カートで [+] を 4 回。

**Q4.** テイクアウト注文はテーブル選択不要?
**A4.** その通り。代わりに顧客名・電話・受取時刻が必須。

**Q5.** 注文送信後にカートが空にならない
**A5.** 通信エラーで未送信の可能性。画面再読込して再送信。

### B. オプション関連

**Q6.** オプションの組み合わせを保存できる?
**A6.** 現状不可。毎回選択が必要。

**Q7.** 複数オプションを同時選択したい
**A7.** オプショングループが複数ある場合は可。同じグループ内での複数選択は未対応 (ラジオボタン)。

**Q8.** オプションに数量を付けたい (チャーシュー 3 枚など)
**A8.** 未対応。選択肢として「チャーシュー 2 枚」「3 枚」「4 枚」を別々に登録。

**Q9.** 注文後にオプションだけ変更したい
**A9.** 注文履歴の編集機能で可能。元の品目を削除して新しいオプションで追加。

### C. テーブル状況関連

**Q10.** 着席ボタンと QR 開放の違いは?
**A10.** 着席 = スタッフが手動で人数などを入力して即着席登録。QR 開放 = お客様がスマホ QR で自動着席するための 5 分開放。

**Q11.** 空席なのに「着席」できないテーブルがある
**A11.** 別店舗のテーブルか、is_active=0 になっている可能性。テーブル管理で確認。

**Q12.** テーブル状況の更新が遅い
**A12.** 5 秒ポーリングなので最大 5 秒ラグあり。手動更新は画面下スワイプ。

**Q13.** 超過中 (赤) のテーブルはどう対応?
**A13.** 食べ放題の時間超過。追加注文停止 or 追加料金案内。

### D. 予約関連 (L-9 新機能)

**Q14.** 予約タブと注文履歴タブの違い
**A14.** 予約 = これから来る客、注文履歴 = 既に注文が入った客。

**Q15.** 予約客が時間通りに来ない
**A15.** 予約時刻 +15 分経過で状態確認。連絡なければ no-show 候補 (30 分経過で記録推奨)。

**Q16.** 予約の人数を変更したい
**A16.** 管理画面の予約台帳で ✏️ 変更 → 人数修正。ハンディからは変更不可。

### E. 注文履歴・編集関連

**Q17.** 会計済の注文を編集できる?
**A17.** 不可。返金が必要な場合はレジ画面の返金機能 (C-R1)。

**Q18.** キャンセル注文を復活させたい
**A18.** 不可。同じ内容を新規注文として再入力。

**Q19.** 注文履歴を昨日分も見たい
**A19.** ハンディでは当日のみ。昨日以前は管理画面の「注文履歴」タブ。

### F. トラブル・技術

**Q20.** ハンディが固まった
**A20.** ブラウザタブを閉じて再度開く。カート内容は失われる。

**Q21.** Wi-Fi 切断時の動作
**A21.** オフライン警告バナー表示。注文送信は再接続後に自動再送 (order-sender.js の pending queue)。

**Q22.** ログアウトボタンを押したらどうなる?
**A22.** 即ログアウト。未送信カートは失われる。

**Q23.** 別のスタッフの注文履歴を見たい
**A23.** ハンディでは全員分が同じリストに表示 (フィルタなし)。管理画面で「スタッフ別」レポート参照。

**Q24.** ハンディで会計できる?
**A24.** 不可。会計はレジ画面 (kds/cashier.html) で行う。

**Q25.** iPad でも動作する?
**A25.** 動作する。Safari 最新版推奨。Chrome でも可。

### G. ラストオーダー関連

**Q26.** ラストオーダー時刻を過ぎたらハンディから注文できる?
**A26.** 不可。サーバー側 (`api/customer/orders.php` および `api/store/handy-order.php`) で `LAST_ORDER_PASSED` (HTTP 409) を返してブロックします。判定は (1) 店舗全体の `store_settings.last_order_time` (2) `store_settings.last_order_active=1` (3) プランセッションの `table_sessions.last_order_at` の 3 経路。

**Q27.** ラストオーダー時刻だけ一時的に延長したい
**A27.** 「設定」タブ →「ラストオーダー時刻」から `last_order_time` を編集 (manager 以上)。延長 → 即座にサーバー側判定に反映。

**Q28.** ラストオーダー後でも追加注文を取りたい (例: 常連客)
**A28.** 「設定」タブの `last_order_active` を `0` (オフ) に切り替え → その間は通常受付。終了後に必ず戻す。

### H. テイクアウト関連

**Q29.** 受取時刻が過ぎても客が来ない
**A29.** ハンディの注文履歴で該当注文を確認 → 電話連絡 → 来店未定なら `cancelled` ではなく状態を保留。会計済の場合のみレジで返金処理。

**Q30.** 同じ顧客が複数注文した場合
**A30.** 別注文として個別に作成される (顧客名は同じでも UUID が異なる)。レジで一括会計したい場合は、注文 ID をまとめて選択して合算決済 (8 章レジ参照)。

**Q31.** テイクアウトに座席指定はできる?
**A31.** 不可。`order_type='takeout'` は `table_id=NULL` で記録されます。店内席で食べる場合は店内注文タブを使用。

### I. 多店舗・権限関連

**Q32.** 複数店舗のハンディを 1 端末で切り替えて使える?
**A32.** 可能。ヘッダー右の「店舗切替」リンクから再選択。ただし切替前のカートは破棄されます。

**Q33.** 自分の所属店舗以外のテーブルが選択肢に出てしまう
**A33.** バグ。`api/store/tables.php` が `require_store_access()` で店舗境界を強制しているため通常は出ない。ブラウザキャッシュをクリアして再ログイン。

**Q34.** スタッフ A が入力した注文をスタッフ B が編集できる?
**A34.** 可能。ハンディは「誰でも全注文を編集可」設計。ただし監査ログ (`audit_logs`) に編集者 ID が記録される。

### J. 監査・データ確認

**Q35.** ハンディからの注文と顧客セルフ注文の見分け方
**A35.** `orders.order_type` で判別。`handy` = ハンディから入力、`self` = 顧客 QR メニューから、`takeout` = テイクアウト、`kds` = レジから直接。

**Q36.** 一日の総注文数を確認したい
**A36.** ハンディからは不可 (当日リストのみ表示)。管理画面 (`dashboard.html`) → 「レポート」タブ → 「日次サマリー」を参照。

### 6.16.1 関連テーブル

#### orders (注文本体)

主要カラム:
- `id`, `tenant_id`, `store_id`, `table_id` (nullable for takeout)
- `order_type` ENUM('handy','self','takeout','kds')
- `items` JSON ([{name, qty, price, options, memo}])
- `status` ENUM('pending','preparing','ready','served','paid','cancelled')
- `memo` TEXT — 注文全体メモ
- `payment_method` ENUM('cash','card','qr')
- `customer_name`, `customer_phone`, `pickup_at` — takeout 用
- `created_by_user_id` — 入力スタッフ
- `created_at`, `updated_at`, `paid_at`

#### order_items (品目単位ステータス管理、C-4)

- `id`, `order_id`, `menu_item_id`
- `item_name`, `quantity`, `unit_price`
- `options` JSON
- `memo`
- `status` ENUM('pending','preparing','ready','served','cancelled')

#### table_sessions (テーブルセッション)

- `id`, `store_id`, `table_id`, `reservation_id` (L-9 で追加)
- `status` ENUM('seated','eating','bill_requested','paid','closed')
- `guest_count`
- `started_at`, `closed_at`
- `plan_id`, `course_id`, `time_limit_min`, `expires_at`
- `memo`, `session_pin` (S6)

#### call_alerts (スタッフ呼び出し)

- `id`, `store_id`, `table_id`
- `alert_type` ENUM('staff_call','product_ready','kitchen_call')
- `body` TEXT
- `status` ENUM('pending','acknowledged','resolved')
- `created_at`, `acknowledged_at`

#### cart_events (カート操作ログ、C5)

迷った品目分析・離脱分析用 (顧客セルフメニュー側で主に書き込み、ハンディからは将来用)。

| カラム | 型 | 用途 |
|---|---|---|
| `id` | BIGINT AUTO_INCREMENT | PK |
| `store_id` | VARCHAR(36) | テナント境界 |
| `table_id` | VARCHAR(36) NULL | NULL = テイクアウト or 開放前 |
| `session_token` | VARCHAR(64) NULL | テーブル QR セッション識別 |
| `item_id` | VARCHAR(36) | 操作対象品目 |
| `item_name` | VARCHAR(100) | 表示名 (スナップショット) |
| `item_price` | INT | 価格 (スナップショット) |
| `action` | ENUM('add','remove') | カートに追加 / 削除 |
| `created_at` | DATETIME | 操作日時 |

インデックス: `(store_id, created_at)` / `(item_id, action)` — ABC 分析や「カート追加 → 削除」率の計算に使用。

API: `POST /api/customer/cart-event.php` (認証なし、レートリミット 60 回/10 分/IP)。

### 6.16.2 API エンドポイント

#### 注文系
- `POST /api/customer/orders.php` — 注文作成 (客側も使用、**レートリミット `customer-order` 20 req / 10 分**)
- `POST /api/store/handy-order.php` — ハンディ専用の注文作成 (manager 以上)
- `PATCH /api/store/handy-order.php` — 注文の修正 (編集モード時、paid/cancelled は不可)
- `GET /api/kds/orders.php?store_id=xxx&view=accounting` — 当日注文一覧（ハンディの【注文履歴】タブ用）

#### メニュー系
- `GET /api/customer/menu.php?store_id=xxx` — カテゴリ + 品目 + オプション
- `GET /api/store/menu-version.php?store_id=xxx` — バージョン番号のみ (軽量)
