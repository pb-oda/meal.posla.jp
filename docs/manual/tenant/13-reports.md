# 13. レポート

管理ダッシュボードの「レポート」タブでは、売上・レジ・回転率・スタッフ評価・併売分析・**予約** (L-9 新機能) のレポートを閲覧できます。マネージャー以上のロールで利用できます。

---

## 13.0 はじめにお読みください

このページは「POSLA で取得できる全レポートを、初めて触る人でも 1 ステップずつ実行できる」ように、画面遷移・入力フィールド・実例つきで解説した完全ガイドです。

::: tip この章を読む順番（おすすめ）
1. **13.0.1 レポート全体マップ** で、どのレポートがどの判断に使えるかを把握
2. **13.0.2 共通の前提条件** で必須チェックリストを確認
3. **日次運用** → 13.2 売上 → 13.3 レジ → 13.4 回転率 → 13.5 スタッフ
4. **週次・月次レビュー** → 13.6 併売 + 13.7 ABC（owner）
5. **監査・障害対応** → 13.8 監査ログ + 13.9 エラー監視
6. うまくいかなかったら → 13.Y FAQ + 13.Z トラブルシューティング
:::

各レポートを開く前に、必ず以下を確認してください。

| # | 確認項目 | 確認方法 | NG 時の対応 |
|---|---|---|---|

## 13.1 共通操作

### 13.1.1 期間プリセットボタン

全レポートの上部に同じプリセット 4 ボタンが並びます。クリックすると `from` / `to` の日付フィールドが即座に書き換わり、自動で再読込されます。

| プリセット | 設定される from | 設定される to | 用途 |
|---|---|---|---|
| 今日 | 当日 (営業日基準) | 当日 | 営業中のリアルタイム確認 |
| 昨日 | 前営業日 | 前営業日 | 締め後の振り返り |
| 今週 | 月曜日 | 当日 | 週次レビュー |
| 今月 | 1 日 | 当日 | 月次レビュー |

::: tip 営業日基準
「営業日区切り時刻」（例: 03:00）を考慮。深夜営業店では「2026-04-19 02:30」の注文は 4/18 の売上として集計（実装: `api/lib/business-day.php`）。
:::

### 13.1.2 カスタム期間 / CSV / 店舗フィルタ

- **カスタム期間**: from / to 日付ピッカー → 「表示」ボタン（6 ヶ月超は遅延注意）
- **CSV エクスポート**: 各 API に `?format=csv` を付与。文字コード UTF-8 with BOM（Excel 自動認識）
- **店舗フィルタ (owner)**: ABC・クロス店舗レポートに「全店舗合算 / 個別店舗」ドロップダウン。manager は自店舗固定

### 13.1.5 画面遷移

```
[dashboard] → 「レポート」タブ → サブタブ → プリセット + 日付 → 「表示」
   ↓
[サマリーカード] + [詳細テーブル / バーチャート]
```

## 13.2 売上レポート

「レポート」タブ → 「売上レポート」サブタブ

### 13.2.1 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | 店舗が選択されている |
| 2 | 期間に少なくとも 1 件の `paid` ステータスの注文がある |
| 3 | レジ締めが完了している（締め前は `paid_at` が未設定の注文が含まれる） |

### 13.2.2 操作手順（クリック単位）

1. dashboard.html → 上部タブ **「レポート」** をクリック
2. サブタブ **「売上レポート」** をクリック
3. 既定で「今日」のデータが表示される
4. 期間を変えたい場合はプリセットボタン or 日付ピッカー
5. **「表示」** ボタンをクリック（プリセット選択時は自動再読込）
6. 1〜3 秒後にデータが描画される

### 13.2.3 表示されるセクション

#### 13.2.3.1 サマリーカード

| カード | 値 | 計算式 |
|---|---|---|
| 注文数 | 期間内の `status='paid'` 注文件数 | `COUNT(*)` |
| 売上合計 | 期間内の総売上 | `SUM(total_amount)` |
| 平均注文額 | 売上 ÷ 注文件数 | `AVG(total_amount)` |

#### 13.2.3.2 提供時間分析（service time analysis）

注文受付（`created_at`）から提供（`served_at` または `ready_at`）までの秒数。

| カード | 説明 |
|---|---|
| 平均 | 全注文の平均提供時間 |
| 中央値 | データの中央 50% の境界値 |
| 最短 | 最も早かった提供時間 |
| 最長 | 最も遅かった提供時間 |

::: tip 平均と中央値が乖離している場合
特定の遅い注文（厨房トラブル等）が平均値を引き上げています。中央値で実態を把握してください。
:::

#### 13.2.3.3 品目ランキング（売上順 TOP 20）

| 列 | 説明 |
|---|---|
| 順位 | 1〜3 は 🥇🥈🥉、4 位以降は数字 |
| 品名 | メニュー名 |
| 数量 | 期間内の販売数 |
| 売上 | その品目の合計売上 |

#### 13.2.3.4 時間帯別売上（バーチャート）

| 列 | 説明 |
|---|---|
| 時間帯 | 0時〜23時 |
| 注文数 | その時間帯の注文件数 |
| 売上 | その時間帯の売上金額 |

棒の長さは「最大売上の時間帯 = 100%」で正規化。

### 13.2.4 入力フィールドの詳細表

| # | フィールド | 入力例 | 必須 | 形式 | 補足 |
|---|---|---|---|---|---|
| 1 | from | `2026-04-01` | ✅ | YYYY-MM-DD | 営業日基準 |
| 2 | to | `2026-04-19` | ✅ | YYYY-MM-DD | from <= to |
| 3 | プリセット | today/yesterday/week/month | 任意 | クリック | from/to を上書き |

### 13.2.5 実例（1 日サマリー）

```
注文 87 / 売上 ¥184,200 / 平均 ¥2,117
提供時間 平均 11:32 / 中央値 9:12 / 目標 10 分超過
1 位 焼き魚定食 28 食 / ピーク 12 時 24 件
```

→ 12 時集中は厨房ボトルネック。14 章「需要予測」で人員追加を検討。

## 13.3 レジ分析

「レポート」タブ → 「レジ分析」サブタブ

### 13.3.1 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | 当日のレジ開け (`cash_log.type='open'`) が記録されている |
| 2 | (締め後の確認なら) レジ締め (`type='close'`) が記録されている |
| 3 | manager 以上のロール |

### 13.3.2 操作手順

1. dashboard.html → **「レポート」** タブ
2. サブタブ **「レジ分析」** をクリック
3. 既定で「今日」が表示される
4. プリセットまたは日付ピッカーで期間を変更
5. **「表示」** ボタン

### 13.3.3 サマリーカード

| カード | 説明 | 計算式 |
|---|---|---|
| レジ開け | 営業開始時のつり銭準備金 | `cash_log` 最新の `type='open'` |
| 現金売上 | 現金決済の合計 | `payments WHERE method='cash'` |
| 入金 | 営業中の `cash_in` 合計 | — |
| 出金 | 営業中の `cash_out` 合計 | — |
| 予想残高 | 開始 + 現金売上 + 入金 − 出金 | 自動算出 |
| レジ締め | 営業終了時の実際残高 | `type='close'` の amount |
| 過不足 | レジ締め − 予想残高 | 自動算出 |

#### 過不足の色分け

| 状態 | 色 | 意味 | 対応 |
|---|---|---|---|
| マイナス（不足） | 赤 (#e53935) | 実際残高 < 予想残高 | 釣り銭ミス・抜き取り疑惑 → 監査ログ確認 |
| ゼロ | 黒 | 完全一致 | 正常 |
| プラス（過剰） | 青 (#1565c0) | 実際残高 > 予想残高 | 入金ログ漏れ・釣り銭返し忘れ |

::: warning 500 円以上の過不足
スタッフ評価レポート (13.6) で行ハイライトされます。3 日連続で発生したら次の対策を検討:
- レジ担当 PIN（CB1d、18 章 18.Z 参照）の運用徹底
- レジカメラ設置
- スタッフ研修
:::

### 13.3.4 支払方法別内訳

| 列 | 説明 |
|---|---|
| 方法 | 現金 / カード / QR (`payment_method` ENUM) |
| 件数 | 決済件数 |
| 金額 | 合計金額 |

### 13.3.5 キャッシュログ詳細（時系列）

| 列 | 説明 |
|---|---|
| 時刻 | 操作日時 |
| 種別 | レジ開け / 締め / 入金 / 出金 / 現金売上 |
| 担当者 | スタッフ名（PIN 認証されたユーザー） |
| 金額 | 操作金額 |
| メモ | 自由入力（出金理由など） |

### 13.3.6 実例（1 日のレジ）

```
開け ¥30,000 / 現金売上 ¥86,400 / 出金 ¥1,200 / 予想 ¥115,200 / 締め ¥114,800
過不足 -¥400 (赤)
```

→ 18 章「PIN 監視」+ 監査ログで時間帯特定。

## 13.4 注文履歴

「レポート」タブ → 「注文履歴」サブタブ

### 13.4.1 用途

- お客様クレーム時の個別注文調査
- 「11:42 のテーブル A の注文内容」を特定
- キャンセル注文の詳細確認

### 13.4.2 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | 検索したい時間帯 / テーブル / 品名のいずれかが分かっている |
| 2 | manager 以上 |

### 13.4.3 操作手順

1. dashboard.html → **「レポート」** → **「注文履歴」**
2. プリセットまたは日付ピッカー
3. **ステータス** ドロップダウンで絞り込み（任意）
4. **検索ボックス** に品名を入力（任意、JSON LIKE 検索）
5. **「表示」** ボタン
6. 1〜3 秒で 20 件ずつページネーション表示

### 13.4.4 入力フィールドの詳細表

| # | フィールド | 入力例 | 必須 | 形式 | 補足 |
|---|---|---|---|---|---|
| 1 | from | `2026-04-15` | ✅ | YYYY-MM-DD | |
| 2 | to | `2026-04-15` | ✅ | YYYY-MM-DD | |
| 3 | status | `paid` / `cancelled` 等 | 任意 | ENUM | 6 値から選択 |
| 4 | q (検索) | `焼き魚` | 任意 | 部分一致 | items JSON LIKE |
| 5 | page | `1` | 任意 | 整数 | 既定 1 |
| 6 | limit | `20` | 任意 | 10-100 | 既定 20、最大 100 |

### 13.4.5 ステータスの種類

| status | 表示 | 意味 |
|---|---|---|
| `pending` | 受付 | KDS 投入直後 |
| `preparing` | 調理中 | 厨房着手 |
| `ready` | 提供待ち | 完成・配膳待ち |
| `served` | 提供済み | お客様に渡した |
| `paid` | 会計済み | 集計対象 |
| `cancelled` | キャンセル | 売上から除外 |

### 13.4.6 表示テーブル列

| 列 | 説明 |
|---|---|
| 時刻 | `created_at` |
| テーブル | `table_code` |
| 品目 | `name × qty` のカンマ区切り |
| 削除品 | `removed_items`（取り消された品目、赤色取り消し線） |
| 金額 | `total_amount` |
| 提供時間 | `served_at - created_at` |
| ステータス | バッジ表示 |

### 13.4.7 実例（クレーム調査）

「12:00 / C-3 / 焼き魚」→ 検索ボックス「焼き魚」+ 今日 → 結果「12:03 / C-3 / ¥1,100 / 提供時間 28 分」→ 調理時間異常を 13.5 と比較、必要なら 13.10 監査ログで status 遷移確認。

## 13.5 回転率・客層レポート

「レポート」タブ → 「回転率・客層」サブタブ（実装: `turnover-report.js`）

### 13.5.1 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | テーブルセッション機能（QR 入店）が有効 |
| 2 | 当該期間に `table_sessions` レコードがある |
| 3 | manager 以上 |

### 13.5.2 操作手順

1. **「レポート」** → **「回転率・客層」** サブタブ
2. プリセット（既定: 今日）
3. **「表示」** ボタン

### 13.5.3 表示セクション

#### サマリー（4 カード）

| カード | 説明 |
|---|---|
| 平均滞在 | 全セッションの平均滞在分 |
| 平均回転 | テーブルあたりの平均利用回数 |
| ピーク時間帯 | 最も組数が多い時間帯 |
| 調理超過率 | 目標調理時間を超えた品目の割合（≥20% で赤） |

#### テーブル別回転率

| 列 | 説明 |
|---|---|
| テーブル | テーブルコード |
| 回転数 | 利用回数 |
| 平均滞在 | 分単位 |
| 合計客数 | — |

#### ランチ / ディナー比較

| 時間帯 | 範囲 |
|---|---|
| ランチ | 10:00〜15:00 |
| ディナー | 17:00〜23:00 |
| その他 | 上記以外 |

各時間帯について：組数、客数、平均滞在、売上

#### 時間帯別効率（棒グラフ）

1 時間ごとの組数・客数・売上・平均滞在

#### 調理時間分析

- **品目別**: 各品目の平均・最大調理秒数、超過時はオレンジ
- **全体サマリー**: 平均、中央値、目標時間（店舗設定）、超過率

#### 組人数別分析

| 列 | 説明 |
|---|---|
| 人数帯 | 1名 / 2名 / 3名 / 4名以上 |
| 組数 | — |
| 客数 | — |
| 組平均単価 | — |
| 客単価 | — |

#### 価格帯別注文分布（バーチャート）

注文金額を価格帯で 5-10 段階に分けて表示。

#### 時間帯別客層

各時間帯の組数・客数・平均人数・客単価。

### 13.5.4 実例（週次サマリー）

```
平均滞在 42 分 / 平均回転 4.2 / ピーク 19 時 / 超過率 8%
ランチ:   142 組 / 198 人 / 32 分 / ¥197,400
ディナー: 186 組 / 412 人 / 58 分 / ¥643,200
```

→ ディナー 58 分は業界平均 (45-60 分) の上限。回転率向上に「90 分制」導入の余地あり。

## 13.6 スタッフ評価レポート

「レポート」タブ → 「スタッフ評価」サブタブ（実装: `staff-report.js`、API: `staff-report.php`）

### 13.6.1 操作前のチェックリスト

| # | 確認項目 |
|---|---|
| 1 | 各スタッフが個別アカウント（共有禁止、CLAUDE.md 基本ルール） |
| 2 | ハンディ POS / レジ PIN を運用中 |
| 3 | manager 以上 |

### 13.6.2 操作手順

1. **「レポート」** → **「スタッフ評価」**
2. プリセットまたは日付ピッカー
3. **「表示」** ボタン

### 13.6.3 アラートバナー（最上部）

| 表示 | 意味 |
|---|---|
| 緑「異常なし」 | 期間内にアラートなし |
| 赤「アラート N 件」 | severity=danger / warning が検出 |

severity=danger 例:
- レジ過不足が連続 3 日マイナス
- 同一スタッフのキャンセル率 ≥30%

### 13.6.4 スタッフパフォーマンス一覧

| 列 | 説明 |
|---|---|
| スタッフ | 表示名 |
| 役職 | オーナー / マネージャー / スタッフ |
| 注文数 | ハンディ POS で投入した件数 |
| 売上 | 担当注文の合計 |
| 客単価 | 売上 ÷ 注文数 |
| キャンセル率 | キャンセル ÷ 全注文。20% 以上は赤太字 |
| 満足度 | お客様評価平均（評点 + 件数）。3.0 未満赤、4.0 以上緑 |

::: tip device ロールはスタッフ集計から除外
KDS / レジの端末アカウント（`role='device'`）はスタッフではないため、評価レポートに含まれません。シフト管理・人件費・スタッフレポート全てから除外（CLAUDE.md「device ロール」参照）。
:::

## 13.Y FAQ 25 件

**Q1.** 売上が会計伝票と合わない → 営業日区切り時刻 (設定 > 営業日) を確認。`api/lib/business-day.php`。

**Q2.** 過去データはいつまで → 全期間保存。削除機能なし。

**Q3.** CSV が文字化け → Excel「データ → テキストファイル → UTF-8」。最新 Excel は BOM 自動認識。

**Q4.** ABC 比率 (70/20/10) を変えたい → 現状固定。Phase 2 で設定 UI 予定。

**Q5.** スタッフ評価の基準 → 注文 + 売上 + キャンセル率 + 満足度 + 出勤の総合（`staff-report.php` 参照）。

**Q6.** クロス店舗 vs 通常レポート → クロス = 複数店舗比較 (owner)。通常 = 単一店舗詳細 (manager+)。

**Q7.** リアルタイム売上 → 即時反映（キャッシュなし）。「表示」で都度再取得。

**Q8.** スマレジ連携 → 現状不可。POSLA 単独運用。

**Q9.** グラフ画像 DL → 現状テーブル / CSS バー。Phase 2 で Chart.js 予定。

**Q10.** 予約 no-show 業界水準 → 3-5%。10% 超は予約金導入検討。

**Q11.** カスタムレポート → 不可。CSV → Excel ピボット推奨。

**Q12.** staff にレポート権限 → 不可。manager 以上のみ。

**Q13.** 月次定期メール → Phase 2 予定。現状は手動 CSV → 外部ツール。

**Q14.** バスケット分析の最小データ量 → 100 取引以上が目安。

**Q15.** AI 分析と通常レポートの違い → レポート = 数値。AI = Gemini が文章で解釈・提案 (14 章)。

**Q16.** device ロールはレポートに出る → スタッフ評価から除外。監査ログ・注文履歴には記録。

**Q17.** 監査ログが重い → 期間 + 操作種別で必ず絞り込む。owner は 1 日単位推奨。

**Q18.** エラー監視「未作成」 → CB1 マイグレーション未実行。POSLA 運営連絡。

**Q19.** 過不足が常にプラス → 釣り銭戻し忘れ / 過剰受領。スタッフ評価で傾向確認。

**Q20.** 提供時間が異常に長い → 注文履歴で個別確認。`served_at` 押し忘れも含む。

**Q21.** 平均滞在が極端に長い → table_sessions が `session_end` 未設定。25 章参照。

**Q23.** ABC 合算と個別で結果違う → 合算では主力品が薄まる。経営=合算 / 店舗運営=個別。

**Q24.** クロス店舗の「キャンセル」項目 → 2026-04 以降の追加。金額 / 件数 / 率を可視化。

**Q25.** 監査ログ device_create が増えている → P1a の正常動作。新規 KDS / レジ追加で発生。
