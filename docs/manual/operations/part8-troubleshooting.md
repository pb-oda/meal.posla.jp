# Part 8 — トラブルシューティング（困ったとき）

このページでは、営業中によく起きるトラブルと、**その場での対処法** を順序立てて説明します。

::: tip 基本の復旧 3 ステップ
何が起きても、まず以下の 3 つを試してください。

1. **ブラウザをリロード**（Cmd+Shift+R / Ctrl+Shift+F5）
2. **Wi-Fi 接続を確認**（画面右上のアイコン）
3. **1 分待って再試行**

これで 6 割のトラブルは解消します。それでもダメなら下の詳細手順へ。
:::

---

## 8-0. エラーが出たら（2026-04-19 追加）

POSLA でエラーが出ると、画面に **以下のような形式で「番号付きメッセージ」**が出ます。

```
[E3024] PIN が一致しません
```

### 8-0-1. やること（バイトスタッフ向け）

1. **番号（`Exxxx`）を控える**（紙にメモ・スクショ・声に出す）
2. **マネージャーに伝える**（口頭でOK）
3. マネージャーが管理画面の **オレンジ ? ボタン（右下）** で AI に質問して対処方法を引き出す

### 8-0-2. スタッフは AI ヘルプデスクを使わない

::: warning スタッフ用端末（KDS / レジ / ハンディ）には AI ヘルプデスクのボタン（FAB）はありません
- スタッフが操作で困ったときは **必ずマネージャーに聞く** のが POSLA の運用ルールです
- AI ヘルプデスクは管理画面（dashboard / owner-dashboard）にだけ設置されています
- 業務中の操作集中度を保つため、KDS・レジ・ハンディの画面には何も置きません
:::

### 8-0-3. マネージャー向け：エラー番号の調べ方

1. 管理画面（`dashboard.html` や `owner-dashboard.html`）右下の **オレンジ色 ? ボタン** をタップ
2. チャットパネルが開いたら、`[E3024]` のように番号をそのまま貼り付ける
3. AI がエラーカタログ（章 99）と該当章を照合して対処方法を返す

エラー番号の桁の意味（参考）:

| 範囲 | 系統 |
|---|---|
| E1xxx | 認証・セッション |
| E2xxx | 入力ミス |
| E3xxx | 会計・PIN・決済 |
| E4xxx | 注文・KDS |
| E5xxx | 在庫・メニュー |
| E6xxx | 予約 |
| E7xxx | 印刷 |
| E8xxx | 外部 API |
| E9xxx | サーバー側（POSLA 運営マター） |

## 8-1. ログインできない

### 8-1-1. エラーメッセージ別の対処

| エラー文言 | 原因 | 対処法 |
|---|---|---|
| 「ユーザー名またはパスワードが正しくありません」 | 入力ミスの可能性が最も高い | 大文字小文字、スペース、似た文字（`I/l/1`、`O/0`）の見間違いを確認。再試行は何度でも可（自動ロック機能は未実装、2026-04-18 時点） |
| 「このアカウントは無効化されています」 | 店長がアカウントを停止 | 店長に連絡して有効化してもらう |
| 「このテナントは無効化されています」 | 契約が止まっている | POSLA 運営（info@posla.jp）に連絡 |
| 「ユーザー名とパスワードは必須です」 | 片方が空欄 | 両方を入力 |
| 「通信エラーが発生しました」 | Wi-Fi 切れ or サーバー障害 | Wi-Fi 確認、1 分後再試行 |
| 「サーバーが空のレスポンスを返しました」 | POSLA サーバー側の問題 | 5 分待って再試行。繰り返す場合 POSLA 運営に連絡 |
| 画面真っ白 | ブラウザ or JavaScript エラー | Cmd+Shift+R で強制リロード |
| URL が違う | URL タイプミス | 正しい URL: `https://eat.posla.jp/public/admin/index.html`（`/public/` プレフィックス必須） |

### 8-1-2. パスワードを忘れた場合

**スタッフ（staff）が忘れた**:
1. 店長に依頼
2. 店長が管理画面（dashboard → スタッフ管理）で **【パスワードリセット】**
3. 新しい仮パスワードをスタッフに伝える
4. スタッフは次回ログイン後に自分で好きなパスワードに変更

**店長（manager）が忘れた**:
1. オーナーに依頼
2. オーナーが owner-dashboard → ユーザー管理 で **【パスワードリセット】**

**オーナーが忘れた**:
1. `info@posla.jp` に連絡
2. 本人確認のため **契約時の代表者名 + 契約時のメール** を送る
3. POSLA 運営が 24 時間以内にリセット対応

### 8-1-3. ログイン画面が開かない

**症状**: URL を開いても「管理画面」と表示されない

**対処**:
1. URL を再確認（`https://eat.posla.jp/public/admin/`）
2. Wi-Fi 接続確認
3. Chrome を完全終了してもう一度起動
4. 違うブラウザで試す（Safari 等）
5. 別のタブレット・スマホで試す（問題の切り分け）
6. 全部ダメなら POSLA サーバー障害の可能性 → `info@posla.jp` に連絡

## 8-2. 注文が入らない / 画面が動かない

### 8-2-1. ハンディでの注文送信トラブル

| 症状 | 原因 | 対処 |
|---|---|---|
| ボタンが反応しない | Wi-Fi 切れ or JavaScript エラー | Wi-Fi 確認 → Cmd+R リロード |
| 「カートが空です」エラー | メニュー未選択で送信を押した | メニューをタップしてカートに追加してから送信 |
| 「テーブルを選択してください」 | テーブル未選択 | 画面上部のドロップダウンからテーブル選択 |
| 「品切れが含まれています」 | カート内に品切れ品目がある | 該当品目をカートから削除して再送信 |
| 「セッションが無効です」 | 4 時間経過で切れた | 再ログイン |
| 「注文上限に達しています」 | 1 時間 / 1 テーブルの上限（20 件/10 分）超過 | 10 分待って再送信 |
| タイムアウト | 通信遅い | 1 分待って再試行、別端末で試す |

### 8-2-2. セルフオーダーでお客様が注文できない

**症状**: お客様から「メニューで注文ボタンが押せない」と言われた

| 症状 | 原因 | 対処 |
|---|---|---|
| 「ラストオーダー時刻を過ぎました」 | 設定されたラストオーダー時刻超過 | マネージャーが店舗設定で LO 時刻を確認、必要なら延長 |
| 「セッションが期限切れです」 | QR 読み込みから 4 時間経過 | スタッフが新しい QR を発行（ハンディで 📲 QR 開放） |
| QR コードが読めない | QR が汚れ / 折れている | 管理画面で QR 再印刷して貼り直し |
| 画面が真っ白 | スマホブラウザのバグ | お客様のスマホで Chrome / Safari を再起動してから QR を読み直す |

### 8-2-3. 注文の送信後、キッチンに届いていない

**症状**: ハンディで注文送信したのに KDS 画面に現れない

**対処**:
1. KDS 画面をリロード（Cmd+Shift+R）
2. ステーション設定で **「全品目」** に切替（ステーション別に絞られている可能性）
3. ハンディの **【注文履歴】** タブで該当注文があるか確認
4. あるのに KDS に出ない → POSLA サーバー側の問題 → 運営に連絡

## 8-3. KDS に注文が表示されない

### 8-3-1. 症状別対処

| 症状 | 原因 | 対処 |
|---|---|---|
| 新規注文が全く入らない | ポーリング停止 | ブラウザリロード。音声の「タップして開始」を実行していないと通知音が鳴らないので見落としている可能性 |
| 一部品目だけ表示されない | ステーション設定が絞り込んでいる | 画面上部のステーションバーで **【全品目】** をタップ |
| 音が鳴らない | 音声未許可 | 初回 KDS 起動時の **【タップして開始】** が実行されていない。再ログインして必ずタップ |
| 更新が遅い | 3 秒間隔のポーリング | 仕様通り。手動リフレッシュは不要だが、心配なら Cmd+R |
| 調理ボタンが反応しない | 通信エラー | 3 秒待って状態が戻るか確認。戻らないならリロード |
| カードが古い情報のまま | ブラウザキャッシュ | Cmd+Shift+R で強制更新 |

### 8-3-2. KDS 画面が赤く点滅

**症状**: 注文カードが赤く点滅している

**意味**: 注文から **20 分以上経過**（`kds-renderer.js` L18-19）

**対処**: 
1. 急いで調理・提供
2. お客様にお詫び
3. 遅延が続くなら人員配置・調理フローの見直し

## 8-4. カードリーダーが反応しない

### 8-4-1. 手順で確認

1. **電源ランプ**（本体正面）が緑色点灯しているか確認
   - 消えている → 本体の電源ボタンを 2 秒長押し
2. **Wi-Fi ランプ**（青色）を確認
   - 消えている → ルーターの電源を確認、必要なら再起動
3. POSレジ画面右上の **「Terminal 接続中」** 表示を確認
   - 未接続の場合 → 画面右上の **【再検出】** ボタン
4. それでも無反応 → **USB-C ケーブルでコンセントに接続**（バッテリー切れ）
5. すべてダメなら Stripe 側の障害の可能性 → Stripe Dashboard を確認

### 8-4-2. カードが通らない

**症状**: お客様のカードを当てても「決済失敗」

**対処**:
1. **別のカード** で試す（カード側の問題切り分け）
2. 挿入式カードなら **IC チップ面を正しい向き** で挿入
3. タッチ式なら **カードをリーダーから離して再タッチ**
4. それでも失敗: 現金 / QR 決済に切り替え
5. お客様に無理強いしない（失敗は Stripe 側の判断）

### 8-4-3. 「[STRIPE_MISMATCH]」エラーが出た（2026-04-19 S3 修正）

**症状**: カード決済確定時に画面に `[STRIPE_MISMATCH]` または「決済結果が一致しません」と表示

**意味**: Stripe Terminal から取得した決済結果（`payment_intent_id`）が、POSLA 側 DB に記録された ID と **一致しなかった**。決済確定の直前に何らかの異常（同時操作・Terminal 再接続・Stripe Dashboard での手動操作）が発生した可能性。

**対処**（必須順序）:
1. **会計を確定しない**（現金 / QR 等への切り替えも一旦止める）
2. Stripe Dashboard → Payments で該当時刻の決済記録を確認
3. POSLA 運営（`info@posla.jp`）に連絡:
   - 店舗名・日時
   - お客様への請求金額（カード末尾 4 桁）
   - Stripe Dashboard の event_id（手元にあれば）
4. 運営側で Stripe ↔ POSLA の整合性を確認してから、改めて会計可否を判断

::: warning このエラーは「サーバー側の防御が効いた」サイン
2026-04-19 の S3 セキュリティ修正で導入した検証です。Stripe 側の決済結果と POSLA 側の記録が食い違ったまま会計を確定すると、二重請求や売上集計ズレが発生します。**`STRIPE_MISMATCH` が出たら必ず手を止めて運営に連絡** してください。
:::

詳細: [tenant 18 章 18.W S3 セキュリティ修正一式](../tenant/18-security.md#18w-s3-セキュリティ修正一式2026-04-19) / [tenant 19 章 19.S3](../tenant/19-troubleshoot.md#19s3-s3-セキュリティ修正で追加された新エラーコード2026-04-19)

## 8-5. Wi-Fi が切れた / インターネットに繋がらない

### 8-5-1. 症状の確認

1. タブレット画面右上の Wi-Fi アイコン確認
   - 扇形 3 本 → 強い
   - 扇形 1 本 → 弱い
   - バツマーク → 切れている
2. ルーターのランプ確認（緑点灯が正常、赤は異常）

### 8-5-2. 対処（順番に試す）

1. **タブレットの Wi-Fi を OFF → ON**（設定アプリから）
2. **ルーターを再起動**:
   - ルーターの電源コンセントを **30 秒間抜く**
   - 再度挿す
   - 緑ランプが全部点灯するまで待つ（約 1 分）
3. **違う端末**（スマホ等）で Wi-Fi に繋いでみる
4. **ブラウザで `google.com` を開いてみる**（Wi-Fi は繋がっているけどインターネットがない場合）
5. それでもダメなら **プロバイダ（NTT / ソフトバンク等）に連絡**

### 8-5-3. オフライン時の POSLA の動き

Wi-Fi が切れている間の POSLA の挙動:

- **ハンディの注文**: 自動的にブラウザに保存（`mt_pending_orders`）→ 再接続時に自動送信
- **KDS**: 画面上部にオフラインバナー表示 + 30 秒ポーリング
- **POSレジ**: 会計は失敗する（サーバーに届かない）
- **セルフオーダー**: お客様のスマホにも同様に保留注文が残る（24 時間以内に接続復旧すれば自動送信）

## 8-6. お客様の QR コードが読めない

### 8-6-1. 対処フロー

1. **QR コードの物理確認**:
   - 汚れていないか（拭く）
   - 折れ目・シワがないか
   - 印字がかすれていないか
2. **QR 再印刷**:
   - dashboard → 【テーブル・フロア】 → 該当テーブルの【QR 印刷】
   - 新しい QR をテーブルに貼る
3. **それでも読めない場合のフォールバック**:
   - スタッフがハンディで **【+ Walk-in】** から手動着席
   - スタッフがお客様の注文を代わりに取る

### 8-6-2. お客様のスマホ側の問題

| 症状 | 原因 | 対処 |
|---|---|---|
| カメラが QR 認識しない | 古い iOS / Android | iOS 14+, Android 9+ 推奨 |
| QR を読んでも URL が出ない | QR リーダーアプリの問題 | カメラアプリで直接読み取り、または Chrome で読み取り |
| URL をタップしても開かない | ブラウザのポップアップブロック | Chrome のプライバシー設定を確認 |

## 8-7. 間違えて注文を入れてしまった

### 8-7-1. 注文履歴から編集（会計前）

1. ハンディ画面の **【注文履歴】** タブ
2. 該当注文をタップ
3. **【編集】** ボタン
4. 品目を追加・削除・数量変更
5. **【注文送信】** で上書き（**PATCH** リクエスト）

### 8-7-2. 注文をキャンセル（会計前）

1. KDS 画面で該当注文カードをタップ
2. **【取消】** ボタン（赤）
3. 確認ダイアログ → **【OK】**
4. 注文が `cancelled` ステータスに

### 8-7-3. 会計済みの注文の修正（返金対応）

会計後は編集不可。8-8 返金を参照。

## 8-8. 返金が必要なとき

### 8-8-1. 対象者
manager 以上のロール。staff はできない。

### 8-8-2. 手順

1. POSレジ画面右上の **【📋 取引履歴】** ボタン
2. 該当取引をタップ
3. **【返金】** ボタン（赤）
4. 返金額 + 理由を入力
5. **【返金を確定】**
6. **確認ダイアログ** で最終確認

### 8-8-3. 返金処理の中身

- **Stripe カード決済**: Stripe Refund API で自動返金。**お客様のカードへ 3〜5 営業日で戻る**
- **現金**: POSLA は記録のみ。現金はスタッフが手渡しで返却
- **QR 決済**: PayPay / d払いの管理画面から別途返金（POSLA は記録のみ）

### 8-8-4. 部分返金

全額ではなく一部だけ返す場合:

1. 【返金】モーダルで **金額** 欄に返したい金額を入力（例: ¥1,000 のうち ¥500 だけ返す）
2. 理由入力
3. **【返金を確定】**

## 8-10. 画面が真っ白 / エラーが出る

### 8-10-1. 対処（順番に試す）

1. **ブラウザリロード**（Cmd+Shift+R / Ctrl+Shift+F5）
2. **ブラウザのキャッシュクリア**:
   - Chrome: 設定 → プライバシー → 閲覧履歴データの削除
   - Safari: 設定 → Safari → 履歴と Web サイトデータを消去
3. **違うブラウザ**で試す（Chrome 推奨）
4. **タブレットを再起動**（電源ボタン長押し → 再起動）
5. **違う端末**で同じ操作を試す（問題の切り分け）
6. それでもダメなら `info@posla.jp` に連絡
   - エラー画面のスクリーンショット添付
   - 使用ブラウザ + 端末情報（iPad Pro 12.9, iOS 17 等）

## 8-11. プリンタが印刷しない

### 8-11-1. チェックリスト

1. **プリンタの電源ランプ** 緑点灯か確認
2. **用紙残量** 確認（紙がなくなっていないか）
3. **紙詰まり** 確認（紙送りゼロだと印字できない）
4. **プリンタ本体のカバー** が正しく閉まっているか
5. **Wi-Fi 接続** 確認（プリンタ画面の Wi-Fi マーク）
6. POSLA 管理画面で **「プリンタ設定」→ 【🖨 テスト印刷】** を実行

### 8-11-2. テスト印刷で何も出ない場合

1. プリンタ電源を切る → 10 秒待つ → 再度電源
2. プリンタの **IP アドレスが変わっていないか確認**（FEED ボタン長押しで再確認）
3. POSLA 管理画面でプリンタ設定の IP を更新

詳細は **26 章** 参照。

## 8-12. 予約が反映されない / 自動着席しない

### 8-12-1. お客様が QR を読んでも自動着席しない

**原因の切り分け**:
1. ガント台帳でその予約を確認
   - テーブル割当済みか？ → 未割当ならスタッフが手動割当
   - 予約時刻 ±30 分以内か？ → 範囲外なら手動着席
   - ステータス（confirmed / pending）か？ → キャンセルされていないか
2. お客様が読んだ QR が **正しいテーブル** のものか？
3. 上記クリアなら手動着席ボタン（ガントブロック → 🪑 着席）

### 8-12-2. 予約が LP で作れない

**原因**:
- 予約設定で **「予約受付」** が OFF になっている
- スロット容量を使い切っている（その時刻の席が埋まっている）
- 予約金設定ありで Stripe 未設定（決済画面で止まる）

**対処**:
1. 管理画面の【予約管理】→【予約設定】で受付 ON を確認
2. Stripe 設定が完了しているか確認（21 章）

## 8-13. 緊急連絡先

### 8-13-1. トラブル種別と連絡先

| トラブル種別 | 連絡先 |
|---|---|
| POSLA 全般のトラブル | `info@posla.jp`（24 時間以内に返答） |

## 8-17. 付録 D — AI ヘルプデスクの使い方（2026-04-19 更新）

わからないことがあれば AI に聞くのが最も早い。**マネージャー以上が管理画面で利用**します。

1. 管理画面（`dashboard.html` / `owner-dashboard.html`）の **右下にあるオレンジ色 ? ボタン**（円形 FAB）を 1 タップ
2. チャットパネルが開く
3. 質問を入力（例: 「パスワードを変更する方法は？」「`[E3024]` とは？」）
4. 送信ボタンまたは Enter キー
5. 数秒で AI がこのマニュアル全章 + エラーカタログから該当部分を探して回答

::: warning スタッフ用端末には AI ボタンはありません
KDS・POSレジ・ハンディには AI ヘルプデスク FAB を設置していません（業務集中のため）。スタッフが困ったらマネージャーに聞くフローです。
:::

お客様側（セルフメニュー）にも AI があり、**「このメニューに卵は入っていますか？」** などの質問に回答します（AI ウェイター機能、こちらは別の AI）。
