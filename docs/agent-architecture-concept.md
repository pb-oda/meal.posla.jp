# POSLA 運用支援エージェント — アーキテクチャ構想

## コンセプト

お客様（テナント）からエラーや不具合の連絡が来たとき：
1. **エージェントが自動調査** — ログ・DB・コードを確認して原因を特定
2. **Hiroに通知** — エラー箇所・原因・影響範囲を報告
3. **将来：承認後にAIが自動修正** — Hiroが「直して」と承認すれば、コード修正 → テスト → デプロイ

---

## フェーズ1：監視 + 自動調査 + 通知

### 仕組み

```
[お客からの連絡]        [自動監視]
      |                     |
      v                     v
  ┌─────────────────────────────┐
  │   POSLA Agent（常駐）         │
  │                               │
  │  ① エラーログ監視              │
  │  ② DB異常検知                 │
  │  ③ API応答時間監視            │
  │  ④ Stripe Webhook失敗検知    │
  └───────────┬─────────────────┘
              │
              v
  ┌─────────────────────────────┐
  │   調査エンジン                │
  │                               │
  │  - エラーログの解析            │
  │  - 関連コードの特定            │
  │  - DB状態の確認               │
  │  - 再現手順の推定             │
  └───────────┬─────────────────┘
              │
              v
  ┌─────────────────────────────┐
  │   Hiroへ通知                  │
  │                               │
  │  - Slack / メール / LINE      │
  │  - エラー概要                 │
  │  - 原因の推定                 │
  │  - 影響範囲                   │
  │  - 推奨対応                   │
  └─────────────────────────────┘
```

### 具体的な監視項目

| 項目 | 検知方法 | 通知内容 |
|------|----------|----------|
| PHPエラー | error_log監視 | ファイル名、行番号、エラー内容 |
| API 500エラー | アクセスログ解析 | エンドポイント、パラメータ、エラー原因 |
| DB接続エラー | 定期ヘルスチェック | 接続失敗時間、影響テナント |
| Stripe Webhook失敗 | subscription_events監視 | 失敗イベント、テナント、リトライ状況 |
| 注文データ不整合 | 定期バリデーション | 不整合レコード、影響テーブル |
| セッション異常 | audit_log分析 | 異常ログイン試行、ロックアウト |
| API応答遅延 | レスポンスタイム計測 | 遅延エンドポイント、平均/P99 |
| 在庫データ矛盾 | 在庫 vs 売上突合 | 矛盾食材、理論値 vs 実値 |

### 技術的な実現方法

**選択肢A：Claude Agent SDK（推奨）**
- Anthropic の Claude Agent SDK を使用
- PHPシステムとは独立したPythonサービスとして動作
- MySQL直接参照 + ログファイル監視 + Stripe API確認
- 通知はSlack Webhook / LINE Notify / メール

**選択肢B：自前PHPエージェント**
- cronで定期実行するPHPスクリプト
- Gemini APIで分析
- シンプルだが、分析能力はClaude/GPTに劣る

**選択肢C：外部監視サービス + AI分析**
- Datadog / Sentry / New Relic でエラー収集
- Webhook → Claude API で分析
- インフラコストは増えるが信頼性は高い

## フェーズ2：承認付き自動修正（将来）

### ワークフロー

```
[エージェントがエラー検知]
        |
        v
[原因特定 + 修正案を生成]
        |
        v
[Hiroに通知]
  「〇〇.phpの△△行でNullPointerError。
   修正案：nullチェックの追加。
   影響範囲：該当API1本のみ。
   承認しますか？」
        |
    ┌───┴───┐
    v       v
 [承認]  [却下]
    |       |
    v       v
[自動修正]  [手動対応へ]
  - git branch作成
  - コード修正
  - テスト実行
  - PRを作成
  - Hiro最終確認後マージ
```

### 安全装置

| 安全装置 | 説明 |
|----------|------|
| 承認必須 | 修正前にHiroの明示的承認が必要 |
| ブランチ分離 | 修正は別ブランチで実施。mainには直接触れない |
| 影響範囲表示 | 変更ファイル・関数・影響APIを事前表示 |
| テスト必須 | 修正後にテスト実行、パスしなければ通知のみ |
| ロールバック | デプロイ後に異常検知したら即ロールバック |
| 修正範囲制限 | 1タスク = 最小変更範囲（CLAUDE.mdルール準拠） |
| 禁止操作 | auth.php変更禁止、schema.sql変更禁止、UI削除禁止 |

## まず始めるなら

### 最小構成（フェーズ1の最小版）

1. **PHPエラーログ監視スクリプト**（cron 5分間隔）
   - error_logを tail → 新規エラーを抽出
   - Gemini APIでエラー分析
   - Slack Webhookで通知

2. **Stripe Webhook失敗監視**
   - subscription_eventsテーブルを定期チェック
   - 失敗イベントを通知

3. **API ヘルスチェック**
   - 主要エンドポイントに定期リクエスト
   - 応答時間・ステータスコードを記録
   - 異常時に通知

これだけでも「お客から連絡が来る前に問題を把握できる」状態になります。

## POSLAとの統合ポイント

### 既存コードへの影響（最小限）

| 変更 | 内容 | 影響 |
|------|------|------|
| エラーログ形式統一 | error_log()の出力フォーマットを統一 | api/lib/response.php に1行追加 |
| ヘルスチェックAPI | /api/public/health.php を新規追加 | 新規ファイル。既存に影響なし |
| 監視テーブル | agent_alerts テーブル追加 | 新規マイグレーション。既存に影響なし |
| 通知設定 | posla_settings に通知先URL追加 | INSERTのみ |

### CLAUDE.mdルール厳守
- 既存ファイルへの変更は最小限（エラーログ形式統一の1行のみ）
- 新規ファイルとして追加
- agent関連コードはPOSLA本体とは別ディレクトリ（/agent/）
